KY 16 PROJECT — TECHNICAL REFERENCE

KY16Project 기술문서
소스 구조 · 설계 · 빌드

C++ 콘솔 게임 모음집 KY16Project의 개발자용 레퍼런스. 하나의 main() 메뉴 아래 절차적 → 객체지향 두 계층이 공존하는 구조를 파일 맵 · 상속 계층 · 게임 루프 · 직렬화 포맷 · 기술 부채까지 소스 기준으로 정리한다.

소스 30개 파일 · 약 4,700 LOC 기간 9/2 ~ 9/27 빌드 VS · v141 · MultiByte 타깃 Windows x64 콘솔
01 · Build & Runtime

빌드 · 실행 환경

외부 라이브러리 없이 표준 라이브러리 + Win32 콘솔 API만 사용한다. 문자 집합이 MultiByte라는 점이 인코딩 특성(§06)을 결정한다.

항목값
IDE / 빌드KY16Project.sln · MSBuild (.vcxproj)
플랫폼 툴셋v141 (Visual Studio 2017)
Windows SDK10.0.18362.0
구성Debug / Release × Win32 / x64
문자 집합MultiByte (CharacterSet=MultiByte) — CP949 기반
서브시스템Console
배포 빌드64-bit, 런타임 정적 링크(/MT) → 재배포 패키지 없이 단독 실행
의존성 — 표준 라이브러리 + Win32 콘솔 API
<iostream> <string> <fstream> <ctime>/<time.h> <conio.h> <limits.h>
<Windows.h>   // SetConsoleCursorPosition, WriteFile, COORD, Sleep 등
🧩
MultiByte 문자집합 — 유니코드가 아니라 코드페이지(CP949) 기반으로 한글 리터럴을 처리한다. 이것이 소스가 CP949로 저장된 이유이며, UTF-8 편집기에서 열면 일부 파일이 깨져 보인다.
02 · Architecture

전체 아키텍처

프로젝트는 두 개의 패러다임 계층을 한 실행 파일에 담는다. 학습 진행(절차적 → 객체지향)이 그대로 코드 구조에 드러난다.

모듈 계층 구조
                     main.cpp  (메뉴 디스패처)
                          │
        ┌─────────────────┼──────────────────────────┐
        │                 │                           │
  [기초 예제]        [심화 게임 · 절차적]         [OOP 게임]
  Basic01~04        HeroIsProcedural            Gamble
  (전역 함수)        DirectRope                  └ GameMain ─ USER / DEAL
                    EscapeMirror                Alcanoid
                    (단일 함수+지역 struct)     └ GameManager ─ GameObject
                                                             ├ Actor
                                                             ├ Ball
                                                             └ Block

HKYModule.h — 공용 헤더

  • 표준 include를 한데 집약
  • 메뉴가 호출하는 모든 진입 함수를 전방 선언
  • 각 게임 .cpp는 이 헤더 하나만 포함
  • 선언·정의 분리의 교과서적 예

Mmain.cpp — 디스패처

  • 최상위 메뉴 while 루프
  • 숫자 입력 → switch로 모듈 분기
  • Basic01/02는 하위 메뉴 한 단계 더
  • stage==0 또는 ESC로 종료
실행 흐름 — main()
DrawOpenning()
while (true):
    DrawIndex()               // 1~9 메뉴 출력
    cin >> stage
    system("cls")
    switch (stage) → 각 모듈 진입 함수 호출
    if stage==0 or ESC → break
    getchar()                // 모듈 종료 후 대기, 메뉴 복귀

Module 1 · Fundamentals

기초 예제 — Basic01~04

전역 함수의 집합. 각 파일이 하나의 "테스트 모듈" 함수를 노출하고, 그 안에서 개별 실습 함수를 호출한다.

Basic 03

함수 · 오버로딩 · 포인터 · 참조

Basic03.cpp
  • 함수 오버로딩 — NumAvg(int,int) vs NumAvg(int*,int), studyRandNum() 2종을 대비
  • 포인터를 통한 값 변경 — ChangeNum(int*)로 원본을 직접 수정
  • 참조 전달 — func_by_ref(int&)로 포인터 없이 원본 수정
Basic 04

템플릿 · 참조 · Swap

Basic04.cpp
  • 템플릿 함수 — template<typename T> AddTemp, 단일/이종 타입 파라미터, typeid().name()
  • 포인터 Swap vs 참조 Swap — 같은 동작을 두 메커니즘으로 오버로딩해 나란히 제시
Module 2 · Procedural

심화 게임 — 절차적

클래스 없이, 한 함수 안에서 지역 변수·지역 struct·while 루프로 게임 전체를 구현한다.

심화 1

HeroIsProcedural — 텍스트 RPG

01. HeroIsProcedural.cpp
  • hero_* / monster_* 스칼라 변수로 스탯 관리
  • 탐험 시 rand()로 몬스터 선택 → 가위바위보(GA/BA/BO) 승패로 데미지 교환
  • day 카운터가 0 이하로 내려가면 서브보스 → 보스 순차 등장
  • 경험치 누적 → 레벨업 루프(while(hero_exp > hero_up_exp))로 스탯 상승, 승리 시 아스키 아트 연출
심화 2

DirectRope — 2D 타일맵 이동

02. DirectRope.cpp
  • int map[10][10]에 0(길)/1(영웅)/2(탈출구)을 심음
  • WASD로 좌표 갱신, system("cls") 후 전체 재출력하는 전면 리드로우 방식
  • 탈출구 좌표 도달 시 승리
심화 3

EscapeMirror — 미로·채굴 탈출

03. EscapeMirror.cpp
  • TILE / PLAYER / ITEM / KEY 4종 struct를 지역 배열로 운용
  • stage 변수를 상태머신으로 사용 — 1=초기화, 2=타이틀, 3=게임오버, 4=상점, 그 외=플레이
  • _kbhit()/_getch() 실시간 입력, 지형 타입별 스태미나 소모
  • 채굴(SPACE)로 벽 → 길 변환, 열쇠 수집 후 탈출 조건 검사
⚠️
변수·struct·플래그가 한 함수에 폭증한다. "절차적 방식의 한계"를 체감시키는 지점 — 다음 단계에서 클래스로 정리된다.
Module 3 · OOP 도입

Gamble — 매니저 + 데이터 모델

클래스로 책임을 분리한 첫 게임. 두 슬롯 카드 사이(min < 카드 < max)에 다음 카드가 들어오면 이기는 베팅 게임.

DUSER — 데이터 모델

  • private + Get/Set 캡슐화
  • _money, _slot_min/max, _choice(폴드)
  • 플레이어 1 + AI 4 = 5인 게임

CDEAL — 카드 딜러

  • MixCard() — 중복 제거 셔플
  • ShareCardNum() — share_count 커서 순차 분배
  • 소진 시 재셔플

GGameMain — 매니저

  • Betting()/Ado()(올인) 판정
  • NextReceiver() 턴 이양
  • AiProcess() — 슬롯 간격 기반 콜/올인/폴드
  • OnDraw() — 표 형태 상태 렌더링
Module 4 · OOP 종합

Alcanoid — 상속 계층 + 게임 루프 + 영속화

가장 정형화된 구조이자 학습의 최종 도착점. 상속·다형성·캡슐화·게임루프·파일 저장을 종합한다.

상속 계층 — GameObject
GameObject  // 기반: _posX/Y, _dirX/Y, _speedPower, _shape, _isAlive, _isCollide + Get/Set
   ├── Actor : 패들.  Move(key, Ball*), Shoot(Ball*), _isBallOn(공 부착 여부)
   ├── Ball  : 공.    Move()에서 cos/sin 각도로 방향벡터, 벽/블록/패들 반사, bornTime/deadTime(점수)
   └── Block : 벽돌.  위치·생존만 갖는 최소 파생 클래스
게임 루프 — Alcanoid.cpp (표준 실시간 루프)
Initialize()                       // 객체 생성·배치
while (true):
    startFrame = clock()
    key = GetKeyEvent()            // _kbhit 논블로킹 입력
    InputProcess(key)             // 이동/발사/저장/불러오기
    Update()                       // 공 이동·충돌·점수·생명
    DrawScreen()                   // 씬별 렌더링
    FrameDelay(startFrame)         // 30 FPS 고정 (1000/FPS 보정)
  • 씬 관리 — enum SCENE { GAME_START, GAME_PLAY, GAME_OVER }로 상태 전환
  • 스테이지 — enum BLOCKNUM { STAGE1=30, STAGE2=40, STAGE3=50 }. 블록 전멸 시 NextGame()이 delete[] 후 다음 스테이지 배열 재할당
  • 저장/불러오기 — save_point.txt 텍스트 직렬화 (§05 참고)
05 · Design & Algorithms

핵심 설계 · 알고리즘

모듈을 가로지르는 공통 기법 — 렌더링 · 입력 · 충돌 · 직렬화 · 셔플.

R콘솔 렌더링

  • 전면 리드로우 — system("cls") 후 전체 재출력 (Basic·절차적·Gamble). 단순하나 깜빡임
  • 커서 직접 이동 — SetConsoleCursorPosition+WriteFile로 오브젝트만 갱신 (Alcanoid). mode con 창 고정, 커서 숨김

I입력 처리

  • 블로킹 — cin, getchar() (메뉴·턴제)
  • 논블로킹 — _kbhit()+_getch() (실시간)

C충돌 판정

  • float 좌표를 ×107 정수화해 비교
  • 도형 너비(std::size) + Y 인접(posY==other.posY+1)
  • 부동소수 비교를 정수화로 회피

S카드 셔플

  • rand()%52+1 후 중복이면 i-- 재시도
  • 52장 유일 값 보장 (Fisher–Yates 계열 변형)
저장 포맷 — save_point.txt (Alcanoid)

06 · Technical Debt

기술 이슈 · 개선점

학습 결과물 특성상 남아 있는 부채. 소스를 읽으며 확인한 실제 항목으로, 재사용/리팩터링 시 우선 검토 대상.

#이슈위치내용 / 개선안
1소스 인코딩 혼재프로젝트 전반대부분 CP949(EUC-KR), 일부(pch.h, Alcanoid.cpp)는 UTF-8. UTF-8 편집기에서 CP949 파일이 깨져 보임 → UTF-8 일괄 변환 + .gitattributes 고정 권장
2상수 매크로 중복config.h ↔ pch.hGAME_FPS, GAME_WIDTH, enum SCENE/BLOCKNUM 등 양쪽 중복 → 재정의 경고 소지. 한 곳으로 단일화
3대입/비교 혼동 버그GameMain::OnDrawif (game_start = true) — 비교가 아니라 대입이라 항상 참. ==로 수정 필요
4메모리 해제 누락GameMain, GameManager 소멸자new한 객체를 delete 하지 않아 누수 → 소멸자 정리 또는 스마트 포인터
5미완성 함수AiChoice(), Actor::Shoot()AiChoice()는 빈 함수, Shoot()은 양 분기 동일 동작 → 죽은 코드 정리
6매직 넘버Collide(), Ball::Move()×10000000 스케일, 고정 angle=120 → 명명 상수/주석화
7전역 상태Basic03.cpp _worldNum전역 변수 의존 → 재사용 시 지역화 권장
8플랫폼 종속전반Win32 콘솔 API·system("cls")·CP949 전제 → Windows 전용, 이식성 낮음(학습용으론 무방)
07 · Extend & Reuse

확장 · 재사용 가이드

모듈을 떼어 쓰거나 엔진 골격을 재활용할 때의 지침.

  • 개별 게임 분리 — 각 게임은 KYModule.h 하나에 의존하고 진입 함수 하나만 노출한다. 해당 .cpp(+OOP 게임은 매니저/모델 헤더 세트)와 KYModule.h만 가져오면 분리 이식 가능
  • 엔진 골격 재사용 — Alcanoid의 GameObject 계층 + Initialize/Input/Update/Draw/FrameDelay 루프는 다른 콘솔 게임의 최소 프레임워크로 재활용 가능. 렌더링을 커서 이동 방식으로 통일하면 확장이 쉬움
우선 정리 순서

소멸자 메모리 정리 — GameMain·GameManager 누수 제거
