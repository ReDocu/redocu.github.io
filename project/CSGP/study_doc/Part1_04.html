D 모듈 · 9–10장 — 구조체와 포인터: 데이터에 모양을 주다

모듈 D · 9–10장 · 게임 버전 v0.5 · 예상 학습 시간 4–6시간 처음 배우는 기술 — 구조체(struct) typedef 구조체 배열 포인터 주소·역참조 함수 인자로 포인터 전달 -> 연산자 NULL

지금까지(1–8장) 우리 회사는 흩어진 변수 수십 개로 굴러왔다. 이 모듈에서 그 변수들을 구조체로 묶어 "직원"과 "프로젝트"라는 개념에 모양을 주고, 포인터로 함수가 그 데이터를 직접 고치게 만든다. 이 두 문법이 손에 붙으면, 비로소 게임 개발 사이클(기획 → 개발 → 출시 → 판매)이 완성된다.

이 모듈의 학습 목표

이 모듈을 마치면 다음을 할 수 있다.

  1. 서로 관련된 여러 값을 하나의 구조체로 묶고, . 로 멤버에 접근할 수 있다.
  2. typedef 로 구조체에 짧은 이름을 붙이고, 구조체 배열로 같은 종류의 데이터 여럿을 다룰 수 있다.
  3. 변수의 주소(&) 와 역참조(*) 의 의미를 그림으로 설명할 수 있다.
  4. 함수에 포인터를 넘겨 함수 안에서 호출자의 원본 값을 바꿀 수 있고, 값 전달과의 차이를 안다.
  5. 구조체 포인터에서 -> 로 멤버에 접근하고, NULL 을 안전하게 검사할 수 있다.
  6. 왜 우리 게임의 모든 시스템 함수가 Company *c 하나를 받도록 설계됐는지 이해한다.

왜 지금, 구조체와 포인터인가

배움에는 순서가 있다. 이 책은 그 순서를 게임의 필요로 정한다. 여기까지 걸어온 길을 잠깐 돌아보자.

  • A 모듈(1–4장) 에서 우리는 자본금·명성 같은 값을 담고, 계산하고, 조건에 따라 갈라지게 했다.
  • B 모듈(5–6장) 에서 while 루프가 생기며 게임이 "돌기" 시작했고, 커진 코드를 함수로 정리했다.
  • C 모듈(7–8장) 에서 배열로 직원 여럿을, 문자열로 이름을 다뤘다.

그런데 C 모듈까지 온 우리 코드에는 슬슬 곪는 상처가 하나 있다. 직원 한 명을 표현하는 데 변수가 여러 개 필요한데, 그것들이 서로 묶여 있지 않다는 것. 지금 우리 직원 정보는 이렇게 생겼다.

/* 8장까지의 방식 — "병렬 배열"의 지옥 */
char staffName[4][20];   /* 직원 이름 4명분 */
int  staffDev[4];        /* 직원 개발력 4명분 */
int  staffCre[4];        /* 직원 창의력 4명분 */
int  staffSalary[4];     /* 직원 월급 4명분 */
int  staffMonths[4];     /* 직원 근속 4명분 */
int  staffCount;         /* 현재 직원 수 */

staffName[2], staffDev[2], staffCre[2] … 세 번째 직원 한 명의 정보가 배열 세 개에 흩어져 있다. 이 방식은 곧바로 문제를 일으킨다.

  • 직원을 한 명 해고하려면 다섯 개 배열을 모두 한 칸씩 당겨야 한다. 하나라도 빠뜨리면 이름과 능력치가 어긋난다.
  • 함수에 "직원 한 명"을 넘기려면 인자를 다섯 개 넘겨야 한다.
  • 직원에게 속성 하나(예: 사기)를 추가하려면 배열 선언·복사·인자를 쓰는 곳을 전부 손봐야 한다.

이것이 구조체가 필요한 이유다. 구조체는 흩어진 값들을 "직원"이라는 하나의 이름으로 묶어 준다. 그리고 그렇게 묶고 나면, 이번엔 "그 묶음을 함수가 어떻게 고치게 할 것인가"라는 질문이 자연스럽게 따라온다 — 그 답이 포인터다.

순서를 다시 보라. 구조체 없이 병렬 배열로 고생을 먼저 겪은 뒤에 구조체를 배운다. 이건 우연이 아니라 설계다. "왜 이게 필요한가"를 몸으로 느낀 다음에 도구를 손에 쥐면, 그 도구는 잊히지 않는다.

9장 · 구조체 — 흩어진 값에 이름을 주다

9.1 구조체 정의하기

구조체(structure) 는 서로 다른 자료형의 값 여러 개를 하나로 묶는 사용자 정의 자료형이다. 직원 한 명을 이렇게 정의한다.

struct Employee {
    char name[20];   /* 이름  */
    int  dev;        /* 개발력 (1~10) */
    int  cre;        /* 창의력 (1~10) */
    int  salary;     /* 월급 (만원) */
    int  months;     /* 근속 개월 */
};

이제 struct Employee 는 int, double 처럼 하나의 자료형이 되었다. 이 자료형으로 변수를 만들고, .(점 연산자)로 각 멤버에 접근한다.

struct Employee kim;            /* 직원 한 명 선언 */
strcpy(kim.name, "김개발");     /* 문자열 멤버는 strcpy 로 대입 */
kim.dev = 7;
kim.cre = 5;
kim.salary = (kim.dev + kim.cre) * 30;   /* 능력에 비례한 월급 = 360 */
kim.months = 0;

kim 이라는 변수 하나 안에 이름·개발력·창의력·월급·근속이 한 덩어리로 들어 있다. 병렬 배열에서 흩어져 있던 것들이 마침내 한집에 산다.

9.2 typedef — struct 를 매번 쓰지 않으려면

변수를 선언할 때마다 struct Employee 라고 두 단어를 쓰는 건 번거롭다. typedef 로 짧은 별명을 붙이면, 앞으로는 Employee 한 단어로 쓸 수 있다.

typedef struct {
    char name[20];
    int  dev, cre;     /* 같은 자료형은 콤마로 함께 선언 */
    int  salary;
    int  months;
} Employee;            /* 이 이름이 곧 자료형 이름이 된다 */

Employee kim;          /* 이제 struct 없이 이렇게 쓴다 */

우리 게임은 이 typedef struct { … } Name; 형태를 표준으로 쓴다. 앞으로 나오는 모든 구조체가 이 모양이다.

9.3 구조체 배열 — 직원 "목록"

직원은 한 명이 아니다. 7장에서 배운 배열을 구조체에 그대로 적용하면, 직원 목록이 된다. 병렬 배열 다섯 개가 구조체 배열 하나로 접힌다.

Employee staff[4];       /* 직원 4명분 — 8장의 병렬 배열 5개를 대체 */
int      staffCount = 0; /* 현재 직원 수 */

/* 두 번째 직원의 개발력에 접근: 배열 인덱스 뒤에 점 */
staff[1].dev = 8;

staff[i].name, staff[i].dev … 인덱스 i 하나로 그 직원의 모든 정보에 닿는다. 이제 직원을 해고할 때 배열을 한 칸씩 당기는 것도 훨씬 안전하다. 다섯 줄을 각각 당기는 게 아니라, 구조체 하나를 통째로 당기면 되기 때문이다.

/* idx 번 직원을 목록에서 제거 — 뒤 직원들을 한 칸씩 앞으로 */
void QuitEmployee(Company *c, int idx)
{
    int i;
    for (i = idx; i < c->staffCount - 1; i++)
        c->staff[i] = c->staff[i + 1];   /* 구조체 통째 복사 — 한 줄이면 끝 */
    c->staffCount--;
}

c->staff[i] = c->staff[i + 1]; 이 한 줄이 이름·개발력·창의력·월급·근속을 한꺼번에 복사한다. 병렬 배열이었다면 다섯 줄이 필요했고, 하나라도 빠뜨리면 데이터가 어긋났을 것이다. (c-> 의 화살표는 곧 10장에서 설명한다. 지금은 "회사 c 안의 staff 배열" 정도로 읽으면 된다.)

9.4 우리 게임의 데이터 설계

이 원리로 우리 게임의 핵심 데이터를 모두 구조체로 정의한다. 이것이 게임의 뼈대다.

/* --- 이름 붙인 정수 목록 (enum) — 정식 설명은 15장, 지금은 '라벨'로 이해하면 충분 --- */
typedef enum { G_RPG, G_ACTION, G_PUZZLE, G_SIM, G_SPORTS, GENRE_COUNT } Genre;
typedef enum { S_SMALL, S_MEDIUM, S_LARGE, SCALE_COUNT } Scale;

typedef struct {          /* 직원 */
    char name[20];
    int  dev, cre;        /* 개발력 / 창의력 (1~10) */
    int  salary;          /* 만원/월 */
    int  months;          /* 근속 */
} Employee;

typedef struct {          /* 개발 중인 게임 프로젝트 */
    char  title[30];
    Genre genre;
    Scale scale;
    int   progress, need; /* 진행도 / 필요치 */
    int   quality, fun;   /* 품질 / 재미 */
    int   devMonths;
    int   active;         /* 0 = 진행 중인 프로젝트 없음 */
} Project;

typedef struct {          /* 회사 — 게임 전체 상태를 담는 큰 구조체 */
    char      name[20];
    int       money, fame;
    int       year, month;
    Employee *staff;      /* 직원 목록 (11장에서 동적 배열로) */
    int       staffCount, staffCap;
    Project   proj;       /* 구조체가 다른 구조체를 멤버로 가진다 */
    Genre     trend;      /* 이번 분기 유행 장르 */
} Company;

세 가지를 눈여겨보자.

  • 구조체가 구조체를 품는다. Company 안에 Project proj; 가 있다. c.proj.progress 처럼 점을 두 번 찍어 안쪽 멤버에 닿는다.
  • Company 하나가 게임의 모든 상태다. 자본금, 날짜, 직원 목록, 진행 중 프로젝트, 유행 장르 — 게임을 저장하려면 이 구조체 하나만 파일에 쓰면 된다(13장). 이 "상태를 한 곳에 모으는" 설계가 이후 모든 장을 편하게 만든다.
  • Employee *staff 의 별(*). 이건 포인터다. 지금 당장은 Employee staff[4]; 같은 배열로 생각해도 좋지만, 11장에서 직원 수 제한을 풀 때 이 별이 중요해진다. 10장에서 이 별의 정체를 밝힌다.

⚠️ enum은 아직 맛보기다. Genre, Scale 은 typedef enum 으로 정의했지만, enum 의 정식 설명은 15장(캡스톤)에서 한다. 지금은 G_RPG 가 0, G_ACTION 이 1 … 처럼 이름을 붙인 정수 목록 이라고만 이해하면 충분하다. 장르에 0, 1, 2 대신 G_RPG, G_ACTION 이라는 이름을 쓰면 코드가 훨씬 읽기 쉬워진다는 것만 느끼면 된다.

10장 · 포인터 — 함수가 원본을 고치게 하다

구조체로 데이터에 모양을 줬으니, 이제 함수가 그 데이터를 고치게 만들 차례다. 여기서 포인터가 등장한다. 1부 전체에서 가장 어려운 고비이지만, 한 번 넘으면 프로그래밍을 보는 눈이 달라진다.

10.1 문제: 함수는 왜 원본을 못 바꾸는가

먼저 포인터 없이 벽에 부딪혀 보자. "직원의 근속을 1 늘리는" 함수를 순진하게 짜면 이렇게 된다.

void AgeOne(Employee e)   /* 값을 그대로 받는다 */
{
    e.months++;           /* e 의 근속을 늘린다 */
}

int main(void)
{
    Employee kim = { "김개발", 7, 5, 360, 0 };
    AgeOne(kim);
    printf("%d\n", kim.months);   /* ??? */
}

출력은 1 이 아니라 0 이다. 왜일까?

C에서 함수에 값을 넘기면, 함수는 그 값의 복사본을 받는다. AgeOne 안의 e 는 kim 이 아니라 kim 을 베낀 딴 상자다. e.months++ 는 그 복사본의 근속만 늘리고, 함수가 끝나면 복사본은 사라진다. 원본 kim 은 손도 대지 않은 채 그대로다.

이걸 값에 의한 전달(pass by value) 이라 한다. 안전하지만(함수가 실수로 원본을 망칠 수 없다), 원본을 고쳐야 할 때는 무력하다. 그리고 게임에서는 원본을 고쳐야 할 일이 대부분이다 — 고용은 직원 목록을 늘리고, 개발은 프로젝트 진행도를 올리고, 정산은 자본금을 바꾼다.

10.2 주소와 역참조 — & 와 *

해결의 열쇠는 "복사본 말고 원본이 사는 위치를 알려 주는 것"이다. 모든 변수는 메모리 어딘가에 살고, 그 위치에는 주소(address) 가 있다. 집집마다 번지수가 있는 것과 같다.

int hp = 30;
printf("%d\n", hp);    /* 30  — 값 */
printf("%p\n", &hp);   /* 0x7ffe... — hp 가 사는 주소 (& = 주소 연산자) */
  • &hp — 변수 hp 의 주소. "이 값이 어디 사는지."
  • 포인터 — 주소를 담는 변수. int *p 는 "int 가 사는 주소를 담는 변수 p".
  • *p (역참조) — 포인터가 가리키는 그 주소에 사는 값. "번지수를 따라가 실제 집을 여는" 동작.
int  hp = 30;
int *p  = &hp;   /* p 는 hp 의 주소를 담는다 = "p 가 hp 를 가리킨다" */

printf("%d\n", *p);   /* 30 — p 를 따라가면 hp 의 값 */
*p = 100;             /* p 가 가리키는 곳(=hp)에 100 을 쓴다 */
printf("%d\n", hp);   /* 100 — 원본 hp 가 바뀌었다! */

마지막 두 줄이 핵심이다. *p = 100; 은 hp 를 직접 바꾼다. p 를 통하면, p 가 가리키는 원본에 손을 댈 수 있다. 이 힘을 함수에 넘겨주면 아까의 문제가 풀린다.

void AgeOne(Employee *e)   /* 값이 아니라 '주소'를 받는다 */
{
    (*e).months++;         /* e 가 가리키는 원본의 근속을 늘린다 */
}

int main(void)
{
    Employee kim = { "김개발", 7, 5, 360, 0 };
    AgeOne(&kim);          /* kim 자체가 아니라 kim 의 '주소'를 넘긴다 */
    printf("%d\n", kim.months);   /* 1 — 이번엔 진짜 바뀐다 */
}

AgeOne(&kim) — 호출할 때 & 를 붙여 주소를 넘긴다. 함수는 복사본이 아니라 kim 이 사는 위치를 받았으므로, (*e).months++ 로 원본을 늘린다. 이것이 주소에 의한 전달(pass by pointer) 이다.

10.3 -> — 구조체 포인터의 지름길

(*e).months 는 매번 쓰기 번거롭다. "역참조하고 멤버 접근하기"는 워낙 자주 하는 일이라 C는 전용 연산자 ->(화살표)를 준다.

(*e).months++;   /* 이 둘은 완전히 같다 */
e->months++;     /* 화살표: '포인터가 가리키는 것의 멤버' */

규칙은 간단하다.

  • 구조체 변수 는 점(.) → kim.months
  • 구조체 포인터 는 화살표(->) → e->months

앞으로 함수는 대부분 구조체 포인터를 받으므로, 우리 게임 코드에는 -> 가 가득하다. c->money, p->progress, e->salary — 전부 "포인터가 가리키는 원본의 멤버를 직접 만진다"는 뜻이다.

10.4 게임에 적용: 모든 시스템 함수가 Company *c 를 받는다

이제 우리 게임의 실제 코드를 보자. 10장부터 우리는 전역 변수를 버린다. 대신 게임의 모든 상태를 Company 구조체 하나에 담고, 모든 시스템 함수가 그 회사의 주소(Company *c) 를 받아 직접 고친다.

첫 번째 예 — 매달 개발을 진행시키는 함수다.

/* 매월: 진행 중인 프로젝트의 진행도·품질·재미를 누적한다 */
void ProgressProject(Company *c)
{
    Project *p = &c->proj;          /* 회사 안 프로젝트의 '주소'를 p 에 담는다 */
    if (!p->active)                 /* 진행 중인 프로젝트가 없으면 */
        return;                     /* 아무것도 안 하고 나간다 */

    p->progress += BASE_PROGRESS + DevSum(c);   /* 진행도 += 15 + 개발력 합 */
    p->quality  += DevSum(c);                   /* 품질 += 개발력 합 */
    p->fun      += CreSum(c) + rand() % 3;      /* 재미 += 창의력 합 + 약간의 랜덤 */
    p->devMonths++;                             /* 개발 개월 수 +1 */
}

한 줄 한 줄 뜯어 보자.

  • Project *p = &c->proj; — c 는 회사의 주소다. c->proj 는 그 회사 안의 프로젝트(구조체)이고, &c->proj 는 그 프로젝트가 사는 주소다. 그 주소를 p 에 담았으니, 이제 p 를 통해 프로젝트 원본을 고칠 수 있다. (매번 c->proj.progress 라고 길게 쓰지 않으려는 지름길이기도 하다.)
  • p->progress += … — p 가 가리키는 프로젝트의 진행도를 직접 올린다. 함수가 끝나도 이 변화는 남는다. 회사 c 안의 진짜 프로젝트가 바뀌었기 때문이다.
  • return; — 값을 돌려주지 않는(void) 함수라도, 조건이 안 맞으면 일찍 빠져나올 수 있다. if (!p->active) return; 은 "프로젝트 없으면 그냥 나가라"는 가드(guard) 다.

값 전달이었다면 이 함수는 아무 일도 못 했을 것이다. ProgressProject 가 회사의 복사본을 받았다면, 진행도를 올려 봤자 함수가 끝나는 순간 복사본과 함께 사라졌을 테니까. 포인터를 받기 때문에, 이 함수는 진짜 회사를 키운다.

두 번째 예 — 읽기만 하는 함수도 포인터를 받는다. 단, const 를 붙여 "나는 읽기만 하고 고치지 않겠다"고 약속한다.

/* 직원들 개발력 합계 (+ 사장 능력치) — 읽기만 하므로 const */
int DevSum(const Company *c)
{
    int sum = BOSS_DEV, i;              /* 사장의 개발력부터 시작 */
    for (i = 0; i < c->staffCount; i++)
        sum += c->staff[i].dev;         /* 각 직원의 개발력을 더한다 */
    return sum;
}
  • const Company *c — "c 가 가리키는 회사를 바꾸지 않겠다"는 선언이다. 실수로 c->money = 0; 같은 코드를 쓰면 컴파일러가 막아 준다. 읽기 전용 함수에 const 를 붙이는 건 좋은 습관이다.
  • c->staff[i].dev — 화살표와 점이 함께 나온다. c->staff 는 (포인터가 가리키는 회사의) 직원 배열, c->staff[i] 는 그중 i 번째 직원(구조체), .dev 는 그 직원의 개발력. 화살표 한 번, 인덱스, 점 한 번 을 차례로 따라가면 된다.

세 번째 예 — 앞서 본 해고 함수를 다시 보자. 이제 왜 Company *c 를 받는지 분명하다.

void QuitEmployee(Company *c, int idx)
{
    int i;
    for (i = idx; i < c->staffCount - 1; i++)
        c->staff[i] = c->staff[i + 1];   /* 뒤 직원을 한 칸씩 앞으로 (구조체 통째 복사) */
    c->staffCount--;                     /* 직원 수를 줄인다 — 원본 회사에 반영된다 */
}

c->staffCount--; 가 원본 회사의 직원 수를 실제로 줄인다. 함수가 회사의 주소를 받았기 때문에 가능한 일이다.

10.5 NULL — "아무것도 가리키지 않는" 포인터

포인터가 아직 아무 데도 가리키지 않을 때, 우리는 NULL(빈 주소)을 담는다. 포인터를 쓰기 전에 NULL 인지 확인하는 습관은 프로그램이 죽는 것을 막는다. Employee *staff 는 게임 시작 시 아직 직원이 없으면 NULL 일 수 있고, 11장에서 malloc 이 실패해도 NULL 을 돌려준다. 파일을 못 열어도 fopen 은 NULL 을 준다(13장).

Employee *staff = NULL;      /* 아직 아무 직원 배열도 없음 */

if (staff != NULL)           /* NULL 이 아닐 때만 안전하게 접근 */
    printf("%s\n", staff[0].name);

NULL 인 포인터를 역참조하면(*staff, staff->name) 프로그램이 그 자리에서 죽는다. 그래서 "쓰기 전에 검사"가 철칙이다. 이 습관은 11장(동적 메모리)과 13장(파일)에서 계속 등장한다.

10.6 배열과 포인터, 그리고 다음 장으로의 복선

마지막으로 Company 구조체의 Employee *staff; 를 다시 보자. 왜 Employee staff[4];(고정 배열)가 아니라 포인터일까?

7장에서 우리 회사는 직원 4명까지만 고용할 수 있었다. 5번째를 뽑으려 하면 "사무실이 좁습니다"라며 거절당했다. 고정 배열의 크기는 컴파일 시점에 못 박히기 때문이다. 하지만 회사는 계속 커져야 한다.

C에서 배열의 이름은 사실 그 배열의 첫 원소를 가리키는 포인터처럼 동작한다. 그래서 Employee *staff 는 "직원 배열을 가리키는 포인터"가 되고 — 그 배열을 실행 중에 원하는 만큼 크게 만들 수 있게 된다. 그 방법이 바로 다음 장(11장)의 동적 메모리(malloc/realloc) 다. 이번 장에서 포인터를 손에 넣었기 때문에, 다음 장에서 "직원 수 무제한"이라는 선물을 풀 수 있다.

/* 11장 미리보기 — 직원이 꽉 차면 배열을 2배로 늘린다 */
if (c->staffCount == c->staffCap) {
    Employee *tmp = realloc(c->staff, sizeof(Employee) * c->staffCap * 2);
    if (tmp == NULL) return;      /* 확장 실패 — NULL 검사 (10.5) */
    c->staff = tmp;               /* staff 포인터가 더 큰 배열을 가리키게 */
    c->staffCap *= 2;
}

지금은 이 코드를 다 이해하지 못해도 좋다. 다만 staff 가 포인터라서 이런 일이 가능하다는 것 — 이번 장의 포인터가 다음 장의 문을 연다는 것만 기억하자.

코드 워크스루 — 실제 소스에서 보는 구조체·포인터

지금까지의 조각들이 실제 게임 코드에서 어떻게 맞물리는지, Study/Tycoon/ 의 실제 소스로 확인하자. 아래는 매달 개발이 진행되는 흐름이다.

실제 파일: Study/Tycoon/company.h · Study/Tycoon/employee.c · Study/Tycoon/project.c

① 데이터 설계 (company.h) — 게임의 모든 상태가 Company 구조체 하나에 모인다. (9.4에서 본 그대로다.)

② 매달 개발 진행 (project.c)

void ProgressProject(Company *c)
{
    Project *p = &c->proj;
    if (!p->active) return;
    p->progress += BASE_PROGRESS + DevSum(c);   /* ← DevSum 이 직원 능력을 합산 */
    p->quality  += DevSum(c);
    p->fun      += CreSum(c) + rand() % 3;
    p->devMonths++;
}

③ 능력치 합산 (employee.c) — ProgressProject 가 부르는 DevSum 은 const Company *c 를 받아 직원 배열을 순회한다.

int DevSum(const Company *c)
{
    int sum = BOSS_DEV, i;
    for (i = 0; i < c->staffCount; i++)
        sum += c->staff[i].dev;
    return sum;
}

이 세 조각의 연결을 보라. ProgressProject(c) 는 회사의 주소를 받아, 그 회사의 프로젝트 진행도를 직접 올린다. 그 값은 DevSum(c) 가 같은 회사의 직원 배열을 훑어 계산한다. 회사(Company)라는 하나의 구조체를, 여러 함수가 포인터로 나눠 들고 각자 자기 역할을 한다. 이것이 구조체 + 포인터가 만드는 프로그램의 짜임새다.

전역 변수를 썼다면 어땠을까? 함수들이 같은 전역 상태를 몰래 공유했을 것이고, "이 함수가 무엇을 건드리는지"가 코드에 드러나지 않았을 것이다. Company *c 를 명시적으로 주고받는 지금 방식은, 함수의 서명(signature)만 봐도 "이 함수는 회사를 만진다" 는 것을 알려 준다. 포인터는 힘일 뿐 아니라, 설계를 정직하게 만드는 문서이기도 하다.

그림으로 정리 — 값 전달 vs 포인터 전달

📊 웹 교육문서에서는 이 개념이 인터랙티브 애니메이션(V-P1-D)으로 재생된다. 함수 호출 시 상자(변수)가 복사되는지, 화살표(주소)가 원본을 가리키는지를 단계별로 보여 준다. 책에서는 아래 도식으로 대신한다.

값 전달 — 복사본을 고친다 (원본은 그대로)

  main:  kim = [ 김개발 | dev 7 | ... | months 0 ]
                       │  (값을 복사)
                       ▼
  함수:  e   = [ 김개발 | dev 7 | ... | months 0 ]  ← 딴 상자
                                          months++ → 1
              함수 끝 → e 소멸.  kim.months 는 여전히 0

포인터 전달 — 주소로 원본을 고친다

  main:  kim = [ 김개발 | dev 7 | ... | months 0 ]  ← 원본
                 ▲
                 │  &kim (주소를 넘김)
  함수:  e ──────┘   e->months++  →  원본 kim.months 가 1 이 된다
              함수 끝 → e(주소)만 소멸.  kim 의 변화는 남는다

한 문장으로: 값 전달은 사진을 주는 것, 포인터 전달은 집 주소를 주는 것이다. 사진에 낙서해 봤자 집은 안 바뀌지만, 주소를 알면 그 집에 찾아가 벽을 칠할 수 있다.

구조체 메모리 레이아웃 — Employee 하나가 메모리에 놓이는 모습.

  Employee kim;
  ┌───────────────┬─────┬─────┬────────┬────────┐
  │ name[20]      │ dev │ cre │ salary │ months │
  └───────────────┴─────┴─────┴────────┴────────┘
   0             20    24    28       32       36 (바이트, 대략)
   ← 멤버들이 한 덩어리로 연속해 있다. 그래서 c->staff[i]=c->staff[i+1] 한 줄이
     이 블록 전체를 통째로 복사할 수 있다.

체크포인트 — 스스로 점검하기

각 문항을 먼저 스스로 답해 본 뒤, 아래 해설을 펼쳐 확인하자.

Q1. 함수 void heal(Player p) { p.hp += 10; } 를 heal(hero); 로 불렀는데 hero.hp 가 안 바뀐다. 왜인가?

Q2. (*e).salary 를 화살표 연산자로 다시 쓰면?

Q3. const Company *c 에서 const 는 무엇을 막는가?

Q4. 아래 코드에서 잘못된 줄을 찾고 고쳐라.

Employee *e = NULL;
printf("%s\n", e->name);

Q5. Company 구조체 안에 Project proj; 가 있을 때, 회사 c(포인터)를 통해 프로젝트 진행도에 접근하는 표현은?

해설 펼치기

A1. Player p 는 값 전달이라 hero 의 복사본을 받는다. p.hp += 10 은 복사본만 고치고 함수가 끝나면 사라진다. 원본을 고치려면 포인터를 받아야 한다: void heal(Player *p) { p->hp += 10; } 로 정의하고 heal(&hero); 로 호출한다.

A2. e->salary. (*포인터).멤버 는 언제나 포인터->멤버 와 같다.

A3. c 가 가리키는 회사를 수정하는 것을 막는다. c->money = 0; 같은 쓰기는 컴파일 에러가 된다. 읽기(c->money 값을 읽는 것)는 허용된다. "이 함수는 읽기만 한다"는 약속이자 실수 방지 장치다.

A4. e 가 NULL(아무것도 안 가리킴)인데 e->name 으로 역참조하면 프로그램이 죽는다. 접근 전에 검사해야 한다:

if (e != NULL)
    printf("%s\n", e->name);

A5. c->proj.progress. c 는 포인터라 화살표(->)로 회사에 닿고, proj 는 값 멤버(구조체)라 점(.)으로 그 안의 progress 에 닿는다. 화살표 다음 점 이 나오는 전형적인 패턴이다.

실습 미션 — 턴제 RPG 배틀 battle.c ⭐

본편(타이쿤)이 "한 게임을 계속 키우는 세로 축"이라면, 실습 미션은 이 모듈의 새 문법만 콕 집어 연습하는 독립 미니게임(가로 축) 이다. 먼저 직접 짜 보고, 막히면 힌트를, 그래도 막히면 정답을 펼쳐 대조하자.

드릴 문법 — struct · typedef · 포인터로 원본 HP 직접 수정 · 구조체를 함수 인자로 전달 만드는 것 — 용사 vs 슬라임 턴제 전투. 한쪽 HP가 0이 되면 종료. 클리어 조건 — 매 턴 행동(강공격/일반/방어)을 골라 상대 HP를 깎고, 승패를 판정해 출력한다. 규모 — 약 45줄, 파일 1개. 표준 C만(printf/scanf). 아직 난수는 쓰지 않는다(난수는 E 모듈에서 등장).

문제 정의

이 게임의 핵심은 딱 하나다. attackUnit 함수가 대상의 HP를 "직접" 깎는다. 즉, 함수에 유닛의 주소를 넘겨(&hero, &slime), 함수 안에서 그 원본의 hp 를 줄인다. 만약 값 전달로 짰다면, 아무리 공격해도 HP가 줄지 않는 버그를 만나게 된다 — 그것이 바로 이 미션이 겨냥하는 깨달음이다(10.1의 함정과 정확히 같다).

목표 / 클리어 조건

  1. Unit 구조체를 정의한다: 이름(char name[20]), HP(int hp), 공격력(int atk).
  2. void attackUnit(Unit *attacker, Unit *target, int power) 를 만든다 — 포인터를 받아 target->hp 를 직접 깎는다.
  3. while 루프로 두 유닛의 HP가 모두 양수인 동안 턴을 반복한다.
  4. 매 턴 플레이어가 행동(1:강공격 2:일반 3:방어)을 고르면 power 가 달라진다.
  5. 한쪽 HP가 0 이하가 되면 승패를 출력하고 종료.

힌트

힌트 1 — 데미지 공식

power 를 강공격 15 / 일반 10 / 방어 5 로 두고, dmg = attacker->atk * power / 10; 으로 계산하면 방어(5)는 절반, 강공격(15)은 1.5배가 된다. 최소 데미지 1은 보장하자(if (dmg < 1) dmg = 1;).

힌트 2 — 삼항 연산자로 power 고르기 
power = (sel == 1) ? 15 : (sel == 3) ? 5 : 10;

4장에서 배운 삼항 연산자 조건 ? A : B 를 겹쳐 쓰면 세 갈래를 한 줄로 고를 수 있다.

힌트 3 — 죽음 판정 위치

용사가 슬라임을 때린 직후 슬라임이 죽었다면, 슬라임의 반격 전에 break 로 루프를 빠져나가야 한다. "때린다 → 죽었나 확인 → 살아 있으면 반격" 순서가 자연스럽다.

정답 코드

정답 펼치기 — battle.c 
#define _CRT_SECURE_NO_WARNINGS
#include <stdio.h>

typedef struct {
    char name[20];
    int  hp;
    int  atk;
} Unit;

/* 포인터로 대상의 HP를 '직접' 깎는다 — D 모듈의 핵심 */
void attackUnit(Unit *attacker, Unit *target, int power)
{
    int dmg = attacker->atk * power / 10;   /* power: 강공격15 / 일반10 / 방어5 */
    if (dmg < 1) dmg = 1;
    target->hp -= dmg;                       /* 원본 HP 를 깎는다 */
    if (target->hp < 0) target->hp = 0;
    printf("  %s -> %s : %d 피해!  (%s HP %d)\n",
           attacker->name, target->name, dmg, target->name, target->hp);
}

int main(void)
{
    Unit hero  = { "용사",   30, 8 };
    Unit slime = { "슬라임", 24, 6 };
    int sel, power;

    printf("=== 턴제 RPG 배틀 ===\n");
    while (hero.hp > 0 && slime.hp > 0) {
        printf("\n[용사 HP %d | 슬라임 HP %d]\n", hero.hp, slime.hp);
        printf("행동 (1:강공격 2:일반 3:방어): ");
        scanf("%d", &sel);
        power = (sel == 1) ? 15 : (sel == 3) ? 5 : 10;

        attackUnit(&hero, &slime, power);           /* 주소를 넘긴다 */
        if (slime.hp <= 0) break;                    /* 죽었으면 반격 전에 종료 */

        attackUnit(&slime, &hero, (sel == 3) ? 5 : 10);  /* 방어 시 반격 절반 */
    }

    printf("\n== 결과 ==\n");
    if (hero.hp > 0) printf(">> 승리! 슬라임을 물리쳤다.\n");
    else             printf(">> 패배... 용사가 쓰러졌다.\n");
    return 0;
}

빌드: cl battle.c (MSVC) 또는 gcc battle.c -o battle

함정/체크포인트

  • attackUnit(&hero, &slime, …) 에서 & 를 빼고 값으로 넘기면(그리고 함수도 값으로 받게 바꾸면) HP가 절대 줄지 않는다. 원본을 못 고치기 때문이다 — 10.1의 함정과 똑같다. 일부러 한 번 그렇게 짜 보고 버그를 눈으로 확인하면, 포인터의 의미가 몸에 남는다.
  • -> (포인터의 멤버)와 . (변수의 멤버)를 헷갈리지 말자. 함수 안 attacker 는 포인터라 ->, main 의 hero 는 변수라 . 다.

확장 과제

  • E 모듈(난수) 에서 크리티컬(확률로 데미지 2배)을 추가한다.
  • 몬스터 여러 마리를 구조체 배열(Unit party[3])로 만들어 파티 전투로 키운다.

이 모듈을 마치며

얻은 것 다음에 쓰는 곳
구조체로 데이터에 모양 주기 모든 후속 장 — 게임의 모든 데이터가 구조체다
포인터로 함수가 원본 수정 11장(동적 배열), 13장(세이브), 그리고 게임 전체
Company *c 를 주고받는 설계 14장 파일 분리 시 함수 서명이 그대로 인터페이스가 됨

구조체와 포인터는 1부의 고비였다. 이 둘을 넘었다면, 사실상 C의 가장 어려운 개념을 통과한 것이다. 남은 장들(동적 메모리·랜덤·파일·모듈)은 모두 이 두 도구를 응용하는 이야기다.
