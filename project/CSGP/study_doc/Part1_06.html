F 모듈 · 13–15장 — 파일·모듈·캡스톤: 회사를 저장하고, 게임을 출시하다

모듈 F · 13–15장 · 게임 버전 v0.7 → v1.0 · 예상 학습 시간 5–7시간 처음 배우는 기술 — fopen fclose fprintf fscanf 파일 모드 텍스트 vs 바이너리 파일 열기 실패(NULL) 헤더/소스 분리 인클루드 가드(#ifndef) extern enum(열거형) #define 상수화 게임 엔딩(상태 enum)

우리 게임은 이제 살아 움직인다. 회사는 자라고, 시장은 매번 다른 얼굴을 보인다. 그런데 프로그램을 껐다 켜면 모든 게 사라진다. 그리고 main.c 는 어느새 수백 줄로 부풀어, 무엇이 어디 있는지 찾기 어렵다. 마지막 모듈에서 이 둘을 해결한다. 파일로 회사를 저장해 이어하기를 만들고, 코드를 여러 파일로 분리해 프로젝트답게 정리하며, enum 으로 흩어진 숫자에 이름을 주고 엔딩을 붙여 — 게임을 v1.0으로 출시한다. 1부의 종착점이다.

이 모듈의 학습 목표

이 모듈을 마치면 다음을 할 수 있다.

  1. fopen/fclose 로 파일을 열고 닫으며, 파일 모드("r"·"w")를 구분해 쓸 수 있다.
  2. fprintf/fscanf 로 게임 상태를 파일에 저장하고 다시 읽어올 수 있다.
  3. 파일 열기 실패(NULL)를 검사해, 세이브가 없으면 새 게임으로 안전하게 넘어갈 수 있다.
  4. 하나의 소스를 헤더(.h)와 소스(.c) 로 나누고, 인클루드 가드로 중복 포함을 막을 수 있다.
  5. enum 으로 장르·엔딩 같은 "이름 붙인 정수 집합"을 정의하고, switch 와 함께 쓸 수 있다.
  6. #define 으로 밸런스 상수를 한곳에 모아, 게임을 조율하고 마무리할 수 있다.

왜 지금, 파일·모듈·종합인가

E 모듈까지 우리 게임은 기능적으로 거의 완성됐다. 남은 것은 "게임을 게임답게, 프로젝트를 프로젝트답게" 마감하는 일이다.

첫째, 영속성. 지금 우리 회사는 프로그램이 끝나면 흔적도 없이 사라진다. 3년을 키운 회사도 창을 닫으면 끝이다. 진행 상황을 파일에 남겨 다음에 이어할 수 있어야 비로소 "저장되는 게임"이 된다 — 13장 파일 입출력이다.

둘째, 구조. main.c 하나에 타이틀·루프·정산·직원·판매·저장이 다 들어가 이제 수백 줄이다. 관련된 코드끼리 파일로 묶으면(직원은 employee.c, 판매는 market.c) 찾기도 고치기도 쉬워진다 — 14장 파일 분리다. 이건 문법이라기보다 큰 프로그램을 다루는 태도다.

셋째, 마감. 지금까지 우리는 G_RPG 같은 이름을 "맛보기"로 써 왔다(D 모듈에서 예고했듯 정식 설명을 미뤄 뒀다). 15장에서 enum 을 제대로 배워 흩어진 숫자에 이름을 주고, #define 으로 밸런스를 모으고, 엔딩 3종을 붙여 게임을 출시 버전으로 마무리한다.

이 모듈에서 만드는 것. 세이브/로드(v0.8)로 이어하기가 가능하고, 코드가 7개 모듈로 깔끔히 나뉘며(v0.9), enum 과 엔딩·밸런스 정리로 개발 사이클과 승패가 온전히 굴러가는 — GDD 마일스톤 v1.0 "출시" 다. 이 모듈을 마치면 1부의 게임이 완성되고, 우리는 2부(C++ 프레임워크)로 넘어갈 준비를 마친다.

13장 · 파일 입출력 — 회사를 저장하다

13.1 fopen 과 fclose — 파일을 열고 닫기

파일을 다루는 첫걸음은 fopen 으로 여는 것이다. fopen(파일명, 모드) 은 파일을 열어 그 파일 포인터(FILE *) 를 돌려주고, 다 쓰면 fclose 로 닫는다.

#include <stdio.h>

FILE *fp = fopen("save.txt", "w");   /* "save.txt" 를 쓰기 모드로 열기 */
if (fp == NULL) {                    /* 열기 실패 검사 (13.3) */
    printf("파일을 열 수 없습니다.\n");
    return;
}
/* ... fp 로 읽거나 쓰기 ... */
fclose(fp);                          /* 반드시 닫는다 */

모드가 무엇을 할지 정한다.

모드 뜻 파일이 없으면 파일이 있으면
"r" 읽기 실패(NULL) 처음부터 읽음
"w" 쓰기 새로 만듦 기존 내용 덮어씀
"a" 이어쓰기 새로 만듦 끝에 덧붙임

"w" 는 기존 내용을 지우고 새로 쓴다. 세이브는 매번 최신 상태로 덮어써야 하니 "w", 로드는 읽기니 "r" 이다. fclose 를 빼먹으면 쓴 내용이 파일에 다 반영되지 않을 수 있으니(버퍼 문제) 반드시 닫는다.

13.2 fprintf — 파일에 쓰기

fprintf 는 printf 와 똑같다. 다른 점은 첫 인자로 어디에 쓸지(FILE *) 를 준다는 것뿐이다. printf 가 화면에 찍었다면, fprintf 는 파일에 찍는다.

fprintf(fp, "%d %d %d %d\n", money, fame, year, month);  /* 파일에 한 줄 */

값들 사이를 공백으로, 줄 끝을 \n 으로 구분해 두면, 나중에 fscanf 로 같은 순서로 읽어올 수 있다. 세이브 파일은 결국 "게임 상태를 정해진 순서로 적어 둔 텍스트" 다.

13.3 fscanf — 파일에서 읽기, 그리고 NULL 검사

fscanf 는 scanf 의 파일 버전이다. 저장할 때와 똑같은 순서로 읽어 변수에 담는다.

FILE *fp = fopen("save.txt", "r");
if (fp == NULL)
    return 0;               /* 세이브 파일이 없다 → 새 게임으로 (실패로 처리) */

fscanf(fp, "%d %d %d %d", &money, &fame, &year, &month);
fclose(fp);

여기서 fopen 의 NULL 검사가 결정적이다. 세이브 파일이 아직 없는 첫 실행에서 fopen("save.txt", "r") 은 NULL 을 돌려준다. 이걸 검사하지 않고 fscanf(NULL, ...) 를 부르면 프로그램이 죽는다. E 모듈에서 배운 NULL 검사가 여기서 또 실전이 된다 — 자원을 얻는 함수(malloc·fopen)는 실패할 수 있고, 그 결과는 늘 검사한다.

읽기와 쓰기의 순서는 반드시 같아야 한다. fprintf 로 money fame year month 순서로 썼다면, fscanf 도 정확히 그 순서로 읽어야 한다. 순서가 어긋나면 명성 자리에 자본금이 들어가는 식으로 데이터가 뒤섞인다. 세이브 포맷은 "쓰는 쪽과 읽는 쪽이 지키기로 한 약속"이다.

13.4 텍스트 파일 vs 바이너리 파일

파일에 값을 저장하는 방법은 두 가지다.

  • 텍스트 파일 (fprintf/fscanf) — 값을 사람이 읽을 수 있는 글자로 적는다. 4920 은 문자 '4' '9' '2' '0'. 메모장으로 열어 볼 수 있고, 손으로 고칠 수도 있다.
  • 바이너리 파일 (fwrite/fread) — 메모리의 바이트를 그대로 쏟아붓는다. 작고 빠르지만 메모장으로 열면 깨져 보인다.

우리 게임은 텍스트를 골랐다. 세이브 파일을 메모장으로 열어 자본금을 확인하거나(디버깅), 일부러 고쳐 치트를 쓸 수도 있어서(학습 효과) 입문 단계에 알맞다. 바이너리 세이브는 15장 이후의 선택 과제다.

13.5 게임에 적용 — 세이브와 로드

이제 회사 전체를 파일에 저장한다. 매직 문자열(이게 우리 세이브 파일이 맞는지 확인용)과 버전을 먼저 쓰고, 자본금·직원·프로젝트·출시작을 순서대로 적는다.

int SaveGame(const Company *c)
{
    FILE *fp = fopen("save.txt", "w");
    if (fp == NULL) return 0;                 /* 열기 실패 → 저장 실패 */

    fprintf(fp, "CSGP_TYCOON_SAVE 1\n");      /* 매직 + 버전 */
    fprintf(fp, "%s\n", c->name);
    fprintf(fp, "%d %d %d %d\n", c->money, c->fame, c->year, c->month);

    fprintf(fp, "%d\n", c->staffCount);       /* 직원 수, 그다음 직원들 */
    for (int i = 0; i < c->staffCount; i++)
        fprintf(fp, "%s %d %d %d %d\n", c->staff[i].name, c->staff[i].dev,
                c->staff[i].cre, c->staff[i].salary, c->staff[i].months);
    /* ... 프로젝트·출시작·유행도 같은 방식으로 ... */

    fclose(fp);
    return 1;                                 /* 저장 성공 */
}

로드는 정확히 같은 순서로 읽되, 파일이 없거나 매직이 안 맞으면 실패로 처리해 새 게임으로 넘긴다.

int LoadGame(Company *c)
{
    char magic[32]; int ver;
    FILE *fp = fopen("save.txt", "r");
    if (fp == NULL) return 0;                 /* 세이브 없음 → 새 게임 */

    if (fscanf(fp, "%31s %d", magic, &ver) != 2 ||
        strcmp(magic, "CSGP_TYCOON_SAVE") != 0 || ver != 1) {
        fclose(fp);
        return 0;                             /* 우리 세이브가 아니거나 버전 불일치 */
    }
    /* ... 자본금·직원·프로젝트를 저장 때와 같은 순서로 fscanf ... */
    fclose(fp);
    return 1;
}

여기 C 모듈에서 만난 함정이 되돌아온다. 회사명 "EQMENT STUDIO"에는 공백이 있는데, fscanf("%s") 는 공백에서 멈춘다(C 모듈 8.3). 그러면 "EQMENT"만 읽고 "STUDIO"는 다음 필드로 새어 들어가 포맷이 통째로 어긋난다. 해결은 저장할 때 공백을 '_' 로 바꿔 두고, 읽은 뒤 되돌리는 것이다.

/* 저장 전: 공백 → '_' (fscanf 가 공백에서 안 끊기게) */
void Encode(char *dst, const char *src, int size)
{
    int i;
    for (i = 0; i < size - 1 && src[i] != '\0'; i++)
        dst[i] = (src[i] == ' ') ? '_' : src[i];
    dst[i] = '\0';
}
/* 읽은 후: '_' → 공백 (원래 이름 복원) */
void Decode(char *s)
{
    for (; *s != '\0'; s++)
        if (*s == '_') *s = ' ';
}

C 모듈에서 "scanf %s 는 공백에서 잘린다"를 배울 때는 그저 성가신 성질이었는데, 파일을 다루는 지금은 그것이 설계를 바꿔야 하는 실제 문제가 된다. 문법의 작은 성질 하나가 어떻게 실전에서 발목을 잡는지 — 이것이 배운 것을 끝까지 밀어붙일 때 얻는 통찰이다.

로드 실패 시 안전하게. 로드가 중간에 실패하면(파일이 손상됐거나 형식이 어긋나면), 이미 잡은 메모리를 free 하고 깨끗이 새 게임으로 돌아가야 한다. 실제 코드는 goto fail; 로 한 곳에 모아 이 정리를 처리한다 — 실패 지점이 여럿일 때 뒷정리를 한 군데로 모으는 실용적 패턴이다.

14장 · 여러 파일로 분리 — 프로젝트답게

세이브까지 붙자 main.c 는 수백 줄이 되었다. 이제 문법이 아니라 큰 코드를 다루는 법을 배운다. 관련된 코드끼리 파일로 나누는 것이다.

14.1 왜 파일을 나누나

한 파일에 모든 게 있으면, 직원 코드를 고치려 해도 판매·저장 코드를 헤치고 다녀야 한다. 관련된 것끼리 파일로 묶으면 —

  • 찾기 쉽다. 직원 관련은 employee.c 에만 있다.
  • 고치기 안전하다. 판매 코드를 만질 때 직원 코드를 건드릴 일이 없다.
  • 나눠 일할 수 있다. 여럿이 서로 다른 파일을 동시에 작업할 수 있다.

우리 게임은 7개 모듈로 나눈다(GDD §8).

파일 담는 것
main.c main, 타이틀, 게임 루프
company.h/.c Company 정의, 월 정산, 승패 판정
employee.h/.c 지원자·고용·해고·월급
project.h/.c 기획·개발 진행·출시(평점)
market.h/.c 판매·유행·랜덤 이벤트
savefile.h/.c 세이브/로드
ui.h/.c 화면 출력·입력 공용 함수

14.2 헤더(.h)와 소스(.c) — 선언과 정의를 나누기

파일을 나누면 문제가 생긴다. main.c 가 employee.c 의 TotalSalary 함수를 부르려면, 그 함수가 "어떻게 생겼는지"(반환형·매개변수)를 알아야 한다. 이 선언을 담는 것이 헤더 파일(.h)이다.

  • 헤더(.h) — "이런 함수/자료형이 있다"는 선언(명세). 함수의 몸통은 없다.
  • 소스(.c) — 그 함수의 실제 정의(구현).
/* employee.h — 선언 (이 파일을 include 하면 함수 사용법을 알게 된다) */
int  TotalSalary(const Company *c);
void QuitEmployee(Company *c, int idx);
/* employee.c — 정의 (실제 구현) */
#include "employee.h"
int TotalSalary(const Company *c) { /* ... 실제 코드 ... */ }
/* main.c — employee.h 를 include 하면 TotalSalary 를 쓸 수 있다 */
#include "employee.h"

6장에서 배운 함수의 "선언"과 "정의"가, 파일 단위로 커진 셈이다. 헤더는 목차, 소스는 본문이다.

extern — 파일 사이에서 변수를 공유하기. 함수뿐 아니라 전역 변수를 여러 파일이 공유해야 할 때는 extern 을 쓴다. 헤더에 extern int g_score; 라고 선언만 두고, 한 소스 파일에서 int g_score; 로 딱 한 번 정의한다. 다른 파일들은 헤더를 include 해 그 변수에 접근한다. (우리 게임은 상태를 전역 대신 Company *c 로 넘겨 다녀서 extern 을 거의 안 쓰지만, 큰 프로젝트에서 자주 만나는 도구다.)

14.3 인클루드 가드 — 같은 헤더를 두 번 포함하지 않기

헤더를 여러 파일이 include 하다 보면, 한 파일에 같은 헤더가 두 번 들어가는 일이 생긴다(예: main.c 가 employee.h 와 savefile.h 를 include 하는데, 둘 다 내부에서 company.h 를 include 하는 경우). 그러면 Company 구조체가 두 번 정의되어 컴파일 에러가 난다. 이를 막는 것이 인클루드 가드다.

#ifndef COMPANY_H        /* 만약 COMPANY_H 가 아직 정의 안 됐으면 */
#define COMPANY_H        /* 정의하고, 아래 내용을 포함한다 */

typedef struct { /* ... Company ... */ } Company;
/* ... 선언들 ... */

#endif                   /* COMPANY_H 끝 */

동작은 단순하다 — 이 헤더가 처음 포함되면 COMPANY_H 를 정의하고 내용을 읽는다. 두 번째로 포함되려 하면 COMPANY_H 가 이미 정의돼 있으니 #ifndef 가 거짓이 되어 내용을 통째로 건너뛴다. 그래서 몇 번을 include 해도 실제 정의는 한 번만 들어간다. 모든 헤더 파일은 이 #ifndef / #define / #endif 삼단 구조로 감싼다 — C 헤더의 관례다.

14.4 게임에 적용 — 나눠도 동작은 그대로

파일 분리는 동작을 전혀 바꾸지 않는 리팩터링이다. main.c 하나였던 게임을 7개 파일로 쪼개도, 컴파일해 실행하면 완전히 똑같이 돈다. 달라진 건 오직 "코드가 사는 집"뿐이다. 각 소스는 자기가 필요한 헤더를 include 해 다른 모듈의 함수를 부른다.

/* market.c — 판매·이벤트. 직원 능력치가 필요하니 employee.h 를 include */
#include "market.h"
#include "employee.h"     /* DevSum, CreSum 을 쓰려고 */
#include "project.h"      /* 출시 처리를 쓰려고 */

void UpdateSales(Company *c) { /* ... */ }

이렇게 나눠 두면, 컴파일은 각 .c 를 따로 컴파일한 뒤 하나로 링크한다. build.bat 이 cl *.c 로 모든 소스를 함께 빌드하는 이유다.

15장 · 캡스톤 — enum 과 마무리

마지막 장이다. 지금까지 미뤄 둔 enum 을 정식으로 배우고, 밸런스를 정리하고, 엔딩을 붙여 게임을 출시 버전으로 마감한다.

15.1 enum — 이름 붙인 정수 (드디어 정식으로)

D 모듈부터 우리는 G_RPG, S_SMALL 같은 이름을 "맛보기"로 써 왔다. 그 정체가 바로 enum(열거형) 이다. enum 은 서로 관련된 정수들에 이름을 붙여 묶은 것이다.

typedef enum { G_RPG, G_ACTION, G_PUZZLE, G_SIM, G_SPORTS, GENRE_COUNT } Genre;
/*              0       1         2         3      4         5           */

enum 은 첫 이름부터 0, 1, 2… 순서로 정수를 자동 부여한다. G_RPG 는 그냥 0 이지만, 코드에 0 대신 G_RPG 라고 쓰면 무슨 장르인지 한눈에 읽힌다. 장르를 int 로 두고 if (genre == 2) 라고 쓰는 것과, Genre 로 두고 if (genre == G_PUZZLE) 라고 쓰는 것의 차이다.

Genre trend = G_RPG;                 /* 0 대신 의미 있는 이름 */
if (proj.genre == trend)             /* "장르가 유행과 같으면" — 읽힌다 */
    sales = sales * 3 / 2;

GENRE_COUNT 트릭. enum 의 마지막에 GENRE_COUNT 를 두면, 그것은 자동으로 "앞 항목들의 개수"(여기선 5)가 된다. 그래서 rand() % GENRE_COUNT 로 무작위 장르를 뽑거나(E 모듈), for (g = 0; g < GENRE_COUNT; g++) 로 모든 장르를 순회할 수 있다. 장르를 추가해도 이 코드들은 자동으로 맞춰진다 — enum 하나만 고치면 된다. 흩어진 숫자에 이름을 주는 것을 넘어, 개수까지 이름으로 관리하는 기법이다.

15.2 #define 으로 밸런스를 한곳에

게임을 조율하려면 자본금·임대료·월급률 같은 수많은 숫자를 만져야 한다. 이 값들이 코드 곳곳에 흩어져 있으면 조율이 악몽이 된다. 그래서 밸런스 상수를 #define 으로 한곳에 모은다.

/* company.h 맨 위 — 밸런스는 전부 여기서 조율 (15장) */
#define START_MONEY   5000    /* 시작 자본 */
#define OFFICE_RENT   100     /* 임대료 */
#define WIN_MONEY     10000   /* 승리 자산 */
#define SALARY_RATE   30      /* 월급 = (dev+cre)*30 */
#define BASE_PROGRESS 15      /* 월 진행도 = 15 + Σdev */
#define EVENT_CHANCE  30      /* 이벤트 확률 % */

A 모듈의 const RENT 를 기억하는가? #define 은 그 정신의 확장이다 — "매직 넘버(맥락 없는 숫자)를 이름 있는 상수로." 게임이 너무 쉬우면 START_MONEY 를 줄이고, 이벤트가 잦으면 EVENT_CHANCE 를 낮춘다. 밸런싱이 파일 맨 위 몇 줄을 만지는 일이 된다. (const 와 #define 은 쓰임이 겹치지만, 배열 크기나 여러 파일이 공유하는 상수엔 전통적으로 #define 을 많이 쓴다.)

15.3 엔딩 — 상태를 enum 으로

B 모듈에서 게임의 끝은 gameOver(0/1) 한 변수였다. 이제 파산·거상·명작을 구분해야 하니, enum 으로 상태를 넓힌다.

typedef enum { END_NONE, END_BANKRUPT, END_RICH, END_MASTERPIECE } Ending;

매달 판정에서 조건에 맞는 엔딩을 설정하고, 게임 루프가 그것을 보고 끝낸다.

void CheckEnding(Company *c)
{
    if (c->money < 0)                  c->ending = END_BANKRUPT;   /* 자본금 음수 */
    else if (c->bestRating >= 9.0f)    c->ending = END_MASTERPIECE;/* 명작 출시 */
    else if (c->money >= WIN_MONEY)    c->ending = END_RICH;       /* 자산 1억 */
}

엔딩 화면은 switch (c->ending) 로 갈라 각기 다른 결말을 보여 준다 — 4장의 switch 가 게임의 마지막을 장식한다.

switch (c->ending) {
case END_BANKRUPT:    printf("파산... 회사의 문을 닫습니다.\n");     break;
case END_RICH:        printf("거상! 업계의 큰손이 되었습니다.\n");   break;
case END_MASTERPIECE: printf("명작! 당신의 게임이 역사에 남았습니다.\n"); break;
default:              printf("이야기는 여기서 잠시 멈춥니다.\n");    break;
}

B 모듈의 gameOver 플래그가 Ending enum으로 자랐다. 뿌리는 같다 — 게임의 상태를 변수에 담고, 그 변수로 흐름을 정한다. 이 발상은 2부의 실시간 게임에서 _EPhase{ TITLE, INGAME } 페이즈로 다시 이어진다.

15.4 마무리 — v1.0 출시

이제 모든 조각이 제자리를 찾았다. 기획(제목·장르·규모) → 개발(매달 진행) → 출시(평점) → 판매(감쇠하는 수익)의 개발 사이클이 완성되고, 랜덤 이벤트가 회사를 흔들며, 세이브로 이어할 수 있고, 세 엔딩이 게임의 끝을 정한다. 코드는 7개 모듈로 정돈됐고, 밸런스는 #define 한곳에서 조율된다. 우리의 게임 회사 운영 게임이 v1.0으로 출시됐다.

돌이켜 보면, 우리는 1장에서 변수 하나(int money = 5000;)로 시작했다. 그 한 줄이 자라 이제 직원을 고용하고, 게임을 만들어 팔고, 파산과 명작 사이를 오가는 살아있는 프로그램이 되었다. 문법을 배우는 순서가 곧 게임을 완성하는 순서였다 — 이 책이 약속했던 그대로다.

코드 워크스루 — 실제 소스에서 보는 파일·모듈·enum

우리가 배운 세 가지가 Study/Tycoon/ 완성본에서 어떻게 맞물리는지 마지막으로 확인하자.

실제 파일: Study/Tycoon/savefile.c(세이브/로드) · Study/Tycoon/company.h(헤더·enum·상수) · Study/Tycoon/main.c(엔딩)

① 세이브/로드 (savefile.c) — 13.5의 실전판. 매직·버전으로 파일을 검증하고, 실패하면 goto fail; 로 정리한 뒤 새 게임으로 돌아간다. 로드 후엔 저장 안 한 런타임 값(최고 평점 등)을 다시 계산해 복구한다.

if (fscanf(fp, "%31s %d", magic, &ver) != 2 ||
    strcmp(magic, SAVE_MAGIC) != 0 || ver != SAVE_VER) {
    fclose(fp); return 0;              /* 우리 세이브가 아니다 → 새 게임 */
}
/* ... 순서대로 fscanf ... */
for (i = 0; i < c->shelfCount; i++)   /* 저장 안 한 값 복구 */
    if (c->shelf[i].rating > c->bestRating)
        c->bestRating = c->shelf[i].rating;

로드 중 직원을 읽을 때도 E 모듈의 realloc 확장이 그대로 쓰인다 — 저장된 직원이 초기 용량보다 많으면 배열을 키우며 읽는다. 한 모듈이 앞선 모듈들 위에 선다.

② 헤더 구조 (company.h) — 14장의 인클루드 가드와 15장의 enum·#define 이 한 파일에 모여 있다. 밸런스 상수, 세 개의 enum, 구조체, 함수 선언 — 이 헤더가 게임 전체의 "명세서"다.

#ifndef COMPANY_H
#define COMPANY_H
#define START_MONEY 5000            /* 밸런스 상수 (15장) */
/* ... */
typedef enum { G_RPG, ..., GENRE_COUNT } Genre;       /* enum (15장) */
typedef enum { END_NONE, END_BANKRUPT, ... } Ending;
typedef struct { /* ... */ } Company;                 /* 구조체 (9장) */
void AdvanceMonth(Company *c);                         /* 함수 선언 (14장) */
#endif

③ 엔딩 (main.c · EndingScreen) — 15.3의 switch (c->ending). 게임의 마지막 화면이 A~F 모듈의 모든 문법을 한자리에 모은다 — 변수·서식 출력·조건 분기·배열 순회·구조체·enum 이 이 한 함수 안에 함께 있다.

switch (c->ending) {
case END_BANKRUPT:    printf("문을 닫았습니다.\n");    break;
case END_RICH:        printf("업계의 큰손이 되었습니다.\n"); break;
case END_MASTERPIECE: printf("이름을 게임 역사에 남겼습니다.\n"); break;
}
for (i = 0; i < c->shelfCount; i++)   /* 배열 순회로 총 판매량 집계 */
    totalSold += c->shelf[i].totalSales;

이 세 조각이 1부 전체를 요약한다 — 파일로 남기고, 모듈로 나누고, enum 으로 이름 붙인다. 1장의 변수 하나에서 시작한 여정이, 저장되고 정돈된 완성 프로그램으로 끝난다.

그림으로 정리 — 세이브 파일과 모듈 지도

📊 웹 교육문서에서는 세이브→로드 왕복과 모듈 의존 그래프가 애니메이션(V-P1-F)으로 재생된다. 책에서는 아래 도식으로 대신한다.

세이브 파일 — 게임 상태를 순서대로 적어 둔 텍스트

  save.txt (메모장으로 열림 — 텍스트 파일)
  ┌─────────────────────────────────┐
  │ CSGP_TYCOON_SAVE 1              │ ← 매직 + 버전 (로드 시 검증)
  │ EQMENT_STUDIO                   │ ← 회사명 (공백은 '_' 로 인코딩)
  │ 4920 12 2026 3                  │ ← money fame year month
  │ 3                               │ ← 직원 수
  │ 김개발 7 5 360 8                 │ ← 직원 × N
  │ ...                             │
  └─────────────────────────────────┘
   fprintf 로 이 순서대로 쓰고, fscanf 로 이 순서대로 읽는다 (약속).

모듈 의존 지도 — 헤더로 이어진 7개 파일

                      company.h  (Company·enum·상수 — 모두가 포함)
                     ╱    │    ╲
             employee.h project.h market.h   savefile.h   ui.h
                 │        │        │             │          │
             employee.c project.c market.c   savefile.c   ui.c
                     ╲    │    ╱  ╲    │    ╱
                        main.c  (게임 루프 — 모든 모듈을 엮는다)
   각 .c 는 필요한 .h 를 include. #ifndef 가드가 중복 포함을 막는다.

한 문장으로: 파일은 게임을 시간 너머로 저장하고, 모듈은 코드를 공간적으로 정돈하며, enum 은 흩어진 숫자에 이름을 준다.

체크포인트 — 스스로 점검하기

Q1. fopen("save.txt", "w") 로 열었을 때, 그 파일에 이미 내용이 있었다면 어떻게 되나? "r" 과의 차이는?

Q2. FILE *fp = fopen("save.txt", "r"); 뒤에 왜 if (fp == NULL) 검사가 반드시 필요한가?

Q3. 회사명 "EQMENT STUDIO"를 fprintf(fp, "%s\n", name) 로 저장하고 fscanf(fp, "%s", name) 로 읽으면 무엇이 문제인가? 어떻게 푸나?

Q4. 인클루드 가드(#ifndef COMPANY_H / #define COMPANY_H / #endif)는 무엇을 막는가?

Q5. enum { G_RPG, G_ACTION, G_PUZZLE, G_SIM, G_SPORTS, GENRE_COUNT } 에서 G_PUZZLE 과 GENRE_COUNT 의 값은 각각 얼마인가? GENRE_COUNT 는 어디에 쓰나?

해설 펼치기

A1. "w" 모드는 기존 내용을 전부 지우고 새로 쓴다(덮어쓰기). 세이브는 매번 최신 상태로 갈아써야 하니 "w" 가 맞다. "r" 은 읽기 전용이라 쓸 수 없고, 파일이 없으면 NULL 을 반환한다.

A2. 세이브 파일이 아직 없는 첫 실행에서 fopen(..., "r") 은 NULL 을 돌려준다. 검사 없이 fscanf(NULL, ...) 를 부르면 프로그램이 죽는다. NULL 이면 "세이브 없음 → 새 게임"으로 안전하게 넘겨야 한다.

A3. fscanf("%s") 는 공백에서 멈추므로 "EQMENT"만 읽고 "STUDIO"는 다음 필드로 새어 들어가 포맷이 어긋난다. 저장할 때 공백을 '_' 로 바꿔(EQMENT_STUDIO) 쓰고, 읽은 뒤 '_' 를 공백으로 되돌린다(Encode/Decode). — C 모듈의 scanf %s 공백 함정이 파일에서 되돌아온 것.

A4. 같은 헤더가 한 파일에 두 번 이상 포함되는 것을 막는다. 두 번째 포함 때는 COMPANY_H 가 이미 정의돼 있어 #ifndef 가 거짓이 되고 내용을 통째로 건너뛴다. 덕분에 구조체·선언이 중복 정의되어 나는 컴파일 에러를 예방한다.

A5. G_PUZZLE 은 2(0부터 세어 세 번째), GENRE_COUNT 는 5(앞의 5개 다음). GENRE_COUNT 는 "장르 개수"라 rand() % GENRE_COUNT 로 무작위 장르를 뽑거나 for (g = 0; g < GENRE_COUNT; g++) 로 전체를 순회하는 데 쓴다. 장르를 추가해도 이 값이 자동으로 맞춰진다.

실습 미션 — 최고 기록 저장 highscore.c ⭐

본편(타이쿤)이 "한 게임을 계속 키우는 세로 축"이라면, 실습 미션은 이 모듈의 새 문법만 콕 집어 연습하는 독립 미니게임(가로 축) 이다. 먼저 직접 짜 보고, 막히면 힌트를, 그래도 막히면 정답을 펼쳐 대조하자.

드릴 문법 — fopen/fscanf/fprintf/fclose · 파일 열기 실패(NULL) 검사 · rand(E 모듈) · while 루프(B 모듈) 만드는 것 — 하이·로우(다음 수가 클지 작을지) 게임. 연승 기록을 파일에 저장해, 다시 실행하면 이전 최고 기록을 불러온다. 클리어 조건 — 프로그램을 껐다 켜도 최고 연승 기록이 유지된다. 갱신 시 파일에 새로 저장. 규모 — 약 50줄, 파일 1개. 1부에서 배운 파일·랜덤·반복·조건이 한자리에 모이는 마무리 미션이다.

문제 정의

이 게임의 핵심은 "프로그램이 꺼져도 기록이 남는다" 는 것이다. 그러려면 최고 기록을 파일에 저장(fprintf)하고, 시작할 때 다시 읽어야(fscanf) 한다. 첫 실행에는 기록 파일이 없으니 fopen 이 NULL 을 돌려주는데, 이때 "기록 없음(0)"으로 처리하는 것이 이 미션의 관문이다 — 13.3의 NULL 검사가 그대로 필요하다.

목표 / 클리어 조건

  1. 시작 시 "best.txt" 에서 최고 연승을 읽는다. 파일이 없으면(NULL) 0으로.
  2. 하이·로우 게임: 현재 수(1~9)를 보여 주고, 다음 수가 클지(>) 작을지(<) 맞힌다.
  3. 맞히면 연승 +1, 틀리면 게임 종료. (같으면 무승부로 계속)
  4. 이번 연승이 최고 기록을 넘으면 "best.txt" 에 새로 저장한다.
  5. 껐다 다시 켜면 이전 최고 기록이 그대로 보여야 한다.

힌트

힌트 1 — 기록 불러오기 (NULL 검사) 
int LoadBest(void)
{
    int best = 0;
    FILE *fp = fopen("best.txt", "r");
    if (fp == NULL)               /* 첫 실행: 파일 없음 → 0 */
        return 0;
    fscanf(fp, "%d", &best);
    fclose(fp);
    return best;
}
힌트 2 — 기록 저장하기 
void SaveBest(int best)
{
    FILE *fp = fopen("best.txt", "w");   /* "w": 매번 최신 기록으로 덮어쓰기 */
    if (fp == NULL) return;
    fprintf(fp, "%d\n", best);
    fclose(fp);
}
힌트 3 — 하이·로우 판정

현재 수 cur 와 새로 뽑은 next 를 비교한다. 사용자가 "크다(1)"를 골랐는데 실제로 next > cur 이면 정답, "작다(2)"인데 next < cur 이면 정답이다.

if ((sel == 1 && next > cur) || (sel == 2 && next < cur)) {
    streak++; cur = next;      /* 맞힘: 연승 +1, 현재 수를 next 로 갱신 */
} else if (next == cur) {
    /* 무승부: 아무 변화 없이 다시 */
} else {
    break;                     /* 틀림: 게임 종료 */
}

정답 코드

정답 펼치기 — highscore.c 
#define _CRT_SECURE_NO_WARNINGS
#include <stdio.h>
#include <stdlib.h>
#include <time.h>

/* 최고 기록 불러오기 — 파일이 없으면(첫 실행) 0 */
int LoadBest(void)
{
    int best = 0;
    FILE *fp = fopen("best.txt", "r");
    if (fp == NULL) return 0;          /* NULL 검사 — 세이브 없음 */
    fscanf(fp, "%d", &best);
    fclose(fp);
    return best;
}

/* 최고 기록 저장하기 — "w" 로 덮어쓰기 */
void SaveBest(int best)
{
    FILE *fp = fopen("best.txt", "w");
    if (fp == NULL) return;
    fprintf(fp, "%d\n", best);
    fclose(fp);
}

int main(void)
{
    int best, streak = 0, cur, next, sel;

    srand((unsigned)time(NULL));       /* 난수 초기화 (한 번만) */
    best = LoadBest();                 /* 이전 최고 기록 불러오기 */

    printf("=== 하이·로우 (역대 최고 연승: %d) ===\n", best);
    cur = 1 + rand() % 9;              /* 현재 수 1~9 */

    for (;;) {
        printf("\n현재 수: %d\n", cur);
        printf("다음 수는? [1] 더 크다(>)   [2] 더 작다(<) : ");
        scanf("%d", &sel);

        next = 1 + rand() % 9;

        if (next == cur) {             /* 무승부 → 계속 */
            printf("  같은 수(%d)! 무승부, 다시.\n", next);
            continue;
        }
        if ((sel == 1 && next > cur) || (sel == 2 && next < cur)) {
            streak++;                  /* 맞힘 */
            printf("  다음 수 %d — 정답! 연승 %d\n", next, streak);
            cur = next;
        } else {                       /* 틀림 */
            printf("  다음 수 %d — 틀렸습니다. %d 연승에서 종료.\n", next, streak);
            break;
        }
    }

    if (streak > best) {               /* 신기록이면 파일에 갱신 */
        printf("\n>> 신기록! %d 연승 (이전 기록 %d)\n", streak, best);
        SaveBest(streak);
    } else {
        printf("\n>> %d 연승. 최고 기록 %d 은(는) 못 넘었습니다.\n", streak, best);
    }
    return 0;
}

빌드: cl highscore.c (MSVC) 또는 gcc highscore.c -o highscore

함정/체크포인트

  • 첫 실행에는 best.txt 가 없다. fopen(..., "r") 이 NULL 을 돌려주는데, 이걸 검사하지 않고 fscanf 를 부르면 프로그램이 죽는다. NULL 이면 "기록 없음(0)"으로 처리하는 것이 이 미션의 핵심이다.
  • 저장은 "w" 모드다. "a"(이어쓰기)로 열면 기록이 지워지지 않고 계속 쌓여, 다음 로드에서 첫 줄만 읽혀 엉킨다. "최신 기록 하나만 유지"에는 "w" 가 맞다.
  • 프로그램을 껐다 다시 켜서 기록이 유지되는지 꼭 확인하자. 그것이 "파일에 저장했다"는 유일한 증거다 — 메모리 변수와 파일의 차이를 눈으로 보는 순간이다.

확장 과제

  • 최고 기록과 함께 날짜(time)나 플레이어 이름(문자열)도 저장해, 세이브 포맷을 여러 필드로 늘려 보자(본편 세이브 파일의 축소판).
  • 기록 파일을 일부러 메모장으로 열어 숫자를 고쳐 보자. 텍스트 세이브는 이렇게 "치트"가 가능하다 — 13.4에서 말한 텍스트 포맷의 양면성을 체감할 수 있다.

1부를 마치며 — 변수 하나에서 완성된 게임까지

돌아보자. 우리는 A 모듈 1장에서 int money = 5000; 이라는 단 한 줄로 시작했다. 그 변수 하나가 —

  • A 모듈에서 계산되고 갈라졌고(값·연산·조건),
  • B 모듈에서 반복되며 게임이 "돌기" 시작했고(루프·함수),
  • C 모듈에서 사람과 이름을 얻었고(배열·문자열),
  • D 모듈에서 구조체로 모양을 갖추고 포인터로 함수에 넘겨졌고(1부의 고비),
  • E 모듈에서 회사를 무한히 키우고 시장을 살아나게 했고(동적 메모리·랜덤),
  • F 모듈에서 파일에 저장되고, 모듈로 정돈되고, 엔딩을 맞았다(파일·분리·캡스톤).

문법을 배우는 순서가 곧 게임을 완성하는 순서였다. 우리는 "이 문법을 어디에 쓰지?"라고 물을 필요가 없었다 — 배운 즉시 회사의 새 기능이 되었으니까. 그것이 이 책 1부의 약속이었고, 회사는 그 약속대로 자랐다.

이제 우리는 C의 뼈대를 손에 쥐었다. 값·분기·반복·함수·데이터의 모양·메모리·파일 — 프로그래밍의 기본기가 모두 하나의 살아있는 게임 안에서 익혀졌다.

하지만 우리 게임에는 아직 없는 것이 있다. 화면은 scanf 앞에서 멈춰 사람을 기다리고, 매 화면은 system("cls") 로 지웠다 다시 그려 깜빡인다. 뱀이 스스로 기어가고, 블록이 떨어지고, 탱크가 서로를 쏘는 — 매 순간 스스로 움직이는 실시간 세계는 여기에 없다. "깜빡임 없는 화면"이 탐나기 시작한다면, 그것이 바로 2부로 넘어갈 때가 됐다는 신호다.

2부에서 우리는 C를 C++ 로 넓히고, Windows 콘솔 API 위에 게임 루프·더블 버퍼링·씬 시스템을 얹어, 실시간으로 돌아가는 게임 7종을 만든다. 1부의 구조체는 클래스가 되고, 함수는 메서드가 되며, Company *c 를 주고받던 감각은 "객체가 자기 데이터를 스스로 관리한다"는 감각으로 자란다. 1부에서 세운 뼈대 위에, 2부가 살아 움직이는 세계를 짓는다.

수고했다. 첫 번째 강을 건넜다. 이제 두 번째 강으로 간다.
