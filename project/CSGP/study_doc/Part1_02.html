B 모듈 · 5–6장 — 반복문과 함수: 회사에 심장을 달다

모듈 B · 5–6장 · 게임 버전 v0.2 → v0.3 · 예상 학습 시간 3–5시간 처음 배우는 기술 — while for do-while break continue 무한 루프 함수 정의·호출 매개변수·반환값 void 지역 변수 전역 변수 스코프 함수 분해(리팩터링)

A 모듈에서 만든 게임은 딱 한 번만 돌고 끝났다. 한 달을 정산하면 프로그램이 종료됐다. 이 모듈에서 그 한 달을 반복문으로 감싸 "다음 달, 또 다음 달…"이 이어지게 만든다. 그 순간 우리 회사는 비로소 살아 움직이기 시작한다 — 게임에 심장이 달리는 것이다. 그리고 루프가 생기며 부풀어 오른 main 을 함수로 나눠, 다시 읽을 수 있는 코드로 정리한다.

이 모듈의 학습 목표

이 모듈을 마치면 다음을 할 수 있다.

  1. while · for · do-while 세 반복문의 차이를 알고, 상황에 맞게 고를 수 있다.
  2. break 로 반복을 즉시 끝내고, continue 로 이번 회차만 건너뛸 수 있다.
  3. 게임의 메인 루프(while (!gameOver))를 짜서, 한 턴짜리 프로그램을 "계속 도는 게임"으로 만들 수 있다.
  4. 반복되는 코드 조각을 함수로 묶고, 매개변수로 값을 넘기며 반환값으로 결과를 받을 수 있다.
  5. 지역 변수와 전역 변수의 수명·범위(스코프) 차이를 설명하고, 왜 게임 상태를 전역에 두는지(그리고 그게 왜 임시방편인지) 안다.
  6. 비대해진 main 을 역할별 함수로 분해(리팩터링) 해, 같은 동작을 더 읽기 좋은 구조로 바꿀 수 있다.

왜 지금, 반복문과 함수인가

A 모듈의 도착점을 떠올려 보자. 우리는 현황판을 그리고, 메뉴를 받고, 한 달을 정산하고, 파산을 판정했다. 완벽한 게임의 한 순간이다. 문제는 딱 하나 — 그 순간이 한 번밖에 일어나지 않는다는 것이다.

A 모듈:   [현황판 → 메뉴 → 정산 → 판정]   ← 여기서 프로그램 끝
B 모듈:   ┌─────────────────────────────┐
          │ [현황판 → 메뉴 → 정산 → 판정] │ ← 반복!
          └──────────────┬──────────────┘
                    파산할 때까지 계속

게임이란 결국 같은 흐름을 조건이 끝날 때까지 반복하는 것이다. 뱀은 매 순간 한 칸씩 움직이고(2부), 우리 회사는 매달 정산을 한다. 이 "매 순간"을 만드는 문법이 반복문이다. 5장에서 반복문을 배우는 순간, 우리 게임은 처음으로 "게임"이라 부를 만한 것이 된다.

그런데 반복문을 main 에 그냥 쑤셔 넣으면, 안 그래도 길던 main 이 감당 못 할 만큼 부푼다. 현황판 출력, 메뉴 분기, 월 정산이 전부 한 함수 안에 뒤엉킨다. 그래서 6장에서 함수를 배워, 이 덩어리를 PrintHeader(), MainMenu(), AdvanceMonth() 같은 역할별 조각으로 나눈다. 반복이 코드를 키우고, 함수가 그 코드를 다시 정리한다 — 5장과 6장이 짝을 이루는 이유다.

이 모듈에서 만드는 것. 실행하면 회사 현황판이 뜨고, 메뉴에서 "다음 달 진행"을 고를 때마다 한 달씩 흐르며, 고정 수입에서 임대료가 빠져 자본금이 오르내리고, 자본금이 음수가 되면 파산 엔딩으로 게임이 끝나는 — GDD 마일스톤 v0.3 "돌아가는 회사", 즉 처음으로 끝까지 플레이되는 최소 게임이다.

5장 · 반복문 — 게임에 심장을 달다

5.1 while — 조건이 참인 동안 반복

while 은 가장 기본적인 반복문이다. "괄호 안 조건이 참인 동안, 블록을 계속 실행하라."

int month = 1;
while (month <= 3) {          /* month 가 3 이하인 동안 */
    printf("%d월\n", month);  /* 실행 */
    month++;                  /* 그리고 month 를 늘린다 */
}
/* 출력: 1월  2월  3월 */

동작을 단계로 보면 이렇다.

  ① 조건 검사(month <= 3)  ──참──▶  ② 블록 실행  ──▶  ③ ①로 돌아감
            │
           거짓
            ▼
        루프 밖으로 (반복 종료)

while 에서 가장 흔한 실수는 조건을 바꾸는 코드를 빠뜨리는 것이다. 위에서 month++ 를 빼면 month 는 영원히 1이라 조건이 늘 참 — 무한 루프에 빠진다. "이 루프는 언제 끝나는가?"를 항상 자문하자.

의도한 무한 루프. 반대로, 일부러 끝나지 않게 만들고 안에서 break 로 빠져나오는 방식도 흔하다. while (1) { … } 또는 for (;;) { … } 는 "조건이 항상 참"인 무한 루프다. 우리 게임의 실제 완성본이 바로 이 for(;;) 형태를 쓴다(5.5·워크스루에서 다시 본다).

5.2 for — 횟수가 정해진 반복

반복 횟수가 정해져 있을 땐 for 가 깔끔하다. while 에 흩어져 있던 세 요소 — 초기화·조건·증감 — 를 한 줄에 모은다.

int i;
for (i = 0; i < 3; i++) {     /* 초기화; 조건; 증감 */
    printf("%d\n", i);        /* 0  1  2 */
}

for (i = 0; i < 3; i++) 는 정확히 아래 while 과 같다. 셋을 한 줄에 모아 두니, "이 루프가 몇 번 도는지"가 한눈에 보인다.

i = 0;              /* 초기화 */
while (i < 3) {     /* 조건 */
    printf("%d\n", i);
    i++;            /* 증감 */
}

언제 무엇을 쓰나? — "정해진 횟수"면 for(직원 4명 순회, 게이지 20칸 그리기), "조건이 끝날 때까지"면 while(파산할 때까지 월 진행). 게임 루프는 후자라 while 계열이다.

5.3 do-while — 최소 한 번은 실행

do-while 은 조건을 끝에서 검사한다. 그래서 블록이 최소 한 번은 실행된다. 메뉴 입력처럼 "일단 한 번은 보여 주고, 잘못 고르면 다시"인 상황에 맞는다.

int sel;
do {
    printf("메뉴 (0~6): ");
    scanf("%d", &sel);
} while (sel < 0 || sel > 6);   /* 범위 밖이면 다시 입력받는다 */

세 반복문을 한 표로 정리하면.

반복문 조건 검사 시점 최소 실행 잘 맞는 상황
while 시작 전 0번 가능 조건이 끝날 때까지 (월 루프)
for 시작 전 0번 가능 정해진 횟수 (배열 순회)
do-while 끝난 후 최소 1번 일단 한 번은 하고 재시도 (입력 검증)

5.4 break 와 continue — 흐름을 끊고 건너뛰기

반복 안에서 흐름을 세밀히 제어하는 두 키워드가 있다.

  • break — 반복문을 즉시 완전히 빠져나온다.
  • continue — 이번 회차의 남은 부분만 건너뛰고, 다음 회차로 넘어간다.
int i;
for (i = 1; i <= 10; i++) {
    if (i == 5) break;        /* i 가 5 가 되면 루프 자체를 끝냄 → 1 2 3 4 */
    printf("%d ", i);
}

for (i = 1; i <= 5; i++) {
    if (i % 2 == 0) continue; /* 짝수면 출력만 건너뜀 → 1 3 5 */
    printf("%d ", i);
}

break 는 4장의 switch 에서도 만났다(각 case 를 끝내는 그 break). 반복문 안의 break 도 같은 정신 — "여기서 그만" 이다. 우리 게임에선 "사용자가 종료를 고르면 break 로 게임 루프를 빠져나온다".

5.5 게임에 적용 — 월 루프 (while (!gameOver))

이제 5장을 게임에 적용한다. A 모듈의 "한 턴"을 while 로 감싸, 파산할 때까지 매달 반복되게 만든다. 게임의 상태는 아직 구조체가 없으므로(구조체는 9장) 전역 변수로 둔다. 그리고 게임이 끝났는지를 나타내는 gameOver 플래그 하나가 루프의 조건이 된다.

#define _CRT_SECURE_NO_WARNINGS
#include <stdio.h>

#define RENT 100

/* --- 게임 상태 : 전역 변수 (구조체는 9장, 포인터로 정리는 10장) --- */
int money = 5000, fame = 0, year = 2026, month = 1;
int gameOver = 0;              /* 0 = 진행 중, 1 = 게임 끝 */

int main(void)
{
    int sel, income;

    while (!gameOver) {                 /* ★ 게임의 심장 — 파산 전까지 매달 반복 */
        /* 현황판 */
        printf("\n== %d년 %2d월 | 자본금 %d만원 | 명성 %d ==\n",
               year, month, money, fame);
        printf("  [6] 다음 달 진행   [0] 게임 종료\n  >> 선택: ");
        scanf("%d", &sel);

        if (sel == 0)
            break;                       /* 사용자 종료 → 루프 탈출 */

        if (sel == 6) {
            /* 한 달 진행: 고정 수입 − 세금 − 임대료 */
            printf("  이번 달 수입(만원): ");
            scanf("%d", &income);
            int tax    = income * 10 / 100;
            int profit = income - tax - RENT;
            money += profit;

            printf("  순이익 %+d만원  →  자본금 %d만원\n", profit, money);

            month++;                     /* 달력 넘김 */
            if (month > 12) { month = 1; year++; }

            if (money < 0)               /* 파산 판정 */
                gameOver = 1;            /* → 다음 조건 검사에서 while 이 끝난다 */
        }
    }

    /* 루프를 빠져나온 이유에 따라 엔딩 */
    if (gameOver)
        printf("\n>> 파산했습니다. [EQMENT STUDIO] 의 문을 닫습니다.\n");
    else
        printf("\n>> 게임을 종료합니다. 수고하셨습니다.\n");
    return 0;
}

이 코드가 A 모듈과 결정적으로 다른 점은 단 하나 — 전체가 while (!gameOver) 로 감싸여 있다는 것. sel == 6 을 고를 때마다 한 달이 지나고, 자본금이 음수가 되면 gameOver = 1 이 되어 다음 조건 검사에서 루프가 끝난다. 처음으로 우리 게임은 "한 번 하고 끝"이 아니라 끝날 때까지 이어지는 무언가가 되었다.

gameOver 가 상태의 전부다. 지금 이 게임의 "지금 어떤 상황인가"는 오직 gameOver 한 변수로 표현된다. 2부의 실시간 게임들은 이걸 _EPhase{ TITLE, INGAME } 같은 여러 페이즈로 확장하지만(2부 Snake), 뿌리는 같다 — 상태를 변수에 담고, 루프가 그 변수를 본다. 15장에서는 이 gameOver 를 enum Ending { NONE, BANKRUPT, RICH, MASTERPIECE } 로 키워 파산·승리를 구분한다.

6장 · 함수 — 커진 코드를 정리하다

5장에서 게임이 돌기 시작했지만, main 하나에 현황판·메뉴·정산이 다 들어가 벌써 답답하다. 앞으로 직원·개발·판매가 붙으면 main 은 수백 줄이 된다. 그래서 함수로 나눈다. 함수는 "이름 붙인 코드 묶음"이다 — 한 번 정의해 두고, 이름으로 여러 번 부른다.

6.1 함수 정의·호출·반환

함수는 반환형 · 이름 · 매개변수 · 몸통 으로 이루어진다.

/* 반환형   이름     (매개변수)          */
   int      Square  (int n)
   {
       return n * n;      /* 결과를 돌려준다 */
   }

int main(void)
{
    int a = Square(5);    /* 호출 — a 는 25 */
    printf("%d\n", Square(a));   /* 625 */
    return 0;
}
  • return 은 값을 돌려주며 함수를 끝낸다.
  • 돌려줄 값이 없는 함수는 반환형을 void 로 한다. void PrintHeader(void) 는 "받는 것도 돌려주는 것도 없이, 화면만 그리는 함수".
void PrintLine(void)      /* 받지도 돌려주지도 않음 */
{
    printf("--------------------------------------------------\n");
}

6.2 매개변수와 반환값 — 함수에 값을 주고받기

매개변수는 함수에 넘기는 입력, 반환값은 함수가 주는 출력이다. 게임의 세금 계산을 함수로 빼 보자.

/* 수입을 받아 세금(10%)을 돌려준다 */
int CalcTax(int income)
{
    return income * 10 / 100;
}

int tax = CalcTax(820);   /* 82 */

여기서 중요한 사실 하나 — 매개변수는 값의 복사본이다. CalcTax(820) 을 부르면 income 에 820의 복사본이 들어간다. 함수 안에서 income 을 바꿔도 바깥엔 영향이 없다. 이것을 값에 의한 전달(pass by value) 이라 한다. 대부분의 경우 이걸로 충분하지만 — "함수가 바깥의 원본을 직접 고쳐야 할 때"는 이야기가 달라진다. 그 문제와 해법(포인터)은 D 모듈의 핵심 주제다. 지금은 "함수에 넘긴 값은 복사된다"만 기억하자.

6.3 지역 변수 vs 전역 변수 — 스코프

변수는 어디서 태어났느냐에 따라 살 수 있는 범위(스코프)와 수명이 다르다.

  • 지역 변수 — 함수(또는 블록) 안에서 선언. 그 함수가 실행되는 동안만 살고, 함수가 끝나면 사라진다. 다른 함수에서는 보이지 않는다.
  • 전역 변수 — 모든 함수 바깥에서 선언. 프로그램 내내 살아 있고, 모든 함수가 접근할 수 있다.
int money = 5000;         /* 전역 — 어느 함수에서든 보인다 */

void PayRent(void)
{
    int rent = 100;       /* 지역 — 이 함수 안에서만 산다 */
    money -= rent;        /* 전역 money 를 직접 줄인다 */
}

int main(void)
{
    PayRent();
    printf("%d\n", money); /* 4900 — PayRent 가 바꾼 전역이 반영됨 */
    /* printf("%d\n", rent);  ← ✗ 에러! rent 는 PayRent 안에서만 산다 */
    return 0;
}

우리 게임은 아직 자본금·명성·날짜를 전역 변수로 둔다. 그래야 PayRent(), AdvanceMonth() 같은 함수들이 매개변수·반환값 없이도 게임 상태를 직접 고칠 수 있기 때문이다. 값 전달(6.2)로는 원본을 못 고치니, 지금 단계에선 전역이 현실적인 선택이다.

⚠️ 전역 변수는 임시방편이다 — D 모듈의 복선. 전역은 편하지만 위험하다. 어느 함수가 money 를 언제 바꾸는지 코드만 봐선 알 수 없다(모두가 만질 수 있으니까). 프로그램이 커질수록 이건 버그의 온상이 된다. 10장에서 우리는 전역 변수를 버리고, 게임 상태를 하나로 묶은 뒤(구조체·9장) 그 주소를 함수에 넘기는(포인터·10장) 방식으로 갈아탄다. 그러면 함수 서명만 봐도 "이 함수가 회사를 만진다"가 드러난다. 지금의 전역은 그 개선을 향한 디딤돌이다.

6.4 게임에 적용 — main 을 함수로 분해하고, 자본금에 콤마를 달다

이제 5.5의 비대한 main 을 역할별 함수로 나눈다. 동작은 그대로, 구조만 바꾸는 것 — 이것이 리팩터링이다.

#define _CRT_SECURE_NO_WARNINGS
#include <stdio.h>

#define RENT 100

int money = 5000, fame = 0, year = 2026, month = 1;
int gameOver = 0;

/* ── 정수를 3자리 콤마로 출력 (반복문 + 함수 예제) ──
   A 모듈에서 예고한 '자본금 콤마'. 자릿수를 세고, 앞자리부터 찍으며 3자리마다 콤마.
   완성본은 이 로직을 문자열 버퍼에 담는 MoneyStr 로 리팩터링한다(C 모듈). */
void PrintMoney(int v)
{
    int digits = 1, t, place = 1, i;
    if (v < 0) { putchar('-'); v = -v; }

    t = v;
    while (t >= 10) { t /= 10; digits++; }        /* 자릿수 세기 */
    for (i = 1; i < digits; i++) place *= 10;     /* place = 10^(digits-1) */

    for (i = digits; i >= 1; i--) {
        putchar('0' + (v / place) % 10);          /* 맨 앞자리부터 한 글자씩 */
        if (i > 1 && (i - 1) % 3 == 0) putchar(','); /* 3자리마다 콤마 */
        place /= 10;
    }
}

/* ── 현황판 출력 ── */
void PrintHeader(void)
{
    printf("\n==================================================\n");
    printf("  %-16s   %d년 %2d월     명성 ★ %d\n", "EQMENT STUDIO", year, month, fame);
    printf("  자본금 ");  PrintMoney(money);  printf("만원\n");
    printf("==================================================\n");
}

/* ── 한 달 정산 (전역 money/month 를 직접 갱신) ── */
void AdvanceMonth(void)
{
    int income, tax, profit;
    printf("  이번 달 수입(만원): ");
    scanf("%d", &income);

    tax    = income * 10 / 100;
    profit = income - tax - RENT;
    money += profit;

    printf("  순이익 %+d만원  →  자본금 ", profit);
    PrintMoney(money);  printf("만원\n");

    month++;
    if (month > 12) { month = 1; year++; }
    if (money < 0) gameOver = 1;
}

/* ── 메뉴: 사용자가 종료를 고르면 1 을 반환 ── */
int MainMenu(void)
{
    int sel;
    printf("  [6] 다음 달 진행   [0] 게임 종료\n  >> 선택: ");
    scanf("%d", &sel);
    switch (sel) {
    case 6: AdvanceMonth(); break;
    case 0: return 1;              /* 종료 신호 */
    default: printf("  잘못된 선택입니다.\n"); break;
    }
    return 0;
}

int main(void)
{
    while (!gameOver) {
        PrintHeader();
        if (MainMenu())            /* 1 을 돌려주면(사용자 종료) */
            break;                 /* 루프 탈출 */
    }
    printf(gameOver ? "\n>> 파산! 회사의 문을 닫습니다.\n"
                    : "\n>> 게임을 종료합니다.\n");
    return 0;
}

같은 게임이지만 main 이 놀랄 만큼 짧아졌다. 이제 main 은 "반복하며 현황판을 그리고 메뉴를 부른다"는 큰 흐름만 말하고, 세부는 각 함수에 숨었다. 새 기능(직원·개발)이 붙어도 main 은 거의 그대로고, 함수만 늘어난다. 함수 분해는 코드를 "읽을 수 있는 이야기"로 만든다.

main 이 짧아진다는 것의 의미. 좋은 main 은 목차 같다. 큰 흐름(제목)만 보여 주고, 자세한 내용(본문)은 각 함수로 내려보낸다. 방금 우리가 만든 PrintHeader → MainMenu → AdvanceMonth 구조는, 실제 완성본의 TitleScreen → GameLoop → MainMenu → AdvanceMonth 구조와 사실상 같은 모양이다(바로 아래 워크스루에서 확인한다). 우리는 지금 진짜 게임의 뼈대를 짜고 있다.

코드 워크스루 — 실제 소스에서 보는 루프와 함수

우리가 만든 "루프 + 함수 분해" 구조가, Study/Tycoon/ 완성본에서 어떻게 나타나는지 확인하자. 완성본은 구조체·포인터(Company *c, D 모듈)를 쓰지만, 반복문과 함수 분해라는 뼈대는 지금 배운 그대로다.

실제 파일: Study/Tycoon/main.c(게임 루프·메뉴) · Study/Tycoon/company.c(월 정산)

① 게임 루프 (main.c · GameLoop) — 우리의 while (!gameOver) 에 해당한다. 완성본은 무한 루프 for(;;) 를 쓰고, 끝낼 이유(사용자 종료·파산·승리)를 만나면 return 으로 빠져나온다.

static void GameLoop(Company *c)
{
    for (;;) {                          /* 우리의 while(!gameOver) 와 같은 심장 */
        if (MainMenu(c))                /* 메뉴 — 사용자가 종료를 고르면 1 */
            return;
        if (c->ending == END_BANKRUPT)  /* 파산이면 루프 종료 */
            return;
        if (c->ending == END_RICH || c->ending == END_MASTERPIECE) {
            /* ... 승리 축하 + '계속하기' 선택 ... */
            return;
        }
    }
}

우리의 gameOver(0/1)가 완성본에선 c->ending(enum Ending)으로 자라 파산·거상·명작을 구분한다(15장). 하지만 "루프가 상태 변수를 보고 끝낼지 정한다"는 원리는 5.5에서 배운 그대로다.

② 메뉴 분기 (main.c · MainMenu) — 우리의 MainMenu() 와 똑같이, 사용자가 종료를 고르면 1 을 돌려준다(6.4의 반환값 신호). 그리고 각 메뉴를 함수 호출로 분기한다 — 6장의 함수 분해가 그대로 보인다.

static int MainMenu(Company *c)     /* 종료 시 1 반환 */
{
    ClearScreen();
    PrintHeader(c);                 /* 현황판 — 함수로 분리 (우리 PrintHeader) */
    printf("  [1] 게임 개발  [2] 직원 관리  ...  [6] 다음 달 진행 >>\n");

    switch (ReadInt("  >> 선택: ", 0, 6)) {
    case 1: DevMenu(c);      break;   /* 각 화면이 저마다의 함수 */
    case 2: StaffMenu(c);    break;
    case 6: AdvanceMonth(c); break;   /* 다음 달 진행 */
    case 0:
        if (ReadInt("  정말 종료할까요? [1] 예 [0] 아니요 : ", 0, 1) == 1)
            return 1;
        break;
    }
    return 0;
}

③ 월 정산 (company.c · AdvanceMonth) — 우리 AdvanceMonth() 의 확장판. GDD가 정한 한 달 처리 8단계(개발→출시→판매→이벤트→월급→리포트→판정→달력)를, 각 단계를 함수 하나씩 불러 처리한다.

void AdvanceMonth(Company *c)
{
    ResetReport();
    ProgressProject(c);              /* (1) 개발 진행 */
    if (ProjectDone(c)) ReleaseNow(c, 0);  /* (2) 출시 */
    UpdateSales(c);                  /* (3) 판매 정산 */
    RollApplicants(c);  MonthlyEvent(c);   /* (4) 지원자·이벤트 */
    if (c->month % 3 == 0) UpdateTrend(c, 1);  /* 분기마다 유행 재추첨 */
    Report("직원 월급", -TotalSalary(c));  /* (5) 지출 */
    Report("사무실 임대료", -OFFICE_RENT);
    PrintMonthlyReport(c);           /* (6) 리포트 + 자본금 반영 */
    CheckEnding(c);                  /* (7) 파산/승리 판정 */
    c->month++;                      /* (8) 달력 진행 */
    if (c->month > 12) { c->month = 1; c->year++; }
}

한 줄 한 줄이 함수 호출이다. AdvanceMonth 는 "한 달 동안 무슨 일이 일어나는가"의 목차이고, 실제 계산은 ProgressProject, UpdateSales, CheckEnding 같은 함수에 나뉘어 있다. 이것이 6장이 말한 "좋은 함수는 목차"의 완성형이다. c->month % 3 == 0(분기 판정)에는 3장의 나머지 연산자가, 함수를 줄줄이 부르는 구조에는 6장의 함수 분해가 함께 들어 있다.

그림으로 정리 — 루프와 함수 호출

📊 웹 교육문서에서는 게임 루프의 한 바퀴가 애니메이션(V-P1-B)으로 재생된다. 현황판 → 메뉴 → 정산 → 판정이 돌고, 자본금 변수가 매 바퀴 갱신되다 음수가 되는 순간 루프가 멈추는 과정을 보여 준다. 책에서는 아래 도식으로 대신한다.

게임 루프 — 조건이 끝날 때까지 도는 심장

        ┌───────────── while (!gameOver) ──────────────┐
        │                                              │
   ┌────▼─────┐   ┌──────────┐   ┌───────────┐   ┌─────┴─────┐
   │ 현황판    │──▶│ 메뉴 선택 │──▶│ 월 정산    │──▶│ 파산 판정  │
   │PrintHeader│   │MainMenu  │   │AdvanceMonth│  │money<0?   │
   └──────────┘   └────┬─────┘   └───────────┘   └─────┬─────┘
                       │ [0] 종료                       │
                       ▼                        gameOver=1이면
                   break로 탈출                   다음 검사에서 종료

함수 호출과 스코프 — 지역 변수는 함수와 함께 태어나고 죽는다

  전역:   money = 5000   ← 프로그램 내내 살아 있음. 모든 함수가 만질 수 있다
            ▲    ▲
            │    │ 직접 수정
   ┌────────┘    └──────────┐
   │                        │
  main()               AdvanceMonth()
   sel (지역)            income, tax (지역)
   └ main 끝나면 소멸      └ AdvanceMonth 끝나면 소멸

한 문장으로: 반복문은 게임을 '계속 돌게' 하는 심장이고, 함수는 그 심장이 하는 일을 '이름 붙은 조각'으로 나눈 근육이다.

체크포인트 — 스스로 점검하기

Q1. 아래 while 루프는 무한 루프다. 왜이고, 어떻게 고치나?

int n = 1;
while (n <= 5) {
    printf("%d ", n);
}

Q2. for (i = 0; i < 3; i++) 와 정확히 같은 동작을 하는 while 문을 써라.

Q3. break 와 continue 의 차이는? for (i=1;i<=5;i++) { if (i==3) continue; printf("%d",i); } 의 출력은?

Q4. do-while 이 while 과 다른 결정적 차이 하나는? 어떤 상황에 do-while 이 더 맞나?

Q5. 함수 void addTax(int money) { money += 100; } 를 addTax(cash); 로 불러도 바깥 cash 가 안 변한다. 왜인가? (전역이 아니라 매개변수로 받았다고 가정)

해설 펼치기

A1. 조건 n <= 5 를 바꾸는 코드(n++)가 없어 n 이 영원히 1이다. 조건이 늘 참이라 무한 반복한다. 블록 안에 n++; 를 넣어 매 회차 n 을 늘려야 언젠가 조건이 거짓이 되어 끝난다.

A2.

i = 0;
while (i < 3) {
    /* 블록 */
    i++;
}

for 의 초기화(i=0)는 루프 앞으로, 증감(i++)은 블록 끝으로 옮긴다.

A3. break 는 루프를 즉시 완전히 끝내고, continue 는 이번 회차만 건너뛰고 다음 회차로 간다. 출력은 3 만 빠진 1 2 4 5.

A4. do-while 은 조건을 끝에서 검사해 블록이 최소 한 번은 실행된다. "일단 한 번 실행하고, 조건에 따라 반복"인 상황 — 메뉴 입력을 한 번 받고 잘못됐으면 다시 받는 입력 검증 등에 맞는다.

A5. 매개변수 money 는 cash 값의 복사본이다(값에 의한 전달). 함수 안에서 복사본 money 를 늘려도 원본 cash 와는 무관하고, 함수가 끝나면 복사본은 사라진다. 원본을 고치려면 주소를 넘겨야 한다 — 그 방법(포인터)이 D 모듈의 주제다. (이것이 우리 게임이 지금 상태를 전역 변수로 두는 이유이기도 하다.)

실습 미션 — 숫자 맞히기 updown.c ⭐

본편(타이쿤)이 "한 게임을 계속 키우는 세로 축"이라면, 실습 미션은 이 모듈의 새 문법만 콕 집어 연습하는 독립 미니게임(가로 축) 이다. 먼저 직접 짜 보고, 막히면 힌트를, 그래도 막히면 정답을 펼쳐 대조하자.

드릴 문법 — while 반복 · break · 비교 연산자 · 함수(판정 함수) · 지역 변수 만드는 것 — 1~100 중 정해진 수를 맞힐 때까지 반복하며 "업/다운" 힌트를 주는 게임. 클리어 조건 — 정답을 맞히면 시도 횟수를 출력하고 끝낸다. 정해진 최대 횟수를 넘기면 실패. 규모 — 약 40줄, 파일 1개. 표준 C만. 아직 난수는 안 쓴다(정답은 상수로 고정 — E 모듈에서 rand 로 매번 다르게).

문제 정의

이 게임의 핵심은 while 루프 하나 다. 정답을 맞힐 때까지 "입력받고 → 크다/작다 판정하고 → 다시" 를 반복하다가, 맞히면 break 로 빠져나온다. A 모듈의 게임이 "한 번 하고 끝"이었다면, 이 게임은 5장의 반복문 덕분에 "맞힐 때까지 계속"이 된다 — B 모듈의 깨달음을 그대로 담은 미션이다.

목표 / 클리어 조건

  1. 정답을 #define ANSWER 42 로 정한다(난수는 E 모듈).
  2. while 로 "추측 입력 → 판정 → 반복"을 돌린다.
  3. 추측이 정답보다 크면 "다운!", 작으면 "업!", 같으면 성공 후 break.
  4. 판정을 함수로 뺀다: int Judge(int guess) 가 −1(작다)/0(정답)/1(크다)을 반환.
  5. 최대 시도 횟수(예: 7회)를 넘기면 실패 메시지 후 종료 — 시도 횟수를 세는 지역 변수.

힌트

힌트 1 — 판정 함수

정답과 비교해 세 갈래 중 하나를 돌려주는 작은 함수를 만든다.

#define ANSWER 42
int Judge(int guess)
{
    if (guess < ANSWER) return -1;   /* 더 크게 (업) */
    if (guess > ANSWER) return  1;   /* 더 작게 (다운) */
    return 0;                        /* 정답 */
}
힌트 2 — 반복과 탈출

시도 횟수를 세며 최대 횟수까지 for 로 돌리고, 정답이면 break.

int tries, cleared = 0;
for (tries = 1; tries <= MAX_TRY; tries++) {
    /* 입력받기 */
    int r = Judge(guess);
    if (r == 0) { cleared = 1; break; }   /* 맞히면 탈출 */
    printf(r < 0 ? "  업!\n" : "  다운!\n");
}

루프가 끝난 뒤 cleared 로 성공/실패를 가른다.

힌트 3 — 삼항으로 힌트 한 줄

Judge 결과 r 이 음수면 "업", 양수면 "다운". 4장의 삼항 연산자로 한 줄에.

printf(r < 0 ? "  업! 더 큰 수\n" : "  다운! 더 작은 수\n");

정답 코드

정답 펼치기 — updown.c 
#define _CRT_SECURE_NO_WARNINGS
#include <stdio.h>

#define ANSWER   42     /* 정답 (E 모듈에서 rand()%100+1 로 매번 다르게) */
#define MAX_TRY   7     /* 최대 시도 횟수 */

/* 추측을 판정: -1=정답보다 작다, 0=정답, 1=정답보다 크다 */
int Judge(int guess)
{
    if (guess < ANSWER) return -1;
    if (guess > ANSWER) return  1;
    return 0;
}

int main(void)
{
    int tries, guess, cleared = 0;

    printf("=== 숫자 맞히기 (1~100, %d번 안에) ===\n", MAX_TRY);

    for (tries = 1; tries <= MAX_TRY; tries++) {
        printf("[%d/%d] 추측: ", tries, MAX_TRY);
        scanf("%d", &guess);

        int r = Judge(guess);
        if (r == 0) {                       /* 맞혔다! */
            cleared = 1;
            break;                          /* 루프 탈출 */
        }
        printf(r < 0 ? "  업! 더 큰 수\n" : "  다운! 더 작은 수\n");
    }

    if (cleared)
        printf(">> 정답! %d번 만에 맞혔습니다.\n", tries);
    else
        printf(">> 실패... 정답은 %d 였습니다.\n", ANSWER);
    return 0;
}

빌드: cl updown.c (MSVC) 또는 gcc updown.c -o updown

함정/체크포인트

  • break 위치가 핵심이다. 정답일 때 break 하지 않으면 맞히고도 루프가 계속 돌아 시도 횟수를 낭비한다. "맞히면 즉시 탈출"이 이 게임의 규칙이다.
  • 성공 시 tries 를 그대로 출력하는 게 맞는가? break 는 tries++ 가 실행되기 전에 루프를 끊으므로, tries 는 정확히 "몇 번째 시도에 맞혔는지"를 담고 있다. for 의 증감 시점을 이해하면 이게 왜 맞는지 보인다.
  • 최대 횟수까지 못 맞히면 for 가 정상 종료되고 cleared 는 0인 채다 — 루프 밖에서 성공/실패를 가르는 cleared 플래그가 5.5의 gameOver 와 똑같은 발상이다.

확장 과제

  • E 모듈(난수) 에서 ANSWER 를 rand() % 100 + 1 로 바꿔 매판 다른 수로.
  • 이분 탐색 힌트: 최선을 다하면 1~100은 7번 안에 반드시 맞힐 수 있다(매번 절반을 노리면). 왜 7인지 생각해 보자(2^7 = 128 > 100).

반복문으로 우리 회사는 처음 "돌아가는" 게임이 되었고, 함수로 그 코드를 다시 읽을 수 있게 정리했다. 이제 게임에는 심장(루프)과 근육(함수)이 생겼다. 다음 C 모듈에서는 직원 여럿을 배열로, 회사명·게임 제목을 문자열로 다루며, 회사에 사람과 이름을 채운다.
