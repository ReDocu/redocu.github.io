7단계 · Battle City — 종합 캡스톤 (맵 생성 + 풀링)

단계 7 🏁 · 게임 Battle City · 씬 클래스 BattleCityContent : public IGameContent 처음 배우는 기술 — 두 기술 축의 결합 절차적 전장 생성(대칭 배치) 연결성 검증(flood fill) 파괴 지형(타일 상호작용) 기지 방어(승패 축) 적 AI(기지 편향/BFS) 파워업 아이템 6종 얼음 관성·숲 시야 가림 예상 학습 시간 — 8–10시간

커리큘럼의 졸업 과제다. 지금까지 우리는 두 개의 큰 기술 축을 따로 배웠다 — 5단계 Galaga의 오브젝트 풀과 6단계 Maze의 절차적 생성·경로탐색. Battle City는 그 둘을 한 게임에서 수렴시킨다. 절차적으로 생성된 전장 위에서, 풀로 관리되는 총알·적 탱크·폭발이 싸우고, 벽돌은 부서지고, 적은 기지를 향해 몰려오고, 플레이어는 아군 기지(이글)를 지킨다. 절차적 맵·대량 오브젝트·파괴 지형·적 AI가 동시에 얽히는 — 이 책이 쌓아 온 모든 것이 만나는 마지막 무대다.

이 단계의 학습 목표

이 단계를 마치면 다음을 할 수 있다.

  1. 두 기술 축(오브젝트 풀 + 절차적 생성)을 한 게임에 결합하는 구조를 설계할 수 있다.
  2. 대칭 배치로 전장을 절차적으로 생성하고, 다층 파이프라인으로 타일을 채울 수 있다.
  3. 연결성 검증(flood fill) 으로 "모든 적 스폰이 기지에 도달 가능한가"를 검사할 수 있다.
  4. 파괴 가능한 지형 — 총알이 타일에 닿을 때 종류별로 다르게 반응하는 상호작용을 구현할 수 있다.
  5. 기지 방어라는 승패 축과, 목표를 향해 편향 이동하는 적 AI를 짤 수 있다.
  6. 여러 시스템(맵·풀·충돌·AI·아이템)이 한 프레임에 수렴하는 캡스톤의 구조를 읽을 수 있다.

왜 Battle City가 마지막인가

이 책의 2부는 게임마다 새 기술을 한 겹씩 쌓아 왔다. 자료구조(Snake) → 알고리즘(Tetris) → 물리(Dino) → 시스템(Road Fighter) → 성능/풀(Galaga) → 알고리즘 심화/생성·탐색(Maze). 각 게임은 한 축에 집중했다.

캡스톤의 임무는 그 축들을 하나로 수렴시키는 것이다. 특히 마지막 두 게임이 세운 두 기둥이 여기서 만난다.

  • Galaga의 오브젝트 풀 → 탱크전은 슈팅보다 총알·폭발이 훨씬 잦다. 총알·적 탱크·폭발·아이템을 전부 풀로 돌려, 초당 수십 개가 오가도 힙을 건드리지 않는다.
  • Maze의 절차적 생성 + 경로탐색 → 스테이지마다 전장을 절차적으로 생성하고(벽돌·철벽·물·숲·얼음·기지), flood fill로 "적이 기지까지 갈 수 있는지" 검증하며, 적 AI가 기지를 향해 길을 찾는다.

여기에 캡스톤만의 새 요소가 더해진다 — 파괴 가능한 지형(벽돌이 부서지고 길이 열린다), 기지 방어(지켜야 할 것이 있는 승패 축), 파워업(별·헬멧·시계·삽·목숨·폭탄). 실시간 액션·절차적 맵·대량 오브젝트·타일 규칙·AI가 동시에 얽힌다.

커리큘럼의 수렴점. Galaga(성능)와 Maze(알고리즘)라는 두 봉우리가 여기서 하나의 계곡으로 흘러든다. 그래서 Maze를 캡스톤 바로 앞에 뒀다 — flood fill과 BFS를 배우고 즉시 전장 검증과 적 AI에 쓴다. 커리큘럼 전체가 이 마지막 게임을 향해 설계됐다.

개념 1 · 두 기술의 결합 — 타일 맵 위의 풀

Battle City의 데이터는 두 세계로 나뉜다 — 정적인 타일 맵과 동적인 오브젝트 풀.

// 세계 ① 절차적으로 생성되는 정적 전장 (Maze의 격자)
enum class _ETile { EMPTY, BRICK, STEEL, WATER, FOREST, ICE, BASE };
_ETile tile[FIELD_H][FIELD_W];

// 세계 ② 풀로 관리되는 동적 오브젝트 (Galaga의 풀)
Pool<Bullet, MAX_PBULLET> pBullets;   // 8
Pool<Bullet, MAX_EBULLET> eBullets;   // 24
Pool<Tank,   MAX_ENEMY>   enemies;    // 8
Pool<Fx,     MAX_FX>      fxs;         // 24
Pool<Item,   MAX_ITEM>    items;       // 4

이 둘이 맞물려 게임을 만든다. 타일 맵은 오브젝트가 움직일 무대를 정하고(탱크는 벽을 못 지나고, 총알은 벽돌을 부순다), 풀은 그 무대 위 배우들(총알·탱크·폭발·아이템)을 관리한다. Galaga의 Pool<T,N>(5단계의 FrameWork/Object/Pool.h)을 그대로 재사용하고, Maze의 격자 좌표계를 타일 맵으로 가져온다. 두 게임에서 손으로 만든 도구가 여기서 조립된다.

스테이지가 바뀌면 두 세계를 함께 리셋한다 — 풀은 Clear()(재할당 없이), 맵은 GenerateMap()(새로 생성).

void BattleCityContent::StartStage(int s)
{
    pBullets.Clear();  eBullets.Clear();  enemies.Clear();  fxs.Clear();  items.Clear();  // 풀 비우기
    GenerateMap(s);                    // 전장 새로 생성
    enemiesToSpawn = STAGE_TOTAL;      // 이번 스테이지 적 20마리
    /* ... 플레이어·기지·상태 리셋 ... */
}

개념 2 · 절차적 전장 생성 — 대칭 배치 파이프라인

전장은 여러 단계를 거쳐 채워진다. 빈 맵에서 시작해, 지형을 층층이 얹는 다층 파이프라인이다.

void BattleCityContent::GenerateMap(int s)
{
    // 0) 전부 EMPTY 로 초기화
    for (…) tile[y][x] = _ETile::EMPTY;

    // 1) 지형 군집 — 좌우 대칭 랜덤 배치 (스테이지로 양 조절)
    for (n < brickN)  PlaceRectSym(BRICK, 랜덤위치, 랜덤크기);   // 벽돌
    for (n < steelN)  PlaceRectSym(STEEL, …);                    // 철벽
    for (n < waterN)  PlaceRectSym(WATER, …);                    // 물
    for (n < forestN) PlaceRectSym(FOREST, …);                   // 숲
    for (n < iceN)    PlaceRectSym(ICE,  …);                     // 얼음

    // 2) 철벽 테두리 (파괴 불가 경계)
    for (…) tile[0][x] = tile[FIELD_H-1][x] = STEEL; /* 상하 */
    for (…) tile[y][0] = tile[y][FIELD_W-1] = STEEL; /* 좌우 */

    // 3) 기지 + 벽돌 요새
    tile[baseY][baseX] = BASE;
    SetFortress(BRICK);            // 기지 3면을 벽돌로 감쌈

    // 4) 스폰존 확보 + 연결성 검증
    ClearSpawnZones();             // 스폰 지점·플레이어 시작칸을 EMPTY 로
    CarveConnectivity();           // 막힌 곳을 뚫어 도달 가능 보장 (개념 3)
}

핵심은 대칭 배치 PlaceRectSym 이다. Battle City 특유의 좌우 대칭 맵을 위해, 사각형을 놓을 때 그 거울상도 함께 놓는다.

void BattleCityContent::PlaceRectSym(_ETile t, int x0, int y0, int w, int h)
{
    for (int yy = y0; yy < y0+h; yy++)
        for (int xx = x0; xx < x0+w; xx++) {
            if (경계 밖) continue;
            tile[yy][xx] = t;                    // 원본
            int mx = FIELD_W - 1 - xx;           // ★ 좌우 미러 x
            if (mx 유효) tile[yy][mx] = t;        // 거울상도 함께
        }
}

mx = FIELD_W - 1 - xx 가 좌우 대칭점이다. 왼쪽에 벽돌 덩어리를 놓으면 오른쪽에도 똑같이 놓인다. 이 한 줄이 전장에 균형 잡힌 대칭미를 준다 — 원작 배틀 시티의 그 느낌이다. 지형의 양(brickN, steelN…)을 스테이지에 따라 조절하면, 맵 생성 파라미터가 곧 난이도 다이얼이 된다(Dino·Road Fighter의 게이팅과 같은 발상, 이번엔 지형에 적용).

as-built 참고 — 대칭 배치 vs 셀룰러 오토마타. GDD는 물·숲을 셀룰러 오토마타(노이즈 + 이웃 다수결 스무딩)로 유기적으로 만들자고 설계했지만, 실제 구현은 더 단순한 대칭 사각형 배치를 택했다. "동작하는 단순함을 먼저"(Galaga의 교훈)의 또 한 예다. CA는 폴리시로 남았다 — 캡스톤이라고 모든 걸 최고급으로 만들 필요는 없다. 무엇을 단순하게 둘지 판단하는 것도 완성의 일부다.

개념 3 · 연결성 검증 — flood fill (Maze 재사용)

절차적으로 지형을 뿌리면 문제가 생긴다 — 적 스폰 지점이나 기지가 벽에 둘러싸여 갇힐 수 있다. 적이 기지까지 못 가면 게임이 성립하지 않는다. 그래서 생성 후 반드시 "모든 중요 지점이 서로 연결됐는가"를 검증한다. 이때 쓰는 것이 6단계 Maze에서 배운 flood fill(BFS/DFS로 연결 영역 채우기)이다.

void BattleCityContent::CarveConnectivity()
{
    bool vis[FIELD_H][FIELD_W] = { false };
    int stackX[…], stackY[…], sp = 0;      // 명시적 스택 (Maze의 그것)

    // ① 플레이어 시작칸에서 flood fill — 통행 가능 타일을 전부 방문
    vis[startY][startX] = true;
    stackX[sp] = startX; stackY[sp] = startY; sp++;
    while (sp > 0) {
        int cx = stackX[--sp], cy = stackY[sp];
        for (int d = 0; d < 4; d++) {
            int nx = cx + dxs[d], ny = cy + dys[d];
            if (InField(nx,ny) && !vis[ny][nx] && Passable(nx,ny)) {   // 갈 수 있고 미방문이면
                vis[ny][nx] = true;                                    // 방문 표시
                stackX[sp] = nx; stackY[sp] = ny; sp++;                // 스택에 쌓기
            }
        }
    }

    // ② 스폰 지점·기지 앞이 방문 안 됐으면(고립) → 시작칸 쪽으로 길을 뚫는다
    for (각 중요 지점) {
        while (도달 안 됨) {
            tile[cy][cx] = EMPTY;              // 벽 뚫기
            /* 시작칸 방향으로 한 칸 전진 */
        }
    }
}

flood fill의 원리는 단순하다 — 시작칸에서 갈 수 있는 이웃으로 계속 번져 나가며 방문한 칸을 표시한다. 다 번지고 나면, vis 가 true 인 칸이 곧 "시작칸에서 도달 가능한 영역"이다. 스폰 지점이나 기지 앞이 그 영역 밖(vis == false)이면 고립된 것이므로, 시작칸 방향으로 벽을 뚫어 길을 낸다.

이것이 Maze에서 배운 것의 직접 응용이다. Maze의 ShortestPath(BFS)가 "미로가 풀리는가"를 검증했듯, 여기선 flood fill이 "전장이 연결됐는가"를 검증한다. 같은 알고리즘이 다른 게임에서 다시 일한다 — 알고리즘은 한 번 익히면 어디서든 재사용되는 재산이다.

Passable 이 통행 규칙을 한곳에 모은다. flood fill이 "갈 수 있는가"를 묻는 Passable(tx,ty) 은 EMPTY/FOREST/ICE 만 참을 준다(벽돌·철벽·물·기지는 못 감). 이 한 함수에 통행 규칙을 모아 두니, 연결성 검증도·탱크 이동도·적 AI도 전부 같은 규칙을 공유한다(Tetris의 CanMove 와 같은 "판단을 한곳에" 정신).

개념 4 · 파괴 가능한 지형 — 타일 상호작용

배틀 시티의 손맛은 벽돌을 부수며 길을 여는 것이다. 총알이 타일에 닿을 때, 타일 종류마다 다르게 반응한다. 이 상호작용이 StepBullet 한 함수에 모여 있다.

// 총알을 한 칸 전진시키고 타일 충돌을 처리. false = 총알 소멸.
bool BattleCityContent::StepBullet(Bullet& b)
{
    int nx = b.x + dx, ny = b.y + dy;          // 다음 칸
    if (!InField(nx, ny)) return false;         // 전장 밖 → 소멸

    _ETile t = tile[ny][nx];
    if (t == BRICK) { tile[ny][nx] = EMPTY; return false; }        // 벽돌: 부수고 소멸
    if (t == STEEL) {
        bool border = (경계);
        if (b.power && !border) tile[ny][nx] = EMPTY;              // 철벽: 강화탄만 부숨
        return false;                                              //       (아니면 총알만 소멸)
    }
    if (t == BASE)  { baseAlive = false; SpawnFx(nx,ny); return false; }   // 기지: 게임 오버!
    // EMPTY / WATER / FOREST / ICE → 통과
    b.x = nx; b.y = ny;
    return true;
}

타일 종류가 곧 규칙이다.

  • BRICK(벽돌) — 총알에 부서져 EMPTY 가 된다(길이 열린다). 캡스톤의 핵심 손맛.
  • STEEL(철벽) — 보통 총알은 못 뚫고, 강화탄(파워업 최대치)만 부순다. 단 경계 철벽은 파괴 불가.
  • WATER(물) — 탱크는 못 지나지만 총알은 통과한다(타일 상호작용이 탱크와 총알에 다르게 적용).
  • FOREST(숲) — 통과 가능하지만 시야를 가린다(렌더에서 오브젝트 위에 다시 덮어 그림).
  • ICE(얼음) — 통과하되 미끄러진다(관성).
  • BASE(기지) — 총알이 닿으면 즉시 게임 오버.

"정적인 맵이 동적으로 변한다" 는 것이 캡스톤을 앞선 게임과 가르는 지점이다. Tetris·Maze의 격자는 고정이었지만, Battle City의 전장은 총알에 부서지고 삽 아이템에 강화된다. 맵과 오브젝트가 서로 영향을 주고받는 — 살아있는 전장이다.

개념 5 · 적 AI — 기지를 향해

적 탱크는 단순히 헤매지 않는다. 기지를 향해 편향되어 움직인다. 40% 확률로 기지 방향으로, 나머지는 랜덤으로 방향을 정한다.

void BattleCityContent::ChooseEnemyDir(Tank& e)
{
    if (rand() % 100 < 40) {                        // 40%: 기지 쪽으로 편향
        int ddx = baseX - e.x, ddy = baseY - e.y;   // 기지까지의 방향
        if (abs(ddx) > abs(ddy))                    // 더 먼 축부터 좁힌다 (그리디)
            e.dir = (ddx > 0) ? _EDir::RIGHT : _EDir::LEFT;
        else
            e.dir = (ddy > 0) ? _EDir::DOWN : _EDir::UP;
    } else {
        e.dir = (_EDir)(rand() % 4);                // 60%: 랜덤 (예측 불가·벽 회피)
    }
}

이것은 그리디(greedy) 접근 — "기지까지 x와 y 중 더 먼 축을 먼저 좁힌다". 완벽한 최단경로는 아니지만(벽을 못 피한다), 랜덤 40:60 섞임 덕분에 적들이 기지로 몰려오되 벽에 갇히지 않고 자연스럽게 흩어진다. 이동은 Passable + TankAt(다른 탱크 점유)을 검사해, 못 가면 방향을 다시 고른다.

여기가 Maze의 BFS가 빛날 자리다. 그리디 편향은 벽을 우회하지 못한다 — 기지 앞에 벽돌 요새가 있으면 적이 벽에 부딪혀 헤맬 수 있다. GDD는 정밀한 추적을 원하면 Maze의 BFS로 기지까지 다음 한 칸을 계산하라고 남겨 뒀다(폴리시). 6단계에서 만든 ShortestPath 를 여기 붙이면, 적이 벽을 돌아 기지로 최단 접근하는 진짜 추적 AI가 된다. 알고리즘 하나(BFS)가 게임의 지능을 바꾼다 — 커리큘럼이 Maze를 캡스톤 앞에 둔 이유가 여기서 완성된다.

개념 6 · 기지 방어와 파워업 — 지킬 것과 뒤집을 수

캡스톤에는 앞선 게임에 없던 지켜야 할 것이 있다 — 아군 기지(이글, ★). 승패 축이 둘로 늘었다: 적 전멸(승리) 또는 기지 파괴/생명 0(패배).

// 패배: 기지 파괴 또는 생명 0
if (!baseAlive) { gameState = GAMEOVER; return; }
// 승리: 스폰할 적이 없고, 살아있는 적도 없으면 → 스테이지 클리어
if (enemiesToSpawn <= 0 && ActiveEnemyCount() == 0) {
    score += 1000;
    gameState = STAGECLEAR;
}

기지는 벽돌 요새로 보호되고(SetFortress), 삽 아이템으로 잠깐 철벽 방어막을 두른다. 기지를 지키며 적을 막는 이 공수(攻守) 동시성이 배틀 시티의 긴장감이다.

파워업 아이템 6종은 판을 뒤집는 수다. 점멸(flashing) 적을 잡으면 드랍되고, 주우면 즉시 발동한다.

void BattleCityContent::ApplyItem(_EItemType type)
{
    switch (type) {
    case STAR:   powerLevel++;   break;                 // 동시탄↑, 최대치=강화탄
    case HELMET: invincible = true; invTimer = 300; break;  // 잠깐 무적
    case CLOCK:  freezeTimer = 360; break;              // 모든 적 정지
    case SHOVEL: shieldTimer = 600; SetFortress(STEEL); break;  // 기지 철벽 보호
    case LIFE:   life++; break;                          // 목숨 +1
    case BOMB:   /* 활성 적 전멸 */ break;               // 화면 청소
    }
}

freezeTimer, invTimer, shieldTimer — 전부 4단계 Road Fighter에서 배운 "상태에 수명을 주는" 타이머다(매 프레임 감소, 0이 되면 해제). 시계로 적을 얼리고, 삽으로 기지를 지키고, 폭탄으로 화면을 청소하는 — 이 극적인 순간들이 아이템 풀(items)과 타이머로 구현된다. 앞 게임들에서 배운 조각들이 캡스톤에서 하나하나 제자리를 찾는다.

코드 워크스루 — OnInGameUpdate, 모든 것이 수렴하는 한 프레임

캡스톤의 한 프레임은 이 책이 배운 모든 시스템이 순서대로 도는 파이프라인이다.

실제 파일: Main/Content/BattleCityContent.cpp

void BattleCityContent::OnInGameUpdate()
{
    // ... STAGESTART · PAUSE · STAGECLEAR · GAMEOVER 처리 ...

    // ── PLAYING : 모든 시스템 수렴 ──
    UpdatePlayer();      // ① 이동(관성·벽) · 회전 · 발사(pBullets 풀)   [Dino 물리·Galaga 풀]
    UpdateEnemies();     // ② 스폰 · 기지 편향 AI · 사격(eBullets 풀)      [Maze 경로·RF 엔티티]
    UpdateBullets();     // ③ 총알 이동 · 타일 상호작용(StepBullet)         [타일 파괴]
    UpdateItems();       // ④ 아이템 점멸
    UpdateFx();          // ⑤ 폭발 애니메이션(fxs 풀)                       [Galaga 풀]
    HandleCollisions();  // ⑥ 풀 × 풀 충돌 → 파괴·득점·아이템·피격          [Galaga 충돌]

    if (freezeTimer > 0) freezeTimer--;    // 상태 타이머들 (RF)
    if (shieldTimer > 0) shieldTimer--;

    if (!baseAlive) { gameState = GAMEOVER; return; }              // 패배: 기지
    if (enemiesToSpawn <= 0 && ActiveEnemyCount() == 0)            // 승리: 전멸
        { score += 1000; gameState = STAGECLEAR; }
}

각 줄 옆에 그 시스템이 어느 게임에서 왔는지 적어 두었다. 캡스톤은 새 기법을 발명하기보다, 배운 것을 조립한다.

  • ① 이동은 Dino의 float 물리 감각(여기선 얼음 관성)과 벽 충돌, 발사는 Galaga의 풀.
  • ② 적 스폰·AI는 Road Fighter의 엔티티 관리와 Maze의 경로 발상.
  • ③ 총알-타일 상호작용은 이 게임의 새 요소(파괴 지형).
  • ⑥ 충돌은 Galaga의 풀 × 풀 판정 그대로.

HandleCollisions 하나가 [내탄↔︎적탱크(격추·득점·아이템 드랍) / 적탄↔︎플레이어(피격) / 내탄↔︎적탄(상쇄) / 플레이어↔︎아이템(획득)]을 전부 처리한다 — 전부 활성 슬롯끼리의 좌표 비교(Galaga 개념 5). 렌더도 계층적이다: 맵 → 아이템 → 탱크 → 총알 → 폭발 → 숲 오버레이(오브젝트 위에 다시 덮어 시야 가림) → HUD. 숲을 맨 나중에 그리는 그 한 수가 "숲에 숨은 탱크"를 만든다.

이것이 캡스톤의 본질이다 — 화려한 새 기술이 아니라, 여러 시스템을 정확한 순서로 엮어 하나의 완결된 게임을 만드는 것. 그 엮음을 읽을 수 있다면, 당신은 이 책을 졸업한 것이다.

알려진 이슈에서 배우기 (as-built)

캡스톤이라고 모든 걸 최고급으로 만들지 않았다. 무엇을 완성하고 무엇을 단순화할지의 판단이, 여기 고스란히 남아 있다.

이슈 as-built 상태 무엇을 배우나
적 AI 그리디 기지 방향 편향(BFS 아님) 벽 우회 못 함. Maze BFS 붙이면 진짜 추적(개념 5) — 남겨 둔 성장 여지
물/숲 대칭 배치 셀룰러 오토마타 대신 대칭 사각형 GDD의 CA는 폴리시로. "동작하는 단순함 먼저"
연결성 = flood fill 뚫기 고립되면 시작칸 쪽으로 직선 carve 최단 경로는 아니지만 도달성은 보장 — 실용적 안전망
충돌 O(총알×적) 이중 루프 전수 비교 풀이 작아 충분. 타일 점유 맵은 폴리시
하이스코어 세션 한정 hiScore 멤버 파일 저장(1부 F)은 향후 확장

"AI가 BFS를 안 쓰는 것"이 눈에 띈다 — 그런데 이건 결함이 아니라 다음 단계로 가는 문이다. 당신이 이 책을 다 읽었다면, 6단계의 ShortestPath 를 여기 붙여 적을 똑똑하게 만들 수 있다. 완성된 게임에도 언제나 개선의 여지가 있고, 그 여지를 볼 수 있는 것이 곧 개발자가 된다는 것이다. as-built의 정직함은, 마지막까지 "게임은 완벽한 설계가 아니라 계속 자라는 것"이라는 이 책의 메시지를 전한다.

Q1. Battle City가 결합하는 두 기술 축은 무엇이고, 각각 어느 게임에서 왔나?

Q3. 전장을 절차적으로 생성한 뒤 flood fill로 무엇을 검증하나? 검증에 실패하면 어떻게 하나?

Q4. 총알이 BRICK·STEEL·WATER·BASE에 닿았을 때 각각 어떻게 되나? 이 규칙은 어디에 모여 있나?

Q5. 적 AI는 40% 확률로 기지 방향으로 편향된다. 이 그리디 방식의 한계는? 어떻게 개선하나?

해설 펼치기

A1. 오브젝트 풀(5단계 Galaga — 총알·탱크·폭발·아이템을 Pool<T,N>로 관리, 힙 할당 0회)과 절차적 생성·경로탐색(6단계 Maze — 전장을 알고리즘으로 생성, flood fill로 연결성 검증, BFS로 적 AI). 두 봉우리가 캡스톤에서 하나로 수렴한다.

A2. FIELD_W - 1 - xx 는 x좌표의 좌우 대칭점이다. 지형을 왼쪽에 놓을 때 그 거울상을 오른쪽에도 함께 놓아, 배틀 시티 특유의 좌우 대칭 맵을 만든다. 대칭은 양쪽 플레이의 공정함과 시각적 균형을 준다.

A3. "모든 적 스폰 지점과 기지가 플레이어 시작칸에서 도달 가능한가" 를 검증한다. 시작칸에서 통행 가능 타일로 flood fill해 방문 영역을 구하고, 중요 지점이 그 밖(고립)이면 시작칸 방향으로 벽을 뚫어 길을 낸다. 적이 기지에 못 가는 "성립 불가 맵"을 방지하는 안전망이다.

A4. BRICK → 부서져 EMPTY가 되고 총알 소멸. STEEL → 강화탄이면 부수고 아니면 총알만 소멸(경계는 파괴 불가). WATER → 총알 통과(탱크는 못 감). BASE → 게임 오버. 이 규칙은 StepBullet 한 함수에 모여 있다("판단을 한곳에").

A5. 그리디 편향은 "기지 쪽으로 한 칸"만 볼 뿐 벽을 우회하지 못한다 — 기지 앞 요새에 막혀 헤맬 수 있다. 개선책은 6단계 Maze의 BFS(ShortestPath)로 기지까지 최단경로를 계산해 다음 한 칸을 정하는 것. 그러면 적이 벽을 돌아 기지로 정확히 접근하는 진짜 추적 AI가 된다.

실습 미션 — 연결성 검증 flood.c ⭐

이 마지막 미션은 캡스톤의 심장 flood fill 연결성 검증을 손으로 만든다. 전장에서 "모든 스폰이 기지에 닿는가"를 확인하는 것 — Maze의 flood fill이 Battle City에서 어떻게 일하는지 그대로 재현한다. bfs.c(6단계)와 짝을 이루는, 책을 닫는 알고리즘이다.

드릴 문법 — 2D 타일 격자 · flood fill(스택/DFS) · 4방향 · 도달성 판정 만드는 것 — 벽/통로로 된 전장에서 시작칸으로부터 flood fill해, 각 스폰 지점과 기지가 연결됐는지 판정. 클리어 조건 — 도달 가능한 칸을 정확히 표시하고, 고립된 지점을 찾아낸다. 규모 — 약 55줄, 파일 1개. 표준 C만. 책의 마지막 알고리즘.

문제 정의

flood fill은 "시작칸에서 갈 수 있는 이웃으로 계속 번져 방문 영역을 채우는" 알고리즘이다(BFS/DFS와 뼈대가 같다). 전장에서 플레이어 시작칸부터 통로를 타고 번지면, 방문된 칸이 "도달 가능 영역"이다. 스폰 지점(X)이나 기지(B)가 그 밖이면 고립 — 적이 갈 수 없는 성립 불가 맵이다. 이 검증을 통과해야 게임이 성립한다.

목표 / 클리어 조건

  1. 전장을 문자 격자로 둔다(#=벽, .=통로, S=시작, X=적 스폰, B=기지).
  2. S에서 flood fill: 스택에 넣고 꺼내며, 벽이 아닌 미방문 이웃을 스택에 넣는다.
  3. 방문 영역을 표시한다.
  4. 각 X 와 B 가 방문됐는지(연결됐는지) 판정해 보고한다.
  5. 하나라도 고립됐으면 "재생성 필요"를 출력.

힌트

힌트 1 — flood fill (명시적 스택) 
int stackX[H*W], stackY[H*W], sp = 0;
int visited[H][W] = {{0}};

stackX[sp] = sx; stackY[sp] = sy; sp++;  visited[sy][sx] = 1;
while (sp > 0) {
    int cx = stackX[--sp], cy = stackY[sp];   /* 스택에서 꺼내기 */
    for (int d = 0; d < 4; d++) {
        int nx = cx + DX[d], ny = cy + DY[d];
        if (범위 밖 || 벽 || 방문) continue;
        visited[ny][nx] = 1;
        stackX[sp] = nx; stackY[sp] = ny; sp++;   /* 스택에 넣기 */
    }
}
힌트 2 — 연결성 판정

flood fill 후, 각 스폰/기지 칸의 visited 를 확인한다.

int allConnected = 1;
for (각 X, B 칸) if (!visited[y][x]) allConnected = 0;  /* 하나라도 고립되면 실패 */

정답 코드

정답 펼치기 — flood.c 
#define _CRT_SECURE_NO_WARNINGS
#include <stdio.h>
#include <string.h>

#define H 9
#define W 16

/* # = 벽, . = 통로, S = 시작, X = 적 스폰, B = 기지 */
char field[H][W + 1] = {
    "################",
    "#X....#..X.....#",
    "#.###.#.####.#.#",
    "#...#...#....#.#",
    "###.#.###.###.##",
    "#.....S...#....#",
    "#.#####.#.#.##.#",
    "#.....#.#.B..#.#",   /* B = 기지 */
    "################",
};

static const int DX[4] = { 0, 0, -1, 1 };
static const int DY[4] = { -1, 1, 0, 0 };

int visited[H][W];

int main(void)
{
    int stackX[H * W], stackY[H * W], sp = 0;
    int sx = -1, sy = -1;

    memset(visited, 0, sizeof(visited));
    for (int y = 0; y < H; y++)
        for (int x = 0; x < W; x++)
            if (field[y][x] == 'S') { sx = x; sy = y; }

    /* --- flood fill (시작칸에서 번지기) --- */
    stackX[sp] = sx; stackY[sp] = sy; sp++;
    visited[sy][sx] = 1;
    while (sp > 0) {
        int cx = stackX[--sp], cy = stackY[sp];
        for (int d = 0; d < 4; d++) {
            int nx = cx + DX[d], ny = cy + DY[d];
            if (nx < 0 || nx >= W || ny < 0 || ny >= H) continue;
            if (field[ny][nx] == '#' || visited[ny][nx]) continue;   /* 벽·방문 제외 */
            visited[ny][nx] = 1;
            stackX[sp] = nx; stackY[sp] = ny; sp++;
        }
    }

    /* --- 도달 영역 표시 + 연결성 판정 --- */
    int allConnected = 1;
    for (int y = 0; y < H; y++) {
        printf("  ");
        for (int x = 0; x < W; x++) {
            char c = field[y][x];
            if ((c == 'X' || c == 'B') && !visited[y][x]) allConnected = 0;   /* 고립! */
            /* 도달 가능한 통로는 소문자로 표시 */
            if (visited[y][x] && c == '.') c = 'o';
            printf("%c", c);
        }
        printf("\n");
    }

    printf("\n>> %s\n", allConnected
        ? "모든 스폰/기지가 연결됨 — 유효한 전장!"
        : "고립된 지점 발견 — 재생성 필요!");
    return 0;
}

빌드: cl flood.c (MSVC) 또는 gcc flood.c -o flood

함정/체크포인트

  • flood fill = BFS/DFS의 사촌. 스택으로 꺼내면 DFS(깊이), 큐로 꺼내면 BFS(너비)지만, "연결 영역을 채운다"는 목적엔 어느 쪽이든 결과가 같다. 여기선 스택(DFS)을 썼다 — 6단계의 bfs.c 와 비교해 보자.
  • 벽 안에 갇힌 X 를 찾아라. 맵의 X(스폰) 하나를 벽으로 완전히 둘러싸면, flood fill이 거기 못 닿아 allConnected = 0 이 된다. 이것이 Battle City가 "성립 불가 맵"을 걸러내는 방식이다.
  • visited 표시는 스택에 넣을 때. 꺼낼 때 표시하면 같은 칸이 스택에 여러 번 들어가 중복된다(bfs.c 와 같은 주의).

확장 과제

  • 고립된 지점을 찾으면, 시작칸 쪽으로 벽을 뚫는 CarveConnectivity 를 구현해 보자(개념 3) — 그러면 Battle City의 실제 검증-보정 로직이 완성된다.
  • 스택을 큐로 바꿔 BFS로 만들면, 각 칸의 "시작칸으로부터의 거리"도 함께 얻을 수 있다.

Battle City로 우리는 이 책이 쌓아 온 모든 것을 하나로 조립했다. 절차적 맵 위에서 풀링된 오브젝트가 싸우고, 지형이 부서지고, 적이 기지를 노리고, 플레이어가 그것을 지킨다. 새 기법을 배우기보다, 배운 것을 엮어 완결된 게임을 만드는 것 — 그것이 졸업 과제의 의미였다.

커리큘럼의 일곱 단계는 여기서 끝난다. 하지만 이 프레임워크로 지을 수 있는 게임이 하나 더 남아 있다 — 다음 확장 챕터에서, 유령 넷이 쫓아오는 고전 Pac-Man을 만들며 격자 위 추적 AI와 상태머신을 다루고, 그 끝에서 이 책의 두 여정을 함께 돌아본다.
