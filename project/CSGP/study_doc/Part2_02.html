2단계 · Tetris — 알고리즘과 그리드

단계 2 · 게임 Tetris · 씬 클래스 TetrisContent : public IGameContent 처음 배우는 기술 — 2D 그리드 보드(board[H][W]) 사전 정의 회전 테이블 충돌 판정(CanMove) 회전·월킥 라인 클리어(배열 shift) 7-bag 랜덤(Fisher-Yates) DAS(자동 반복 이동) 홀드·고스트·하드드롭 내부 상태 enum 예상 학습 시간 — 4–6시간

Snake가 "직관적 충돌"의 게임이었다면, Tetris는 명시적 규칙과 알고리즘의 게임이다. 뱀은 그저 머리가 무엇에 닿았는지만 보면 됐지만, 테트리스는 회전할 때 벽을 밀어내고(월킥), 가득 찬 줄을 찾아 지우고 위를 끌어내리고(라인 클리어), 7개 피스를 공정하게 섞어(7-bag) 내보낸다. 여기서 배열은 게임의 메인 자료구조로 승격하고, 그 배열 위에서 돌아가는 알고리즘을 본격적으로 설계한다.

이 단계의 학습 목표

이 단계를 마치면 다음을 할 수 있다.

  1. int board[20][10] 2D 그리드로 게임판을 표현하고, 보드 좌표와 화면 좌표(x*2)를 구분할 수 있다.
  2. 테트로미노의 4가지 회전 상태를 사전 정의 테이블로 저장하는 이유를 설명하고, (dx, dy) 오프셋으로 피스를 표현할 수 있다.
  3. CanMove 충돌 판정 하나가 어떻게 이동·회전·낙하·고정 모두의 심판이 되는지 안다.
  4. 회전 시 벽·블록에 막히면 밀어내는 월킥(wall kick) 을 구현할 수 있다.
  5. 라인 클리어 — 가득 찬 행을 찾아 지우고 위 행을 끌어내리는 배열 조작을 짤 수 있다.
  6. 7-bag Fisher-Yates 셔플로 "무작위를 공정하게" 만들고, DAS 로 입력 반복을 제어할 수 있다.

왜 Tetris가 이 자리인가

1단계 Snake에서 우리는 "살아있는 게임"의 최소 완성형을 만들었다. 하지만 Snake의 규칙은 직관적이었다 — 머리가 무엇과 같은 칸에 있는가, 그것뿐이었다. 자료구조도 "자라나는 목록"(vector) 하나로 충분했다.

Tetris는 다르다. 여기서 게임은 설계된 규칙과 알고리즘의 집합이 된다.

  • "이 피스를 여기 놓을 수 있는가?"는 4칸을 보드와 대조하는 충돌 판정이고,
  • "회전했더니 벽에 걸린다"는 후보 위치를 순서대로 시도하는 월킥이며,
  • "한 줄이 꽉 찼다"는 배열을 스캔해 지우고 끌어내리는 라인 클리어이고,
  • "다음 피스는 무엇인가"는 7개를 공정하게 섞는 셔플 알고리즘이다.

그리고 이 모든 것의 무대가 2D 배열이다. 1부 C 모듈에서 배운 board[H][W] 가, Snake의 동적 리스트를 밀어내고 게임의 메인 자료구조로 올라선다. Snake가 "자료구조 위의 상호작용"이었다면, Tetris는 "자료구조 위의 알고리즘" 이다 — 커리큘럼이 한 계단 오르는 지점이다.

자료구조의 진화. Snake는 몸통을 vector<Snake*> 로 관리하며 new/delete 로 마디를 늘렸다(순진한 관리). Tetris는 정반대다 — 보드도, 7-bag도, 피스도 전부 값 타입·고정 배열이라 동적 할당이 거의 없다(board[20][10], struct Tetromino). 그래서 OnRelease 가 거의 빈다. 이 "값 타입으로 단순하게"가 Dino·Road Fighter로 이어지고, 결국 Galaga의 오브젝트 풀로 완성된다.

개념 1 · 2D 그리드 보드 — 배열이 주인공이 되다

테트리스의 심장은 게임판을 나타내는 2차원 배열 하나다.

static const int BOARD_W = 10;   // 가로 10칸
static const int BOARD_H = 20;   // 세로 20칸

int board[BOARD_H][BOARD_W];     // 0 = 빈칸, 1~7 = 고정된 블록(색 = 종류)

board[r][c] 의 값이 그 칸의 상태다 — 0 이면 비어 있고, 1~7 이면 그 색의 블록이 고정돼 있다. 게임의 거의 모든 판단이 이 배열을 읽고 쓰는 일이다. 떨어지는 피스가 놓일 수 있는지, 한 줄이 꽉 찼는지, 어디까지 내려갈 수 있는지 — 전부 board 를 들여다본다.

보드 좌표와 화면 좌표를 구분하라. 게임 로직은 보드 인덱스 (col, row) 로만 계산한다. 화면에 그릴 때만 0단계에서 배운 변환을 거친다.

void TetrisContent::DrawCell(int col, int row, int color)
{
    if (row < 0) return;
    SCREEN->OnDrawColor((ORIGIN_X + col) * 2, ORIGIN_Y + row, "■", color);
    //                   └─ 보드 원점 더하고 ─┘ └ 전각이라 ×2 (0단계 개념 3)
}

보드 셀 (col, row) → 논리 좌표 (ORIGIN_X + col, ORIGIN_Y + row) → 화면 ((ORIGIN_X+col)*2, ...). 보드 인덱스와 화면 좌표를 혼동하지 않는 것 이 2D 게임의 기본 규율이다.

개념 2 · 테트로미노와 사전 정의 회전 테이블

떨어지는 피스(테트로미노)는 작은 구조체 하나다.

struct Tetromino {
    int type;   // 0~6 = I,O,T,S,Z,J,L
    int rot;    // 0~3 회전 상태
    int x, y;   // 보드 좌표 (피스의 기준점)
};

문제는 "회전한 모양"을 어떻게 아느냐다. 회전을 실시간으로 계산(좌표 회전 공식)할 수도 있지만, 콘솔 게임에선 4가지 회전 상태를 미리 표로 저장하는 것이 가장 안전하고 단순하다. 각 회전 상태는 4×4 상자 안 블록 4칸의 (dx, dy) 오프셋 목록이다.

// SHAPES[피스종류][회전상태][블록4개] = { dx, dy }
static const int SHAPES[7][4][4][2] = {
    // I 피스
    {
        { {0,1},{1,1},{2,1},{3,1} },   // rot 0 : 가로 막대
        { {2,0},{2,1},{2,2},{2,3} },   // rot 1 : 세로 막대
        { {0,2},{1,2},{2,2},{3,2} },   // rot 2
        { {1,0},{1,1},{1,2},{1,3} },   // rot 3
    },
    // O 피스 — 4상태가 모두 같다(정사각형은 회전해도 그대로)
    {
        { {1,0},{2,0},{1,1},{2,1} }, { {1,0},{2,0},{1,1},{2,1} },
        { {1,0},{2,0},{1,1},{2,1} }, { {1,0},{2,0},{1,1},{2,1} },
    },
    // T, S, Z, J, L ... (SRS 표준 4상태를 같은 방식으로 하드코딩)
};

SHAPES[current.type][current.rot] 하나로 "지금 이 피스의 지금 회전 상태에서 블록 4칸이 어디에 있는지"를 얻는다. 피스의 실제 보드 위치는 여기에 current.x, current.y 를 더한 것이다.

// 현재 피스의 i번째 블록의 보드 좌표
int c = current.x + SHAPES[current.type][current.rot][i][0];
int r = current.y + SHAPES[current.type][current.rot][i][1];

왜 테이블인가. 회전 공식((x,y) → (-y,x))으로 매번 계산하면 코드가 복잡하고, SRS 표준의 미묘한 회전 중심을 맞추기 어렵다. 미리 계산된 표를 읽는 것이 빠르고, 정확하고, 디버깅하기 쉽다. "계산 대신 조회(lookup)"는 게임 프로그래밍의 흔한 최적화이자 단순화 기법이다 — 1부 F 모듈의 enum 도, 결국 "숫자에 이름을 붙인 표"였다.

개념 3 · 충돌 판정 CanMove — 모든 조작의 심판

테트리스에서 가장 많이 불리는 함수 하나가 CanMove 다. "이 피스를 (dx, dy) 만큼 옮기고 newRot 로 회전하면, 놓을 수 있는가?"를 답한다. 이동·회전·낙하·하드드롭·고스트·게임오버 판정이 전부 이 함수 하나를 쓴다.

bool TetrisContent::CanMove(const Tetromino& p, int dx, int dy, int newRot)
{
    for (int i = 0; i < 4; i++)   // 피스의 4칸을 각각 검사
    {
        int c = p.x + dx + SHAPES[p.type][newRot][i][0];
        int r = p.y + dy + SHAPES[p.type][newRot][i][1];

        if (c < 0 || c >= BOARD_W || r < 0 || r >= BOARD_H)
            return false;         // ① 보드 밖으로 나감
        if (board[r][c] != 0)
            return false;         // ② 이미 고정된 블록과 겹침
    }
    return true;                  // 4칸 모두 이상 없음 → 놓을 수 있다
}

핵심은 "옮겨 본 뒤 판단한다" 는 발상이다. 실제로 피스를 옮기기 전에, dx/dy/newRot 를 가정해 4칸의 최종 위치를 계산하고, 그 자리가 (a)보드 안이고 (b)비어 있는지 확인한다. 하나라도 걸리면 false. 이 "가정 → 검사 → 통과하면 실제 적용" 패턴이 모든 조작의 뼈대다.

void TetrisContent::MoveHorizontal(int dx)
{
    if (CanMove(current, dx, 0, current.rot))   // 옮길 수 있으면
        current.x += dx;                        // 실제로 옮긴다
}

CanMove 하나에 규칙을 모아 둔 덕분에, 이동·회전·낙하가 전부 "먼저 물어보고 움직인다"로 통일된다. 판단 로직을 한 곳에 모으는 것 — 이것이 Snake의 흩어진 충돌 검사보다 한 단계 성숙한 설계다.

개념 4 · 회전과 월킥 — 벽을 밀어내며 돌기

회전은 회전 상태 인덱스를 +1(시계)/+3(반시계, = −1) 하는 것이다. 그런데 벽이나 다른 블록에 붙어 회전하면 4칸 중 일부가 벽을 뚫는다. 그럴 때 그냥 포기하지 않고, 피스를 살짝 밀어 놓을 수 있는 자리를 찾아 주는 것이 월킥(wall kick) 이다.

// 회전 시 시도할 밀어내기 오프셋들 (0,0 부터 순서대로)
static const int KICKS[8][2] = {
    {0,0}, {-1,0}, {1,0}, {-2,0}, {2,0}, {0,-1}, {-1,-1}, {1,-1}
};

void TetrisContent::Rotate(int dir)
{
    int newRot = (current.rot + (dir > 0 ? 1 : 3)) % 4;   // CW=+1, CCW=+3

    for (int k = 0; k < 8; k++)   // 킥 후보를 순서대로 시도
    {
        if (CanMove(current, KICKS[k][0], KICKS[k][1], newRot))
        {
            current.x += KICKS[k][0];   // 처음 성공한 자리로 밀어 회전 확정
            current.y += KICKS[k][1];
            current.rot = newRot;
            return;
        }
    }
    // 8개 모두 실패하면 회전을 취소(아무 일도 안 일어남)
}

동작은 이렇다 — 먼저 제자리 회전({0,0})을 시도하고, 안 되면 왼쪽 1칸, 오른쪽 1칸, 왼쪽 2칸… 순서로 밀며 CanMove 로 확인한다. 처음 성공하는 자리로 밀어 회전을 확정하고, 8개 모두 실패하면 회전을 없던 일로 한다. 벽에 붙은 I 피스가 회전하며 살짝 튕겨 세워지는 그 감각이 이 표에서 나온다.

as-built 참고 — 기본 킥 vs 정식 SRS. 이 코드의 KICKS 는 SRS(Super Rotation System)의 단순화 버전이다. 정식 SRS는 피스 종류와 회전 방향마다 다른 킥 테이블을 쓰지만, 콘솔 MVP에서는 이 8개 오프셋으로도 충분히 자연스럽다. "완벽한 규격보다 동작하는 단순함을 먼저" — GDD가 정식 킥을 폴리시 단계로 미룬 이유다.

개념 5 · 라인 클리어 — 배열 조작의 정석

한 줄이 가득 차면 지우고, 그 위의 모든 줄을 한 칸씩 끌어내린다. 이것이 테트리스의 카타르시스이자, 2D 배열 조작의 교과서적 알고리즘이다.

int TetrisContent::ClearLines()
{
    int cleared = 0;

    for (int r = BOARD_H - 1; r >= 0; r--)   // 아래 행부터 위로 스캔
    {
        bool full = true;
        for (int c = 0; c < BOARD_W; c++)
            if (board[r][c] == 0) { full = false; break; }   // 빈칸 있으면 안 참

        if (full)
        {
            // r행 위의 모든 행을 한 칸씩 아래로 끌어내린다
            for (int rr = r; rr > 0; rr--)
                for (int c = 0; c < BOARD_W; c++)
                    board[rr][c] = board[rr - 1][c];
            for (int c = 0; c < BOARD_W; c++)
                board[0][c] = 0;   // 맨 위 행은 빈칸으로

            cleared++;
            r++;   // ★ 끌어내린 새 행을 같은 위치에서 다시 검사
        }
    }
    return cleared;
}

두 곳을 눈여겨보자.

  • 끌어내리기(shift). board[rr][c] = board[rr-1][c] 로 위 행을 아래 행에 복사하며 내려온다. 1부 C 모듈에서 직원을 해고할 때 배열을 한 칸씩 당기던 그 조작 — 그것의 2차원 버전이다.
  • r++ 의 함정 회피. 한 줄을 지우고 위를 끌어내리면, 방금 r 자리에 새로운 행(원래 r-1이던 행)이 내려온다. 그 새 행도 꽉 찼을 수 있으니, r++ 로 인덱스를 되돌려 같은 자리를 다시 검사한다(다음 루프의 r-- 와 상쇄되어 제자리). 이 한 줄을 빠뜨리면, 4줄을 동시에 지우는 "테트리스"에서 줄이 새어 나간다. 배열을 조작하며 순회할 때, 인덱스가 어긋나지 않는지 항상 의심하라.

지운 줄 수에 따라 점수가 붙는다 — 많이 한 번에 지울수록 이득이 크다(도박의 유혹).

static const int lineScore[5] = { 0, 100, 300, 500, 800 };  // 0·1·2·3·4줄
score += lineScore[cleared] * (level + 1);   // 레벨이 높을수록 배점 ↑
lines += cleared;
level = lines / 10;                          // 10줄마다 레벨업 → 빨라진다

개념 6 · 7-bag 랜덤 — 무작위를 공정하게

피스를 매번 순수 랜덤(rand() % 7)으로 뽑으면, 운 나쁘면 I 피스가 20번 안 나올 수도 있다. 그래서 현대 테트리스는 7-bag 을 쓴다 — 7개 피스를 봉지에 넣고 섞어 하나씩 꺼내고, 봉지가 비면 다시 채운다. 한 사이클에 각 피스가 정확히 한 번씩 나오니, 무작위이면서도 공정하다.

void TetrisContent::RefillBag()
{
    bag.clear();
    for (int i = 0; i < 7; i++)
        bag.push_back(i);           // 0~6 (7종) 을 봉지에 넣고

    // Fisher-Yates 셔플 — 뒤에서부터 임의의 앞 원소와 교환
    for (int i = 6; i > 0; i--)
    {
        int j = rand() % (i + 1);   // 0 ~ i 중 하나
        int tmp = bag[i]; bag[i] = bag[j]; bag[j] = tmp;   // 교환
    }
}

int TetrisContent::DrawNext()
{
    if (bag.empty()) RefillBag();   // 봉지 비면 다시 채워 섞는다
    int t = bag.back();             // 뒤에서 하나 꺼내기
    bag.pop_back();
    return t;
}

Fisher-Yates 셔플은 배열을 공정하게 섞는 표준 알고리즘이다. 뒤에서부터 각 원소를, 자기 포함 앞쪽 임의의 원소와 교환한다. 이렇게 하면 모든 순열이 같은 확률로 나온다 — 카드를 섞는 정석이다. 1부 E 모듈의 rand() 가, 여기서 "공정한 섞기"라는 알고리즘으로 자란다.

nextQueue 에 미리 몇 개를 뽑아 두면 NEXT 미리보기가 된다.

void TetrisContent::SpawnPiece()
{
    while ((int)nextQueue.size() < 4)     // 큐를 4개 이상으로 유지
        nextQueue.push_back(DrawNext());

    int t = nextQueue.front();            // 맨 앞을 꺼내 스폰
    nextQueue.erase(nextQueue.begin());

    current = { t, 0, 3, 0 };             // 상단 중앙(col 3, row 0)에
    if (!CanMove(current, 0, 0, current.rot))   // 스폰 자리가 막혔으면
        gameState = _EGameState::GAMEOVER;      // → 게임 오버
}

스폰하자마자 CanMove 로 놓을 수 있는지 확인한다. 이미 블록이 쌓여 스폰 자리가 막혔으면 그것이 곧 게임 오버다.

개념 7 · DAS — 누르고 있으면 미끄러지는 이동

방향키를 한 번 누르면 한 칸, 꾹 누르고 있으면 처음엔 잠깐 멈췄다가 이후 빠르게 반복 이동한다. 이 "초기 지연 후 자동 반복"을 DAS(Delayed Auto Shift) 라 한다. 키보드로 문자를 꾹 누를 때 "ㄱ … ㄱㄱㄱㄱ" 하는 그 감각이다.

bool downL = INPUT->OnKeyDown(VK_LEFT);    // 막 눌림(엣지)
bool stayL = INPUT->OnKeyStay(VK_LEFT);    // 눌려 있음(레벨)
float now  = TIMER->GetContentTime();

if (downL) {
    MoveHorizontal(-1);              // 즉시 한 칸
    dasDir = -1;
    dasNextMove = now + 0.16f;       // 초기 지연 0.16초 (첫 반복까지 뜸)
}
else if (dasDir == -1 && stayL) {    // 계속 누르고 있으면
    if (now >= dasNextMove) {
        MoveHorizontal(-1);          // 반복 이동
        dasNextMove = now + 0.05f;   // 반복 간격 0.05초 (빠르게)
    }
}
else {
    dasDir = 0;                      // 손 떼면 리셋
}

0단계에서 배운 엣지(OnKeyDown)와 레벨(OnKeyStay) 이 여기서 함께 쓰인다. 막 눌린 순간(엣지)엔 즉시 한 칸 옮기고 초기 지연을 걸고, 눌려 있는 동안(레벨)엔 정해진 간격마다 반복한다. 초기 지연(0.16초)이 있어야 "한 칸만 톡" 과 "쭉 미끄러짐"을 손가락이 구분할 수 있다 — 이 미세한 타이밍이 조작감을 좌우한다.

"입력 주기 ≠ 상태 변화 주기"의 심화. 0단계에서 "입력은 매 프레임, 이동은 틱마다"를 배웠다. DAS는 여기서 한 걸음 더 나아가, 같은 키라도 "막 눌림"과 "계속 눌림"에 다른 타이밍을 준다. 이런 입력 타이밍 설계가 Dino의 점프 버퍼, Road Fighter의 조작으로 이어진다.

코드 워크스루 — OnInGameUpdate 한 프레임

이제 조각들이 한 프레임에서 어떻게 맞물리는지, 게임의 심장 OnInGameUpdate 를 따라가자.

실제 파일: Main/Content/TetrisContent.cpp — OnInGameUpdate()

void TetrisContent::OnInGameUpdate()
{
    if (INPUT->OnKeyDown(VK_ESCAPE)) { currentPhase = _EPhase::TITLE; return; }  // 타이틀로
    if (INPUT->OnKeyDown('P')) { /* PLAYING ↔ PAUSE 토글 */ }
    if (gameState == _EGameState::PAUSE)    return;                  // 일시정지면 멈춤
    if (gameState == _EGameState::GAMEOVER) { /* Enter → 타이틀 */ return; }

    // ── PLAYING ──
    // ① 좌우 이동 + DAS (개념 7)
    // ② 회전 (개념 4)
    if (INPUT->OnKeyDown(VK_UP) || INPUT->OnKeyDown('X')) Rotate(1);
    if (INPUT->OnKeyDown('Z')) Rotate(-1);
    // ③ 홀드 / 하드드롭
    if (INPUT->OnKeyDown('C')) Hold();
    if (INPUT->OnKeyDown(VK_SPACE)) { HardDrop(); return; }

    // ④ 중력 낙하 (레벨별 속도, 소프트 드롭이면 빠르게)
    bool soft = INPUT->OnKeyStay(VK_DOWN);
    float speed = soft ? 0.04f : GRAVITY[level < 14 ? level : 14];
    if (TIMER->GetTickTimer(speed))
    {
        if (CanMove(current, 0, 1, current.rot))   // 아래로 갈 수 있으면
        {
            current.y++;                            // 한 칸 낙하
            if (soft) score += 1;                   // 소프트 드롭 보너스
        }
        else
            LockPiece();                            // 못 내려가면 고정 + 라인 검사 + 다음 피스
    }
}

한 프레임의 순서가 곧 설계다.

  • 입력이 먼저, 중력이 나중. 이동·회전·홀드·하드드롭을 다 처리한 뒤, 마지막에 중력으로 한 칸 떨어뜨린다. 그래서 "떨어지기 직전에 옮기고 돌리는" 조작이 자연스럽게 반영된다.
  • 중력만 틱 타이머 안에. 조작(①~③)은 매 프레임 즉각 반응하지만, 낙하(④)는 GetTickTimer(speed) 로 레벨별 속도에 맞춰서만 일어난다. 0단계의 "입력은 매 프레임, 이동은 틱마다"가 그대로다.
  • LockPiece 가 연쇄를 부른다. 못 내려가면 LockPiece 하나가 [보드에 고정 → 라인 클리어 → 점수·레벨 갱신 → 홀드 리셋 → 다음 피스 스폰]을 줄줄이 처리한다(6장 함수 분해의 정신). 그 스폰이 막히면 게임 오버로 이어진다.

렌더(OnInGameRender)는 이 상태를 그리기만 한다 — 프레임 벽 → 고정 블록(board 순회) → 고스트 피스(DARKGRAY, 착지 위치 미리보기) → 현재 피스 → UI 패널(NEXT·HOLD·SCORE). 고스트는 매 프레임 GetGhost() 로 "현재 피스를 끝까지 떨어뜨린 위치"를 계산해 옅게 그린다.

TetrisContent::Tetromino TetrisContent::GetGhost()
{
    Tetromino g = current;
    while (CanMove(g, 0, 1, g.rot)) g.y++;   // 더 못 내려갈 때까지 낙하
    return g;                                 // 착지 위치
}

GetGhost 도, HardDrop 도 결국 CanMove 를 반복 호출할 뿐이다 — 하나의 충돌 판정이 온 게임을 지탱한다(개념 3).

그림으로 정리 — 회전 테이블과 라인 클리어

📊 웹 교육문서에서는 회전·월킥·라인 클리어가 단계별 애니메이션(V-P2-2)으로 재생된다. 책에서는 아래 도식으로 대신한다.

회전 테이블 — 미리 저장된 4상태 (T 피스)

  rot0      rot1      rot2      rot3
  . ■ .     . ■ .     . . .     . ■ .
  ■ ■ ■     . ■ ■     ■ ■ ■     ■ ■ .
  . . .     . ■ .     . ■ .     . ■ .
  SHAPES[T][rot] 이 이 4칸의 (dx,dy) 를 담는다. 계산 대신 조회.

라인 클리어 — 지우고 끌어내리기

  꽉 찬 행 발견 (r):          지운 뒤 위를 끌어내림:
  . ■ . . ■                  . . . . .   ← board[0] 은 빈칸
  ■ ■ ■ . ■        →         . ■ . . ■   ← 위 행들이
  ■ ■ ■ ■ ■ ← full           ■ ■ ■ . ■      한 칸씩 내려옴
  ┌─────────────────────────────────────────┐
  │ board[rr][c] = board[rr-1][c]  (rr: r→1) │
  │ 그리고 r++ 로 같은 자리 재검사 (4줄 대비) │
  └─────────────────────────────────────────┘

한 문장으로: 2D 배열이 게임판이고, CanMove 가 그 위의 모든 움직임을 심판하며, 회전 테이블·월킥·라인 클리어·7-bag은 그 배열을 다루는 알고리즘들이다.

알려진 이슈에서 배우기 (as-built)

Tetris도 Snake처럼 "완벽한 규격"이 아니라 동작하는 MVP로 구현됐다. GDD가 설계한 것 중 일부는 폴리시 단계로 미뤄졌다. 이 간극을 감추지 않는 이유는, 무엇을 먼저 만들고 무엇을 미룰지 판단하는 것도 개발 기술이기 때문이다.

이슈 as-built 상태 무엇을 배우나
락 딜레이(lock delay) 없음 — 낙하 틱에 못 내려가면 즉시 고정 바닥에서 좌우로 미끄러뜨리는 여유가 없다. "즉시 고정"이 가장 단순한 구현
라인 클리어 연출 없음 — 즉시 지우고 끌어내림 GDD의 LINECLEAR 상태(깜빡임)는 미구현. 재미 없어도 규칙은 완결
정식 SRS 킥 8개 기본 오프셋으로 단순화 "완벽한 규격보다 동작하는 단순함 먼저"(개념 4)
T-Spin·콤보 없음 GDD §13 향후 확장. MVP는 핵심 규칙에 집중
렌더 최적화 매 프레임 전체 렌더(200셀) 60fps에서 충분. "측정 후 최적화" — 느려지면 그때

이 표는 결함 목록이 아니라 우선순위 결정의 기록이다. "락 딜레이 없이도 게임은 완결된다"는 판단, "SRS 킥은 단순화해도 자연스럽다"는 타협 — MVP를 만든다는 것은 이런 결정의 연속이다. 1부 타이쿤이 v0.1부터 자랐듯, 게임은 완벽한 설계를 한 번에 짓는 게 아니라 동작하는 핵심을 먼저 세우고 살을 붙인다.

체크포인트 — 스스로 점검하기

Q1. board[r][c] 의 값 0 과 1~7 은 각각 무엇을 뜻하나? 왜 bool(찼다/비었다)이 아니라 정수인가?

Q2. CanMove 하나로 이동·회전·낙하·하드드롭·게임오버를 모두 판정할 수 있는 이유는?

Q3. 라인 클리어에서 한 줄을 지운 뒤 r++ 를 하는 이유는? 빼먹으면 어떤 상황에서 버그가 나나?

Q4. 순수 rand() % 7 대신 7-bag을 쓰는 이유는? Fisher-Yates 셔플은 무엇을 보장하나?

Q5. 방향키를 꾹 누르면 "한 칸 → 잠깐 멈춤 → 빠르게 반복"으로 움직인다. OnKeyDown 과 OnKeyStay 를 각각 어디에 쓰나?

해설 펼치기

A1. 0 은 빈칸, 1~7 은 그 칸에 고정된 블록의 종류(색) 다(피스 type + 1). bool 이 아니라 정수인 이유는 "찼는지"뿐 아니라 "무슨 색인지" 까지 담아야 렌더 시 색을 결정할 수 있기 때문이다. 하나의 배열이 상태와 색을 겸한다.

A2. CanMove(p, dx, dy, newRot) 가 "피스를 그렇게 옮기고 돌리면 4칸이 보드 안이고 비어 있는가"라는 하나의 질문에 답하기 때문이다. 이동은 (dx,0), 회전은 newRot, 낙하는 (0,1), 하드드롭은 (0,1) 반복, 게임오버는 스폰 자리 (0,0) — 전부 이 질문의 특수한 경우다. 규칙을 한 곳에 모으면 재사용된다.

A3. 한 줄을 지우고 위를 끌어내리면, 방금 그 자리(r)에 새로운 행이 내려온다. 그 새 행도 꽉 찼을 수 있으므로 r++ 로 인덱스를 되돌려 같은 자리를 다시 검사해야 한다. 빼먹으면 연속으로 붙어 꽉 찬 줄들(특히 4줄 테트리스)에서 아래 줄만 지우고 위 줄을 건너뛰는 버그가 난다.

A4. 순수 랜덤은 특정 피스가 오래 안 나오는 "가뭄"이 생겨 불공정하다. 7-bag은 7개를 섞어 하나씩 꺼내므로 한 사이클에 각 피스가 정확히 한 번씩 나온다(공정). Fisher-Yates는 그 7개를 모든 순열이 같은 확률이 되도록 고르게 섞는 것을 보장한다.

A5. OnKeyDown(엣지) — 키가 막 눌린 순간 즉시 한 칸 옮기고 초기 지연 타이머를 건다. OnKeyStay(레벨) — 키가 눌려 있는 동안, 정해진 간격(0.05초)마다 반복 이동시킨다. 초기 지연(0.16초)이 "한 칸만"과 "쭉 미끄러짐"을 구분해 준다.

실습 미션 — 한 칸 블록 쌓기 drop.c ⭐

Snake의 미션이 "리스트·충돌"이었다면, 이 미션은 2D 배열과 라인 클리어 알고리즘을 콕 집어 연습한다. 회전·피스 없이, 테트리스의 심장인 "쌓고 → 꽉 찬 줄 지우고 → 끌어내리기"만 순수하게 구현한다.

드릴 문법 — 2D 배열 board[H][W] · 배열 순회(이중 for) · 열 낙하(landing 계산) · 라인 클리어 + shift(개념 5의 핵심) 만드는 것 — 열 번호를 입력하면 그 열 맨 아래에 블록 한 칸이 쌓이고, 한 줄이 꽉 차면 지워지며 위가 내려오는 미니 퍼즐. 클리어 조건 — 줄이 채워질 때마다 지워지고 점수가 오른다. 한 열이 넘치면 게임 오버. 규모 — 약 55줄, 파일 1개. 표준 C만. 1부의 배열·반복·조건이 테트리스의 핵심 알고리즘으로 모인다.

문제 정의

이 게임의 심장은 딱 두 가지다. ① 블록이 열의 맨 아래(빈칸 중 가장 아래)에 안착하고, ② 꽉 찬 줄을 지우고 위를 끌어내린다. ②가 바로 개념 5의 ClearLines 를 축소한 것이다. 회전도 피스도 없이 이 알고리즘만 순수하게 짜 보면, 테트리스의 진짜 뼈대가 손에 남는다.

목표 / 클리어 조건

  1. int board[H][W] 를 0 으로 초기화한다(빈칸).
  2. 열 번호(1~W)를 입력받아, 그 열의 가장 아래 빈칸에 블록(1)을 놓는다.
  3. 놓은 뒤, 꽉 찬 행을 찾아 지우고 위 행을 한 칸씩 끌어내린다(지운 수만큼 점수).
  4. 어떤 열이 넘치면(맨 위까지 참) 게임 오버.
  5. 매번 보드를 출력해 상태를 보여 준다.

힌트

힌트 1 — 열 낙하(안착 위치 찾기)

그 열을 아래에서 위로 훑어, 처음 만나는 빈칸이 안착 위치다.

int landRow = -1;
for (int r = H - 1; r >= 0; r--)   // 맨 아래부터
    if (board[r][col] == 0) { landRow = r; break; }
if (landRow < 0) { /* 열이 꽉 참 → 게임 오버 */ }
else board[landRow][col] = 1;
힌트 2 — 라인 클리어 (개념 5 축소판) 
int ClearFull(int board[][W]) {
    int cleared = 0;
    for (int r = H - 1; r >= 0; r--) {
        int full = 1;
        for (int c = 0; c < W; c++) if (!board[r][c]) { full = 0; break; }
        if (full) {
            for (int rr = r; rr > 0; rr--)
                for (int c = 0; c < W; c++) board[rr][c] = board[rr-1][c];
            for (int c = 0; c < W; c++) board[0][c] = 0;
            cleared++; r++;      /* 같은 자리 재검사 */
        }
    }
    return cleared;
}

정답 코드

정답 펼치기 — drop.c 
#define _CRT_SECURE_NO_WARNINGS
#include <stdio.h>

#define W 6
#define H 8

int board[H][W];

void PrintBoard(void)
{
    printf("\n");
    for (int r = 0; r < H; r++) {
        printf("  |");
        for (int c = 0; c < W; c++)
            printf("%c", board[r][c] ? '#' : '.');
        printf("|\n");
    }
    printf("   ");
    for (int c = 0; c < W; c++) printf("%d", c + 1);   /* 열 번호 */
    printf("\n");
}

/* 꽉 찬 행을 지우고 위를 끌어내림 → 지운 수 반환 (개념 5) */
int ClearFull(void)
{
    int cleared = 0;
    for (int r = H - 1; r >= 0; r--) {
        int full = 1;
        for (int c = 0; c < W; c++) if (!board[r][c]) { full = 0; break; }
        if (full) {
            for (int rr = r; rr > 0; rr--)
                for (int c = 0; c < W; c++) board[rr][c] = board[rr - 1][c];
            for (int c = 0; c < W; c++) board[0][c] = 0;
            cleared++;
            r++;                    /* 끌어내린 새 행을 같은 자리에서 재검사 */
        }
    }
    return cleared;
}

int main(void)
{
    int col, score = 0;

    for (int r = 0; r < H; r++)
        for (int c = 0; c < W; c++)
            board[r][c] = 0;

    printf("=== 한 칸 블록 쌓기 (열 1~%d, 0=종료) ===\n", W);
    for (;;) {
        PrintBoard();
        printf("점수 %d | 떨어뜨릴 열: ", score);
        if (scanf("%d", &col) != 1) break;
        if (col == 0) break;
        if (col < 1 || col > W) { printf("  ! 1~%d\n", W); continue; }
        col--;                       /* 1~W → 0~W-1 */

        /* 열 낙하: 맨 아래 빈칸 찾기 */
        int landRow = -1;
        for (int r = H - 1; r >= 0; r--)
            if (board[r][col] == 0) { landRow = r; break; }

        if (landRow < 0) {           /* 열이 꽉 참 → 게임 오버 */
            printf("\n>> %d번 열이 넘쳤습니다. 게임 오버! 최종 점수 %d\n", col + 1, score);
            break;
        }
        board[landRow][col] = 1;

        int cleared = ClearFull();   /* 꽉 찬 줄 처리 */
        if (cleared > 0) {
            score += cleared * 100;
            printf("  >> %d줄 클리어! (+%d)\n", cleared, cleared * 100);
        }
    }
    return 0;
}

빌드: cl drop.c (MSVC) 또는 gcc drop.c -o drop

함정/체크포인트

  • ClearFull 의 r++ 를 빼면, 위아래로 붙어 꽉 찬 두 줄을 한 번에 지울 때 위 줄을 놓친다 — 개념 5의 그 함정이 이 미니게임에도 그대로 있다. 두 줄을 동시에 채워 확인해 보자.
  • 열 낙하는 아래에서 위로 훑어 첫 빈칸을 찾는다. 위에서 아래로 훑으면 이미 쌓인 블록 위에 얹혀 버린다.
  • 2D 배열을 함수에 넘길 때 int board[][W] 처럼 열 크기는 반드시 명시해야 한다(행은 생략 가능). 여기선 전역 배열이라 그냥 접근했다.

확장 과제

  • 블록에 색(1~7)을 주고, 같은 색 3개가 인접하면 지우는 규칙으로 바꾸면 뿌요뿌요에 가까워진다.
  • 열이 아니라 가로 2칸짜리 조각을 떨어뜨리게 하면, 회전 없는 테트리스로 한 발 더 나아간다.
