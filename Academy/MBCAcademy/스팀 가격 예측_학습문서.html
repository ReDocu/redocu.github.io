01

프로젝트 개요

"이 게임은 세일 때 얼마나 할인될까?"를 데이터로 답해보는 프로젝트. 장르·평점·출시년도·가격 같은 특징으로부터 할인가격을 예측하는 모델을 만들었다. 데이터를 그대로 넣으면 모델이 학습하지 못하기 때문에, 이상치 제거·인코딩·정규화 같은 전처리가 이 프로젝트의 중심이었다.

항목내용
데이터스팀 게임 목록 데이터 (장르·태그·평점·정가·할인가 등)
전처리이상치 제거, 원-핫 인코딩, 학습/테스트 분리, 정규화
모델회귀 알고리즘 (선형 회귀 → 트리 계열 순으로 비교)
평가MAE·RMSE·R² 지표 비교 + 하이퍼파라미터 튜닝으로 개선
02

진행 과정

P1EDA

분포·상관관계를 그래프로 탐색하며 어떤 특징이 할인과 관련 있는지 파악

P2전처리

이상치 제거 → 원-핫 인코딩 → 데이터 분리 → 정규화 순서로 학습 준비

P3학습 · 평가

여러 회귀 모델을 학습시켜 MAE·RMSE·R²로 성능 비교

P4튜닝

GridSearch로 하이퍼파라미터를 바꿔가며 최적 조합 탐색

03

핵심 학습 개념

EDA (탐색적 데이터 분석)

  • 히스토그램·산점도·상관계수로 데이터를 먼저 이해
  • 가격 분포의 긴 꼬리(고가 게임)를 보고 이상치 기준 결정
  • 모델링 전에 "무엇이 영향을 주는가"의 가설 세우기

전처리 4종 세트

  • 이상치 제거 — IQR 기준으로 극단값 걸러내기
  • 원-핫 인코딩 — 장르 같은 범주형을 0/1 컬럼으로 변환
  • 데이터 분리 — train/test를 나눠 공정한 평가 준비
  • 정규화 — 스케일이 다른 특징을 같은 범위로 맞춤

회귀 모델 · 평가

  • 연속값(가격)을 예측하는 회귀 문제로 정의
  • MAE·RMSE로 오차 크기, R²로 설명력 확인
  • 모델별 성능을 같은 지표로 비교해 선택

하이퍼파라미터 튜닝

  • 모델이 학습하지 못하는 값(트리 깊이 등)은 사람이 정한다
  • GridSearch + 교차검증으로 조합을 체계적으로 탐색
  • 튜닝 전후 지표를 기록해 개선 폭 확인
04

분석 노트

전처리 → 학습 → 평가 → 튜닝 흐름을 요약한 예시 코드.

In [1]전처리 — 이상치 제거와 원-핫 인코딩
# IQR 기준 이상치 제거
q1, q3 = df["price"].quantile([0.25, 0.75])
iqr = q3 - q1
df = df[(df["price"] >= q1 - 1.5 * iqr) & (df["price"] <= q3 + 1.5 * iqr)]

# 범주형(장르) 원-핫 인코딩
df = pd.get_dummies(df, columns=["genre"])
In [2]분리·정규화 후 회귀 모델 학습
from sklearn.model_selection import train_test_split
from sklearn.preprocessing import StandardScaler
from sklearn.ensemble import RandomForestRegressor

X_train, X_test, y_train, y_test = train_test_split(X, y, test_size=0.2)
scaler = StandardScaler()
X_train = scaler.fit_transform(X_train)   # 테스트 데이터는 transform만!
X_test  = scaler.transform(X_test)

model = RandomForestRegressor().fit(X_train, y_train)
In [3]평가와 하이퍼파라미터 튜닝
from sklearn.metrics import mean_absolute_error, r2_score
from sklearn.model_selection import GridSearchCV

pred = model.predict(X_test)
print("MAE:", mean_absolute_error(y_test, pred), "R²:", r2_score(y_test, pred))

grid = GridSearchCV(RandomForestRegressor(),
                    {"n_estimators": [100, 300], "max_depth": [8, 16, None]},
                    cv=5, scoring="neg_mean_absolute_error")
grid.fit(X_train, y_train)
print(grid.best_params_)
05

배운 점 & 회고

  • 전처리 순서에는 이유가 있다. 분리 전에 정규화하면 테스트 정보가 학습에 새어 들어간다(데이터 누수)는 것을 배웠다.
  • 지표 없이 "좋다"는 없다. MAE·RMSE·R²를 함께 봐야 모델의 성격(큰 오차에 민감한지 등)이 보였다.
  • 튜닝은 마법이 아니라 탐색. GridSearch로 조합을 바꿔도 개선 폭은 전처리 품질만큼 크지 않았다 — 결국 데이터가 우선.
  • EDA가 방향을 정한다. 그래프에서 세운 가설이 특징 선택과 이상치 기준의 근거가 됐다.
💡

한 줄 정리EDA → 전처리 → 학습 → 평가 → 튜닝이라는 머신러닝 프로젝트의 표준 절차를 완주했다. 이 절차는 이후 어떤 데이터·모델을 만나도 그대로 적용할 수 있는 틀이 됐다.
