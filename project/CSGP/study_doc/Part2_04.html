4단계 · Road Fighter — 엔티티 시스템과 리소스

단계 4 · 게임 Road Fighter · 씬 클래스 RoadFighterContent : public IGameContent 처음 배우는 기술 — 엔티티 시스템(이종 집합) type 태그로 분기 충돌의 분기 처리 리소스 관리(연료·목숨) 일시적 상태(무적 타이머) 시간 기반 스테이지 HUD 차선 이동 스크롤 누적 예상 학습 시간 — 4–6시간

지금까지의 게임에는 "한 종류의 주인공"만 있었다 — 뱀의 몸통, 떨어지는 블록, 공룡과 장애물. Road Fighter는 다르다. 하나의 도로 위에 적 차량·고속 차량·트럭·연료·보너스·장애물이 뒤섞여 흐른다. 이 여섯 종류를 어떻게 한꺼번에 다룰까? 답은 엔티티 시스템 — 종류를 type 하나로 구분해 한 리스트에 담고, type 에 따라 다르게 처리하는 것이다. 여기에 연료·목숨이라는 자원, 피격 후 잠깐 무적이 되는 일시적 상태, 시간이 흐를수록 어려워지는 스테이지 가 얽힌다. 여러 시스템이 한 게임 안에서 물려 도는 — 시스템 설계를 배우는 단계다.

이 단계의 학습 목표

이 단계를 마치면 다음을 할 수 있다.

  1. 서로 다른 여러 종류의 오브젝트를 하나의 struct + type 태그로 정의해 한 리스트에서 관리할 수 있다.
  2. 같은 충돌이라도 type 에 따라 다른 결과(획득 vs 피해)로 분기 처리할 수 있다.
  3. 연료·목숨 같은 자원의 소모·회복을 관리하고, 자원 고갈을 실패 조건으로 삼을 수 있다.
  4. 무적 타이머 처럼 상태에 수명을 주어, 시간이 지나면 저절로 풀리게 할 수 있다.
  5. GetContentTime() 기반 시간 스테이지로 난이도(스폰·속도·확률)를 점점 올릴 수 있다.
  6. 여러 시스템(엔티티·자원·스테이지·충돌)이 한 프레임 안에서 순서대로 도는 구조를 설계할 수 있다.

왜 Road Fighter가 이 자리인가

Snake는 자료구조를, Tetris는 알고리즘을, Dino는 물리를 배웠다. 각 게임은 하나의 새 기술에 집중했다. Road Fighter의 임무는 다르다 — 앞선 기술들을 한 게임 안에서 동시에 쓰며, 여러 시스템이 얽혀 도는 경험을 하는 것이다.

그 얽힘의 중심에 엔티티 시스템이 있다. 이전 게임들은 오브젝트가 사실상 한 종류였다. 뱀 몸통은 다 같은 몸통이고, 블록은 색만 다른 같은 블록이었다. 하지만 도로 위에는 성격이 전혀 다른 여섯 종류가 함께 흐른다 — 부딪히면 죽는 적 차량, 주우면 회복되는 연료, 점수를 주는 보너스. 이들을 각각 별도의 리스트로 관리하면(적 리스트, 연료 리스트, 보너스 리스트…) 코드가 여섯 배로 늘고 스폰·이동·충돌·제거를 여섯 번씩 짜야 한다.

엔티티 시스템은 이 여섯을 하나로 묶는다. 종류를 나타내는 type 필드 하나를 두고, 모두를 한 리스트에 담는다. 이동·스크롤·제거는 종류에 상관없이 똑같이 처리하고, 다르게 처리해야 할 곳(충돌의 결과)에서만 type 으로 분기한다. 이것이 "이종 집합(heterogeneous collection)"을 다루는 가장 실용적인 방법이자, 이후 Galaga·Battle City가 수많은 오브젝트를 다루는 토대다.

값 타입 벡터, 그리고 그 다음. Dino에서 이어진 "값 타입 std::vector" 설계가 여기서도 그대로다 — std::vector<Entity> 하나에 모든 것을 담는다. 그런데 Road Fighter는 매 프레임 화면 밖으로 나간 엔티티를 지우려고 벡터를 새로 만들어 교체한다(Dino와 같은 방식). 오브젝트가 적을 땐 괜찮지만, 초당 수십 개가 생겼다 사라지는 Galaga(5단계)에서는 이 비용이 문제가 된다 — 그것이 오브젝트 풀의 동기다. Road Fighter의 이 방식을 기억해 두자.

개념 1 · 엔티티 시스템 — 여러 종류를 한 리스트에

도로 위 모든 오브젝트를 하나의 구조체로 정의한다. 종류는 type 필드가 구분한다.

enum class _EEntity { ENEMY_NORMAL, ENEMY_FAST, TRUCK, FUEL, BONUS, OBSTACLE };

struct Entity {
    _EEntity type;   // ★ 이 하나가 "무엇인지"를 결정한다
    int      lane;   // 소속 차선 (0~4)
    float    y;      // 세로 위치 (스크롤로 내려옴)
    int      w, h;   // 히트박스 크기
};

std::vector<Entity> entities;   // 적·트럭·아이템·장애물 — 전부 여기 한 리스트에

type 이 ENEMY_NORMAL 이면 빨간 적 차량, FUEL 이면 초록 연료 아이템, TRUCK 이면 큰 트럭이다. 같은 구조체가 type 값에 따라 다른 것이 된다. 1부 F 모듈에서 배운 enum 이, 여기서 "오브젝트의 정체를 태그하는" 핵심 도구가 된다.

공통 처리는 종류를 안 가린다. 스크롤로 내리는 것도, 화면 밖으로 나간 것을 지우는 것도, 종류에 상관없이 똑같다.

void RoadFighterContent::UpdateWorld()
{
    float step = speed * SCROLL_K;
    for (size_t i = 0; i < entities.size(); i++)
        entities[i].y += step;          // 모든 엔티티를 똑같이 아래로 (type 안 봄)

    // 화면 밖으로 나간 것 제거 — 단, 차량을 피해 보냈으면 점수
    std::vector<Entity> kept;
    for (size_t i = 0; i < entities.size(); i++) {
        if (entities[i].y > (float)ROAD_BOTTOM) {
            _EEntity t = entities[i].type;
            if (t == ENEMY_NORMAL || t == ENEMY_FAST || t == TRUCK)
                score += 20;            // 차량 회피 성공 보너스
        } else {
            kept.push_back(entities[i]);
        }
    }
    entities = kept;
}

여섯 종류를 한 리스트로 다루니, 스폰·이동·제거 코드가 한 벌이면 된다. 종류가 하나 늘어도(예: 오일 웅덩이) enum 에 값 하나 추가하고 필요한 분기만 더하면 끝이다. 이것이 엔티티 시스템의 힘 — 확장이 쉽다.

개념 2 · 충돌의 분기 처리 — 같은 충돌, 다른 결과

엔티티 시스템의 진가는 충돌에서 드러난다. 플레이어가 무언가에 부딪히는 것은 다 같은 AABB 충돌(Dino에서 배운 그것)이다. 하지만 부딪힌 것이 무엇이냐에 따라 결과가 완전히 다르다 — 연료는 회복이고, 적은 피해다. 여기서 type 분기가 필요하다.

void RoadFighterContent::HandleCollisions()
{
    // 플레이어 히트박스 (차선 중앙, 하단 고정)
    int cx = LaneCenterX(playerLane);
    int pLeft = cx - 1, pRight = cx + 1, pTop = PLAYER_Y - 2, pBottom = PLAYER_Y;

    std::vector<Entity> kept;
    for (size_t i = 0; i < entities.size(); i++) {
        Entity e = entities[i];
        int eLeft = LaneCenterX(e.lane) - e.w / 2;
        int eRight = eLeft + e.w - 1;
        int eTop = (int)(e.y + 0.5f), eBottom = eTop + e.h - 1;

        // ── 공통: AABB 겹침 판정 (Dino 개념 5 그대로) ──
        bool hit = !(pRight < eLeft || pLeft > eRight || pBottom < eTop || pTop > eBottom);

        // ── 분기: 무엇에 부딪혔나? ──
        if (hit && e.type == FUEL)  { fuelF += 25; if (fuelF > 100) fuelF = 100; continue; }  // 연료 획득 → 소비
        if (hit && e.type == BONUS) { score += 500; continue; }                               // 보너스 획득 → 소비
        if (hit && !invincible) {                                                             // 차량/장애물 → 피해
            life -= 1;
            speed = SPEED_MIN;
            invincible = true;  invTimer = 90;   // 무적 시작 (개념 4)
        }
        kept.push_back(e);   // 획득 아이템(continue)은 여기 안 옴 = 리스트에서 사라짐
    }
    entities = kept;
}

핵심 구조를 보라 — 충돌 판정은 하나(AABB), 그 뒤 결과가 type 으로 갈라진다.

  • FUEL → 연료 +25, continue 로 리스트에서 제거(소비됨).
  • BONUS → 점수 +500, 마찬가지로 소비.
  • 그 밖(차량·장애물) → 무적이 아닐 때만 목숨 -1, 속도 리셋, 무적 시작.

continue 로 건너뛴 아이템은 kept.push_back 에 안 담기니 자연히 사라지고, 차량은 담기니 계속 남는다(지나가야 회피 점수를 받으므로). 하나의 충돌 루프가, type 분기로 "획득"과 "피해"라는 정반대 결과를 낸다 — 이것이 이종 집합을 다루는 방식이다.

개념 3 · 리소스 관리 — 연료와 목숨

Road Fighter에는 두 개의 자원(resource) 이 있다 — 소모되다 0이 되면 끝나는 연료, 부딪히면 줄어드는 목숨. 이 자원들이 게임에 "관리해야 할 긴장"을 준다.

연료는 매 프레임 조금씩 준다. 그런데 빠를수록 더 많이 소모된다 — 속도를 올리면 점수는 빨라지지만 연료가 빨리 마른다(트레이드오프).

void RoadFighterContent::UpdateFuel()
{
    // 속도가 빠를수록 소모 배수 ↑ (SPEED_MIN에서 1.0배, SPEED_MAX에서 2.0배)
    float factor = 1.0f + (speed - SPEED_MIN) / (SPEED_MAX - SPEED_MIN);
    fuelF -= FUEL_DRAIN * factor;
    if (fuelF < 0.0f) fuelF = 0.0f;
}

목숨은 충돌로 줄고(개념 2), 두 자원 중 하나라도 바닥나면 게임 오버다.

if (life <= 0 || fuelF <= 0.0f) {
    gameState = _EGameState::GAMEOVER;
    if (score > hiScore) hiScore = score;
}

자원을 회복하는 길도 있다 — 연료 아이템(◆)을 주우면 +25(개념 2). 그래서 플레이어는 끊임없이 저울질한다: 속도를 올려 점수를 벌까, 연료를 아낄까? 위험을 무릅쓰고 연료 아이템을 주우러 차선을 바꿀까? 자원 관리가 곧 게임의 의사결정이다. 1부 타이쿤의 "자본금 관리"(돈이 곧 게임)와 같은 발상 — 숫자 자원이 만드는 긴장이다.

개념 4 · 일시적 상태 — 무적에 수명을 주다

부딪혀 목숨이 줄면, 곧바로 또 부딪혀 연속으로 죽는 건 억울하다. 그래서 피격 직후 잠깐 무적이 되어 깜빡이며, 그동안은 충돌을 무시한다. 이 "잠깐"을 어떻게 구현할까? 상태에 수명(타이머)을 준다.

// 피격 시 (개념 2에서):
invincible = true;
invTimer = 90;      // 90프레임 ≈ 1.5초 동안 무적

// 매 프레임 (UpdatePlayer에서):
if (invincible) {
    invTimer--;                        // 수명을 하나씩 깎는다
    if (invTimer <= 0)
        invincible = false;            // 다 되면 무적 해제
}

invTimer 를 매 프레임 1씩 줄이다 0이 되면 무적이 저절로 풀린다. 그리고 무적인 동안엔 충돌 분기에서 !invincible 조건으로 피해를 건너뛴다(개념 2). 렌더는 이 타이머로 깜빡임까지 만든다.

void RoadFighterContent::DrawPlayer()
{
    if (invincible && (invTimer / 4) % 2 == 0)   // 4프레임 주기로 켜졌다 꺼졌다
        return;                                   // 이 프레임엔 안 그림 → 깜빡
    // ... 플레이어 그리기 ...
}

(invTimer / 4) % 2 는 4프레임마다 0↔︎1을 오가는 값이라, 그리기를 건너뛰며 깜빡이는 효과를 낸다. "상태에 수명을 주고, 매 프레임 깎는다" 는 이 패턴은 게임에서 무수히 쓰인다 — 무적 시간, 파워업 지속 시간, 폭발 이펙트의 잔존, 쿨다운. 상태가 영원하지 않고 시간이 지나면 저절로 사라지게 하는 것이다. Tetris의 락 딜레이, Dino의 점수 깜빡도 같은 발상이었다.

개념 5 · 시간 기반 스테이지 — 갈수록 어려워지기

게임이 시작부터 끝까지 똑같이 어려우면 지루하다. Road Fighter는 시간이 흐를수록 스폰이 잦아지고, 빠른 차·트럭이 늘어난다. 스테이지는 엔진의 GetContentTime()(콘텐츠 시작 후 흐른 초)으로 정한다.

void RoadFighterContent::UpdateStage()
{
    float t = TIMER->GetContentTime();   // 게임 시작 후 흐른 초
    if      (t < 30.0f)  stage = 1;
    else if (t < 60.0f)  stage = 2;
    else if (t < 90.0f)  stage = 3;
    else if (t < 120.0f) stage = 4;
    else                 stage = 5;
}

stage 하나가 여러 시스템을 동시에 조인다. 스폰 간격이 짧아지고,

float interval;
switch (stage) {
case 1:  interval = 1.30f; break;   // 스테이지 1: 1.3초마다
case 2:  interval = 1.05f; break;
case 3:  interval = 0.85f; break;
case 4:  interval = 0.70f; break;
default: interval = 0.60f; break;   // 스테이지 5: 0.6초마다 (더 자주!)
}
if (TIMER->GetTickTimer(interval)) SpawnEntity();

스폰되는 차량의 종류 확률도 스테이지에 따라 바뀐다(초반엔 일반 차만, 후반엔 고속·트럭 증가) — Dino의 "점수 게이팅"이 여기선 "시간 게이팅"이다.

// SpawnEntity 안: 스테이지별 고속·트럭 확률
int fastP = 0, truckP = 0;
if      (stage == 2) { fastP = 30; truckP = 0;  }
else if (stage == 3) { fastP = 30; truckP = 15; }
else if (stage >= 4) { fastP = 35; truckP = 25; }

하나의 변수(stage)가 스폰 빈도·차량 구성·기준 속도를 한꺼번에 끌어올린다. 이것이 난이도 곡선을 설계하는 실전 방법이다 — 여러 손잡이를 하나의 다이얼로 묶는 것.

개념 6 · 스크롤 누적과 차선 — 부드러운 세계를 정수 칸에 얹기

속도는 float(km/h)이고 칸은 정수다. "시속 180으로 달릴 때 한 프레임에 몇 칸 내려가야 하나?"를 잇는 것이 스크롤 누적기 scrollAccum 이다.

float step = speed * SCROLL_K;        // 이번 프레임에 흐를 양 (예: 0.18칸)
scrollAccum += step;                  // 누적
while (scrollAccum >= 1.0f) {         // 1칸이 쌓일 때마다
    scrollAccum -= 1.0f;
    distance += 1;                    // 거리 +1
    score += (speed > SCORE_HI) ? 2 : 1;   // 고속이면 +2 (위험 보상)
}

한 프레임에 0.18칸씩 쌓다가 1을 넘으면 그때 거리·점수를 1 올린다. 연속적인 속도를 정수 칸/점수로 변환하는 이 누적 패턴은, 물리(연속)와 격자(이산)를 잇는 다리다(Dino의 float 위치를 반올림하던 것과 짝을 이룬다).

차선은 정수 인덱스(0~4)로 다루고, 화면 x는 헬퍼로 계산한다.

int RoadFighterContent::LaneCenterX(int lane) const
{
    int laneW = (ROAD_RIGHT - ROAD_LEFT - 1) / LANE_COUNT;   // = 7
    return ROAD_LEFT + 1 + lane * laneW + laneW / 2;         // = 5 + 7*lane
}

플레이어의 차선 이동은 엣지 입력에 클램프(벽)를 건다 — Tetris의 좌우 이동과 같은 정신이다.

if (INPUT->OnKeyDown(VK_LEFT)  && playerLane > 0)              playerLane--;
if (INPUT->OnKeyDown(VK_RIGHT) && playerLane < LANE_COUNT - 1) playerLane++;

하드코딩을 피하는 습관. LaneCenterX 가 laneW 를 상수(LANE_COUNT·도로 폭)에서 계산하는 것을 보라. 차선 수를 바꿔도 이 한 함수가 자동으로 맞춘다. 1부 F 모듈의 #define 밸런스 상수, Tetris의 GENRE_COUNT 트릭과 같은 정신 — 매직 넘버 대신 계산으로.

코드 워크스루 — OnInGameUpdate, 다섯 시스템이 도는 한 프레임

Road Fighter의 한 프레임은 여러 시스템이 정해진 순서로 도는 파이프라인이다. 1부 타이쿤의 "월 정산 8단계"가 실시간 버전으로 자란 셈이다.

실제 파일: Main/Content/RoadFighterContent.cpp — OnInGameUpdate()

void RoadFighterContent::OnInGameUpdate()
{
    if (INPUT->OnKeyDown(VK_ESCAPE)) { currentPhase = _EPhase::TITLE; return; }
    if (INPUT->OnKeyDown('P')) { /* PLAYING ↔ PAUSE */ }
    if (gameState == PAUSE)    return;
    if (gameState == GAMEOVER) { if (SPACE/ENTER) StartGame(); return; }

    // ── PLAYING : 다섯 시스템을 순서대로 ──
    UpdatePlayer();        // ① 차선 이동 · 가속/감속 · 무적 타이머 (개념 4·6)
    UpdateStage();         // ② 시간으로 스테이지 갱신 (개념 5)
    UpdateWorld();         // ③ 스크롤 · 스폰 · 제거 · 회피 점수 (개념 1·5·6)
    UpdateFuel();          // ④ 연료 소모 (개념 3)
    HandleCollisions();    // ⑤ 충돌 → 획득/피해 분기 (개념 2)

    if (life <= 0 || fuelF <= 0.0f) {   // ⑥ 실패 판정 (개념 3)
        gameState = GAMEOVER;
        if (score > hiScore) hiScore = score;
    }
}

이 순서가 곧 설계다.

  • 입력·상태가 먼저(①②). 플레이어를 옮기고 스테이지를 정한 뒤,
  • 세계를 굴리고(③④) — 스크롤·스폰·연료 소모,
  • 그 결과로 충돌을 판정(⑤) 한다. 세계를 다 움직인 뒤에 부딪힘을 봐야 정확하다(Dino와 같은 원리).
  • 마지막에 실패를 확인(⑥). 목숨이든 연료든 하나만 바닥나도 끝.

각 시스템이 자기 일만 하는 함수 하나로 분리돼 있다(1부 6장 함수 분해). OnInGameUpdate 는 "이 프레임에 무슨 시스템이 어떤 순서로 도는가"의 목차이고, 세부는 각 Update* 함수에 숨었다. 여러 시스템이 얽힌 게임일수록, 이 "파이프라인 + 함수 분해" 구조가 코드를 읽을 수 있게 지킨다.

렌더(OnInGameRender)도 계층적이다 — 도로(벽·차선) → 엔티티(type별 색·모양) → 플레이어(무적이면 깜빡) → HUD(SCORE·STAGE·연료 바·목숨). HUD는 자원을 시각화한다: 연료 바는 잔량에 따라 초록→노랑→빨강으로 색이 변해, 숫자를 안 봐도 위험을 느끼게 한다.

그림으로 정리 — 엔티티 시스템과 충돌 분기

📊 웹 교육문서에서는 엔티티 리스트의 type 분기와 무적 타이머가 애니메이션(V-P2-4)으로 재생된다. 책에서는 아래 도식으로 대신한다.

엔티티 시스템 — 한 리스트, 여러 정체

  std::vector<Entity> entities
  ┌──────────┬──────────┬──────────┬──────────┬──────────┐
  │type=ENEMY│type=FUEL │type=TRUCK│type=BONUS│type=FAST │
  │lane 2    │lane 0    │lane 3    │lane 4    │lane 1    │
  └────┬─────┴────┬─────┴────┬─────┴────┬─────┴────┬─────┘
       │  공통 처리(이동·제거)는 type 안 봄  │
       ▼                                    ▼
   충돌 시에만 분기:  FUEL→연료+  BONUS→점수+  ENEMY/TRUCK→목숨-

무적 타이머 — 상태에 수명을 주다

  피격!  invincible=true, invTimer=90
   ▼
  invTimer:  90 → 89 → ... → 2 → 1 → 0
             │(매 프레임 -1)         │
             └ 이 동안 충돌 무시      └ invincible=false (저절로 해제)
             (invTimer/4)%2 로 깜빡깜빡

한 문장으로: 엔티티 시스템은 여러 종류를 한 리스트로 묶고, 충돌 분기는 그 종류에 따라 결과를 가르며, 자원과 타이머와 스테이지가 얽혀 게임의 긴장을 만든다.

알려진 이슈에서 배우기 (as-built)

Road Fighter도 MVP다. 여러 시스템을 얽다 보니 생긴 타협과 거친 부분들이, 실전 시스템 설계의 교훈을 준다.

이슈 as-built 상태 무엇을 배우나
매 프레임 벡터 재생성 제거·충돌마다 kept 벡터를 새로 만들어 교체(2번!) 간단하지만 매 프레임 복사. 엔티티가 많아지면 병목 → Galaga의 풀
차선 스냅 이동 부드러운 보간 없이 즉시 차선 이동 MVP는 스냅으로 단순화. 보간은 폴리시
AABB 코드 중복 충돌 판정식이 Dino와 거의 동일하게 반복 공통 로직을 유틸로 뽑을 여지("반복은 추출 신호")

"매 프레임 벡터 재생성"이 두 곳(UpdateWorld 제거, HandleCollisions)에서 반복된다. 지금은 엔티티가 적어 괜찮지만, 이 방식이 한계에 부딪히는 게 바로 다음 단계다. Road Fighter의 이 비용이 Galaga 오브젝트 풀의 존재 이유 — "동작하는 코드"에서 "성능을 의식하는 코드"로 넘어가는 다리다.

체크포인트 — 스스로 점검하기

Q1. 적·연료·트럭·보너스를 각각 별도의 리스트로 관리하는 것과, 하나의 Entity 리스트에 type 으로 담는 것의 차이는? 후자의 장점은?

Q2. 플레이어가 연료 아이템에 부딪히는 것과 적 차량에 부딪히는 것은 같은 AABB 충돌이다. 그런데 결과는 정반대다. 이를 코드로 어떻게 구현하나?

Q3. 무적 상태를 "5초 뒤 자동 해제"되게 하려면 무엇이 필요한가? invTimer 는 어떻게 동작하나?

Q4. speed 가 오르면 점수는 빨라지지만 연료는 빨리 마른다. 이 트레이드오프는 어느 코드에서 나오나?

Q5. 스테이지가 오를수록 게임이 어려워지는 것을, 단 하나의 변수 stage 로 어떻게 여러 시스템에 반영하나?

해설 펼치기

A1. 별도 리스트면 스폰·이동·제거·충돌·렌더 코드를 종류마다 반복해야 하고, 종류가 늘 때마다 코드가 배로 는다. type 태그로 한 리스트에 담으면 공통 처리(이동·제거)는 한 벌이면 되고, 다르게 처리할 곳(충돌 결과)에서만 type 으로 분기한다. 확장이 쉽고 코드가 준다(엔티티 시스템의 핵심).

A2. 충돌 판정(AABB)은 하나로 두고, 판정 뒤에 type 으로 분기한다. if (hit && type==FUEL) 연료+; else if (hit && type==BONUS) 점수+; else if (hit && !무적) 목숨-;. 같은 "부딪힘"이 종류에 따라 획득/피해로 갈린다.

A3. 상태에 수명(타이머) 을 준다. 피격 시 invincible=true; invTimer=300;(60fps면 5초=300프레임)로 두고, 매 프레임 invTimer-- 해서 0 이 되면 invincible=false. 그 동안 충돌 분기에서 !invincible 로 피해를 무시한다.

A4. UpdateFuel 에서 factor = 1.0 + (speed-SPEED_MIN)/(SPEED_MAX-SPEED_MIN) — 속도가 빠를수록 factor가 1.0→2.0으로 커져 fuelF -= FUEL_DRAIN * factor 소모가 는다. 한편 스크롤 누적에서 고속이면 점수 +2. 두 시스템이 만나 "빠르면 이득도 손해도 크다"는 트레이드오프가 된다.

A5. UpdateStage 가 시간으로 stage 를 정하면, 스폰 간격(switch(stage) 로 interval 감소), 차량 종류 확률(스테이지별 fast/truck 확률), 기준 속도(SPEED_MIN + (stage-1)*20)가 모두 stage 를 읽어 조정된다. 하나의 다이얼이 여러 손잡이를 동시에 돌린다.

실습 미션 — 차선 피하기 dodge.c ⭐

Dino의 미션이 "물리+AABB"였다면, 이 미션은 엔티티 시스템 + 충돌 분기 + 리소스 관리를 콕 집어 연습한다. 여러 종류가 한 배열에 섞인 채 내려오고, 부딪힌 종류에 따라 자원이 변한다 — Road Fighter의 심장을 축소했다.

드릴 문법 — struct + enum type(이종 집합) · 배열 순회 · type 분기(충돌 결과) · 자원(연료·목숨) 관리 · rand 만드는 것 — 5차선 위로 적·연료·보너스가 섞여 내려오고, 매 턴 차선을 골라 좋은 건 줍고 나쁜 건 피하는 미니 레이서. 클리어 조건 — 연료가 마르거나 목숨이 다하면 종료. 종류별로 다른 결과가 정확히 적용된다. 규모 — 약 65줄, 파일 1개. 표준 C만. 1부의 구조체·배열·enum이 엔티티 시스템으로 모인다.

문제 정의

이 게임의 심장은 두 가지다. ① 서로 다른 종류(적·연료·보너스)를 하나의 구조체 배열에 담고, ② 플레이어와 같은 칸에 온 엔티티를 type 으로 분기 처리한다. 적이면 목숨 -1, 연료면 연료 +, 보너스면 점수 +. 매 턴 연료가 줄어드니, "위험을 무릅쓰고 연료를 주우러 갈지"를 판단해야 한다 — 리소스 관리다.

목표 / 클리어 조건

  1. struct Entity { int type; int lane; int row; } 배열로 여러 종류를 함께 관리한다.
  2. 매 턴: 모든 엔티티를 한 행 아래로, 화면 밖은 제거, 새 엔티티를 위에 랜덤 스폰.
  3. 플레이어 차선을 입력받고(0~4), 플레이어 행에 온 엔티티를 type으로 분기: 적→목숨-1, 연료→연료+30, 보너스→점수+100.
  4. 매 턴 연료 -10. 연료 0 또는 목숨 0이면 게임 오버.
  5. 매 턴 격자를 출력해 상태를 보여 준다.

힌트

힌트 1 — 엔티티 배열과 스폰 
typedef enum { ENEMY, FUEL, BONUS } EType;
typedef struct { EType type; int lane; int row; } Entity;
Entity ents[32]; int count = 0;

/* 스폰: 랜덤 차선·종류 (적이 흔하게) */
void Spawn(void) {
    if (count >= 32) return;
    int r = rand() % 100;
    EType t = (r < 60) ? ENEMY : (r < 80) ? FUEL : BONUS;
    ents[count].type = t;
    ents[count].lane = rand() % 5;
    ents[count].row  = 0;
    count++;
}
힌트 2 — 충돌 분기

플레이어 행(맨 아래)에 온 엔티티를 종류로 분기한다.

if (ents[i].row == PLAYER_ROW && ents[i].lane == playerLane) {
    switch (ents[i].type) {
    case ENEMY: life--;        break;
    case FUEL:  fuel += 30;    break;
    case BONUS: score += 100;  break;
    }
    /* 처리한 엔티티는 제거 표시 */
}

정답 코드

정답 펼치기 — dodge.c 
#define _CRT_SECURE_NO_WARNINGS
#include <stdio.h>
#include <stdlib.h>
#include <time.h>

#define LANES 5
#define ROWS  6
#define PLAYER_ROW (ROWS - 1)
#define MAXE 32

typedef enum { ENEMY, FUEL, BONUS } EType;
typedef struct { EType type; int lane; int row; } Entity;

Entity ents[MAXE];
int count = 0;

char Glyph(EType t) { return t == ENEMY ? 'X' : t == FUEL ? 'F' : '$'; }

void Spawn(void)
{
    if (count >= MAXE) return;
    int r = rand() % 100;
    ents[count].type = (r < 60) ? ENEMY : (r < 80) ? FUEL : BONUS;
    ents[count].lane = rand() % LANES;
    ents[count].row  = 0;
    count++;
}

void PrintGrid(int playerLane)
{
    for (int row = 0; row < ROWS; row++) {
        printf("  ");
        for (int lane = 0; lane < LANES; lane++) {
            char ch = '.';
            for (int i = 0; i < count; i++)
                if (ents[i].row == row && ents[i].lane == lane) ch = Glyph(ents[i].type);
            if (row == PLAYER_ROW && lane == playerLane) ch = ch == '.' ? 'A' : ch;
            printf("%c ", ch);
        }
        printf("\n");
    }
    printf("  0 1 2 3 4  (X=적 F=연료 $=보너스 A=나)\n");
}

int main(void)
{
    int playerLane = 2, life = 3, fuel = 100, score = 0;

    srand((unsigned)time(NULL));
    printf("=== 차선 피하기 (A: 내 위치, X 피하고 F/$ 줍기) ===\n");

    while (life > 0 && fuel > 0) {
        Spawn();
        PrintGrid(playerLane);
        printf("목숨 %d | 연료 %d | 점수 %d | 차선(0~4, 0=유지): ", life, fuel, score);

        int in;
        if (scanf("%d", &in) != 1) break;
        if (in >= 0 && in < LANES) playerLane = in;

        // 엔티티 한 칸 아래로
        for (int i = 0; i < count; i++) ents[i].row++;

        // 충돌 분기 + 화면 밖 제거 (kept 방식)
        Entity kept[MAXE]; int nk = 0;
        for (int i = 0; i < count; i++) {
            if (ents[i].row == PLAYER_ROW && ents[i].lane == playerLane) {
                switch (ents[i].type) {              // ★ type 분기
                case ENEMY: life--;       printf("  ! 적 충돌 (목숨 -1)\n"); break;
                case FUEL:  fuel += 30;   printf("  + 연료 획득 (+30)\n");   break;
                case BONUS: score += 100; printf("  + 보너스! (+100)\n");    break;
                }
                continue;                            // 처리한 엔티티는 제거
            }
            if (ents[i].row < ROWS) kept[nk++] = ents[i];   // 화면 안만 유지
        }
        for (int i = 0; i < nk; i++) ents[i] = kept[i];
        count = nk;

        fuel -= 10;   // 매 턴 연료 소모 (리소스 관리)
    }

    printf("\n>> 게임 오버! 최종 점수 %d (%s)\n",
           score, life <= 0 ? "목숨 소진" : "연료 고갈");
    return 0;
}

빌드: cl dodge.c (MSVC) 또는 gcc dodge.c -o dodge

함정/체크포인트

  • 한 배열, 여러 종류. ents[] 하나에 적·연료·보너스가 섞여 있고, 이동·제거는 종류를 안 가린다. 결과가 갈리는 곳(충돌)에서만 type switch — 엔티티 시스템의 핵심이다.
  • 자원 두 개, 실패 조건 두 개. 목숨과 연료 중 하나만 0이 돼도 끝난다(while (life>0 && fuel>0)). 연료가 매 턴 줄어드니, 적을 피하면서도 연료를 주우러 가야 하는 딜레마가 생긴다 — 이것이 리소스 관리의 재미다.
  • kept 배열로 제거. 처리했거나 화면 밖으로 나간 엔티티를 빼고 나머지만 남긴다. Road Fighter의 매 프레임 벡터 재생성과 같은 방식 — 이 비용이 Galaga 풀의 동기임을 기억하자.

확장 과제

  • 무적 상태 추가: 적에 부딪히면 3턴 무적(invTimer). 개념 4를 붙여 보자.
  • 트럭(2차선 차지) 타입을 추가하면, type 하나로 확장되는 엔티티 시스템의 유연함을 체감한다.

Road Fighter로 우리는 "여러 시스템이 얽혀 도는 게임"을 만들었다. 여섯 종류가 한 리스트에 섞이고, 자원과 타이머와 스테이지가 물려 돈다. 그런데 이 게임은 매 프레임 벡터를 새로 만들어 오브젝트를 관리했다 — 오브젝트가 적을 땐 괜찮지만, 다음 5단계 Galaga에서는 초당 수십 개의 총알이 생겼다 사라진다. 그 부담을 정면으로 만나, 오브젝트 풀로 해결한다. "동작하는 코드"에서 "성능을 의식하는 코드" 로 — 엔지니어링이 한 계단 성숙하는 지점이다.
