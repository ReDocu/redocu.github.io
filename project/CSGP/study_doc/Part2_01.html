1단계 · Snake — 자료구조·충돌·상태머신

단계 1 · 게임 Snake · 씬 클래스 SnakeContent : public IGameContent 처음 배우는 기술 — vector<T*> new/delete·소유권 이터레이터 순회 충돌 판정 방향 입력·역주행 차단 상태머신(페이즈) 이동 틱 타이머 예상 학습 시간 — 3–5시간

Snake는 커리큘럼의 출발점이다. "플레이어의 입력이 자료구조를 바꾸고 → 충돌이 규칙을 만드는" 상호작용의 최소 완성형을 만든다. 여기서 익힌 "오브젝트를 리스트로 관리 + 충돌 + 상태머신"이, 이후 모든 게임의 토대가 된다.

이 단계의 학습 목표

이 챕터를 마치면 다음을 할 수 있다.

  1. IContent → IGameContent → SnakeContent 로 이어지는 상속 구조를 읽고, 각 계층이 무엇을 책임지는지 설명할 수 있다.
  2. std::vector<Snake*> 로 뱀의 몸통을 관리하고, new/delete 로 힙 메모리와 소유권을 직접 다룰 수 있다.
  3. "몸통 팔로우 이동" 알고리즘 — 각 마디가 앞 마디의 직전 위치를 물려받는 방식 — 을 그림으로 설명할 수 있다.
  4. 머리 vs (벽 / 몸 / 아이템 / 트랩 / 터널)의 충돌 판정을 좌표 비교로 구현할 수 있다.
  5. lastDirect 로 역주행 입력을 막는 이유와 방법을 안다.
  6. _EPhase(TITLE/INGAME) 와 서브 메뉴로 상태머신을 구성할 수 있다.

왜 Snake가 출발점인가

2부의 첫 게임으로 Snake를 고른 데는 이유가 있다. Snake는 가장 적은 개념으로 "게임이란 무엇인가"의 본질을 담기 때문이다.

게임은 결국 세 가지가 맞물려 돈다. ① 상태(자료구조) 가 있고, ② 입력이 그 상태를 바꾸고, ③ 규칙(충돌)이 승패를 만든다. Snake에서는 이 셋이 가장 순수하게 드러난다 — 뱀의 몸통이 상태(리스트)이고, 방향키가 입력이며, "머리가 무엇에 닿았는가"가 규칙이다. 여기에 화면 전환을 담당하는 상태머신(타이틀 ↔︎ 인게임)을 얹으면, 이후 모든 게임이 공유할 표준 게임 구조가 완성된다.

또 하나 — Snake는 오브젝트를 가장 순진한 방법으로 관리한다. 몸통 마디, 아이템, 트랩을 전부 new 로 힙에 만들고 delete 로 지운다. 이 "순진한 관리"를 먼저 겪어야, 5단계 Galaga에서 배울 오브젝트 풀링의 가치를 체감할 수 있다. Snake는 그 대조 학습의 출발점이기도 하다.

누적의 시작점. Snake의 vector<Snake*> 관리 → Dino·Road Fighter의 값 타입 벡터 → Galaga·Battle City의 오브젝트 풀. 이 발전의 첫 계단이 여기다.

개념 1 · 게임 위에 게임 얹기 — 상속 구조

Snake는 main.cpp 의 게임 루프를 직접 건드리지 않는다. 대신 엔진이 정해 둔 인터페이스를 상속해, 정해진 빈칸(OnUpdate, OnRender …)만 채운다. 이 상속 사슬을 이해하는 것이 첫걸음이다.

맨 아래 — IContent (순수 인터페이스). 모든 씬이 지켜야 할 최소 약속이다. 네 개의 순수 가상 함수와 가상 소멸자뿐, 구현은 하나도 없다.

class IContent abstract       // abstract: 이 클래스 자체로는 객체를 못 만든다
{
public:
    virtual ~IContent() = default;   // 가상 소멸자 — 자식을 안전하게 delete

    virtual void OnInit()   = 0;     // = 0: 순수 가상. 자식이 반드시 구현
    virtual void OnRelease()= 0;
    virtual void OnUpdate() = 0;
    virtual void OnRender() = 0;
};
  • = 0(순수 가상 함수) 은 "선언만 있고 몸통은 없다. 자식이 반드시 채워라"는 뜻이다. 그래서 IContent 는 그 자체로 객체를 만들 수 없는 추상 클래스다. 게임 루프가 요구하는 4개 메서드의 계약서인 셈이다.
  • virtual ~IContent() — 소멸자가 virtual 이어야, IContent* 포인터로 자식(Snake)을 delete 할 때 자식의 소멸자가 제대로 불린다. 힙 멤버를 가진 씬(Snake의 벡터들)에서 이건 필수다.

중간 — IGameContent (게임용 껍데기). 대부분의 게임은 "타이틀 화면 → 인게임"이라는 공통 흐름을 가진다. 그 공통 부분을 여기서 한 번 구현해 둔다.

class IGameContent : public IContent
{
public:
    enum class _EPhase { NONE, TITLE, INGAME };   // 게임의 큰 상태

protected:
    _EPhase currentPhase = _EPhase::NONE;

public:
    virtual void OnUpdate();   // IContent 의 순수 가상을 여기서 '구현'한다
    virtual void OnRender();

    // 게임마다 다른 부분은 다시 순수 가상으로 자식에게 넘긴다
    virtual void OnTitleUpdate()  = 0;
    virtual void OnTitleRender()  = 0;
    virtual void OnInGameUpdate() = 0;
    virtual void OnInGameRender() = 0;
};

IGameContent::OnUpdate 는 무엇을 할까? 현재 페이즈에 따라 적절한 함수로 갈라 주는 교통정리 만 한다.

void IGameContent::OnUpdate()
{
    switch (currentPhase)
    {
    case _EPhase::TITLE:  OnTitleUpdate();  break;   // 타이틀이면 타이틀 갱신
    case _EPhase::INGAME: OnInGameUpdate(); break;   // 인게임이면 인게임 갱신
    default: break;
    }
}

OnRender 도 똑같이 currentPhase 를 보고 OnTitleRender 또는 OnInGameRender 를 부른다. 엔진은 "지금이 타이틀인가 인게임인가"만 판단하고, 그 안에서 무엇을 그릴지는 게임에게 맡긴다.

맨 위 — SnakeContent (실제 게임). 이제 Snake는 IGameContent 를 상속해, 남은 네 개의 빈칸(OnTitleUpdate/Render, OnInGameUpdate/Render)과 초기화/해제(OnInit, OnRelease)만 채우면 된다.

class SnakeContent : public IGameContent
{
    // ... 뱀·아이템·트랩·터널 데이터 ...
public:
    virtual void OnInit();          // 씬 진입 시 1회
    virtual void OnRelease();       // 씬 이탈/종료 시

    virtual void OnTitleUpdate();   // 타이틀 메뉴 조작
    virtual void OnTitleRender();   // 타이틀 화면
    virtual void OnInGameUpdate();  // ★ 게임 로직의 심장
    virtual void OnInGameRender();  // 게임 화면
};

이 구조의 힘. Tetris도, Dino도, Battle City도 전부 이 네 칸을 채우는 방식으로 만든다. 게임 루프도, 페이즈 분기도, 렌더 타이밍도 다시 짤 필요가 없다. "빈칸 채우기"만으로 새 게임이 태어난다. 이것이 1부의 함수·구조체가 2부의 상속·인터페이스로 자란 모습이다.

개념 2 · 상태머신 — 타이틀과 인게임을 오가다

Snake는 두 개의 큰 상태를 가진다. 타이틀(시작/종료 메뉴)과 인게임(실제 플레이). 그 위에 타이틀 안에서만 쓰는 서브 상태 _ESELECT(커서가 START에 있나 EXIT에 있나)가 얹힌다.

enum class _ESELECT { NONE, START, EXIT, END };   // 타이틀 메뉴 커서
enum class _EDIRECT { NONE, RIGHT, LEFT, UP, DOWN }; // 뱀 진행 방향

타이틀에서 위아래 키로 커서를 옮기고, Enter로 확정한다.

void SnakeContent::OnTitleUpdate()
{
    if (INPUT->OnKeyDown(VK_DOWN)) {                 // ↓ : 커서 아래로
        select = (_ESELECT)((int)select + 1);
        if (select == _ESELECT::END) select = _ESELECT::START;  // 끝이면 처음으로 (순환)
    }
    if (INPUT->OnKeyDown(VK_UP)) {                   // ↑ : 커서 위로
        select = (_ESELECT)((int)select - 1);
        if (select == _ESELECT::NONE) select = _ESELECT::EXIT;  // 처음보다 위면 끝으로
    }

    if (INPUT->OnKeyDown(VK_RETURN)) {               // Enter : 확정
        switch (select) {
        case _ESELECT::START:
            currentPhase = _EPhase::INGAME;          // → 인게임으로 전환
            break;
        case _ESELECT::EXIT:
            SCENE->ChangeContentWithLoading(         // → 로딩 연출 거쳐 프레임워크 타이틀 씬으로
                (int)_ECONTENT::LOAD, (int)_ECONTENT::TITLE);
            break;
        default: break;
        }
    }
}

핵심은 currentPhase = _EPhase::INGAME; 한 줄이다. 이 대입 하나로 다음 프레임부터 엔진의 OnUpdate 가 OnTitleUpdate 대신 OnInGameUpdate 를 부르게 된다 — 상태 전환은 곧 변수 하나를 바꾸는 일이다. 1부 타이쿤의 gameOver 플래그가 상태의 전부였던 것과 같은 원리이지만, 이제는 페이즈가 여럿이다.

반대로 인게임에서 타이틀로 돌아가는 길도 두 개다.

void SnakeContent::OnInGameUpdate()
{
    if (INPUT->OnKeyDown(VK_SPACE))
        currentPhase = _EPhase::TITLE;   // SPACE: 수동으로 타이틀 복귀
    // ... (플레이 로직) ...
    if (life <= 0)
        currentPhase = _EPhase::TITLE;   // 생명 0: 게임 오버 → 타이틀
}

상태 전이도

        Enter(START)
  TITLE ───────────────▶ INGAME
    ▲                      │
    │  SPACE / life<=0     │
    └──────────────────────┘
    │
    │ Enter(EXIT)
    ▼
  프레임워크 타이틀 씬 (다른 게임 선택)

개념 3 · 동적 자료구조 — vector<Snake*> 와 소유권

뱀의 몸통은 길이가 변한다. 아이템을 먹을 때마다 한 마디씩 는다. 길이가 실행 중에 변하는 목록 — 이것이 std::vector 를 쓰는 이유다. 1부의 고정 배열(Employee staff[4])과 동적 배열(realloc)을 지나온 우리에게, vector 는 그 자연스러운 다음 단계다. vector 는 "필요하면 알아서 커지는 배열"이다.

Snake는 몸통 마디 하나를 작은 클래스로 정의한다.

class Snake {
public:
    int xPos, yPos;         // 타일 좌표
    std::string shape;      // 글리프 ("■")
    unsigned short color;   // 색
    Snake(int x, int y, std::string s, unsigned short c)
        : xPos(x), yPos(y), shape(s), color(c) {}
};

std::vector<Snake*> snake;  // 몸통: 머리=맨 앞, 꼬리=맨 뒤. 포인터를 담는다

vector<Snake*> — 객체가 아니라 포인터를 담는 벡터다. 각 마디는 new 로 힙에 만들고, 그 주소를 벡터에 넣는다. 이건 1부에서 배운 포인터의 실전이다: 각 Snake* 는 힙 어딘가에 사는 마디 하나를 가리킨다.

void SnakeContent::SnakeInit()
{
    // 기존 몸통을 전부 해제 (delete)
    for (snakeIter = snake.begin(); snakeIter != snake.end(); snakeIter++) {
        delete (*snakeIter);        // 힙 메모리 반납
        (*snakeIter) = nullptr;
    }
    snake.clear();                  // 벡터 비우기

    Snake* head = new Snake(SPAWN_X, SPAWN_Y, "■", GREEN);  // 머리 하나로 리셋
    snake.push_back(head);
    direct = lastDirect = _EDIRECT::NONE;
}

여기서 소유권(ownership) 이라는 개념이 처음 등장한다. new 로 만든 것은 누군가 반드시 delete 해야 한다 — 안 그러면 메모리 누수다. Snake의 규칙은 명확하다: 씬이 자기가 new 한 모든 오브젝트를 책임지고 해제한다. 그래서 씬이 사라질 때 OnRelease 가 벡터를 전부 순회하며 delete 한다.

void SnakeContent::OnRelease()
{
    IGameContent::OnRelease();

    for (snakeIter = snake.begin(); snakeIter != snake.end(); snakeIter++) {
        delete (*snakeIter);        // 몸통 마디마다 해제
        (*snakeIter) = nullptr;
    }
    snake.clear();
    // itemList, trapList, tunnelList 도 같은 방식으로 delete + clear ...
}

⚠️ 이 방식의 한계 — 커리큘럼의 복선. 매 아이템마다 new, 리셋마다 delete … 이 "순진한 관리"는 Snake처럼 오브젝트가 적을 땐 문제없다. 하지만 초당 총알 수십 개가 생겼다 사라지는 Galaga(5단계)에서는 이 방식이 성능 병목이 된다. 그때 배우는 것이 오브젝트 풀 — 미리 잡아 둔 슬롯을 켜고 끄며 재사용해 new/delete 를 0회로 만드는 기법이다. Snake의 new/delete 를 똑똑히 기억해 두자. 5단계에서 정확히 이 지점을 개선한다.

개념 4 · 몸통 팔로우 이동 — 뱀은 어떻게 기어가는가

뱀 이동의 고전 알고리즘은 우아하다. 머리를 한 칸 옮기고, 각 마디는 바로 앞 마디가 방금 전까지 있던 자리로 따라간다. 꼬리부터 옮기는 게 아니라 "앞 마디의 직전 위치를 물려받는" 방식이다.

그림으로 보는 팔로우 이동 (머리 H, 몸통 1 2, 오른쪽으로 이동)

  이동 전:   H 1 2               (H는 (5,5), 1은 (4,5), 2는 (3,5))
                 │
             각 마디가 '앞 마디의 직전 위치'를 계승:
               2 ← 1의 옛 자리(4,5)
               1 ← H의 옛 자리(5,5)
             그 다음 머리만 진행 방향으로 +1:
               H → (6,5)
  이동 후:     1 2 H  →  화면상:   2 1 H   (한 칸 오른쪽으로 전체가 흐른 것처럼 보인다)

코드로는 이렇게 구현된다. 벡터를 머리부터 순회하며 "직전에 저장해 둔 위치"를 다음 마디에 넘겨준다.

int saveXPos = 0, saveYPos = 0;   // '앞 마디의 옛 위치'를 잠시 담아 두는 변수
for (snakeIter = snake.begin(); snakeIter != snake.end(); snakeIter++)
{
    if (snakeIter == snake.begin()) {          // 머리(맨 앞)라면
        saveXPos = (*snakeIter)->xPos;         // 머리의 현재 위치만 저장하고
        saveYPos = (*snakeIter)->yPos;
        continue;                              // 머리 자신은 아직 안 옮긴다
    }
    int xPos = saveXPos, yPos = saveYPos;      // 앞 마디의 옛 위치를 가져와
    saveXPos = (*snakeIter)->xPos;             // 이 마디의 현재 위치를 다음을 위해 저장
    saveYPos = (*snakeIter)->yPos;
    (*snakeIter)->xPos = xPos;                 // 이 마디를 앞 마디의 옛 자리로 이동
    (*snakeIter)->yPos = yPos;
}

MoveSnake();          // 마지막으로 머리를 진행 방향으로 1칸
lastDirect = direct;  // 이번 프레임의 방향을 기록 (역주행 차단에 사용)

MoveSnake() 는 방향에 따라 머리만 옮긴다.

void SnakeContent::MoveSnake()
{
    switch (direct) {
    case _EDIRECT::RIGHT: (*snake.begin())->xPos++; break;
    case _EDIRECT::LEFT:  (*snake.begin())->xPos--; break;
    case _EDIRECT::UP:    (*snake.begin())->yPos--; break;   // 콘솔 y는 아래로 증가
    case _EDIRECT::DOWN:  (*snake.begin())->yPos++; break;
    default: break;
    }
}

(*snake.begin()) 이 머리다. snake.begin() 은 벡터의 첫 원소를 가리키는 이터레이터, * 로 역참조하면 그 Snake*, 다시 ->xPos 로 머리의 좌표에 닿는다. 1부의 포인터·역참조가 여기서 그대로 쓰인다.

개념 5 · 충돌 판정 — 머리가 무엇에 닿았는가

Snake의 규칙은 전부 "머리가 무엇과 같은 칸에 있는가"로 정해진다. 오브젝트는 모두 정수 타일 좌표라, 충돌은 단순한 좌표 일치 비교다.

대상 머리가 닿으면
아이템 ★ 성장(+1마디), 점수 +10, 아이템 재배치
트랩 ♠ 생명 -1, 뱀 리셋
터널 □ 짝지어진 반대쪽으로 순간이동
자기 몸 생명 -1, 뱀 리셋
벽 생명 -1, 뱀 리셋

아이템 — 성장. 머리가 아이템 칸에 들어가면, 아이템을 옮기고 꼬리 위치에 새 마디를 붙인다.

for (itemListIter = itemList.begin(); itemListIter != itemList.end(); itemListIter++)
{
    if ((*itemListIter)->xPos == (*snake.begin())->xPos &&
        (*itemListIter)->yPos == (*snake.begin())->yPos)      // 머리 == 아이템?
    {
        MoveItem(*itemListIter);                              // 아이템 재배치
        snake.push_back(new Snake(                            // 꼬리에 새 마디 추가
            snake.back()->xPos, snake.back()->yPos, "■", DARKGREEN));
        score += 10;
    }
}

snake.back() 이 꼬리다. 새 마디를 꼬리 위치에 new 로 만들어 벡터에 push_back 한다 — 개념 3의 소유권 규칙대로, 이 new 도 언젠가 OnRelease/SnakeInit 에서 delete 된다.

자기 몸 충돌. 머리를 몸통(두 번째 마디부터)과 하나씩 비교한다.

for (snakeIter = snake.begin() + 1; snakeIter != snake.end(); snakeIter++)
{
    if ((*snakeIter)->xPos == (*snake.begin())->xPos &&
        (*snakeIter)->yPos == (*snake.begin())->yPos)
    {
        SnakeInit();   // 길이 1로 리셋
        life--;
        break;
    }
}

snake.begin() + 1 로 머리(자기 자신)는 건너뛴다. 머리를 머리와 비교하면 항상 충돌이니, 두 번째 마디부터 봐야 한다.

벽 충돌. 좌표가 경계를 벗어났는지 비교한다.

if ((*snake.begin())->xPos <= wallPosX || (*snake.begin())->yPos <= wallPosY ||
    (*snake.begin())->xPos >= WALL_X - 1 - wallPosX ||
    (*snake.begin())->yPos >= WALL_Y - 1 - wallPosY)
{
    SnakeInit();
    life--;
}

터널 워프. 각 터널은 입구·출구 한 쌍(TunnelConnect)이다. 머리가 한쪽에 닿으면 반대쪽 좌표로 옮긴 뒤 MoveSnake() 를 한 번 더 불러 터널에서 빠져나오게 한다.

개념 6 · 방향 입력과 역주행 차단

뱀 게임의 고전 규칙 하나 — 진행 방향의 정반대로는 꺾을 수 없다. 오른쪽으로 가는데 왼쪽 키를 누르면 머리가 곧장 자기 몸으로 파고들어 즉사하기 때문이다. Snake는 lastDirect(직전에 실제로 이동한 방향)와 비교해 이를 막는다.

if (INPUT->OnKeyDown(VK_RIGHT)) {
    if (lastDirect != _EDIRECT::LEFT || snake.size() == 1)  // 왼쪽으로 가던 중이면 무시
        direct = _EDIRECT::RIGHT;
}
if (INPUT->OnKeyDown(VK_LEFT)) {
    if (lastDirect != _EDIRECT::RIGHT || snake.size() == 1)
        direct = _EDIRECT::LEFT;
}
// UP / DOWN 도 같은 방식 ...

두 가지를 눈여겨보자.

  • lastDirect 와 비교하지 direct 와 비교하지 않는다. 한 이동 틱 안에 방향키를 두 번 누르면(예: ↑ 그리고 →) direct 는 중간에 바뀔 수 있다. 하지만 실제로 몸이 움직인 마지막 방향은 lastDirect 다. 그래서 "직전에 정말로 간 방향"의 반대만 막아야 안전하다.
  • || snake.size() == 1 — 길이가 1(머리뿐)이면 몸이 없으니 어느 쪽으로 꺾어도 자기 몸에 부딪힐 수 없다. 그래서 이때는 역주행을 허용한다.

입력을 그대로 믿지 않고 걸러 내는 이 태도는, 1부 Snake의 표현을 빌리면 "입력을 그대로 믿지 않는다"는 게임 프로그래밍의 기본 감각이다.

개념 7 · 이동 틱 — 입력은 매 프레임, 이동은 0.1초마다

여기까지의 이동·충돌 코드는 전부 하나의 if 안에 들어 있다.

if (TIMER->GetTickTimer(0.1f))
{
    // 몸통 팔로우 이동 → MoveSnake → 충돌 판정들 ...
}

게임 루프는 초당 60번 도는데, 뱀이 초당 60칸씩 기어가면 눈으로 쫓을 수 없다. 그래서 이동만 GetTickTimer(0.1f) 로 0.1초(초당 10칸)에 한 번으로 제한한다. 반면 방향키 입력(개념 6)은 이 if 바깥에 있어 매 프레임 받는다. 그래서 조작은 즉각 반응하고, 몸은 일정한 속도로 움직인다.

이 분리가 콘솔 게임의 조작감을 만든다. "입력을 받는 주기"와 "상태가 변하는 주기"는 달라도 된다 — 이 감각은 이후 Tetris의 DAS, Dino의 물리 적분으로 이어진다.

코드 워크스루 — OnInGameUpdate 전체 읽기

이제 조각들을 하나로 모아, 게임의 심장인 OnInGameUpdate 의 흐름을 통째로 따라가자.

실제 파일: Main/Content/SnakeContent.cpp — OnInGameUpdate()

void SnakeContent::OnInGameUpdate()
{
    // ── ① 입력: 매 프레임 (역주행 차단 포함) ──────────────
    if (INPUT->OnKeyDown(VK_SPACE)) currentPhase = _EPhase::TITLE;
    // ← → ↑ ↓ : lastDirect 와 비교해 direct 갱신 (개념 6)

    // ── ② 이동 틱: 0.1초마다 한 번 ────────────────────────
    if (TIMER->GetTickTimer(0.1f))
    {
        //   a. 몸통 팔로우 이동 (개념 4)
        //   b. MoveSnake() 로 머리 1칸, lastDirect = direct

        //   c. 충돌 판정 (개념 5) — 순서가 중요하다:
        //      아이템(성장) → 트랩(리셋) → 터널(워프) → 자기 몸(리셋) → 벽(리셋)
    }

    // ── ③ 게임 오버 판정 ──────────────────────────────────
    if (life <= 0) currentPhase = _EPhase::TITLE;
}

세 구역의 위치가 설계다.

  • ① 입력은 틱 바깥 — 매 프레임 즉각 반응(개념 7).
  • ② 이동·충돌은 틱 안 — 일정 속도로만 상태가 변한다. 충돌을 이동 직후 판정하는 것도 중요하다. 머리를 옮긴 그 자리에서 무엇에 닿았는지 봐야 하기 때문이다.
  • ③ 게임 오버는 틱 바깥 — life 가 0이 된 즉시(틱을 기다리지 않고) 타이틀로 넘긴다.

렌더(OnInGameRender)는 이 상태를 화면에 옮기기만 한다. 우측 UI 패널(SCORE·LIFE·TIME)을 그리고, 뱀 → 아이템 → 트랩 → 터널 → 벽 순서로 OnDrawColor((*it)->xPos * 2, (*it)->yPos, …) 를 부른다. 여기서도 xPos * 2 — 전각 글리프의 콘솔 2칸 규칙(0단계 미리보기 ③)이 그대로 적용된다.

알려진 이슈에서 배우기 (as-built)

Snake는 "설계 후 구현"이 아니라 이미 구현된 코드를 문서화한 게임이라, 실제 코드에 남은 거친 부분들이 있다. 이걸 감추지 않는 이유는, 진짜 코드의 미완성에서 배우는 것이 교과서적 완성품보다 많기 때문이다.

이슈 무엇이 문제인가 무엇을 배우나
멤버 이터레이터 snakeIter 등을 클래스 멤버로 둬서, 중첩 순회 시 서로 간섭할 위험 이터레이터/인덱스는 지역 변수로 두거나 범위 기반 for 를 쓰는 게 안전하다
재시작 시 상태 미리셋 life<=0 → TITLE 후 다시 START를 골라도 life/score 가 리셋 안 됨 → 즉시 튕김 상태 전환 시 무엇을 초기화해야 하는지를 명시적으로 챙겨야 한다
stage 미증가 HUD에 STAGE 1 표시만 있고 증가 로직 없음 "표시"와 "로직"은 별개다. 보이는 것이 구현된 것은 아니다
ESC 처리 부재 인게임 ESC를 소비하지 않아 main.cpp 로 전달 → 프로그램 종료 입력을 어느 계층이 소비하는가가 중요하다(이후 게임은 "인게임 ESC → 타이틀" 규칙으로 통일)
오브젝트 풀 미사용 new/delete 방식 커리큘럼상 의도된 "순진한 관리". 5단계에서 Pool<T,N> 로 개선

이 표는 결함 목록이 아니라 다음 게임들의 개선 항목 목록이다. "재시작 시 상태 리셋"은 모든 게임이 챙겨야 할 문제이고, "ESC 규칙 통일"은 Road Fighter 이후 도입되며, "오브젝트 풀"은 Galaga의 주제다. Snake의 미완성이 곧 이후 커리큘럼의 예고편이다.

체크포인트 — 스스로 점검하기

Q1. IContent 의 네 메서드에 = 0 이 붙어 있다. 이것이 뜻하는 바는? 그리고 IContent 로 직접 객체를 만들 수 있는가?

Q2. IGameContent::OnUpdate 는 게임 로직을 직접 실행하지 않는다. 그럼 무엇을 하는가?

Q3. 몸통 팔로우 이동에서, 왜 머리를 먼저 옮기지 않고 몸통부터 "앞 마디의 옛 위치"로 옮기는가?

Q4. 역주행 차단에서 direct 가 아니라 lastDirect 와 비교하는 이유는?

Q5. new 로 만든 뱀 마디는 어디서 delete 되는가? (두 곳)

해설 펼치기

A1. = 0 은 순수 가상 함수 — 선언만 있고 구현이 없으니 자식이 반드시 채워야 한다. 그래서 IContent 는 추상 클래스이고, 직접 객체를 만들 수 없다(abstract 로도 명시돼 있다). 게임 루프가 요구하는 4개 메서드의 계약서 역할을 한다.

A2. 교통정리(dispatch). currentPhase 를 보고 타이틀이면 OnTitleUpdate, 인게임이면 OnInGameUpdate 를 부른다. "지금이 어느 페이즈인가"만 판단하고, 실제 내용은 게임(자식)에게 맡긴다.

A3. 머리를 먼저 옮기면 머리의 옛 위치가 사라져 다음 마디가 물려받을 자리가 없어진다. 그래서 "앞 마디의 직전 위치를 물려받기"를 먼저 하고, 맨 마지막에 머리를 진행 방향으로 옮긴다.

A4. 한 이동 틱 안에 방향키를 여러 번 누르면 direct 는 중간에 바뀔 수 있다. 실제로 몸이 움직인 마지막 방향은 lastDirect 다. "직전에 정말 간 방향"의 반대만 막아야, ↑→↓ 를 빠르게 눌러 몸으로 파고드는 사고를 정확히 막을 수 있다.

A5. ① 씬이 사라질 때 OnRelease(), ② 뱀이 리셋될 때 SnakeInit(). 두 곳 모두 벡터를 순회하며 delete 하고 clear() 한다. new 한 주체가 delete 도 책임진다 — 소유권 규칙.

이 단계를 마치며

얻은 것 다음에 쓰는 곳
IGameContent 상속으로 게임 만들기 7개 게임 전부 — 빈칸 채우기 패턴
vector<T*> + new/delete + 소유권 Dino·RF(값 벡터)로 개선 → Galaga·BC(풀)
충돌 판정(좌표 일치) Dino·RF·BC의 AABB·타일 충돌로 심화
상태머신(페이즈) 모든 게임 + 각자의 서브 상태

Snake로 우리는 "살아 있는 게임"의 최소 완성형을 만들었다. 입력이 자료구조를 바꾸고, 충돌이 규칙을 만들고, 상태머신이 화면을 오간다. 다음 단계 Tetris는 여기에 명시적 규칙과 알고리즘(회전·라인 클리어·7-bag)을 얹는다. 직관적 충돌에서 설계된 알고리즘으로 — 한 계단 오른다.
