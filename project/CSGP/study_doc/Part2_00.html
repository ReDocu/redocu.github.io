2부 — C++ 콘솔 게임 프레임워크: 7개의 게임으로 오르기

2부는 C++와 Windows 콘솔 API 위에서, 실시간으로 돌아가는 게임 7종을 하나씩 만든다. 1부의 타이쿤이 scanf 를 기다리며 멈춰 있는 턴제 게임이었다면, 2부의 게임들은 매 초 60번 스스로 화면을 다시 그리는 살아 있는 세계다. 뱀이 기어가고, 블록이 떨어지고, 공룡이 뛰고, 탱크가 서로를 쏜다.

게임 하나가 곧 하나의 학습 단계다. 이전 단계에서 익힌 기술 위에 새 기술을 한 겹씩 쌓는다. 그래서 게임의 순서가 곧 난이도의 순서다.

2부를 여는 말 — 엔진과 콘텐츠를 나누다

1부에서 우리는 "값·분기·반복·데이터의 모양"이라는 프로그래밍의 뼈대를 세웠다. 2부에서 그 뼈대 위에 두 가지 새로운 세계를 얹는다.

첫째, C++. 구조체는 클래스가 되고, 함수는 메서드가 되며, typedef 는 상속과 인터페이스로 자란다. 1부의 Company *c 를 주고받던 감각이, 2부의 "객체가 자기 데이터를 스스로 관리한다"는 감각으로 이어진다.

둘째, 실시간 게임의 구조. 턴제에는 없던 것들이 필요해진다 — 매 프레임 화면을 다시 그리는 게임 루프, 깜빡임을 없애는 더블 버퍼링, 키를 폴링하는 입력 시스템, 시간을 재는 타이머, 화면을 갈아 끼우는 씬 시스템.

이 실시간 토대는 게임마다 새로 만들지 않는다. 모든 게임이 공유하는 하나의 엔진(프레임워크) 으로 한 번 만들어 두고, 게임은 그 위에서 자기 로직만 채운다. 이것이 2부의 핵심 설계다.

엔진(Framework)과 콘텐츠(Content)를 나눈다.

  • 엔진 (FrameWork/) — 게임 루프·렌더링·입력·타이머·씬. 어느 게임에도 종속되지 않는 공용 계층.
  • 콘텐츠 (Main/Content/) — Snake, Tetris… 각 게임. 엔진이 제공하는 껍데기(IGameContent)를 상속해 "그 게임만의 로직"이라는 빈칸만 채운다.

그래서 엔진을 한 번 이해하면, 이후 모든 게임은 오직 그 게임 고유의 로직에만 집중할 수 있다.

읽는 순서. 2부는 반드시 0단계 엔진을 먼저 읽는다. 1단계 이후 모든 게임은 이 엔진이 제공하는 껍데기(IGameContent)를 상속해 빈칸만 채우는 방식이라, 엔진을 모르면 어느 게임 코드도 읽히지 않는다. 엔진을 이해하고 나면, 이후 게임들은 "그 게임만의 새 기술"에만 집중하면 된다.

0단계 · 엔진 (FrameWork) — 실시간 게임의 심장을 짓다

0단계 · 엔진 계층 FrameWork/ · main.cpp · framework.h 처음 배우는 기술 — 게임 루프 클래스·메서드 템플릿 싱글턴 정적 멤버 더블 버퍼링 전각 좌표(x*2) 폴링 입력(엣지/레벨) 틱 타이머 프레임 제한 씬 시스템·소유권 순수 가상 함수·인터페이스 상태머신(_EPhase)

게임을 만들기 전에 갖춰야 할 공통 토대가 있다. 매 프레임 스스로 도는 심장(게임 루프), 깜빡임 없는 화면(더블 버퍼), 매 순간 키를 묻는 손(폴링 입력), 시간을 재는 시계(타이머), 화면을 갈아 끼우는 무대 장치(씬). 이 다섯을 하나의 엔진으로 만들어 두면, 이후 7개 게임은 이 엔진 위에서 자기 로직만 채운다. 0단계는 그 엔진을 해부하는 장이다. 여기서 배운 것이 Snake부터 Battle City까지 모든 게임의 바닥이 된다.

이 단계의 학습 목표

이 단계를 마치면 다음을 할 수 있다.

  1. 실시간 게임의 게임 루프(입력→갱신→렌더→교체→대기)를 읽고, 각 단계가 왜 그 순서인지 설명할 수 있다.
  2. 템플릿 싱글턴으로 매니저를 하나만 존재하게 만드는 원리를 이해하고, SCREEN/TIMER/INPUT/SCENE 매크로를 쓸 수 있다.
  3. 더블 버퍼링이 깜빡임을 없애는 원리를 그림으로 설명하고, 전각 문자 = 콘솔 2칸이라 x*2 로 변환하는 이유를 안다.
  4. 폴링 입력의 엣지(OnKeyDown)와 레벨(OnKeyStay)을 구분해, 상황에 맞게 쓸 수 있다.
  5. 틱 타이머(GetTickTimer)와 프레임 제한(SetFrame)으로 게임의 속도를 제어할 수 있다.
  6. 씬 시스템의 등록·전환·소유권을 이해하고, 인터페이스(IContent)와 상태머신(IGameContent)이 게임의 뼈대임을 안다.

왜 엔진부터인가

1부의 타이쿤을 떠올려 보자. 그 게임은 scanf 앞에서 멈춰 사람을 기다렸다. 사람이 숫자를 입력해야만 한 달이 흘렀다. 실시간 게임은 정반대다 — 아무도 입력하지 않아도 뱀은 계속 기어가고, 블록은 계속 떨어진다. 시간이 스스로 흐른다.

이 "스스로 흐름"을 만들려면, 턴제에는 없던 장치들이 필요하다. 매 순간 화면을 다시 그려야 하고(그런데 그냥 그리면 깜빡인다 → 더블 버퍼링), 매 순간 키 상태를 확인해야 하며(폴링 입력), 시간의 흐름을 재야 한다(타이머). 그리고 이 모든 것을 초당 수십 번 반복하는 심장(게임 루프)이 있어야 한다.

문제는, 이 장치들을 게임마다 새로 만들면 낭비라는 것이다. Snake도 Tetris도 똑같은 게임 루프, 똑같은 더블 버퍼가 필요하다. 그래서 우리는 이것들을 한 번만 만들어 FrameWork/ 라는 엔진에 넣어 두고, 게임은 그 위에서 "이 게임만의 로직"만 채운다. 엔진을 한 번 이해하면, 이후 모든 게임은 빈칸 채우기가 된다 — 이것이 0단계를 맨 앞에 두는 이유다.

1부와의 다리. 이 엔진은 1부에서 배운 것들의 C++ 버전이다. 1부의 while 월 루프 → 2부의 게임 루프, 1부의 구조체 → 2부의 클래스, 1부의 함수 포인터로 상태 넘기기 → 2부의 "객체가 자기 상태를 가짐", 1부의 enum Ending → 2부의 _EPhase. 낯선 C++ 문법이 나와도, 그 밑에는 1부에서 이미 익힌 발상이 흐른다.

개념 1 · 게임 루프 — 게임의 심장 (main.cpp)

모든 실시간 게임의 심장은 main.cpp 의 while 루프다. 이 루프가 초당 60번 돌며 입력을 받고 → 상태를 갱신하고 → 화면을 그리고 → 잠깐 쉰다. 1부의 월 루프(while (!gameOver))와 같은 심장이지만, 이번엔 사람을 기다리지 않고 시계에 맞춰 스스로 돈다.

int main() {
    srand(time(NULL));

    SCREEN->OnInit();  TIMER->OnInit();  INPUT->OnInit();  SCENE->OnInit();  // ① 매니저 초기화

    MainContent* main = new MainContent();   // 게임 콘텐츠 생성
    main->OnInit();

    bool gameState = true;
    while (gameState)
    {
        SCREEN->ClearBuffer();               // ② 뒷화면(백버퍼)을 지운다
        main->OnUpdate();                    // ③ 입력·로직: 게임 상태를 갱신
        if (INPUT->OnKeyDown(VK_ESCAPE)) break;   // ESC 로 루프 종료
        if (SCENE->IsQuitRequested()) break;      // 타이틀 EXIT → 종료
        main->OnRender();                    // ④ 뒷화면에 새 그림을 그린다
        SCREEN->FlippingBuffer();            // ⑤ 뒷화면 ↔ 앞화면 교체 (한순간에)
        TIMER->SetFrame(1000.0f / 60.0f);    // ⑥ 다음 프레임까지 대기 (초당 60프레임)
    }

    main->OnRelease();  delete main;         // 역순 해제
    SCENE->OnRelease();  SCENE->ReleaseSingleton();
    INPUT->OnRelease();  INPUT->ReleaseSingleton();
    TIMER->OnRelease();  TIMER->ReleaseSingleton();
    SCREEN->OnRelease(); SCREEN->ReleaseSingleton();
    return 0;
}

이 짧은 함수가 게임의 전부다. 세 가지를 눈여겨보자.

  • 생명주기가 대칭이다. 위에서 매니저를 SCREEN → TIMER → INPUT → SCENE 순으로 초기화(OnInit)했다면, 아래에서는 역순으로 해제(OnRelease/ReleaseSingleton)한다. 나중에 만든 것을 먼저 정리하는 이 대칭은, 자원 관리의 기본 예절이다(1부의 malloc/free 짝맞춤과 같은 정신).
  • 루프 안은 다섯 박자다. 지우고(②) → 갱신하고(③) → 그리고(④) → 교체하고(⑤) → 쉰다(⑥). 이 다섯 박자가 모든 게임에서 똑같다. Snake가 하는 일은 결국 ③의 OnUpdate 와 ④의 OnRender 안을 채우는 것뿐이다.
  • main 은 게임을 모른다. 이 main.cpp 어디에도 "뱀"이나 "블록"이 없다. main 은 그저 MainContent 에게 "갱신해", "그려"라고 시킬 뿐이다. 무엇을 갱신하고 그릴지는 각 게임이 정한다 — 이 분리가 엔진의 핵심이다.

SCREEN->OnInit() 의 -> 는 1부 그대로. SCREEN 은 화면 매니저 객체(정확히는 그 포인터)이고, -> 는 "포인터가 가리키는 객체의 메서드를 부른다"는 뜻이다. 1부에서 c->money 로 구조체 포인터의 멤버에 닿았듯, 2부에서는 SCREEN->OnInit() 로 객체 포인터의 메서드를 부른다. 문법은 같고, 멤버가 함수(메서드)로 바뀌었을 뿐이다.

개념 2 · 싱글턴 매니저 — 하나만 존재하는 객체 (singleton.h)

위 코드의 SCREEN, TIMER, INPUT, SCENE 는 각각 화면·시간·입력·씬을 관리하는 매니저다. 이들은 게임 전체에서 딱 하나만 존재해야 한다 — 화면 버퍼가 둘이면 곤란하니까. "전역에서 접근 가능한, 하나뿐인 객체"를 만드는 방법이 싱글턴(singleton) 패턴이다.

CSGP는 이걸 템플릿으로 만들어 모든 매니저가 재사용한다. 여기서 C++의 두 새 문법 — 클래스와 템플릿 — 을 만난다.

template <typename T>        // T 자리에 어떤 타입이든 끼울 수 있는 '틀'
class singleton
{
protected:
    static T* instance;      // 하나뿐인 인스턴스 (정적 멤버 — 클래스당 하나만 존재)
    singleton() {};          // 생성자를 감춰 외부에서 new 못 하게
public:
    static T* GetSingleton();
    void ReleaseSingleton();
};

template <typename T> T* singleton<T>::instance = 0;   // 정적 멤버 초기화 (처음엔 없음)

template <typename T>
T* singleton<T>::GetSingleton()
{
    if (!instance)               // 아직 없으면
        instance = new T;        // 그때 딱 한 번 만든다 (지연 생성)
    return instance;             // 있으면 그걸 그대로 돌려준다
}

핵심은 GetSingleton() 이다. 처음 부르면 객체를 new 로 만들고, 두 번째부터는 이미 만든 그 하나를 돌려준다. 그래서 어디서 몇 번을 불러도 늘 같은 객체에 닿는다. static(정적)은 "클래스마다 하나만 존재하는 멤버"라는 뜻으로, 그 하나뿐인 인스턴스를 담기에 딱 맞는다.

ScreenManager 는 이 singleton<ScreenManager> 를 상속하므로, ScreenManager::GetSingleton() 으로 그 하나뿐인 화면 매니저를 얻는다. 매번 이렇게 길게 쓰긴 번거로우니, framework.h 가 짧은 매크로를 준다.

#define SCREEN ScreenManager::GetSingleton()   // 화면(더블 버퍼)
#define TIMER  TimerManager::GetSingleton()    // 시간·프레임·틱 타이머
#define INPUT  InputManager::GetSingleton()    // 키보드 폴링
#define SCENE  SceneManager::GetSingleton()    // 씬(게임) 등록·전환

이제 코드 어디서든 SCREEN->OnDraw(...) 라고 쓰면, 그 하나뿐인 화면 매니저에 닿는다. #define 은 1부 F 모듈에서 배운 그 매크로 — "긴 것에 짧은 이름을 붙이기"의 확장이다.

템플릿 = 타입을 나중에 정하는 틀. template <typename T> 는 "T 자리에 나중에 아무 타입이나 끼워 넣겠다"는 선언이다. singleton<ScreenManager>, singleton<TimerManager> … 같은 싱글턴 코드를 타입만 바꿔 네 매니저가 공유한다. 1부에서 같은 코드를 반복하지 않으려 함수로 묶었듯, C++에서는 같은 코드를 타입만 바꿔 쓰려고 템플릿으로 묶는다. 2부 Galaga의 Pool<T, N> 도 이 템플릿의 힘을 쓴다.

개념 3 · 더블 버퍼링과 전각 좌표 (ScreenManager)

콘솔에 직접 한 글자씩 그리면 화면이 깜빡인다. 그리는 도중이 눈에 보이기 때문이다. 그래서 엔진은 화면 버퍼를 둘 준비한다.

더블 버퍼링 — 완성된 그림만 보여 주기

안 보이는 백버퍼에 이번 프레임을 다 그린 뒤, 앞뒤를 한순간에 바꿔 끼운다. 그리는 과정은 감춰지고 완성된 그림만 보이니 깜빡임이 없다. ScreenManager 는 콘솔 화면 버퍼 두 개(hBuffer[0], hBuffer[1])와 "지금 그리는 쪽"을 가리키는 nScreenIndex 를 갖는다.

void ScreenManager::ClearBuffer()   // ① 백버퍼를 공백으로 지운다
{
    COORD Coor = { 0, 0 };  DWORD dw;
    FillConsoleOutputCharacter(hBuffer[nScreenIndex], ' ', SCREEN_SIZE_X * SCREEN_SIZE_Y, Coor, &dw);
    FillConsoleOutputAttribute(hBuffer[nScreenIndex], /* 기본색 */, SCREEN_SIZE_X * SCREEN_SIZE_Y, Coor, &dw);
}

void ScreenManager::FlippingBuffer()   // ③ 백버퍼를 화면에 띄우고, 그리는 쪽을 반대로
{
    SetConsoleActiveScreenBuffer(hBuffer[nScreenIndex]);  // 방금 그린 버퍼를 화면에!
    nScreenIndex = !nScreenIndex;                          // 다음엔 반대 버퍼에 그린다
}

nScreenIndex = !nScreenIndex 한 줄이 0↔︎1을 오간다. 이번에 1번 버퍼에 그려 화면에 띄웠으면, 다음 프레임은 0번 버퍼에 그린다 — 화면에 보이는 동안 뒤에서 조용히 다음 장면을 준비하는 것이다. 이것이 더블 버퍼링이다.

그리기와 색

그림은 OnDrawColor 로 그린다. 커서를 옮기고(SetConsoleCursorPosition), 색을 설정하고(SetConsoleTextAttribute), 글자를 쓴다(WriteFile).

SCREEN->OnDrawColor(x, y, "■", GREEN);   // (x, y)에 초록색 ■ 를 그린다

색은 framework.h 의 상수(BLACK~WHITE, 0~15)로 지정한다. 배경색을 생략(255)하면 기존 배경을 유지한다.

전각 문자 = 콘솔 2칸 — x * 2

여기 콘솔 특유의 함정이 있다. 전각 문자(■ ★ ♠)는 콘솔에서 가로 2칸을 차지한다. 그래서 게임 로직은 논리 좌표 (x, y) 로 계산하되, 그릴 때는 x를 2배로 변환한다. ScreenManager 는 이걸 DrawCell 로 묶어 둔다.

void ScreenManager::DrawCell(short lx, short y, const char* glyph, ...)
{
    OnDrawColor(lx * 2, y, glyph, ...);   // ★ 논리 x → 화면 x*2
}

논리 해상도는 GAME_SIZE_X(40) × GAME_SIZE_Y(25) 지만, 전각 글리프 때문에 화면은 가로 80칸이다. framework.h 의 주석이 이를 명확히 한다 — "한글은 2Byte, 영문은 1Byte이므로 가로줄의 폭을 80으로 정의한다."

1부의 그림자가 여기 있다. 1부 C 모듈에서 "한글 한 글자는 여러 바이트라 strlen 이 눈에 보이는 글자 수와 다르다"고 했다. 그 바이트-글자 불일치가, 2부에서는 "전각 1글자가 화면 2칸"이라는 좌표 문제로 나타난다. 인코딩은 1부와 2부를 관통하는 그림자 주제다. 게임 코드에 xPos * 2 가 자주 보이는 이유가 바로 이것이다.

개념 4 · 폴링 입력 — 엣지와 레벨 (InputManager)

턴제 게임은 scanf 로 입력을 기다렸다. 실시간 게임은 기다리지 않는다. 매 프레임 "지금 이 키가 눌려 있나?"를 물어본다(폴링). InputManager 는 GetAsyncKeyState 로 키 상태를 확인한다.

여기서 중요한 구분이 하나 있다 — 엣지(edge) 와 레벨(level) 이다.

bool InputManager::OnKeyDown(int key)     // 엣지: '막 눌린 순간'만 true
{
    if (GetAsyncKeyState(key) & 0x8000)   // 지금 눌려 있는가?
    {
        if (!keyDown[key])                // 그런데 직전엔 안 눌려 있었나?
        {
            keyDown[key] = true;
            return true;                  // → 방금 막 눌렸다! (딱 한 번 true)
        }
    }
    else
        keyDown[key] = false;             // 떼면 상태 리셋
    return false;
}

bool InputManager::OnKeyStay(int key)     // 레벨: '눌려 있는 동안' 계속 true
{
    return (GetAsyncKeyState(key) & 0x8001);
}

둘의 차이가 게임의 조작감을 만든다.

함수 언제 true 쓰는 곳
OnKeyDown (엣지) 키가 막 눌린 순간 딱 한 번 방향 전환, 메뉴 이동, 점프 — "한 번 누르면 한 번"
OnKeyStay (레벨) 키가 눌려 있는 내내 계속 이동 유지, 발사 연사 — "누르는 동안 계속"

메뉴에서 OnKeyDown 을 쓰는 이유가 분명하다 — 방향키를 한 번 눌렀는데 커서가 열 칸 내려가면 곤란하니까. OnKeyDown 은 "직전 프레임엔 안 눌려 있었나"(!keyDown[key])를 확인해, 누르고 있는 동안엔 처음 한 번만 true 를 낸다. 이 "엣지 감지"가 실시간 입력의 핵심 기법이다.

1부의 "입력을 그대로 믿지 않는다". 1부 Snake(2부지만)의 역주행 차단처럼, 실시간 게임은 입력을 가공해서 쓴다. 엣지/레벨 구분도 그 일종이다 — "눌림"이라는 날것의 상태를 "막 눌림/계속 눌림"으로 나눠, 게임이 원하는 반응만 골라 쓴다.

개념 5 · 틱 타이머와 프레임 제한 (TimerManager)

실시간 게임에는 두 종류의 시간 제어가 필요하다. "전체 게임을 초당 60프레임으로 고정"(프레임 제한)과, "이 동작은 0.1초마다 한 번"(틱 타이머)이다. TimerManager 는 GetTickCount64(밀리초 단위 시각)를 바탕으로 둘 다 제공한다.

프레임 제한 — SetFrame

게임 루프가 너무 빨리 돌면 CPU를 낭비하고 게임이 지나치게 빨라진다. SetFrame(ms) 은 한 프레임에 걸린 시간을 재, 목표 시간에 모자란 만큼 Sleep 으로 쉰다.

void TimerManager::SetFrame(float frame)
{
    unsigned long long elapsed = GetTickCount64() - frameTime;  // 이번 프레임에 걸린 시간
    if ((float)elapsed < frame)
        Sleep((DWORD)(frame - (float)elapsed));                 // 모자란 만큼 쉰다
    frameTime = GetTickCount64();
}

SetFrame(1000.0f / 60.0f) 은 "한 프레임을 약 16.7ms로" — 즉 초당 60프레임으로 고정한다.

틱 타이머 — GetTickTimer

뱀이 초당 60칸씩 기어가면 눈으로 못 쫓는다. "이 동작은 0.1초마다 한 번"이 필요하다. GetTickTimer(초) 는 그 간격이 지났을 때만 true 를 돌려준다.

bool TimerManager::GetTickTimer(float timer)
{
    if (timerMap.find(timer) != timerMap.end())          // 이 타이머가 등록돼 있으면
    {
        if ((double)(GetTickCount64() - timerMap[timer]) / 1000.0 > timer)  // 간격이 지났나?
        {
            timerMap[timer] = GetTickCount64();          // 시각 갱신
            return true;                                 // → 지났다!
        }
        return false;
    }
    AddTimer(timer);   // 처음 보는 타이머면 등록만
    return false;
}
if (TIMER->GetTickTimer(0.1f))   // 0.1초마다 한 번 (초당 10칸)
{
    // 뱀을 한 칸 이동 ...
}

timerMap 은 map<float, ...> 이라, 같은 float 값이 곧 타이머의 키다. 그래서 서로 다른 용도엔 서로 다른 값을 써야 한다(0.1f 이동, 0.5f 애니메이션처럼). GetTickCount64 의 해상도는 10~16ms라, 그보다 짧은 간격은 의미가 없다.

"입력 주기 ≠ 상태 변화 주기". 게임 루프는 초당 60번 도는데(입력은 매 프레임 받음), 뱀 이동은 GetTickTimer(0.1f) 로 초당 10번만 일어난다. 이 분리가 콘솔 게임의 조작감을 만든다 — 조작은 즉각 반응하되, 몸은 일정 속도로 움직인다. 이 감각은 2부 Tetris의 DAS, Dino의 물리 적분으로 계속 이어진다.

개념 6 · 씬 시스템과 소유권 (SceneManager)

게임에는 여러 화면(씬) 이 있다 — 인트로, 타이틀, 로딩, 그리고 각 게임. SceneManager 는 이 씬들을 등록해 두고, 필요할 때 갈아 끼운다. 씬들을 map<int, IContent*>(번호 → 씬)에 담아 관리한다.

void SceneManager::AddContent(int key, IContent* content)   // 씬 등록
{
    if (contentMap.find(key) == contentMap.end())
        contentMap.insert(std::make_pair(key, content));
}

void SceneManager::ChangeContent(int key)                   // 씬 전환
{
    if (curContent != nullptr)
        curContent->OnRelease();     // ① 이전 씬을 정리하고
    if (contentMap.find(key) != contentMap.end())
    {
        curContent = contentMap[key];
        curContent->OnInit();        // ② 새 씬을 초기화한다
    }
}

씬 전환의 리듬이 중요하다 — 이전 씬 OnRelease() → 새 씬 OnInit(). 그래서 씬의 멤버 초기화는 생성자가 아니라 OnInit 에서 한다(씬에 다시 들어올 때 다시 불리는 곳이 OnInit 뿐이므로). 1부 F 모듈의 enum Ending 상태 전환이, 여기서는 "화면을 통째로 갈아 끼우는" 씬 전환으로 자랐다.

소유권 — 누가 delete 하는가

AddContent 에 new 로 만든 씬을 넘기고 나면, 그 씬의 소유권은 SceneManager 에게 넘어간다. 그래서 프로그램이 끝날 때 SceneManager 가 등록된 모든 씬을 delete 한다.

void SceneManager::OnRelease()
{
    if (curContent != nullptr) { curContent->OnRelease(); curContent = nullptr; }
    for (auto& content : contentMap)
        delete content.second;      // 등록된 모든 씬을 해제 (소유권)
    contentMap.clear();
}

1부에서 "new 한 것은 누군가 free/delete 해야 한다"고 배웠다(동적 메모리). 그 규칙이 2부에서는 소유권이라는 개념으로 명확해진다 — "이 객체를 만든 자가 아니라, 소유하기로 한 자가 정리한다." SceneManager 가 씬을 소유하고, 각 씬은 자기가 new 한 게임 오브젝트를 소유해 자기 OnRelease 에서 정리한다. 이 소유권 사슬이 2부 전체의 메모리 관리 원칙이다.

개념 7 · 인터페이스와 상태머신 (IContent / IGameContent)

엔진의 마지막 조각이자, 모든 게임이 딛고 설 계약서다. 엔진은 각 게임에게 "너는 이 네 가지 메서드를 반드시 구현해야 한다"고 요구한다. 그 계약이 인터페이스 IContent 다.

class IContent abstract          // abstract: 이 클래스 자체로는 객체를 못 만든다
{
public:
    virtual ~IContent() = default;   // 가상 소멸자 — 자식을 안전하게 delete
    virtual void OnInit()    = 0;    // = 0: 순수 가상. 자식이 반드시 구현
    virtual void OnRelease() = 0;
    virtual void OnUpdate()  = 0;
    virtual void OnRender()  = 0;
};
  • = 0(순수 가상 함수) 은 "선언만 있고 몸통은 없다. 자식이 반드시 채워라"는 뜻이다. 그래서 IContent 는 그 자체로 객체를 만들 수 없는 추상 클래스다. 게임 루프가 요구하는 4개 메서드의 계약서인 셈이다. 1부의 함수 "선언"(헤더)과 "정의"(소스)의 분리가, 여기서는 "인터페이스가 요구하고 자식이 구현한다"로 자랐다.
  • virtual ~IContent() — 소멸자가 virtual 이어야, IContent* 포인터로 자식을 delete 할 때 자식의 소멸자가 제대로 불린다. 힙 멤버를 가진 씬에서 이건 필수다.

그 위에 IGameContent 가 있다. 대부분의 게임은 "타이틀 화면 → 인게임"이라는 공통 흐름을 가지므로, 그 공통 부분을 여기서 한 번 구현해 둔다. 게임의 큰 상태를 _EPhase 로 두고, 현재 페이즈에 따라 적절한 함수로 갈라 준다.

class IGameContent : public IContent
{
public:
    enum class _EPhase { NONE, TITLE, INGAME };   // 게임의 큰 상태 (1부 enum Ending 의 후예)
protected:
    _EPhase currentPhase = _EPhase::NONE;
public:
    virtual void OnUpdate();   // currentPhase 를 보고 OnTitleUpdate/OnInGameUpdate 로 분기
    // 게임마다 다른 부분은 다시 순수 가상으로 자식에게 넘긴다
    virtual void OnTitleUpdate()  = 0;
    virtual void OnInGameUpdate() = 0;
    virtual void OnTitleRender()  = 0;
    virtual void OnInGameRender() = 0;
};

IGameContent::OnUpdate 는 "지금이 타이틀인가 인게임인가"만 판단하는 교통정리만 하고, 실제 내용은 각 게임에게 맡긴다. 1부의 enum Ending 으로 엔딩을 갈랐듯, 여기서는 _EPhase 로 화면을 가른다 — 상태를 변수에 담고, 그 변수로 흐름을 정한다는 발상이 그대로 이어진다.

이 구조의 힘 — "빈칸 채우기". Snake도, Tetris도, Battle City도 전부 IGameContent 를 상속해 네 개의 빈칸(OnTitleUpdate/Render, OnInGameUpdate/Render)을 채우는 방식으로 만든다. 게임 루프도, 페이즈 분기도, 렌더 타이밍도 다시 짤 필요가 없다. 엔진이 뼈대를 주고, 게임은 살을 붙인다. 1단계 Snake에서 이 패턴을 처음 직접 보게 된다 — 이 인터페이스가 어떻게 실제 게임이 되는지는 다음 파일(Snake)이 자세히 다룬다.

코드 워크스루 — main.cpp 한 바퀴

엔진의 일곱 조각이 실제로 어떻게 맞물려 도는지, 게임 루프 한 바퀴를 따라가 보자.

실제 파일: main.cpp · framework.h · FrameWork/ 의 매니저 4종

  [프로그램 시작]
    srand → 매니저 4종 OnInit → MainContent 생성·OnInit
                         │
    ┌────────────────────▼──────────────────── while (gameState) ───────────┐
    │  SCREEN->ClearBuffer()      백버퍼를 지운다            (개념 3)         │
    │  main->OnUpdate()           INPUT 폴링 + 게임 로직     (개념 1·4·7)     │
    │      └ 현재 씬의 OnUpdate → _EPhase 로 분기            (개념 6·7)       │
    │  if (ESC) break             입력으로 종료              (개념 4)         │
    │  main->OnRender()           백버퍼에 이번 장면을 그린다 (개념 3·7)      │
    │      └ SCREEN->OnDrawColor(x*2, y, ...)  전각 좌표 변환 (개념 3)        │
    │  SCREEN->FlippingBuffer()   백↔앞 교체 (깜빡임 없음)   (개념 3)         │
    │  TIMER->SetFrame(16.7ms)    초당 60프레임으로 고정      (개념 5)         │
    └───────────────────────────────────────────────────────────────────────┘
                         │  (루프 탈출)

이 한 바퀴에 일곱 개념이 전부 들어 있다. SCREEN·TIMER·INPUT·SCENE 는 싱글턴(개념 2)이라 어디서든 같은 하나에 닿고, 화면은 더블 버퍼(개념 3)로 깜빡임 없이 그려지며, 입력은 폴링(개념 4)으로 매 프레임 확인되고, 속도는 타이머(개념 5)로 제어된다. MainContent 는 SceneManager(개념 6)를 통해 현재 씬에 갱신·렌더를 위임하고, 각 씬은 IContent/IGameContent(개념 7)의 계약을 구현한다. 엔진이 무대를 돌리고, 게임은 그 무대 위에서 연기한다.

  │  ·■··★  ← 다음 장면 준비   │      │  ·■··★                    │
  └───────────────────────────┘      └────────────────────────────┘
        FlippingBuffer() 로 nScreenIndex = !nScreenIndex  →  0↔1 교체
   그리는 과정은 늘 숨겨진 버퍼에서. 완성된 것만 화면에 → 깜빡임 없음.

전각 좌표 — 논리 40칸, 화면 80칸

  논리 좌표 (게임 로직):   x = 0  1  2  3 ...  (GAME_SIZE_X = 40)
                             │
                DrawCell(x,y) → OnDrawColor(x*2, y)
                             ▼
  화면 좌표 (콘솔):        col = 0  2  4  6 ...  (전각 1글자 = 2칸, 총 80)
     ■ 하나가 두 칸을 먹으므로, 논리 x 를 2배 해야 제자리에 그려진다.

한 문장으로: 게임 루프는 게임을 초당 60번 되살리는 심장이고, 매니저들은 그 심장이 부리는 손·눈·시계이며, 인터페이스는 게임과 엔진이 맺은 계약이다.

체크포인트 — 스스로 점검하기

Q1. 게임 루프의 다섯 박자(Clear→Update→Render→Flip→SetFrame)에서, FlippingBuffer 를 Render 앞에 두면 무슨 일이 생기나?

Q2. SCREEN, TIMER 같은 매니저를 싱글턴으로 만드는 이유는? GetSingleton() 을 두 번 부르면 서로 다른 객체가 나오나?

Q3. 논리 좌표 (5, 3) 에 전각 글리프를 그리려면 실제로 콘솔 어느 열(column)에 써야 하나? 왜인가?

Q4. 메뉴에서 방향키로 커서를 옮길 때 OnKeyDown 과 OnKeyStay 중 무엇을 써야 하나? 반대로 쓰면 어떻게 되나?

Q5. IContent 의 네 메서드에 = 0 이 붙어 있다. 이것이 뜻하는 바와, IContent 로 직접 객체를 만들 수 있는지 답하라.

해설 펼치기

A1. FlippingBuffer 는 "지금까지 백버퍼에 그린 것을 화면에 띄우는" 동작이다. Render 앞에 두면 아직 이번 프레임을 안 그린 백버퍼(직전에 지워진 빈 화면)를 띄우게 되어, 화면이 비거나 한 프레임 늦게 갱신되며 깜빡인다. 반드시 다 그린 뒤(Render) 교체(Flip)해야 한다.

A2. 화면 버퍼·입력 상태 같은 자원은 게임 전체에서 하나만 존재해야 하기 때문이다(둘이면 상태가 갈라진다). GetSingleton() 은 처음 부를 때 딱 한 번 new 하고, 이후엔 그 하나를 돌려주므로, 몇 번을 불러도 늘 같은 객체가 나온다.

A3. 콘솔 열 10(= 5 × 2)에 써야 한다. 전각 글리프 하나가 콘솔 2칸을 차지하므로, 논리 x를 2배 해야 제자리에 그려진다. 엔진의 DrawCell 이 OnDrawColor(lx*2, y, ...) 로 이 변환을 대신해 준다.

A4. OnKeyDown(엣지)을 써야 한다 — 한 번 누르면 커서가 한 칸만 움직인다. OnKeyStay(레벨)를 쓰면 키를 누르고 있는 매 프레임(초당 60번) 커서가 움직여, 한 번 눌러도 커서가 끝까지 훌쩍 넘어가 버린다.

A5. = 0 은 순수 가상 함수 — 선언만 있고 구현이 없어 자식이 반드시 채워야 한다. 그래서 IContent 는 추상 클래스이고 직접 객체를 만들 수 없다(abstract 로도 명시됨). 게임 루프가 요구하는 4개 메서드의 계약서 역할을 한다.

실습 미션 — 엔진 감 잡기: 움직이는 점 DotContent ⭐

1단계 Snake로 뛰어들기 전에, 엔진만으로 점 하나를 움직여 보자. 게임 루프·폴링 입력·전각 좌표·틱 타이머가 어떻게 맞물리는지 손으로 느끼는 게 목표다.

드릴 기술 — IGameContent 상속 · OnInGameUpdate/Render 채우기 · INPUT->OnKeyDown · SCREEN->OnDrawColor(x*2, y) · TIMER->GetTickTimer 만드는 것 — 방향키로 화면 위 점(●) 하나를 움직이는 최소 씬. 벽 밖으로는 못 나간다. 클리어 조건 — 방향키 입력에 점이 반응하고, 깜빡임 없이 그려진다. 규모 — 약 40줄, 씬 클래스 하나. (실제 CSGP 프로젝트에 새 씬으로 붙여 실행.)

문제 정의

이 미션은 "게임"이라기보다 엔진 사용법 드릴이다. 새 씬(IGameContent 상속)을 만들어 OnInGameUpdate 에서 방향키를 읽어 점 좌표를 바꾸고, OnInGameRender 에서 그 점을 x*2 좌표에 그린다. 엔진이 게임 루프·더블 버퍼를 알아서 돌려 주니, 우리는 "이 씬만의 로직"인 점의 이동과 그리기만 채우면 된다 — 그것이 바로 엔진의 힘이다.

목표 / 클리어 조건

  1. DotContent : public IGameContent 를 만들고, 점 좌표 int x, y 를 멤버로 둔다.
  2. OnInit 에서 점을 화면 중앙(20, 12)에 놓고 currentPhase = _EPhase::INGAME 로 설정.
  3. OnInGameUpdate 에서 방향키(OnKeyDown)로 x/y 를 ±1. 벽(0~39, 0~24) 밖으로는 안 나가게.
  4. OnInGameRender 에서 SCREEN->OnDrawColor(x*2, y, "●", YELLOW) 로 그린다.
  5. framework.h 의 _ECONTENT 에 값을 추가하고 MainContent 에서 씬을 등록·시작.

힌트

힌트 1 — 이동과 벽 막기

OnKeyDown 은 "막 눌린 순간"만 true라, 한 번 누르면 한 칸 움직인다. 벽은 좌표를 범위로 자른다.

if (INPUT->OnKeyDown(VK_RIGHT) && x < GAME_SIZE_X - 1) x++;
if (INPUT->OnKeyDown(VK_LEFT)  && x > 0)               x--;
if (INPUT->OnKeyDown(VK_UP)    && y > 0)               y--;
if (INPUT->OnKeyDown(VK_DOWN)  && y < GAME_SIZE_Y - 1) y++;
힌트 2 — 전각 좌표로 그리기

논리 x를 2배 해서 그린다. 이걸 빠뜨리면 점이 절반 위치에 찍혀 어긋난다.

SCREEN->OnDrawColor(x * 2, y, "●", YELLOW);   // ★ x * 2 (개념 3)

정답 코드

정답 펼치기 — DotContent (핵심부) 
// DotContent.h
#pragma once
#include "Content/Interface/IGameContent.h"

class DotContent : public IGameContent
{
    int x, y;
public:
    virtual void OnInit();
    virtual void OnRelease() {}
    virtual void OnTitleUpdate()  {}   // 이 미션은 타이틀 없이 바로 인게임
    virtual void OnTitleRender()  {}
    virtual void OnInGameUpdate();
    virtual void OnInGameRender();
};
// DotContent.cpp
#include "DotContent.h"
#include "../../framework.h"

void DotContent::OnInit()
{
    x = GAME_SIZE_X / 2;      // 중앙 (20, 12)
    y = GAME_SIZE_Y / 2;
    currentPhase = _EPhase::INGAME;   // 바로 인게임으로
}

void DotContent::OnInGameUpdate()
{
    if (INPUT->OnKeyDown(VK_RIGHT) && x < GAME_SIZE_X - 1) x++;
    if (INPUT->OnKeyDown(VK_LEFT)  && x > 0)               x--;
    if (INPUT->OnKeyDown(VK_UP)    && y > 0)               y--;
    if (INPUT->OnKeyDown(VK_DOWN)  && y < GAME_SIZE_Y - 1) y++;
}

void DotContent::OnInGameRender()
{
    SCREEN->OnDrawColor(x * 2, y, "●", YELLOW);   // 전각 → x*2
}
// framework.h 의 _ECONTENT 에 DOT 추가, MainContent::OnInit 에서:
SCENE->AddContent((int)_ECONTENT::DOT, new DotContent());
SCENE->ChangeContent((int)_ECONTENT::DOT);   // 시작 씬으로

빌드: 실제 CSGP 프로젝트(CSGP.sln)에 위 파일을 추가하고 .vcxproj/.filters 에 등록한 뒤 빌드한다. 콘솔 창에서 실행해야 스크린 버퍼가 동작한다.

함정/체크포인트

  • x * 2 를 빠뜨리면 점이 논리 좌표 그대로 찍혀, 화면 왼쪽 절반에만 몰리고 위치가 어긋난다. 전각 = 2칸을 잊지 말자.
  • OnKeyDown 대신 OnKeyStay 를 쓰면 한 번 눌러도 점이 화면 끝까지 훅 날아간다(매 프레임 이동). 한 칸씩 움직이려면 엣지 입력.
  • 씬 초기화(중앙 배치)를 생성자가 아니라 OnInit 에 둔다 — 씬에 다시 들어올 때 다시 불리는 곳이 OnInit 이다(개념 6).

확장 과제

  • TIMER->GetTickTimer(0.05f) 를 걸어 OnKeyStay 로 "누르는 동안 일정 속도로 미끄러지는" 이동으로 바꿔 보자.
  • 점이 지나간 자리에 잔상을 남기면 — 그게 바로 다음 단계 Snake의 몸통이다.

이 단계를 마치며

얻은 것 다음에 쓰는 곳
게임 루프·생명주기 7개 게임 전부 — 모두 이 루프 위에서 돈다
싱글턴 매니저 4종 전 게임 — SCREEN/TIMER/INPUT/SCENE
더블 버퍼·전각 좌표(x*2) 전 게임의 렌더링
폴링 입력·틱 타이머 전 게임의 조작·속도 제어
IContent/IGameContent 상속 전 게임 — "빈칸 채우기"로 새 게임

엔진을 손에 넣었다. 게임 루프가 초당 60번 심장을 뛰고, 매니저들이 화면·입력·시간·씬을 관리하며, 인터페이스가 게임과 엔진의 계약을 맺는다. 이제 이 무대 위에 첫 배우를 올릴 차례다. 다음 1단계 Snake에서, 우리는 IGameContent 를 처음 상속해 — 입력이 자료구조를 바꾸고, 충돌이 규칙을 만드는 — 살아있는 게임의 최소 완성형을 만든다.
