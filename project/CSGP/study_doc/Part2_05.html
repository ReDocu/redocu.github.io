번외 · GameDevTycoon 이식 — C에서 C++로

번외 · 다리(Bridge) · 게임 Tycoon · 씬 클래스 TycoonContent : public IGameContent 처음 배우는 기술 — 구조체+전역 → 클래스 멤버 char[] → std::string malloc/realloc/고정 배열 → std::vector #define → static const enum → enum class 전역 함수(Company *c) → 메서드 턴제를 실시간 프레임워크에(커서 메뉴 + 상태머신) 예상 학습 시간 — 3–5시간

1부에서 우리는 게임 회사 운영 게임을 C로 처음부터 끝까지 만들었다. 2부에서는 그 위에 C++와 프레임워크를 배웠다. 이 번외 챕터는 두 부를 잇는 다리다 — 1부의 그 게임을, 2부의 프레임워크 위에 C++ 씬으로 다시 짓는다. 구조체는 클래스가 되고, char[] 는 std::string, malloc/realloc 은 std::vector, Company *c 를 넘기던 함수는 메서드가 된다. 새 게임을 배우는 게 아니라, 이미 아는 게임을 새 언어로 다시 쓰며 C→C++ 리팩터링을 온몸으로 겪는 장이다. 그리고 이 게임은 2부에서 유일한 턴제 게임이라, "실시간 프레임워크가 턴제도 담을 수 있는가"라는 물음의 답이기도 하다.

이 챕터의 학습 목표

이 챕터를 마치면 다음을 할 수 있다.

  1. C의 구조체 + 전역 변수 + Company *c 전달을, C++의 클래스 멤버 + 메서드로 옮길 수 있다.
  2. char name[20] 을 std::string 으로, snprintf/strcpy 를 + 연결로 바꿀 수 있다.
  3. malloc/realloc/고정 배열 + 수동 당기기를 std::vector 의 push_back/erase/size() 로 바꿀 수 있다.
  4. #define 상수를 static const 로, enum 을 enum class 로 승격할 수 있다.
  5. 블로킹 scanf 없는 턴제 를 커서 메뉴 + 내부 상태머신(_EVIEW)으로 구현할 수 있다.
  6. C→C++ 이식이 왜 "종합 복습"인지 — 1부의 데이터·2부의 문법이 한자리에서 만나는지 이해한다.

왜 이 이식인가

새 게임을 하나 더 배우는 대신, 왜 이미 만든 게임을 다시 만들까?

프로그래밍에서 가장 확실한 학습은 아는 것을 다른 방식으로 다시 해 보는 것이다. 1부의 타이쿤은 우리가 로직을 속속들이 아는 게임이다 — 자본금이 어떻게 도는지, 직원을 어떻게 고용하는지, 평점을 어떻게 매기는지. 그 로직은 그대로 두고 언어의 옷만 갈아입히면, C와 C++의 차이가 선명하게 드러난다. 게임 규칙에 정신을 뺏기지 않고, 오직 "이 C 코드가 C++에서 어떻게 더 나아지는가"에만 집중할 수 있다.

그리고 이 이식은 1부의 고비들이 C++에서 얼마나 편해지는지 보여 주는 보상이기도 하다.

  • 1부 C 모듈의 병렬 배열 지옥(직원 하나에 배열 다섯 개) → C++에선 std::vector<Staff> 하나.
  • 1부 D 모듈의 Company *c 를 모든 함수에 넘기기 → C++에선 클래스가 곧 회사, 메서드가 멤버에 바로 접근.
  • 1부 E 모듈의 malloc/realloc 로 직원 배열 키우기 → C++에선 push_back 이 알아서.
  • 1부의 snprintf/strcpy 문자열 다루기 → C++에선 "* [" + name + "] 해고" 한 줄.

우리가 1부에서 겪은 불편들이, 여기서 하나씩 사라진다. 그것이 C++가 C 위에 얹은 편의의 정체다.

왜 여기(Road Fighter 뒤·Galaga 앞)인가. Road Fighter까지 우리는 실시간 시스템 설계를 익혔고, 곧이어 Galaga·Maze에서 성능·알고리즘이라는 엔지니어링으로 들어간다. 그 사이에 데이터·경제 게임인 타이쿤을 C++로 리팩터링하며 잠깐 숨을 고른다. 실시간 액션이 아닌 턴제·데이터 게임이라, "프레임워크는 실시간만을 위한 게 아니다"라는 사실도 확인한다.

개념 1 · 회사가 클래스가 되다 — 구조체 + 전역 → 멤버

1부의 타이쿤은 게임의 모든 상태를 Company 구조체 하나에 담고, 그 구조체의 주소(Company *c)를 모든 함수에 넘겼다(D 모듈의 핵심). C++ 이식의 첫걸음은 이 발상을 뒤집는 것이다 — 회사 그 자체가 클래스가 되고, 게임 상태는 그 클래스의 멤버 변수가 된다.

// 2부 C++ — TycoonContent.h : 회사가 곧 씬 클래스
class TycoonContent : public IGameContent
{
private:
    int money = 0, fame = 0;          // 1부의 c->money, c->fame
    int year = 0, month = 0;
    int trend = 0;
    std::vector<Staff>   staffList;    // 1부의 Employee *staff (개념 3)
    std::vector<Staff>   applicants;
    Project              proj;
    std::vector<Release> shelf;
    _EENDING ending = _EENDING::NONE;
    float bestRating = 0.0f;
    // ...
};

1부에서 Company 구조체의 멤버였던 것들이, 이제 TycoonContent 클래스의 멤버가 되었다. 차이는 결정적이다 — 1부에선 함수가 회사를 만지려면 Company *c 를 인자로 받아야 했지만, C++에선 메서드가 같은 클래스의 멤버에 그냥 접근한다.

/* 1부 C — 회사의 주소를 넘겨받아야 만질 수 있다 */
int TotalSalary(const Company *c) {
    int sum = 0;
    for (int i = 0; i < c->staffCount; i++)
        sum += c->staff[i].salary;
    return sum;
}
// 2부 C++ — c-> 가 사라진다. 메서드는 자기 클래스의 멤버를 바로 본다
int TycoonContent::TotalSalary() const {
    int sum = 0;
    for (size_t i = 0; i < staffList.size(); i++)   // c->staff → staffList (멤버)
        sum += staffList[i].salary;                 // c-> 없이 직접
    return sum;
}

const Company *c 인자가 사라지고, c->staff 가 그냥 staffList 가 됐다. 1부 D 모듈에서 그토록 공들여 배운 "포인터로 회사를 넘기기"가, C++에선 클래스라는 문법에 흡수되어 보이지 않게 된다. 메서드는 암묵적으로 자기 객체(this)를 알기 때문이다. 포인터 전달의 수고를 몸으로 겪은 사람만이, 이 사라짐의 편안함을 안다.

개념 2 · char[] → std::string

1부에서 직원 이름은 char name[20] 이었다. 문자열을 복사하려면 strcpy, 만들려면 snprintf, 크기는 늘 신경 써야 했고(글자수+1), 공백·널 종료의 함정이 도사렸다(C 모듈). C++의 std::string 은 이 모든 것을 지운다.

struct Staff {
    std::string name;     // 1부: char name[20]  →  이제 길이 걱정 없음
    int dev, cre;
    int salary;
    int months;
};

문자열을 만들고 잇는 것도 + 연산자 하나로 끝난다. 1부의 snprintf(buf, ...) 가 사라진다.

/* 1부 C — 버퍼 잡고 snprintf */
char label[64];
snprintf(label, sizeof(label), "판매 수익 \"%s\"", r->title);
// 2부 C++ — string + string, 버퍼도 크기도 없다
uiMsg = "* [" + staffList[staffCursor].name + "] 해고 (퇴직금 지급)";
reportMsgs.push_back("[이벤트] " + s.name + " 월급 인상 수락 (" + MoneyText(s.salary) + "만원)");

"* [" + name + "] 해고" — 문자열 리터럴과 std::string 을 + 로 잇는다. 버퍼 크기(char[64])도, snprintf 의 서식도, 널 종료도 신경 쓰지 않는다. std::string 이 길이를 스스로 관리하기 때문이다. 1부 C 모듈에서 문자열이 "char 배열 + '\0'"이라 성가셨던 그 모든 것이, 여기선 그냥 값처럼 다뤄진다.

MoneyText — 1부 MoneyStr 의 C++ 버전. 1부에선 3자리 콤마를 char buf[] 에 채우는 MoneyStr(v, buf, size) 였다(B 모듈). C++ 이식에선 std::string MoneyText(int v) — 결과를 문자열로 반환한다. 호출부도 MoneyText(cost) 처럼 값으로 받아 바로 + 로 잇는다. 반환값이 문자열이 되니, 버퍼를 넘겨받던 번거로움이 사라진다.

개념 3 · malloc/realloc/고정 배열 → std::vector

여기가 이식의 하이라이트다. 1부에서 직원 목록은 가장 큰 골칫거리였다 — C 모듈에선 병렬 배열 다섯 개, D 모듈에서 구조체 배열로 묶고, E 모듈에서 malloc/realloc 로 크기를 키웠다. 고용은 용량 확인 후 확장, 해고는 배열을 한 칸씩 당기기. C++의 std::vector 는 이 모두를 삼킨다.

std::vector<Staff> staffList;   // 1부: Employee *staff + staffCount + staffCap + malloc/realloc

고용 — 1부에선 staffCount == staffCap 이면 realloc 로 2배 확장한 뒤 넣었다(E 모듈). C++에선 push_back 한 줄. 벡터가 알아서 커진다.

/* 1부 C — 꽉 차면 직접 realloc 로 확장 (E 모듈) */
if (c->staffCount == c->staffCap) {
    Employee *tmp = realloc(c->staff, sizeof(Employee) * c->staffCap * 2);
    if (tmp == NULL) return;
    c->staff = tmp; c->staffCap *= 2;
}
c->staff[c->staffCount++] = *e;
// 2부 C++ — 한 줄. 크기 관리·재할당·복사를 vector 가 알아서
staffList.push_back(applicants[ai]);
applicants.erase(applicants.begin() + ai);   // 지원자 목록에서 제거도 한 줄

해고 — 1부의 가장 아팠던 코드. C 모듈에선 병렬 배열 다섯 개를 각각 당겼고, D 모듈에서 구조체 덕에 c->staff[i]=c->staff[i+1] 로 줄었지만 여전히 수동 반복이었다(QuitEmployee). C++에선 erase 한 줄.

/* 1부 C — 뒤 직원을 한 칸씩 당기기 (D 모듈 QuitEmployee) */
void QuitEmployee(Company *c, int idx) {
    for (int i = idx; i < c->staffCount - 1; i++)
        c->staff[i] = c->staff[i + 1];
    c->staffCount--;
}
// 2부 C++ — 한 줄. 당기기·크기 감소를 vector 가 처리
staffList.erase(staffList.begin() + staffCursor);

erase(begin() + idx) — "idx 번째를 지우고 뒤를 당겨라"가 한 줄이다. 크기(staffCount)도 벡터가 스스로 줄인다. 순회는 size() 로.

for (size_t i = 0; i < staffList.size(); i++)   // 1부: i < c->staffCount
    staffList[i].months++;                       // 근속 +1

1부 C·D·E 모듈에서 세 장에 걸쳐 씨름한 직원 배열 관리가, std::vector 하나로 압축된다. staffCount(현재 수)와 staffCap(용량)이라는 두 변수가 사라지고, push_back·erase·size() 세 메서드가 그 일을 대신한다. 이것이 C++ 표준 라이브러리(STL)의 힘이다 — 자주 필요한 자료구조를, 검증된 구현으로 제공한다.

1부의 복선이 여기서 회수된다. Snake(1단계)에서 vector<Snake*> 로 벡터를 처음 만났고, Tetris·Dino·Road Fighter에서 값 타입 벡터를 썼다. 그 std::vector 가, 1부의 malloc/realloc 자리에 정확히 들어앉는다. "1부의 동적 배열 = 2부의 vector" — 두 부를 잇는 가장 굵은 실이다.

개념 4 · #define → static const, enum → enum class

작지만 의미 있는 승격이 둘 있다.

밸런스 상수: #define → static const. 1부 F 모듈에서 밸런스를 #define 으로 모았다. C++에선 타입이 있는 클래스 상수로 둔다.

/* 1부 C — 매크로 (타입 없음, 전역) */
#define START_MONEY 5000
#define OFFICE_RENT 100
// 2부 C++ — 타입 있는 클래스 멤버 상수 (스코프 안에)
static const int START_MONEY   = 5000;
static const int OFFICE_RENT   = 100;
static const int SALARY_RATE   = 30;

static const int 는 int 라는 타입을 가지며 클래스 스코프 안에 산다(전역을 오염시키지 않음). 컴파일러가 타입을 확인해 주고, 디버거에서도 보인다 — 매크로의 단순 치환보다 안전하다.

열거형: enum → enum class. 1부의 enum 은 이름이 전역으로 새어 나갔다. C++의 enum class 는 스코프를 가진 열거형이다.

enum class _EVIEW   { MAIN, DEV, STAFF, STATUS, SHELF, EVENT, REPORT, ENDING };
enum class _EENDING { NONE, BANKRUPT, RICH, MASTERPIECE };   // 1부: enum Ending

enum class 의 값은 _EENDING::BANKRUPT 처럼 이름을 앞에 붙여 쓴다. 그래서 서로 다른 enum이 같은 이름(NONE)을 써도 충돌하지 않는다. 2부의 모든 씬이 _EPhase, _EDIRECT, _ESELECT 등에서 이 enum class 를 쓰는 이유다. 1부 F 모듈의 enum Ending 이, 여기서 enum class _EENDING 으로 자란다.

개념 5 · 턴제를 실시간 프레임워크에 — 커서 메뉴 + 상태머신

가장 흥미로운 이식 문제가 이것이다. 1부 타이쿤은 scanf 앞에서 멈춰 사람을 기다리는 턴제 게임이었다. 그런데 2부 프레임워크의 게임 루프는 초당 60번 멈추지 않고 돈다(0단계). scanf 로 입력을 기다리면 게임 루프가 멈춰 버린다. 어떻게 턴제를 실시간 루프 안에 담을까?

답은 블로킹을 쓰지 않는 것이다. 입력을 기다리며 멈추는 대신, 매 프레임 "지금 무슨 키가 눌렸나"를 폴링하고(0단계 개념 4), 화면(뷰) 상태를 내부 상태머신으로 관리한다. 시간은 저절로 흐르지 않고, 플레이어가 "다음 달"을 고를 때만 한 달이 진행된다.

// 인게임 세부 화면 = 상태머신 (1부의 화면=함수를, enum 뷰로)
enum class _EVIEW { MAIN, DEV, STAFF, STATUS, SHELF, EVENT, REPORT, ENDING };
_EVIEW view = _EVIEW::MAIN;

void TycoonContent::OnInGameUpdate()
{
    // ... ESC 처리 (뷰에 따라 뒤로/타이틀) ...
    switch (view) {                       // 현재 뷰에 맞는 갱신으로 분기
    case _EVIEW::MAIN:   UpdateMain();   break;
    case _EVIEW::DEV:    UpdateDev();    break;
    case _EVIEW::STAFF:  UpdateStaff();  break;
    case _EVIEW::REPORT: UpdateReport(); break;
    // ...
    }
}

메인 메뉴는 커서로 조작한다 — 1부에서 scanf("%d", &sel) 로 번호를 받던 것을, 방향키로 커서를 옮기고 Enter로 확정하는 방식으로 바꾼다(2부 Snake의 타이틀 메뉴와 같은 패턴).

void TycoonContent::UpdateMain()
{
    if (INPUT->OnKeyDown(VK_UP))   mainCursor = (mainCursor + 5) % 6;   // 커서 위 (순환)
    if (INPUT->OnKeyDown(VK_DOWN)) mainCursor = (mainCursor + 1) % 6;   // 커서 아래

    if (INPUT->OnKeyDown(VK_RETURN)) {
        switch (mainCursor) {
        case 0: view = _EVIEW::DEV;   break;   // 게임 개발 화면으로
        case 1: view = _EVIEW::STAFF; break;   // 직원 관리 화면으로
        case 4: StartMonth();         break;   // ★ 다음 달 진행 — 이때만 시간이 흐른다
        case 5: currentPhase = _EPhase::TITLE; break;
        }
    }
}

핵심은 case 4: StartMonth(); 다. 한 달은 저절로 흐르지 않는다 — 플레이어가 "다음 달 진행"을 고른 그 순간에만 StartMonth() 가 불려 한 달이 처리된다. 나머지 시간엔 커서를 움직이고 화면을 오갈 뿐, 게임 상태는 그대로다. 렌더는 매 프레임 일어나지만(60fps로 현황판을 다시 그림), 상태 변화는 입력에만 반응한다. 이것이 "실시간 루프 위의 턴제"다.

1부의 "화면 = 함수"가 2부의 "화면 = 뷰 상태"로. 1부 GDD는 타이쿤의 화면 전환을 "함수 호출 구조"(MainMenu() → DevMenu() …)로 표현했다. C++ 이식에선 이것이 _EVIEW 상태머신이 된다 — view 변수 하나가 지금 어느 화면인지 담고, OnInGameUpdate/OnInGameRender 가 그에 따라 분기한다. 1부 F 모듈의 gameOver·enum Ending, 2부 0단계의 _EPhase 로 이어진 "상태를 변수에 담는다"는 발상이, 여기서 게임 내부 화면 전환까지 확장된다.

개념 6 · 월 처리 — 같은 로직, 이제 메서드

게임의 심장인 월 처리는 1부의 8단계 로직 그대로다(GDD §2). 달라진 건 전역 함수 Func(Company *c) 들이 Company *c 없는 메서드가 되었다는 것뿐이다.

void TycoonContent::StartMonth()
{
    reportMsgs.clear();  reportMoney.clear();       // std::vector 초기화

    StepProject();                                  // (1) 개발 진행
    if (proj.active && proj.progress >= proj.need)
        DoRelease(false);                           // (2) 출시
    StepSales();                                    // (3) 판매 정산
    RollApplicants();                               // (4) 지원자
    StepEvent();                                    //     랜덤 이벤트

    if (raiseIdx >= 0) { view = _EVIEW::EVENT; return; }  // 월급 인상 → 선택 화면
    FinishMonth();
}

void TycoonContent::FinishMonth()
{
    if (month % 3 == 0) UpdateTrend(true);          // 분기마다 유행 재추첨
    int sal = TotalSalary();                        // (5) 지출
    // ... reportMoney 에 월급·임대료 추가 ...
    for (size_t i = 0; i < staffList.size(); i++)   // 근속 +1 (vector 순회)
        staffList[i].months++;
    money += net;                                   // (6) 자본금 반영 (멤버 직접)
    CheckEnding();                                  // (7) 파산/승리 판정
    month++;                                        // (8) 달력
    if (month > 12) { month = 1; year++; }
    view = _EVIEW::REPORT;                          // 결산 화면으로
}

StepProject(), DoRelease(), TotalSalary(), CheckEnding() — 1부에서 ProgressProject(c), ReleaseNow(c,...), TotalSalary(c), CheckEnding(c) 였던 것들이다. 함수 이름과 로직은 거의 그대로, Company *c 인자만 사라졌다. 그리고 c->money += net 이 money += net 으로, c->staffCount 가 staffList.size() 로 바뀌었다.

StepProject 안을 1부와 나란히 두면 대응이 선명하다.

/* 1부 C */                          // 2부 C++
void ProgressProject(Company *c) {   void TycoonContent::StepProject() {
    Project *p = &c->proj;               if (!proj.active) return;
    if (!p->active) return;              proj.progress += BASE_PROGRESS + DevSum();
    p->progress += BASE_PROGRESS         proj.quality  += DevSum();
        + DevSum(c);                     proj.fun      += CreSum() + rand() % 3;
    /* ... */                            proj.devMonths++;
}                                    }

p->(포인터)가 proj.(멤버)로, DevSum(c) 가 DevSum() 으로. 로직은 한 줄도 다르지 않다 — 오직 "회사에 어떻게 닿느냐"만 C의 포인터 전달에서 C++의 멤버 접근으로 바뀌었다.

코드 워크스루 — 이식의 지도

1부의 C 코드와 2부의 C++ 코드가 어디서 어떻게 대응되는지, 한 장의 지도로 본다.

실제 파일: Main/Content/TycoonContent.{h,cpp} (C++ 이식) ↔︎ Study/Tycoon/*.c (1부 C 원본)

1부 C (Study/Tycoon/) 2부 C++ (TycoonContent) 무엇이 바뀌었나
struct Company { … } + 전역 클래스 멤버 변수 회사 = 클래스 (개념 1)
Func(Company *c) TycoonContent::Func() c-> → 멤버 직접 (개념 1·6)
char name[20] std::string name 길이 관리 자동 (개념 2)
snprintf/strcpy "a" + str + "b" + 연결 (개념 2)
Employee *staff + malloc/realloc std::vector<Staff> push_back/erase (개념 3)
c->staffCount staffList.size() 크기 자동 (개념 3)
#define START_MONEY static const int 타입·스코프 (개념 4)
enum Ending enum class _EENDING 스코프 (개념 4)
scanf 블로킹 턴제 _EVIEW 상태머신 + 폴링 실시간 루프 위 턴제 (개념 5)
월 8단계 함수들 같은 이름 메서드 로직 동일 (개념 6)

이 표가 이 챕터의 전부다. 왼쪽(1부 C)에서 오른쪽(2부 C++)으로 가는 여덟 가지 변환 — 그것이 C→C++ 리팩터링의 실체다. 게임의 규칙(경제·평점·판매)은 왼쪽과 오른쪽이 완전히 같다. 바뀐 것은 오직 그 규칙을 담는 언어의 그릇이다.

한 가지 더 눈여겨볼 것 — 파일 구조의 변화. 1부는 14장에서 게임을 main.c/company.c/employee.c… 7개 파일로 나눴다. C++ 이식은 이 모두를 하나의 씬 클래스 TycoonContent 로 합친다. 1부에서 파일로 나눴던 모듈들(company·employee·project·market)이, 이제 한 클래스의 메서드 그룹(StepProject/RollApplicants/StepSales/StepEvent)이 된다. 프레임워크가 "게임 하나 = 씬 하나"를 요구하기 때문이다(0단계). 나눔의 축이 "파일"에서 "메서드"로 옮겨간 셈이다.

그림으로 정리 — C의 회사와 C++의 회사

📊 웹 교육문서에서는 C 구조체+포인터 전달과 C++ 클래스+메서드가 나란히 애니메이션(V-P2-B)으로 비교된다. 책에서는 아래 도식으로 대신한다.

회사에 닿는 두 가지 길

  [1부 C]  회사는 구조체, 함수는 주소를 넘겨받는다
     Company c;
        │  &c
        ▼
     ProgressProject(&c) → c->proj.progress += ...   (모든 함수에 c 를 넘김)
     TotalSalary(&c)     → c->staff[i].salary

  [2부 C++]  회사는 클래스, 메서드는 멤버를 직접 본다
     class TycoonContent {  int money;  vector<Staff> staffList;  ... }
        │  (this 는 암묵적)
        ▼
     StepProject()  → proj.progress += ...            (c 인자가 사라진다)
     TotalSalary()  → staffList[i].salary

직원 목록의 진화 (1부 세 모듈 → 2부 한 줄)

  C모듈:  char staffName[4][20]; int staffDev[4]; ... staffCount;   (병렬 배열 5개)
     │
  D모듈:  Employee staff[4]; int staffCount;                        (구조체 배열)
     │
  E모듈:  Employee *staff; malloc/realloc; staffCount, staffCap;    (동적 배열)
     │
  2부:    std::vector<Staff> staffList;                             (push_back/erase/size)
          └ 세 모듈의 고생이 STL 한 줄로 흡수된다

한 문장으로: C→C++ 이식은 게임을 새로 만드는 게 아니라, 같은 게임의 데이터를 더 안전하고 짧은 문법으로 다시 담는 일이다 — 1부에서 겪은 불편이 클래스·string·vector 로 하나씩 사라진다.

체크포인트 — 스스로 점검하기

Q1. 1부의 int TotalSalary(const Company *c) 가 C++ 이식에선 int TycoonContent::TotalSalary() const 가 된다. Company *c 인자는 왜 사라졌나?

Q2. 1부에서 직원을 해고하려면 배열을 한 칸씩 당기고 staffCount-- 를 했다. C++에선 어떤 한 줄로 대체되나?

Q3. 1부의 char name[20] + snprintf(buf, ...) 대신 C++이 쓰는 두 가지는?

Q4. 2부 프레임워크의 게임 루프는 멈추지 않는데, 턴제 타이쿤은 어떻게 "다음 달"에만 시간이 흐르게 하나?

Q5. enum Ending 을 enum class _EENDING 으로 바꾸면 좋은 점은? 값은 어떻게 쓰나?

해설 펼치기

A1. C++에선 회사 그 자체가 클래스(TycoonContent) 이고, 게임 상태가 그 클래스의 멤버 변수다. 메서드는 자기 객체(암묵적 this)의 멤버에 바로 접근하므로, "회사를 넘겨받을" 필요가 없다. 1부에서 포인터로 넘기던 회사가, C++에선 클래스라는 문법에 흡수됐다.

A2. staffList.erase(staffList.begin() + idx); 한 줄. std::vector 의 erase 가 해당 원소를 지우고 뒤를 당기고 크기를 줄이는 것을 모두 처리한다(1부 D 모듈 QuitEmployee 전체를 대체).

A3. std::string name(길이 자동 관리, 크기·널 종료 걱정 없음)과 + 연산자("* [" + name + "] 해고" 처럼 문자열을 잇기). 버퍼·snprintf·strcpy 가 모두 사라진다.

A4. 블로킹 입력(scanf)을 쓰지 않는다. 매 프레임 키를 폴링하고 _EVIEW 상태머신으로 화면만 오가되, 플레이어가 메뉴에서 "다음 달 진행"을 고른 그 순간에만 StartMonth() 를 불러 한 달을 처리한다. 그 외엔 상태가 안 변한다(렌더는 매 프레임, 상태 변화는 입력에만).

A5. enum class 는 스코프를 가져, 값을 _EENDING::BANKRUPT 처럼 이름 앞에 붙여 쓴다. 그래서 서로 다른 enum이 같은 이름(NONE)을 써도 충돌하지 않고, 이름이 전역을 오염시키지 않는다. 타입 안전성도 높다.

실습 미션 — 미니 명부를 C에서 C++로 roster.cpp ⭐

이 미션은 이 챕터의 전부를 압축한다 — 작은 C 스타일 코드를 C++로 리팩터링하며, 구조체+배열을 std::vector로, char[]를 std::string으로 바꾼다.

드릴 문법 — std::vector<struct> · std::string · push_back/erase/size · + 연결 · 범위 순회 하는 것 — C 스타일의 "직원 명부"(고정 배열 + char[] + 수동 삭제)를, 같은 동작의 C++로 다시 쓴다. 클리어 조건 — C 버전과 결과가 같으면서, 배열 크기·strcpy·수동 당기기가 모두 사라진다. 규모 — 약 45줄. 1부의 데이터가 2부의 문법으로 옮겨지는 것을 손으로 겪는다.

문제 정의

아래는 1부 스타일의 C 명부다. 고정 배열 4칸, 이름은 char[20], 삭제는 수동 당기기. 이걸 C++로 이식하라 — std::vector<Staff> + std::string 으로. 고용은 push_back, 해고는 erase, 순회는 size().

/* BEFORE — 1부 스타일 C */
#define MAX 4
char names[MAX][20];  int devs[MAX];  int count = 0;

void hire(const char* n, int d) {
    if (count >= MAX) return;
    strcpy(names[count], n);  devs[count] = d;  count++;
}
void fire(int idx) {                       /* 수동 당기기 */
    for (int i = idx; i < count - 1; i++) {
        strcpy(names[i], names[i+1]);  devs[i] = devs[i+1];
    }
    count--;
}

힌트

힌트 1 — 구조체 + vector

병렬 배열(names[][], devs[])을 하나의 구조체 + 벡터로 묶는다.

struct Staff { std::string name; int dev; };
std::vector<Staff> roster;      // count 변수도 필요 없다 — roster.size()
힌트 2 — 고용/해고 한 줄씩 
void hire(const std::string& n, int d) { roster.push_back({ n, d }); }  // 크기 제한도 자동으로 없앰
void fire(int idx) { roster.erase(roster.begin() + idx); }              // 수동 당기기 → 한 줄

정답 코드

정답 펼치기 — roster.cpp 
#include <cstdio>
#include <string>
#include <vector>

struct Staff {
    std::string name;   // char[20] → 길이 걱정 없음
    int dev;
};

std::vector<Staff> roster;   // 고정 배열 + count → vector 하나

void hire(const std::string& n, int d) {
    roster.push_back({ n, d });          // realloc·크기 확인 없이 그냥 추가
}
void fire(int idx) {
    if (idx < 0 || idx >= (int)roster.size()) return;
    roster.erase(roster.begin() + idx);  // 수동 당기기 → 한 줄
}
void print() {
    printf("-- 명부 (%zu명) --\n", roster.size());
    for (size_t i = 0; i < roster.size(); i++)   // count 대신 size()
        printf("  %zu) %-10s dev %d\n", i + 1, roster[i].name.c_str(), roster[i].dev);
    // 문자열 + 연결도 자유롭게:
    std::string msg = "총 개발력 ";
    int sum = 0;
    for (size_t i = 0; i < roster.size(); i++) sum += roster[i].dev;
    msg += std::to_string(sum);
    printf("  %s\n", msg.c_str());
}

int main() {
    hire("김개발", 7);
    hire("이서버", 5);
    hire("박퍼즈", 8);
    hire("최네트", 6);
    hire("정클라", 9);   // C 버전이면 MAX=4라 거절됐을 5번째 — vector는 그냥 늘어난다
    print();

    fire(1);             // 이서버 해고 (뒤가 한 줄로 당겨짐)
    print();
    return 0;
}

빌드: cl /EHsc roster.cpp (MSVC) 또는 g++ roster.cpp -o roster

함정/체크포인트

  • count 변수가 사라진다. 현재 인원은 roster.size(). 별도 카운터를 두면 벡터와 어긋날 위험이 생긴다 — 크기는 벡터에게 맡긴다.
  • 5번째 고용이 그냥 된다. C 버전(MAX=4)이면 거절됐을 다섯 번째 직원을, 벡터는 알아서 공간을 늘려 받는다 — 1부 E 모듈 realloc 을 벡터가 대신하는 것.
  • roster.push_back({ n, d }) 의 { n, d } 는 Staff 를 즉석에서 만드는 C++ 초기화. printf 에 std::string 을 넣을 땐 .c_str() 로 C 문자열을 꺼낸다(C 함수와의 접점).

확장 과제

  • fire 를 이름으로 찾아 지우게 바꿔 보자(for 순회 + erase) — 1부라면 인덱스를 먼저 찾아야 했다.
  • Staff 에 cre, salary 를 추가해도 배열 선언·삭제 코드가 안 늘어난다(병렬 배열이었다면 다섯 곳을 고쳐야 했다).

이 챕터를 마치며

얻은 것 그 뿌리
구조체+전역 → 클래스+메서드 1부 D 포인터 · 2부 0단계 클래스
char[] → std::string 1부 C 문자열
malloc/realloc → std::vector 1부 E 동적 메모리 · 2부 Snake~ 벡터
턴제를 상태머신으로 1부 상태 플래그 · 2부 _EPhase

이 다리를 건너며 우리는 1부와 2부가 한 몸임을 확인했다 — 1부에서 C로 만든 게임의 데이터가, 2부의 C++ 문법으로 더 짧고 안전하게 다시 담겼다. C에서 C++로 간다는 것은, 새 언어를 배우는 게 아니라 이미 아는 것을 더 나은 도구로 다시 짓는 것이다. 이제 이 데이터·경제 게임의 막간을 지나, 다시 실시간 엔지니어링으로 — 다음 5단계 Galaga에서 오브젝트 풀로 성능을 다룬다.
