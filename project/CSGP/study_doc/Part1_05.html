E 모듈 · 11–12장 — 동적 메모리와 랜덤: 회사가 자라고, 시장이 살아난다

모듈 E · 11–12장 · 게임 버전 v0.5 → v0.7 · 예상 학습 시간 4–6시간 처음 배우는 기술 — malloc calloc free realloc sizeof 힙과 스택 메모리 누수 NULL 검사(실전) rand srand time 범위 난수 확률 분기

D 모듈에서 우리는 구조체와 포인터로 게임의 뼈대를 완성했다. 하지만 두 가지가 아직 게임의 발목을 잡는다. 직원은 4명이 한계이고(고정 배열), 매달 결과가 완전히 예측 가능하다(랜덤이 없다). 이 모듈에서 그 둘을 푼다. 동적 메모리로 직원 수 제한을 없애 회사가 원하는 만큼 커지게 하고, 랜덤으로 평점·지원자·유행·이벤트에 변수를 넣어 시장이 살아 숨 쉬게 한다.

이 모듈의 학습 목표

이 모듈을 마치면 다음을 할 수 있다.

  1. 스택과 힙의 차이를 설명하고, 왜 "실행 중에 크기가 변하는" 배열은 힙이 필요한지 안다.
  2. malloc/free 로 힙 메모리를 잡고 돌려주며, sizeof 로 필요한 바이트를 계산할 수 있다.
  3. realloc 로 배열을 키우고, 반환값을 임시 포인터로 받아 안전하게 교체할 수 있다.
  4. 메모리 누수가 무엇인지 알고, NULL 검사와 free 로 이를 막을 수 있다.
  5. srand(time(NULL)) 로 난수를 초기화하고, rand() % n · min + rand() % k 로 범위 난수를 만들 수 있다.
  6. rand() % 100 < 확률 패턴으로 확률 분기를 짜서 랜덤 이벤트를 구현할 수 있다.

왜 지금, 동적 메모리와 랜덤인가

C 모듈에서 우리는 직원을 4칸 배열에 담았고, 5번째를 뽑으려다 "사무실이 좁습니다"라며 거절당했다. 그 벽이 아직 그대로다. 배열 크기는 컴파일 시점에 못 박히니, 회사가 아무리 성공해도 직원은 4명뿐이다. 실행 중에 필요한 만큼 커지는 저장 공간 — 그것이 11장 동적 메모리다. D 모듈 말미에 우리는 이미 Employee *staff 를 포인터로 선언해 두었다. 그 포인터가 이제 진짜 힘을 발휘한다.

또 하나. 지금까지 우리 게임은 완전히 예측 가능하다. 같은 게임을 만들면 평점이 항상 똑같고, 매달 아무 일도 안 일어난다. 계산기처럼 정확하지만, 그래서 재미가 없다. 현실의 게임 시장은 흥행이 도박이고, 정부 지원금이 뚝 떨어지기도 하며, 유행이 돌변한다. 이 예측 불가능성을 코드로 만드는 것이 12장 랜덤이다. GDD의 표현을 빌리면 — "12장에서 게임이 갑자기 살아난다."

이 모듈에서 만드는 것. 직원을 원하는 만큼 고용할 수 있고(v0.6, realloc 로 배열이 자란다), 게임 평점이 매번 조금씩 흔들리며, 매달 30% 확률로 랜덤 이벤트(정부 지원금·게임쇼 호황·버그 사태·월급 인상 요구·악성 리뷰·스카우트)가 터지는 — GDD 마일스톤 v0.7 "살아있는 시장" 이다. 이 모듈을 지나면 게임은 두 판이 결코 같지 않게 된다.

11장 · 동적 메모리 — 필요한 만큼 자란다

11.1 스택과 힙 — 메모리의 두 방

지금까지 우리가 만든 변수·배열은 모두 스택(stack) 에 살았다. 스택은 함수가 실행되는 동안 지역 변수가 자동으로 잡혔다 사라지는 공간이다. 빠르고 편하지만 두 가지 제약이 있다 — 크기가 컴파일 시점에 정해져야 하고, 함수가 끝나면 사라진다.

int staff[4];    /* 스택 — 크기 4로 고정, main 안에서만 산다 */

힙(heap) 은 다르다. 프로그램이 실행 중에 "이만큼 주세요"라고 요청해서 얻고, "다 썼습니다"라고 돌려줄 때까지 유지되는 공간이다. 크기를 실행 중에 정할 수 있고, 함수가 끝나도 살아 있다. 대신 잡은 사람이 직접 돌려줘야 한다(자동으로 안 사라진다). "실행 중에 크기가 변하는 직원 배열"에는 힙이 필요하다.

스택 (지금까지) 힙 (이번 장)
크기 결정 컴파일 시점(고정) 실행 중(가변)
수명 함수와 함께 free 할 때까지
정리 자동 직접 free
얻는 법 그냥 선언 malloc/realloc

11.2 malloc 과 free — 빌리고 돌려주기

malloc(바이트 수) 은 힙에서 그만큼의 메모리를 잡아 그 주소를 돌려준다. 다 쓰면 free(주소) 로 반납한다. 필요한 바이트 수는 sizeof 로 계산한다.

#include <stdlib.h>       /* malloc, free, rand 등 */

int *arr = (int *)malloc(sizeof(int) * 4);   /* int 4칸 = 16바이트를 힙에 */
if (arr == NULL) {                            /* 할당 실패 검사 (11.4) */
    printf("메모리 부족\n");
    return 1;
}
arr[0] = 7;                                   /* 배열처럼 그대로 쓴다 */
arr[1] = 5;
/* ... 사용 ... */
free(arr);                                    /* 다 쓰면 반드시 반납 */
arr = NULL;                                   /* 반납 후 포인터를 NULL 로 (안전) */

malloc 이 돌려준 것은 힙에 잡힌 배열의 첫 주소다. D 모듈에서 배운 대로, 배열 이름처럼 동작하는 포인터이므로 arr[0], arr[1] 로 배열과 똑같이 쓴다. 그래서 Employee *staff 하나로 "직원 배열"을 가리킬 수 있는 것이다.

sizeof 로 계산하는 습관. malloc(16) 이라고 바이트를 직접 쓰지 말고 malloc(sizeof(int) * 4) 라고 쓰자. int 크기가 환경마다 다를 수 있고, Employee 처럼 구조체는 크기를 눈으로 못 세기 때문이다. sizeof(Employee) * cap 은 "Employee 몇 개분"이라는 의도를 그대로 드러낸다.

11.3 realloc — 배열을 키우다

동적 메모리의 진짜 힘은 realloc 에 있다. 이미 잡은 배열을 더 큰 배열로 늘려 준다(기존 내용은 그대로 유지된다). 직원이 꽉 차면 배열을 2배로 키우는 것 — 이것이 "무제한 고용"의 비밀이다.

/* staff 배열이 꽉 찼을 때 2배로 늘린다 */
Employee *tmp = (Employee *)realloc(staff, sizeof(Employee) * cap * 2);
if (tmp == NULL) {
    /* 실패: 기존 staff 는 그대로 살아 있다. 확장만 포기 */
    printf("확장 실패\n");
} else {
    staff = tmp;      /* 성공: 더 큰 배열의 주소로 교체 */
    cap  *= 2;
}

여기 결정적인 안전 규칙이 있다 — realloc 의 결과를 원래 포인터에 바로 대입하지 말고, 임시 포인터(tmp)로 먼저 받는다. 왜냐하면 realloc 이 실패하면 NULL 을 돌려주는데, 그걸 staff = realloc(staff, ...) 로 바로 받으면 원래 배열의 주소를 잃어버려(NULL로 덮여) 이미 있던 직원 데이터가 통째로 새어 나간다. tmp 로 받아 성공을 확인한 뒤에 교체하는 것이 철칙이다.

이 코드가 바로 D 모듈 마지막에 "11장 미리보기"로 예고했던 그것이다. 그때 "다 이해 못 해도 좋다"고 했던 코드를, 이제 우리는 완전히 읽을 수 있다.

11.4 메모리 누수와 NULL 검사

힙은 직접 돌려줘야 한다고 했다. malloc 으로 잡고 free 를 안 하면, 그 메모리는 프로그램이 끝날 때까지 붙잡힌 채 아무도 못 쓴다 — 이것이 메모리 누수(memory leak) 다. 작은 프로그램에선 티가 안 나지만, 오래 도는 프로그램(서버·게임)에선 누수가 쌓여 결국 메모리가 바닥난다.

Employee *staff = malloc(sizeof(Employee) * 4);
/* ... 게임 내내 사용 ... */
/* 프로그램 종료 전 반드시: */
free(staff);        /* 안 하면 누수 */

그리고 malloc/realloc 은 실패할 수 있다(메모리가 부족하면 NULL 반환). D 모듈에서 배운 NULL 검사가 여기서 실전이 된다. 반환값을 NULL 과 비교해, 실패하면 안전하게 대처한다.

staff = malloc(sizeof(Employee) * 4);
if (staff == NULL) {          /* 잡기 실패 */
    printf("메모리 할당 실패\n");
    exit(1);                  /* 더 진행할 수 없으니 종료 */
}

세 가지를 한 세트로 기억하자 — ① sizeof 로 잡고, ② NULL 인지 확인하고, ③ 다 쓰면 free. 힙을 쓰는 모든 코드가 이 리듬을 탄다.

11.5 게임에 적용 — 무제한 고용

이제 직원 배열을 힙으로 옮긴다. C 모듈의 병렬 배열도, D 모듈의 고정 배열 감각도 넘어, Employee *staff 를 malloc 으로 잡고 꽉 차면 realloc 로 키운다. 회사가 성공하는 만큼 직원 배열이 자란다.

회사를 시작할 때 초기 용량(4명분)을 잡는다.

void InitCompany(Company *c, const char *name)
{
    /* ... 이름·자본금 초기화 ... */
    c->staff = (Employee *)malloc(sizeof(Employee) * START_STAFF_CAP);  /* 4명분 */
    if (c->staff == NULL) {
        printf("메모리 할당 실패\n");
        exit(1);
    }
    c->staffCap   = START_STAFF_CAP;   /* 현재 용량 = 4 */
    c->staffCount = 0;                 /* 현재 인원 = 0 */
}

고용할 때, 인원이 용량에 도달했으면 용량을 2배로 늘린 뒤 새 직원을 넣는다.

void Hire(Company *c, const Employee *e)
{
    if (c->staffCount == c->staffCap) {           /* 꽉 찼다 → 확장 */
        int newCap = c->staffCap * 2;
        Employee *tmp = (Employee *)realloc(c->staff, sizeof(Employee) * newCap);
        if (tmp == NULL) {                        /* 확장 실패: 기존 배열 유지 */
            printf("  ! 메모리가 부족해 사무실을 확장할 수 없습니다.\n");
            return;
        }
        c->staff   = tmp;                         /* 더 큰 배열로 교체 */
        c->staffCap = newCap;
        printf("  * 사무실을 확장했습니다! (최대 %d명)\n", newCap);
    }
    c->staff[c->staffCount++] = *e;               /* 새 직원을 넣고 인원 +1 */
    printf("  * [%s] 을(를) 고용했습니다.\n", e->name);
}

그리고 게임이 끝날 때 반드시 반납한다.

void FreeCompany(Company *c)
{
    free(c->staff);       /* 힙에 잡은 직원 배열 반납 (누수 방지) */
    c->staff = NULL;
    c->staffCount = c->staffCap = 0;
}

이제 5번째 직원을 뽑아도 거절당하지 않는다. 4명이 차면 배열이 8칸으로, 8명이 차면 16칸으로 — 회사가 커지는 만큼 배열이 따라 자란다. C 모듈에서 부딪혔던 "사무실이 좁습니다"의 벽이 마침내 무너졌다.

고정 배열 → 동적 배열, 그리고 그 너머. 커리큘럼의 큰 줄기 하나가 여기서 매듭지어진다. 7장 고정 배열(한계) → 11장 동적 배열(해방). 그런데 2부에서는 이 방향이 다시 뒤집힌다. Galaga(5단계)에서는 "총알이 초당 수십 개씩 malloc/free 되니 오히려 느리다"는 문제를 만나, 미리 잡아 둔 고정 배열을 재활용하는 오브젝트 풀로 되돌아간다. 동적이 항상 옳은 게 아니라, 상황이 도구를 정한다 — 이 감각이 초급과 중급을 가른다.

12장 · 랜덤과 시간 — 게임이 살아나다

직원 문제를 풀었으니, 이제 게임에 예측 불가능성을 넣는다. 12장 전까지 우리 게임은 계산기처럼 정확했다 — 같은 입력이면 항상 같은 결과. 랜덤이 그 정확함을 깨고, 게임을 도박과 놀라움이 있는 살아있는 세계로 만든다.

12.1 rand, srand, time — 난수의 삼총사

C의 난수는 stdlib.h 의 rand() 로 얻는다. rand() 는 0부터 아주 큰 수(RAND_MAX) 사이의 정수를 돌려준다.

#include <stdlib.h>       /* rand, srand */
#include <time.h>         /* time */

int x = rand();           /* 0 ~ RAND_MAX 사이의 '무작위' 정수 */

그런데 rand() 는 사실 가짜 난수(의사난수) 다. 정해진 규칙으로 숫자를 만들어 내는데, 그 출발점을 씨앗(seed) 이라 한다. 씨앗이 같으면 난수 순서도 똑같다 — 그래서 씨앗을 안 정하면 프로그램을 켤 때마다 똑같은 "무작위" 가 나온다. 이걸 막으려고, 프로그램 시작 시 srand 로 씨앗을 심는다. 매번 다른 씨앗을 얻으려면 현재 시각(time(NULL))을 쓴다.

int main(void)
{
    srand((unsigned)time(NULL));   /* ★ 시각을 씨앗으로 — 실행마다 다른 난수 */
    /* ... 게임 ... */
}

⚠️ srand 는 main 에서 딱 한 번. 12장의 단골 실수가 "난수를 뽑을 때마다 srand 를 부르는 것"이다. 그러면 time(NULL) 이 1초 안에는 같은 값이라, 되레 같은 수만 나온다. srand 는 프로그램 전체에서 main 시작 시 한 번만 호출한다. 씨앗을 심는 건 한 번, 열매(rand)를 따는 건 여러 번이다.

12.2 범위 난수 — % 로 구간 자르기

rand() 가 주는 큰 수를 원하는 범위로 접을 땐 나머지 연산자 %(A 모듈!)를 쓴다.

rand() % 6            /* 0 ~ 5  (6가지) */
1 + rand() % 6        /* 1 ~ 6  (주사위) */
rand() % 100          /* 0 ~ 99 (백분율 확률용) */
1 + rand() % 10       /* 1 ~ 10 (직원 능력치!) */

공식은 간단하다 — min + rand() % (개수). rand() % n 이 0 ~ n-1 을 주니, 시작값 min 을 더해 원하는 구간으로 옮긴다. 우리 게임의 지원자 능력치가 정확히 이 패턴이다.

/* 랜덤 지원자 한 명 생성 (C 모듈에선 직접 입력받던 것을 이제 난수로) */
void RandomEmployee(Employee *e)
{
    e->dev    = 1 + rand() % 10;                  /* 개발력 1~10 */
    e->cre    = 1 + rand() % 10;                  /* 창의력 1~10 */
    e->salary = (e->dev + e->cre) * SALARY_RATE;  /* 월급은 능력에 비례 */
    e->months = 0;
}

C 모듈에서 지원자의 능력치를 사람이 직접 입력했다면, 이제 매달 능력치가 다른 지원자가 저절로 나타난다. 이름도 이름 후보 배열에서 NAME_POOL[rand() % 개수] 로 무작위로 고른다.

12.3 확률 분기 — rand() % 100 < 확률

"30% 확률로 무슨 일이 일어난다"는 게임의 단골 표현이다. rand() % 100(0~99)을 뽑아 목표 확률과 비교하면 된다.

if (rand() % 100 < 30)        /* 0~99 중 0~29 → 30% 확률 */
    printf("이벤트 발생!\n");

여러 결과에 서로 다른 확률을 주려면, 구간을 나눠 검사한다. 우리 게임의 랜덤 이벤트가 이 방식이다.

int roll = rand() % 100;       /* 0 ~ 99 */
if      (roll < 15) { /* 0~14  : 15% — 정부 지원금 */ }
else if (roll < 30) { /* 15~29 : 15% — 게임쇼 호황 */ }
else if (roll < 50) { /* 30~49 : 20% — 버그 사태 */ }
else if (roll < 70) { /* 50~69 : 20% — 월급 인상 요구 */ }
else if (roll < 85) { /* 70~84 : 15% — 악성 리뷰 */ }
else                { /* 85~99 : 15% — 스카우트 제의 */ }

roll 하나로 100칸의 눈금을 만들고, 그 눈금을 확률 비중대로 나눠 가진 셈이다. 15+15+20+20+15+15 = 100. 이 구간 나누기가 확률 설계의 기본이다.

12.4 게임에 적용 — 살아나는 시장

랜덤을 세 곳에 넣어 게임을 살린다.

① 평점에 랜덤 항. D 모듈에선 평점이 품질·재미로만 정해져 같은 게임은 늘 같은 점수였다. 이제 -0.5 ~ +1.0 의 랜덤 항을 더해, 같은 게임도 운에 따라 흥행이 갈린다.

float r = base / 10.0f;                 /* 품질·재미 기반 점수 (D 모듈) */
if (p->genre == c->trend)  r += 0.5f;   /* 유행 장르 일치 보너스 */
r += (rand() % 16 - 5) / 10.0f;         /* ★ 랜덤 항: -0.5 ~ +1.0 (12장에서 추가) */
if (r < 0.0f)  r = 0.0f;                /* 0~10 범위로 자르기 */
if (r > 10.0f) r = 10.0f;

rand() % 16 은 0~15, 거기서 5를 빼면 −5~+10, 10으로 나누면 −0.5 ~ +1.0. 평균적으로 살짝 플러스(운이 좋으면 대박)인 이 랜덤 항이, 출시 때마다 손에 땀을 쥐게 한다.

② 유행 장르. 분기마다(3·6·9·12월) 유행 장르를 다시 뽑는다. 유행에 맞춘 게임은 판매가 1.5배가 되니, 유행은 전략의 축이 된다.

void UpdateTrend(Company *c)
{
    c->trend = (Genre)(rand() % GENRE_COUNT);   /* 5개 장르 중 무작위 */
}

③ 랜덤 이벤트. 매달 30% 확률로 사건 하나가 터진다. 12.3의 구간 분기로 6종을 나눈다.

void MonthlyEvent(Company *c)
{
    if (rand() % 100 >= EVENT_CHANCE)     /* 70%는 아무 일 없음 */
        return;

    int roll = rand() % 100;
    if (roll < 15) {                       /* 정부 지원금 +500 */
        printf("\n  [이벤트] 정부 지원금 500만원!\n");
        Report("이벤트: 정부 지원금", 500);
    } else if (roll < 30) {                /* 게임쇼 호황: 이번 달 판매 2배 */
        Report("이벤트: 게임쇼 호황", s_monthRevenue);
    } else if (roll < 50) {                /* 버그 사태: 개발 중 품질 -20 */
        if (c->proj.active) c->proj.quality -= 20;
    }
    /* ... 월급 인상 요구 / 악성 리뷰 / 스카우트 ... */
}

이 세 가지가 들어가는 순간, 우리 게임은 두 판이 결코 같지 않게 된다. 좋은 게임을 만들어도 랜덤 항이 나쁘면 평점이 아쉽고, 파산 직전 정부 지원금이 회사를 살리기도 한다. 예측 불가능성이 곧 재미다.

랜덤은 두 얼굴을 가진다. 랜덤은 재미를 주지만, 게임을 디버깅하기 어렵게 만들기도 한다. "가끔만 재현되는 버그"의 대부분이 랜덤 때문이다. 그래서 실제 게임의 main 은 명령줄 인자로 씨앗을 고정할 수 있게 해 뒀다(srand(atoi(argv[1]))) — 같은 씨앗을 주면 같은 전개가 재현되어 디버깅이 된다. 이 "재현 가능한 랜덤"은 실무의 중요한 기법이다.

코드 워크스루 — 실제 소스에서 보는 힙과 난수

우리가 배운 동적 메모리와 랜덤이, Study/Tycoon/ 완성본에서 어떻게 맞물리는지 보자.

실제 파일: Study/Tycoon/company.c(할당·해제) · Study/Tycoon/employee.c(확장 고용) · Study/Tycoon/market.c(랜덤 이벤트) · Study/Tycoon/project.c(평점)

① 할당과 해제 (company.c) — 회사 생성 시 malloc, 종료 시 free. 11.5에서 본 그대로다. NULL 검사로 실패에 대비하는 것도 빠지지 않는다.

/* ... 게임 내내 사용 ... */
void FreeCompany(Company *c) { free(c->staff); c->staff = NULL; }

if (c->staffCount == c->staffCap) {
    Employee *tmp = (Employee *)realloc(c->staff, sizeof(Employee) * c->staffCap * 2);
    if (tmp == NULL) { printf("  ! 메모리가 부족해...\n"); return; }
    c->staff = tmp;
    c->staffCap *= 2;
}
c->staff[c->staffCount++] = *e;

for (tries = 0; tries < 10; tries++) {

    if (roll < 30) { if (s_monthRevenue <= 0) continue; /* 판매 없으면 호황 무의미 */ ... }
    if (roll < 50) { if (!c->proj.active) continue;     /* 개발 중이 없으면 버그 무의미 */ ... }

}

④ 평점의 랜덤 항 (project.c · CalcRating) — 12.4의 그 한 줄. 품질·재미로 만든 기본 점수에 유행 보너스와 랜덤 항을 더하고, 0~10으로 자른다.

r = base / 10.0f;
if (p->genre == c->trend) r += 0.5f;
r += (rand() % 16 - 5) / 10.0f;        /* -0.5 ~ +1.0 */
if (r < 0.0f) r = 0.0f;
if (r > 10.0f) r = 10.0f;

이 네 조각을 관통하는 사실 — 동적 메모리는 게임의 "크기"를, 랜덤은 게임의 "표정"을 결정한다. 회사는 힙 위에서 원하는 만큼 커지고, 시장은 난수로 매번 다른 얼굴을 보인다.

그림으로 정리 — 힙 배열의 성장과 확률의 눈금

📊 웹 교육문서에서는 realloc 로 배열이 2배 커지는 과정과, 확률 분기의 눈금이 애니메이션(V-P1-E)으로 재생된다. 책에서는 아래 도식으로 대신한다.

동적 배열의 성장 — 꽉 차면 2배로

  staffCap 4, 꽉 참:  [김][이][박][최]
                          │ Hire() → realloc(×2)
                          ▼
  staffCap 8:         [김][이][박][최][정][ ][ ][ ]   ← 기존 4명 유지 + 새 직원
                                        ▲
                           realloc 은 옛 내용을 그대로 옮겨 준다
   ⚠ tmp 로 받고 NULL 확인 후 교체! staff=realloc(staff,..) 로 바로 받으면
     실패 시 옛 주소를 잃어 4명이 통째로 샌다(메모리 누수).

확률 분기 — 100칸 눈금을 비중대로 나누기

  roll = rand() % 100    (0 ─────────────────────────────── 99)
  ┌──────┬──────┬────────┬────────┬──────┬──────┐
  │ 0-14 │15-29 │ 30-49  │ 50-69  │70-84 │85-99 │
  │ 지원금│ 호황 │ 버그   │ 월급   │ 리뷰 │스카우트│
  │ 15%  │ 15%  │  20%   │  20%   │ 15%  │ 15%  │
  └──────┴──────┴────────┴────────┴──────┴──────┘
   roll 이 어느 칸에 떨어지느냐로 이벤트가 정해진다. 합 = 100.

한 문장으로: 동적 메모리는 "얼마나 담을지"를 실행 중에 정하게 하고, 랜덤은 "무엇이 일어날지"를 매번 다르게 정한다.

체크포인트 — 스스로 점검하기

Q1. 스택 배열(int a[4];)과 힙 배열(malloc(sizeof(int)*4))의 결정적 차이 두 가지는?

Q2. staff = realloc(staff, newSize); 처럼 원래 포인터에 바로 대입하면 왜 위험한가?

Q3. malloc 으로 잡은 메모리를 free 하지 않으면 무슨 일이 생기나? 그 현상의 이름은?

Q4. 프로그램을 실행할 때마다 "무작위"가 똑같이 나온다. 무엇을 빠뜨렸나?

Q5. 1~6 주사위를 만드는 식은? 그리고 "25% 확률로 참"인 조건식은?

해설 펼치기

A1. ① 크기 결정 시점 — 스택은 컴파일 시점에 고정, 힙은 실행 중에 정할 수 있고 realloc 로 바꿀 수 있다. ② 수명과 정리 — 스택은 함수가 끝나면 자동으로 사라지고, 힙은 free 할 때까지 살아 있으며 직접 반납해야 한다.

A2. realloc 이 실패하면 NULL 을 반환하는데, staff = realloc(staff, ...) 로 바로 받으면 staff 가 NULL 로 덮여 원래 배열의 주소를 잃는다. 그 배열은 아무도 못 가리키게 되어 통째로 누수된다. tmp 로 먼저 받아 NULL 이 아님을 확인한 뒤 교체해야 한다.

A3. 그 메모리는 프로그램이 끝날 때까지 반납되지 않고 붙잡힌 채 아무도 못 쓴다. 이런 낭비가 쌓이는 현상을 메모리 누수(memory leak) 라 한다. 오래 도는 프로그램에선 결국 메모리가 바닥난다.

A4. srand 로 씨앗을 심는 것을 빠뜨렸다. rand 는 씨앗이 같으면 같은 순서를 내므로, main 시작 시 srand((unsigned)time(NULL)); 로 매번 다른 씨앗을 심어야 한다. (단, srand 는 한 번만.)

A5. 주사위: 1 + rand() % 6(rand()%6 이 0~5, +1 해서 1~6). 25% 확률: rand() % 100 < 25(또는 rand() % 4 == 0).

실습 미션 — 뽑기(가챠) 시뮬레이터 gacha.c ⭐

본편(타이쿤)이 "한 게임을 계속 키우는 세로 축"이라면, 실습 미션은 이 모듈의 새 문법만 콕 집어 연습하는 독립 미니게임(가로 축) 이다. 먼저 직접 짜 보고, 막히면 힌트를, 그래도 막히면 정답을 펼쳐 대조하자.

드릴 문법 — rand 확률 분기 · srand(time) · malloc/realloc/free · NULL 검사 · 동적 배열 성장 만드는 것 — 뽑을 때마다 확률로 등급(R/SR/SSR)이 나오고, 뽑은 결과가 동적 배열 인벤토리에 쌓이는 뽑기 게임. 클리어 조건 — 원하는 만큼 뽑고, 인벤토리를 힙 배열로 관리하며(꽉 차면 2배 확장), 끝에 등급별 통계를 낸 뒤 free. 규모 — 약 55줄, 파일 1개. 이 모듈의 두 기둥(동적 메모리 + 랜덤)이 한 게임에서 만난다.

문제 정의

이 게임은 E 모듈의 두 기술을 정확히 반씩 쓴다. 랜덤으로 매 뽑기의 등급을 정하고(확률 분기), 동적 메모리로 뽑은 결과가 몇 개가 될지 모르는 인벤토리를 관리한다(꽉 차면 realloc). 뽑기 횟수를 미리 알 수 없으니 고정 배열로는 안 되고, 힙 배열이 자라야 한다 — 11장이 필요한 이유가 게임 규칙에 그대로 박혀 있다.

목표 / 클리어 조건

  1. srand(time(NULL)) 로 난수를 초기화한다(한 번만!).
  2. 등급을 확률로 뽑는 함수: SSR 3% / SR 17% / R 80%.
  3. 인벤토리를 malloc 으로 4칸 잡고, 뽑을 때마다 결과를 저장. 꽉 차면 realloc 로 2배.
  4. 사용자가 "그만"을 고를 때까지 반복해서 뽑는다.
  5. 끝에 총 뽑기 수와 등급별 개수를 출력하고 free.

정답 코드

정답 펼치기 — gacha.c 
#define _CRT_SECURE_NO_WARNINGS
#include <stdio.h>
#include <stdlib.h>
#include <time.h>

/* 등급 뽑기: 3=SSR(3%), 2=SR(17%), 1=R(80%) */
int PullOne(void)
{
    int roll = rand() % 100;
    if (roll < 3)  return 3;
    if (roll < 20) return 2;
    return 1;
}

const char *GradeName(int g)
{
    return (g == 3) ? "SSR" : (g == 2) ? "SR" : "R";
}

int main(void)
{
    int *inv, cap = 4, count = 0, i, cmd;
    int cnt[4] = { 0 };

    srand((unsigned)time(NULL));          /* 난수 초기화 — main 에서 한 번만 */

    inv = (int *)malloc(sizeof(int) * cap);   /* 인벤토리 힙 배열 */
    if (inv == NULL) { printf("메모리 할당 실패\n"); return 1; }

    printf("=== 뽑기 시뮬레이터 (SSR 3%% / SR 17%% / R 80%%) ===\n");
    for (;;) {
        printf("[%d회] 뽑기 [1] 한 번   [0] 그만 : ", count);
        scanf("%d", &cmd);
        if (cmd == 0) break;

        int g = PullOne();

        if (count == cap) {                        /* 꽉 차면 2배 확장 */
            int *tmp = (int *)realloc(inv, sizeof(int) * cap * 2);
            if (tmp == NULL) { printf("메모리 부족, 종료\n"); break; }
            inv = tmp;
            cap *= 2;
        }
        inv[count++] = g;                          /* 결과 저장 */

        printf("  >> [%s]%s\n", GradeName(g), g == 3 ? " ★대박!" : "");
    }

    /* 통계 (배열 순회) */
    for (i = 0; i < count; i++)
        cnt[inv[i]]++;

    printf("\n== 결과 : 총 %d회 (배열 용량 %d) ==\n", count, cap);
    printf("  SSR %d개 / SR %d개 / R %d개\n", cnt[3], cnt[2], cnt[1]);

    free(inv);                             /* 반드시 반납 (누수 방지) */
    return 0;
}

빌드: cl gacha.c (MSVC) 또는 gcc gacha.c -o gacha

함정/체크포인트

  • srand 를 뽑기 루프 안에 넣으면(흔한 실수), 1초 안의 뽑기가 같은 씨앗이라 결과가 다양하지 않다. main 시작 시 딱 한 번이어야 한다.
  • realloc 을 inv = realloc(inv, ...) 로 바로 받으면 실패 시 인벤토리를 통째로 잃는다. tmp 로 받아 확인 후 교체하는 이유를 코드에서 확인하자.
  • free(inv) 를 빼먹으면 메모리 누수다. 짧은 프로그램이라 티는 안 나지만, "잡았으면 반드시 반납"의 습관을 여기서 들이자.
  • cnt[4] 를 {0} 으로 초기화하지 않으면 쓰레기 값에 더해져 통계가 엉킨다(C 모듈 배열 초기화).

확장 과제

  • 천장(pity) 시스템: SSR 없이 20회를 넘기면 21회째 SSR을 확정으로. 카운터 하나로 구현.
  • 등급별 확률을 #define 상수로 빼서, 15장의 밸런싱처럼 한곳에서 조절.
