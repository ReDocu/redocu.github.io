01

프로젝트 개요

플레이어 기체를 조작해 아래에서 위로 스크롤되는 맵을 비행하며 몬스터를 격추하는 게임. 팀원들이 인터넷에서 직접 가져온 이미지를 알파 처리해서 게임에 넣을 수 있도록 리소스 파이프라인을 코드로 만들었고, 초기 구조를 먼저 잡은 뒤 기능을 하나씩 더해가는 애자일 방식으로 진행했다.

항목내용
목표파이썬 문법(자료구조·함수·클래스)을 실제 게임 제작에 적용
담당 역할초기 게임 구조 설계, 리소스 관리 코드, 오브젝트 풀링·텍스트 애니메이션 구현
핵심 기능플레이어 이동·발사, 몬스터 스폰, 충돌 판정, 점수·UI, 텍스트 연출
협업 방식뼈대 코드를 먼저 공유하고, 팀원이 각자 기능·리소스를 붙이는 방식
02

진행 과정

P1구조 설계

게임 루프·씬 구조를 먼저 정의하고 팀원이 붙을 수 있는 뼈대 마련

P2리소스 파이프라인

이미지 수집 → 알파 처리 → 딕셔너리 등록까지의 흐름을 코드화

P3기능 구현

이동·발사·충돌·스폰 등 핵심 게임 플레이를 반복적으로 추가

P4폴리싱

텍스트 애니메이션·연출을 더하고 밸런스와 버그 정리

03

핵심 학습 개념

게임 루프 & 이벤트 처리

  • 입력 → 갱신 → 렌더로 이어지는 프레임 단위 루프 구조
  • pygame의 이벤트 큐로 키 입력·종료 이벤트 처리
  • Clock.tick()으로 프레임 속도 고정

딕셔너리 기반 리소스 관리

  • 이미지·사운드를 이름(key)으로 꺼내 쓰는 리소스 딕셔너리
  • 팀원이 파일만 추가하면 코드 수정 없이 리소스가 등록되는 구조
  • convert_alpha()로 배경 투명 처리 일괄 적용

오브젝트 풀링

  • 총알·몬스터를 매번 생성/삭제하지 않고 풀에서 재사용
  • 화면 밖으로 나간 객체는 비활성화 후 풀로 반환
  • 객체 수가 많아져도 프레임이 안정적으로 유지됨을 확인

텍스트 애니메이션

  • 점수·스테이지 안내 텍스트가 커지고 사라지는 연출
  • 경과 시간 기반 보간으로 크기·투명도 계산
  • 연출 하나로 게임의 타격감이 크게 달라지는 것을 경험

구현 노트

프로젝트에서 배운 구조를 요약한 예시 코드. 실제 코드의 핵심 아이디어만 남긴 형태다.

In [1]딕셔너리 리소스 로더 — 파일명을 key로 등록해 어디서든 꺼내 쓴다
import os, pygame

images = {}                          # {"player": Surface, "monster_a": Surface, ...}

def load_images(folder="assets"):
    for name in os.listdir(folder):
        if name.endswith(".png"):
            key = name.rsplit(".", 1)[0]
            surf = pygame.image.load(os.path.join(folder, name))
            images[key] = surf.convert_alpha()   # 알파 처리 일괄 적용
In [2]오브젝트 풀링 — 총알을 재사용해 생성 비용 제거
class BulletPool:
    def __init__(self, size=64):
        self.pool = [Bullet(active=False) for _ in range(size)]

    def fire(self, x, y):
        for b in self.pool:
            if not b.active:          # 쉬고 있는 총알을 찾아 재사용
                b.reset(x, y)
                return b
In [3]메인 루프 — 입력 → 갱신 → 렌더의 반복
while running:
    for event in pygame.event.get():      # ① 입력
        if event.type == pygame.QUIT: running = False

    player.update(); pool.update(); spawner.update()   # ② 갱신

    screen.blit(images["background"], (0, scroll_y))  # ③ 렌더
    all_sprites.draw(screen)

05

배운 점 & 회고

  • 문법은 프로젝트에서 완성된다. 딕셔너리·클래스·리스트 컴프리헨션이 "언제 왜 필요한지"를 게임을 만들며 처음 체감했다.
  • 구조를 먼저 잡으면 협업이 쉬워진다. 뼈대 코드와 리소스 규칙을 먼저 공유하자, 팀원이 각자 기능을 붙여도 충돌이 적었다.
  • 성능은 설계에서 나온다. 오브젝트 풀링 적용 전후의 프레임 차이를 보며 "만들고 지우는 비용"을 처음 의식하게 됐다.
  • 연출은 작은 코드로 큰 효과. 텍스트 애니메이션 하나로 게임의 인상이 달라졌다.
