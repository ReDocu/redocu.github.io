3단계 · Dino — 물리와 스크롤

단계 3 · 게임 Dino Run · 씬 클래스 DinoContent : public IGameContent 처음 배우는 기술 — float 물리(중력·속도) 프레임 기반 고정 스텝 점프 포물선(오일러 적분) 월드 스크롤·시차(parallax) 절차적 스폰(거리 기반·점수 게이팅) AABB 충돌 여러 줄 스프라이트 애니메이션 값 타입 vector 예상 학습 시간 — 4–6시간

Tetris가 "정지된 보드 위의 규칙"이었다면, Dino는 시간과 물리로 스스로 흐르는 세계다. 공룡은 중력을 받아 포물선을 그리며 점프하고, 세계는 왼쪽으로 흘러 달리는 착각을 만들며, 장애물은 거리에 맞춰 저절로 생겨난다. 여기서 우리는 정수 격자를 떠나 float 로 표현하는 연속 운동의 세계로 들어간다. 크롬 공룡 게임의 그 단순한 재미 — 한 버튼으로 점프해 선인장을 넘는 — 뒤에 물리·스크롤·절차적 생성이 맞물려 돈다.

이 단계의 학습 목표

이 단계를 마치면 다음을 할 수 있다.

  1. float 속도·중력으로 부드러운 점프 포물선을 만들고, 왜 정수 격자로는 안 되는지 설명할 수 있다.
  2. 프레임 기반 고정 스텝(매 프레임 물리 갱신)과 틱 타이머(애니메이션)를 구분해 쓸 수 있다.
  3. 월드 스크롤 — 플레이어를 고정하고 세계를 흘려 달리는 느낌을 내고, 시차(parallax) 로 원근을 표현할 수 있다.
  4. 거리 기반 절차적 스폰과 점수 게이팅으로 난이도가 오르는 장애물 생성을 짤 수 있다.
  5. AABB 충돌(축 정렬 사각형 겹침)을 구현하고, 점 충돌과의 차이를 안다.
  6. 여러 줄 스프라이트를 프레임 토글로 애니메이션할 수 있다.

왜 Dino가 이 자리인가

Snake와 Tetris의 세계는 정수 격자 위에 있었다. 뱀은 칸에서 칸으로 순간이동했고, 블록은 한 칸씩 딱딱 떨어졌다. 그 세계에는 "0.5칸"이 없었다.

Dino는 이 격자를 떠난다. 공룡이 점프하면 위로 솟았다가 서서히 느려지고, 정점에서 멈칫한 뒤, 점점 빨라지며 떨어진다 — 이 부드러운 포물선은 정수로는 표현할 수 없다. 그래서 우리는 float 로 위치와 속도를 다루고, 매 프레임 중력을 속도에 더하고, 속도를 위치에 더한다. 이것이 물리 시뮬레이션의 시작이다.

그리고 세계가 움직인다. Snake·Tetris의 화면은 고정된 무대였지만, Dino의 공룡은 제자리에 있고 세계가 흐른다. 카메라가 공룡을 따라간다고 봐도 좋다. 이 "월드 스크롤" 감각은 러너·슈터·플랫포머로 확장되는 모든 게임의 핵심이다.

  • 정수 격자(Tetris) → 연속 운동(float 물리),
  • 고정된 화면 → 흐르는 세계(스크롤),
  • 정해진 배치 → 저절로 생겨나는 장애물(절차적 생성),
  • 칸 일치 충돌 → 사각형 겹침(AABB).

네 가지가 모두 한 계단씩 오른다. Snake·Tetris에서 "규칙과 알고리즘"을 배웠다면, Dino에서 배우는 것은 "살아 움직이는 세계" 다.

자료구조는 계속 값 타입으로. Tetris에서 시작한 "값 타입·고정 배열/벡터" 설계가 Dino에서도 이어진다. 장애물·구름을 std::vector<Obstacle>, std::vector<Cloud> 로 관리하되 포인터가 아니라 값을 담는다 — Snake의 vector<Snake*> 와 new/delete 를 벗어난 것이다. 이 흐름이 Road Fighter를 지나 Galaga의 오브젝트 풀로 완성된다.

개념 1 · float 물리 — 정수를 넘어 연속 운동으로

공룡의 점프는 물리 법칙 하나로 요약된다 — 중력이 매 순간 속도를 아래로 당기고, 속도가 매 순간 위치를 바꾼다. 공룡의 발 높이 dinoY 와 수직 속도 dinoVelY 를 float 로 두고, 매 프레임 이렇게 갱신한다.

static const float G       = 0.06f;   // 중력 (매 프레임 속도에 더해짐)
static const float JUMP_V  = -0.95f;  // 점프 초기 속도 (위로 = 음수, y는 아래로 증가)

// 매 프레임:
dinoVelY += G;          // ① 중력이 속도를 아래로 (속도가 점점 커짐)
dinoY    += dinoVelY;   // ② 속도만큼 위치가 바뀜

이 두 줄이 포물선을 만든다. 점프 순간 dinoVelY 는 큰 음수(위로 빠름)라 공룡이 솟구치고, 중력이 매 프레임 속도를 조금씩 키워 상승이 느려지다가, 정점에서 속도가 0을 지나 양수가 되며 떨어지기 시작한다 — 점점 빠르게. 일정한 힘(중력)이 속도를 바꾸고, 속도가 위치를 바꾸는 이 연쇄가 자연스러운 낙하 운동이다. (수학적으로는 미분방정식의 오일러 적분이지만, 코드로는 그저 "매 프레임 두 줄"이다.)

점프와 착지는 상태를 살핀다.

if (onGround && jumpEdge) {           // 땅에 있을 때만 점프 가능
    dinoVelY = JUMP_V;                // 위로 튀어오르는 초기 속도
    onGround = false;
}
if (!onGround) {
    float g = G;
    if (downStay) g *= FASTFALL;      // 체공 중 ↓ = 빠른 착지 (중력 3배)
    dinoVelY += g;
    dinoY    += dinoVelY;
    if (dinoY >= DINO_FOOT) {         // 발이 지면에 닿으면
        dinoY = DINO_FOOT;            // 정확히 지면에 맞추고
        dinoVelY = 0.0f;
        onGround = true;              // 착지
    }
}

왜 float 인가. 정수라면 dinoVelY 를 0.06씩 더할 수 없다 — 소수가 잘려 늘 0이 되니 중력이 안 걸린다. 부드러운 가감속은 소수의 누적으로만 표현된다. 1부 1장에서 "평점만 double 을 쓴다"고 했던 그 실수형이, 여기서 물리의 언어가 된다. 연속적으로 변하는 값은 실수로 다룬다 — 이것이 정수 격자 게임과 물리 게임을 가르는 선이다.

개념 2 · 프레임 기반 고정 스텝 vs 틱 타이머

Snake·Tetris에서 이동은 GetTickTimer(0.1f) 안에서만 일어났다. 그런데 Dino의 물리는 매 프레임 갱신된다. 왜 다를까?

물리는 부드러워야 한다. 0.1초마다 뚝뚝 끊어 갱신하면 점프가 계단처럼 보인다. 그래서 물리·스크롤은 게임 루프가 도는 매 프레임(초당 60번) 고정 스텝으로 갱신한다. 속도의 단위가 "칸/프레임"인 이유다. 반면 애니메이션(다리 교차, 날개 짓)은 매 프레임 바뀌면 너무 빨라 보이니, 여전히 틱 타이머로 제어한다.

void DinoContent::OnInGameUpdate()
{
    // ... 일시정지·게임오버 처리 ...

    if (TIMER->GetTickTimer(0.1f))  legFrame ^= 1;   // 다리 애니: 0.1초마다 토글
    if (TIMER->GetTickTimer(0.15f)) wingFrame ^= 1;  // 익룡 날개: 0.15초마다

    UpdateDino();     // ★ 물리: 매 프레임 (틱 밖!)
    UpdateWorld();    // ★ 스크롤·스폰: 매 프레임
    UpdateScore();

    if (CheckCollision()) { /* 게임 오버 */ }
}

legFrame ^= 1 은 0↔︎1을 토글하는 관용구다(XOR 1). "부드러워야 할 것(물리)은 매 프레임, 띄엄띄엄이어도 될 것(애니메이션)은 틱 타이머" — 0단계에서 배운 "입력 주기 ≠ 상태 변화 주기"가, 이제 "물리 주기 ≠ 애니메이션 주기"로 확장된다.

프레임 기반의 한계와 타협. 매 프레임 고정 스텝은 "프레임률이 일정하다"고 가정한다(엔진이 SetFrame 으로 60fps를 맞춰 주니 콘솔에선 충분하다). 프레임률이 출렁이면 물리도 출렁이는데, 그걸 막으려면 "실제 흐른 시간(delta time)을 곱하는" 방식을 쓴다 — 하지만 콘솔 MVP엔 과하다. 환경이 단순하면 단순한 해법이 옳다.

개념 3 · 월드 스크롤 — 공룡은 제자리, 세계가 흐른다

공룡의 x좌표(DINO_X = 6)는 절대 안 변한다. 대신 장애물과 배경이 매 프레임 왼쪽으로 흐른다. 그래서 공룡이 오른쪽으로 달리는 것처럼 보인다 — 러닝머신 위를 뛰는 것과 같은 착시다.

void DinoContent::UpdateWorld()
{
    gameSpeed = SPEED_0 + score * SPEED_ACCEL;   // 점수가 오를수록 빨라짐
    if (gameSpeed > SPEED_MAX) gameSpeed = SPEED_MAX;

    for (size_t i = 0; i < obstacles.size(); i++)
        obstacles[i].x -= gameSpeed;             // 장애물: 왼쪽으로

    for (size_t i = 0; i < clouds.size(); i++)
        clouds[i].x -= gameSpeed * 0.3f;         // ★ 구름: 더 느리게 (시차)

    // 화면 왼쪽을 벗어난 장애물 제거
    std::vector<Obstacle> kept;
    for (size_t i = 0; i < obstacles.size(); i++)
        if (obstacles[i].x + obstacles[i].w >= 0.0f)
            kept.push_back(obstacles[i]);
    obstacles = kept;
}

두 가지를 눈여겨보자.

  • 시차(parallax). 구름은 장애물의 0.3배 속도로만 흐른다(gameSpeed * 0.3f). 멀리 있는 것은 느리게, 가까운 것은 빠르게 흐르는 이 속도 차가 깊이감(원근) 을 만든다. 창밖으로 보이는 먼 산과 가까운 가로수의 차이다.
  • 벗어난 것 제거. 화면 왼쪽으로 사라진 장애물은 kept 벡터에 담지 않는 방식으로 지운다. 안 지우면 벡터가 무한히 커지며 x가 한없이 작아진다. 생성한 것은 언젠가 정리한다(1부 메모리 관리의 정신).

gameSpeed = SPEED_0 + score * SPEED_ACCEL — 점수(=달린 거리)에 비례해 속도가 오르니, 오래 살수록 세계가 빨라진다. 이 단순한 한 줄이 크롬 공룡 게임의 그 조여드는 긴장감을 만든다.

개념 4 · 절차적 스폰 — 거리에 맞춰 저절로 생겨나기

장애물을 미리 배치하지 않는다. 거리 기반으로 저절로 생성한다. "다음 스폰까지 남은 거리" spawnGap 을 매 프레임 gameSpeed 만큼 줄이고, 0 이하가 되면 오른쪽 끝에 장애물을 하나 만들고 spawnGap 을 새로 뽑는다.

spawnGap -= gameSpeed;
if (spawnGap <= 0.0f)
    SpawnObstacle();
void DinoContent::SpawnObstacle()
{
    Obstacle o;  o.x = 38.0f;                      // 오른쪽 끝에서 등장

    int roll = rand() % 100;
    if (score < 200 || roll < 70) {               // ★ 점수 게이팅
        // 선인장 (200점 전에는 선인장만)
        if (rand() % 2 == 0) { o.type = CACTUS_SMALL; o.w = 1; o.h = 2; o.topY = GROUND_Y - 2; }
        else                 { o.type = CACTUS_LARGE; o.w = 3; o.h = 3; o.topY = GROUND_Y - 3; }
    } else {
        // 익룡 (200점 이후 30%, 3가지 높이)
        int lvl = rand() % 3;
        o.w = 3; o.h = 2;
        if      (lvl == 0) { o.type = PTERO_LOW;  o.topY = 17; }  // 점프 필수
        else if (lvl == 1) { o.type = PTERO_MID;  o.topY = 15; }  // 숙이기/점프
        else               { o.type = PTERO_HIGH; o.topY = 13; }  // 서서 통과
    }
    obstacles.push_back(o);

    // 다음 스폰까지 최소 간격 (속도에 비례 — 빠를수록 멀리) + 랜덤
    float gapMin = 14.0f + 18.0f * gameSpeed;
    spawnGap = gapMin + (float)(rand() % 10);
}

두 가지 설계가 게임을 공정하게 만든다.

  • 점수 게이팅(난이도 곡선). score < 200 이면 선인장만, 그 이후엔 30% 확률로 익룡이 섞인다. 초반엔 쉽게, 갈수록 복잡하게 — 원작의 난이도 진행을 재현한다. 1부 E 모듈의 확률 분기(rand()%100)가 여기서 "난이도 게이트"로 쓰인다.
  • 속도 비례 최소 간격. gapMin = 14 + 18 * gameSpeed — 빠를수록 장애물 사이가 멀어진다. 안 그러면 고속에서 점프로 넘을 수 없는 간격이 나와 불가능한 상황이 생긴다. "플레이 가능성"을 수치로 보장하는 것이다.

절차적 생성의 씨앗. "미리 만들지 않고, 규칙과 난수로 그때그때 생성"하는 이 발상이 절차적 생성(procedural generation)이다. Dino의 거리 기반 스폰은 그 가장 단순한 형태다. 이것이 Maze의 미로 생성 알고리즘, Battle City의 절차적 전장으로 자란다.

개념 5 · AABB 충돌 — 점에서 면으로

Snake의 충돌은 "머리와 같은 칸인가"라는 점 일치였다. Dino의 공룡과 장애물은 여러 칸을 차지하는 사각형이라, 두 사각형이 겹치는지를 봐야 한다. 이것이 AABB(Axis-Aligned Bounding Box, 축 정렬 경계 상자) 충돌이다.

두 사각형이 겹치지 않는 경우는 딱 네 가지다 — 하나가 다른 하나의 완전히 왼쪽·오른쪽·위·아래에 있을 때. 그 네 경우가 모두 아니면 겹친 것이다.

bool DinoContent::CheckCollision()
{
    int footY = (int)(dinoY + 0.5f);   // float 발 높이를 반올림해 정수 칸으로

    // 공룡 히트박스 (상태에 따라 다름)
    int dl, dr, dt, db;                // left, right, top, bottom
    if (dinoState == DUCK) { dl = DINO_X; dr = DINO_X + 3; dt = footY - 1; db = footY; }
    else                   { dl = DINO_X; dr = DINO_X + 2; dt = footY - 2; db = footY; }
    dl += 1;                           // 왼쪽을 한 칸 관대하게 (플레이어에게 유리)

    for (size_t i = 0; i < obstacles.size(); i++)
    {
        int oLeft = (int)(obstacles[i].x + 0.5f);
        int oRight = oLeft + obstacles[i].w - 1;
        int oTop = obstacles[i].topY;
        int oBottom = obstacles[i].topY + obstacles[i].h - 1;

        // 겹치지 않는 네 경우가 모두 아니면 → 충돌
        if (!(dr < oLeft || dl > oRight || db < oTop || dt > oBottom))
            return true;
    }
    return false;
}

핵심은 그 한 줄이다.

!(dr < oLeft || dl > oRight || db < oTop || dt > oBottom)
  • dr < oLeft — 공룡이 장애물보다 완전히 왼쪽에 (안 닿음)
  • dl > oRight — 완전히 오른쪽에
  • db < oTop — 완전히 위에 (점프로 넘음!)
  • dt > oBottom — 완전히 아래에

이 넷 중 하나라도 참이면 안 겹친 것. !(...) 로 뒤집으면 "넷 다 아니다 = 겹쳤다". 점프가 장애물을 넘는다는 것은 db < oTop(공룡 밑바닥이 장애물 꼭대기보다 위) 조건이 참이 되는 순간이다 — 물리가 만든 높이가 충돌 판정을 바꾼다.

관대한 히트박스. dl += 1 로 공룡의 충돌 상자를 그림보다 한 칸 작게 잡는다. 픽셀이 살짝 스쳐도 죽지 않게 해 억울한 죽음을 줄이는 것이다. 좋은 충돌 판정은 물리적으로 정확한 게 아니라 플레이어가 납득할 수 있게 설계된다 — 게임 느낌(game feel)의 한 축이다.

개념 6 · 스프라이트 애니메이션 — 여러 줄 그림에 생명을

공룡·선인장·익룡은 한 글자가 아니라 여러 줄짜리 그림(스프라이트) 이다. 문자열 배열로 모양을 정의하고, 'X' 는 블록, ' ' 는 빈칸으로 해석해 그린다.

// 스프라이트: 'X' = 블록, ' ' = 빈칸
static const char* DINO_RUN0[3] = { " XX", "XXX", "X X" };   // 달리기 프레임 0
static const char* DINO_RUN1[3] = { " XX", "XXX", "XX " };   // 달리기 프레임 1 (다리 반대)
static const char* DINO_JUMP[3] = { " XX", "XXX", "X X" };   // 점프 (고정)

DrawSpriteRows(cx, cy, rows, h, color) 헬퍼가 이 배열을 훑어 'X' 자리에만 블록을 그린다(내부는 결국 OnDrawColor((cx+col)*2, cy+row, "■", color)). 애니메이션은 프레임을 토글하는 것뿐이다.

void DinoContent::DrawDino()
{
    int footY = (int)(dinoY + 0.5f);
    if (dinoState == DUCK)      DrawSpriteRows(DINO_X, footY-1, DINO_DUCK, 2, color);
    else if (dinoState == JUMP) DrawSpriteRows(DINO_X, footY-2, DINO_JUMP, 3, color);
    else if (dinoState == DEAD) DrawSpriteRows(DINO_X, footY-2, DINO_DEAD, 3, color);
    else  // RUN — 다리 프레임 토글
        DrawSpriteRows(DINO_X, footY-2, legFrame ? DINO_RUN1 : DINO_RUN0, 3, color);
}

달리는 공룡의 다리가 교차하는 것은 legFrame(0↔︎1, 개념 2에서 0.1초마다 토글)에 따라 DINO_RUN0/DINO_RUN1 을 번갈아 그리는 것뿐이다. 두 장의 그림을 빠르게 번갈아 = 애니메이션 — 플립북과 같은 원리다. 밤/낮에 따라 색만 바꾸면(nightPhase) 같은 스프라이트가 다른 분위기를 낸다.

코드 워크스루 — OnInGameUpdate 와 UpdateDino

한 프레임에서 물리·세계·충돌이 어떻게 맞물리는지 따라가자.

실제 파일: Main/Content/DinoContent.cpp — OnInGameUpdate(), UpdateDino()

void DinoContent::OnInGameUpdate()
{
    if (INPUT->OnKeyDown(VK_ESCAPE)) { currentPhase = _EPhase::TITLE; return; }
    if (INPUT->OnKeyDown('P')) { /* PLAYING ↔ PAUSE */ }
    if (gameState == PAUSE)    return;
    if (gameState == GAMEOVER) { if (SPACE/ENTER) StartGame(); return; }

    // ── PLAYING ── (매 프레임)
    if (TIMER->GetTickTimer(0.1f))  legFrame ^= 1;    // 애니메이션 (틱)
    if (TIMER->GetTickTimer(0.15f)) wingFrame ^= 1;

    UpdateDino();     // ① 물리: 점프/중력/착지 (개념 1)
    UpdateWorld();    // ② 스크롤·스폰·제거·속도 (개념 3·4)
    UpdateScore();    // ③ 점수·마일스톤·밤낮

    if (CheckCollision()) {              // ④ AABB (개념 5)
        dinoState = DEAD;
        gameState = GAMEOVER;
        if (score > hiScore) hiScore = score;   // 최고 점수 갱신
    }
}

한 프레임의 순서가 곧 인과다 — 입력을 물리에 반영(①) → 세계를 흘리고 장애물을 만들고(②) → 점수를 매기고(③) → 그 결과로 충돌을 검사(④). 물리로 공룡을 움직인 뒤, 세계를 흘린 뒤에 충돌을 보는 순서라, "지금 이 순간 공룡과 장애물이 겹쳤는가"를 정확히 판정한다.

UpdateDino 안이 개념 1의 물리 그대로다 — 땅에 있으면 점프 입력을 받고, 공중이면 중력을 속도에 더하고 속도를 위치에 더하며, 지면에 닿으면 착지한다. "매 프레임 두 줄"(v+=g; y+=v;)이 크롬 공룡의 그 점프감을 통째로 만든다.

렌더(OnInGameRender)는 이 상태를 그리기만 한다 — 배경(구름·별·달) → 지면(스크롤 질감) → 장애물 → 공룡(상태별 스프라이트) → 점수(HI·현재). 밤/낮은 nightPhase 로 색 팔레트만 바꾼다.

그림으로 정리 — 점프 포물선과 AABB

📊 웹 교육문서에서는 점프 포물선과 AABB 겹침이 애니메이션(V-P2-3)으로 재생된다. 책에서는 아래 도식으로 대신한다.

점프 포물선 — 매 프레임 v += g; y += v

  속도(v):  -0.95  -0.89  ...  -0.05  +0.01  ...  +0.89  +0.95   (중력 G가 매 프레임 +0.06)
  높이(y):   솟구침 ──────▶ 정점(v≈0) ──────▶ 낙하 (점점 빠르게)
            ●
           ╱ ╲          중력이 속도를 꾸준히 아래로 당겨
          ╱   ╲         상승→정점→가속 낙하의 부드러운 곡선.
    ─────●─────●─────   float 이라서 이 곡선이 매끄럽다.

AABB — 두 사각형이 겹치는가

  안 겹침(넘김):              겹침(충돌):
   공룡                        공룡
   ┌──┐                        ┌──┐
   │  │  ← db < oTop           │ ┌┼─┐
   └──┘     (완전히 위)         └─┼┘ │  ← 네 분리 조건이 모두 거짓
      ┌───┐                      └──┘     → 겹쳤다 → 충돌
      │장 │                    ┌───┐
      └───┘                    │장 │
  !(dr<oL || dl>oR || db<oT || dt>oB)  ← 이 식이 참이면 충돌

한 문장으로: float 물리가 부드러운 움직임을 만들고, 스크롤이 세계를 흐르게 하며, 절차적 스폰이 세계를 채우고, AABB가 그 세계의 부딪힘을 판정한다.

알려진 이슈에서 배우기 (as-built)

Dino도 완벽한 시뮬레이터가 아니라 동작하는 러너로 구현됐다. 몇 가지 타협을 들여다보면, 실전 게임 개발의 판단이 보인다.

이슈 as-built 상태 무엇을 배우나
Beep() 블로킹 100점 마일스톤에 Beep(1200, 20) — 20ms 동안 프레임이 멈춤 소리를 짧게(≤20ms)만. 콘솔 사운드는 게임 루프를 막는 대가가 있다
despawn 시 벡터 재생성 매 프레임 kept 벡터를 새로 만들어 교체 간단하지만 매 프레임 복사 비용. 장애물이 적어 문제없음("측정 후 최적화")
프레임 기반 물리 delta time 없이 "칸/프레임" 고정 60fps 가정. 프레임률이 흔들리면 물리도 흔들림 — 콘솔엔 충분
정수 반올림 충돌 float 위치를 (int)(x+0.5f) 로 반올림해 AABB 물리는 연속, 충돌은 격자. 이 변환 지점을 의식해야 함
최고 점수 세션 한정 hiScore 는 멤버 변수 — 프로그램 끄면 사라짐 파일 저장(1부 F 모듈!)은 향후 확장. "이어지는 기록"은 파일이 필요

"최고 점수 세션 한정"이 눈에 띈다 — 1부 F 모듈에서 배운 파일 I/O를 여기 붙이면 세션을 넘어 기록이 남는다. 1부의 지식이 2부의 게임을 개선하는 재료가 된다. 커리큘럼은 앞으로만 흐르지 않는다.

체크포인트 — 스스로 점검하기

Q1. 공룡의 높이·속도를 int 가 아니라 float 로 두는 이유는? int 로 하면 점프가 어떻게 되나?

Q2. 물리(UpdateDino)는 매 프레임 갱신하는데, 다리 애니메이션은 GetTickTimer(0.1f) 로 제어한다. 왜 다르게 다루나?

Q3. 공룡의 x는 고정(DINO_X=6)인데 어떻게 "달리는" 것처럼 보이나? 구름이 장애물보다 느리게 흐르는 건 무엇을 표현하나?

Q4. AABB 충돌식 !(dr < oLeft || dl > oRight || db < oTop || dt > oBottom) 에서, 점프로 장애물을 넘는 순간 참이 되는 부분 조건은 어느 것인가?

Q5. 장애물 스폰에서 gapMin 을 gameSpeed 에 비례시키는 이유는? 고정값으로 두면 무슨 문제가 생기나?

해설 펼치기

A1. 중력(0.06)·점프 속도(-0.95) 같은 소수 값을 누적해야 부드러운 포물선이 나오기 때문이다. int 면 dinoVelY += 0.06 이 소수점이 잘려 늘 0이 되어 중력이 안 걸리고, 점프가 계단처럼 뚝뚝 끊긴다. 연속적으로 변하는 값은 실수로 다룬다.

A2. 물리는 부드러워야 하므로 매 프레임(초당 60번) 갱신해 매끄러운 곡선을 만든다. 애니메이션은 매 프레임 바꾸면 너무 빨라 보이므로 0.1초마다(초당 10번) 토글한다. "부드러워야 할 것은 매 프레임, 띄엄띄엄이어도 될 것은 틱 타이머".

A3. 공룡은 제자리에 있고 장애물·배경이 매 프레임 왼쪽으로 흐른다(월드 스크롤). 러닝머신처럼 세계가 지나가 달리는 착시를 만든다. 구름이 더 느리게 흐르는 것은 시차(parallax) — 멀리 있는 것은 느리게 보여 원근(깊이)을 표현한다.

A4. db < oTop — 공룡의 밑바닥(db)이 장애물 꼭대기(oTop)보다 위에 있는 것. 즉 공룡이 장애물보다 높이 떠서 "완전히 위"에 있으면 이 조건이 참이 되어 !(...) 가 거짓 → 충돌 아님(넘김). 점프 물리가 만든 높이가 이 조건을 참으로 만든다.

A5. 속도가 빠를수록 장애물이 빨리 다가오므로, 간격이 좁으면 점프로 넘을 수 없는 불가능한 배치가 생긴다. gapMin = 14 + 18*gameSpeed 로 속도에 비례해 간격을 벌리면, 어떤 속도에서도 "넘을 수 있는" 배치가 보장된다. 게임의 공정성(플레이 가능성)을 수치로 지키는 것이다.

실습 미션 — 점프로 장애물 넘기 jump.c ⭐

Tetris의 미션이 "2D 배열 알고리즘"이었다면, 이 미션은 중력 물리 + AABB 충돌을 콕 집어 연습한다. 다가오는 장애물을 점프 물리로 넘길 수 있는지, 두 개념을 한자리에서 시뮬레이션한다.

드릴 문법 — float(또는 정수) 물리(속도·중력 적분) · 프레임 루프 · AABB 겹침 판정(개념 5) · 조건 분기 만드는 것 — 장애물이 다가오는 동안, 정한 시점에 점프해 넘길 수 있는지 판정하고 궤적을 출력하는 시뮬레이터. 클리어 조건 — 점프 타이밍에 따라 공룡의 포물선이 장애물을 넘는지(AABB) 정확히 판정한다. 규모 — 약 50줄, 파일 1개. 표준 C만. 실시간 입력 없이(결정론적) 물리와 충돌을 눈으로 확인한다.

문제 정의

이 미션의 심장은 두 가지다. ① 매 프레임 v += G; h += v; 로 점프 포물선을 만들고, ② 매 프레임 공룡 상자와 장애물 상자의 AABB 겹침을 검사한다. 실시간 입력은 콘솔 표준 C에선 까다로우니, "몇 프레임째에 점프할지"를 미리 입력받아 결정론적으로 시뮬레이션한다 — 물리와 충돌의 인과를 또렷이 볼 수 있다.

목표 / 클리어 조건

  1. 장애물 시작 거리와 "점프할 프레임"을 입력받는다.
  2. 매 프레임: 장애물을 1칸 왼쪽으로, 점프 중이면 v += G; h += v; 로 공룡 높이 갱신, 착지 처리.
  3. 매 프레임 공룡 상자 vs 장애물 상자를 AABB 로 검사. 겹치면 충돌.
  4. 장애물이 공룡을 완전히 지나가면 "통과", 도중에 겹치면 "충돌".
  5. (선택) 각 프레임의 공룡 높이를 한 줄씩 출력해 포물선을 눈으로 본다.

힌트

힌트 1 — 점프 물리 (정수 버전으로 크게)

실제 게임은 G=0.06 처럼 작지만, 눈에 보이게 정수로 키워 쓴다.

int h = 0, v = 0, onGround = 1;
const int JUMP = 5, G = 1;      /* 점프 초속 +5, 중력 -1 */
/* 점프 순간: */  v = JUMP; onGround = 0;
/* 매 프레임 공중: */  h += v; v -= G;  if (h <= 0) { h = 0; v = 0; onGround = 1; }

높이 h는 0→5→9→12→14→15→15→14→12→9→5→0 의 포물선을 그린다.

힌트 2 — AABB 겹침 함수

두 사각형 [ax1,ay1~ax2,ay2], [bx1,by1~bx2,by2] 의 겹침.

int overlap(int ax1,int ay1,int ax2,int ay2, int bx1,int by1,int bx2,int by2)
{
    return !(ax2 < bx1 || ax1 > bx2 || ay2 < by1 || ay1 > by2);
}

공룡 상자: x[0,2], y[h, h+2] (높이 h에서 3칸). 장애물 상자: x[obsX, obsX+1], y[0, 2] (지면 높이 3).

정답 코드

정답 펼치기 — jump.c 
#define _CRT_SECURE_NO_WARNINGS
#include <stdio.h>

/* 두 축정렬 사각형의 겹침 (개념 5의 AABB) */
int overlap(int ax1, int ay1, int ax2, int ay2,
            int bx1, int by1, int bx2, int by2)
{
    return !(ax2 < bx1 || ax1 > bx2 || ay2 < by1 || ay1 > by2);
}

int main(void)
{
    int obsDist, jumpAt;
    const int JUMP = 5, G = 1;      /* 점프 초속 / 중력 (눈에 보이게 정수) */
    const int OBS_W = 2, OBS_H = 3; /* 장애물 크기 */
    const int DINO_W = 3, DINO_H = 3;

    printf("=== 점프로 장애물 넘기 ===\n");
    printf("장애물까지 거리(프레임): ");  scanf("%d", &obsDist);
    printf("몇 프레임째에 점프?     : "); scanf("%d", &jumpAt);

    int h = 0, v = 0, onGround = 1, hit = 0;

    for (int f = 0; f <= obsDist; f++)
    {
        /* 점프 입력 */
        if (f == jumpAt && onGround) { v = JUMP; onGround = 0; }
        /* 물리 (한 프레임 고정 스텝) */
        if (!onGround) {
            h += v; v -= G;
            if (h <= 0) { h = 0; v = 0; onGround = 1; }
        }
        /* 장애물 위치: 매 프레임 1칸씩 다가옴 (공룡은 x=0 고정) */
        int obsX = obsDist - f;

        /* AABB: 공룡[0..2, h..h+2] vs 장애물[obsX.., 0..OBS_H-1] */
        if (overlap(0, h, DINO_W - 1, h + DINO_H - 1,
                    obsX, 0, obsX + OBS_W - 1, OBS_H - 1))
            hit = 1;

        /* 궤적 출력: 공룡 높이를 막대로 */
        printf("f%2d | 장애물 x=%2d | 공룡 h=%d ", f, obsX, h);
        for (int i = 0; i < h; i++) printf("^");
        printf("\n");
    }

    printf(hit ? "\n>> 충돌! 타이밍을 놓쳤습니다.\n"
               : "\n>> 통과! 멋지게 넘었습니다.\n");
    return 0;
}

빌드: cl jump.c (MSVC) 또는 gcc jump.c -o jump

함정/체크포인트

  • 점프 타이밍이 핵심. 너무 일찍 점프하면 장애물이 오기 전에 착지해 버리고, 너무 늦으면 이미 겹친다. 여러 jumpAt 값을 넣어 "넘는 구간"을 찾아보자 — 이것이 리듬 게임의 타이밍 창(window)이다.
  • AABB의 y 조건. 공룡이 충분히 높이(h ≥ OBS_H) 떠 있으면 ay1 > by2(공룡 밑바닥이 장애물 꼭대기보다 위)가 되어 안 겹친다. 개념 5의 "점프로 넘음 = db < oTop"이 여기선 이 조건이다.
  • v += G 가 아니라 v -= G 인 이유: 이 미션은 높이 h 를 위로 양수로 뒀다(실제 게임은 y가 아래로 증가해 부호가 반대). 좌표계의 방향을 먼저 정하고 부호를 맞추는 습관.

확장 과제

  • G 를 0.5(float)로 줄이고 JUMP 를 키워, 실제 게임처럼 더 부드러운 긴 포물선으로.
  • 장애물을 여러 개 배열로 두고, 각각 AABB 검사하면 진짜 러너에 가까워진다.

Dino로 우리는 "시간과 물리로 스스로 흐르는 세계"를 만들었다. 공룡은 중력으로 점프하고, 세계는 흐르며, 장애물은 저절로 생겨나 사각형으로 부딪힌다. 다음 4단계 Road Fighter는 여기에 여러 시스템을 동시에 얹는다 — 한 리스트에 적·트럭·아이템·장애물을 함께 담는 엔티티 시스템, 연료·목숨이라는 자원, 무적 상태, 시간 기반 스테이지. 하나의 게임 안에서 여러 시스템이 물려 도는 시스템 설계를 경험한다.
