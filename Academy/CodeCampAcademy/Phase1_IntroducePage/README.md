## Day 1 · 일하는 방식이 바뀐다, 그리고 PRD

**배우는 것**

- 과거의 분업(기획자·디자이너·개발자·마케터) → 이제는 **웹/앱 만들기가 모든 직군의 공통 업무**
- AI 발전 3단계: 생성형 AI(웹) → AI 에이전트(설치형) → 에이전틱 AI
- 웹 제작 도구 생태계(홈페이지 → 프로토타입/MVP → 프로덕트)와 **Claude Code 하나로 전 단계 커버**
- AI 에이전트의 5가지 핵심 요소: **컨텍스트 · 모델/에포트 · 스킬 · 훅스 · MCP**
- 클로드코드 실행 환경(CLI / GUI)

**실습·과제** — 나만의 PRD 만들기: "누가 / 무엇을 / 왜"를 문서로 정리한다.

**이걸 할 수 있으면 통과**

- [ ] 만들고 싶은 것을 PRD(누가/무엇을/왜)로 정리할 수 있다
- [ ] 에이전트 5요소를 각각 한 문장으로 설명할 수 있다

## Day 2 · 웹의 구조 — 뒷단과 앞면, 그리고 GDD

**배우는 것**

- 홈페이지 vs 웹 서비스의 차이, 백엔드를 빌려 쓰는 **BaaS**와 **Supabase**
- 프론트엔드의 진화: **MPA vs SPA**, html/css/js → **React** → **Next**
- 현대 웹 개발의 세 조각: **Claude Code + Vercel + Supabase**
- 제품 문서 **PRD** → 게임 문서 **GDD** 확장(소코반 예시)

**실습·과제** — 게임 만들기 + 컨텍스트 관리: GDD를 쓰고 클로드코드로 게임을 만들어 본다.

**이걸 할 수 있으면 통과**

- [ ] 홈페이지와 웹 서비스를 "데이터가 오가는가"로 구분할 수 있다
- [ ] Claude Code · Vercel · Supabase가 각각 무엇을 담당하는지 말할 수 있다

## Day 3 · 클로드코드를 다루는 법 — 컨텍스트와 스킬

**배우는 것**

- AI의 작업 기억 **컨텍스트**: 매 대화마다 재주입(비용 반복 지불), 처음과 끝은 잘 기억하고 중간은 잘 잊음
- **슬래시 명령어**(클로드코드의 리모컨)와 사용량·요금제 읽는 법
- 여러 작업을 동시에 돌리는 **터미널 멀티플렉서**(tmux / cmux / wmux)
- 플러그인의 3요소(**스킬 · 훅스 · MCP**)와 프로젝트 폴더 구조
- **스킬** — 반복 프롬프트를 자산으로 만드는 법

**실습·과제** — 스킬로 소코반 게임 만들기.

**이걸 할 수 있으면 통과**

- [ ] 컨텍스트가 왜 비용이고 왜 중간을 잊는지 설명할 수 있다
- [ ] 자주 쓰는 프롬프트를 스킬로 등록해 재사용할 수 있다

## Day 4 · 서버와 데이터 — Supabase와 MCP

**배우는 것**

- 서버 프로그램의 원리: **호스트와 포트**, 실서비스 포트 80(HTTP)/443(HTTPS), 접속=새로고침
- 데이터는 어디에: **로컬스토리지 vs 외부 DB**, **NoSQL(문서) vs SQL(표)**, Firebase vs Supabase
- **Supabase 실전** — 프로젝트 생성과 제공 기능
- AI를 외부 세계에 연결하는 열쇠, **MCP**

**실습·과제** — DB를 연동하여 게임의 기능과 연결하기(Supabase + MCP).

**이걸 할 수 있으면 통과**

- [ ] 로컬스토리지 저장과 외부 DB 저장의 차이를 설명할 수 있다
- [ ] 클로드코드에 Supabase MCP를 연결해 테이블을 다룰 수 있다

## Day 5 · 배포 — 내가 만든 것을 세상에

**배우는 것**

- 배포의 세 가지 준비물과 주소(URL) 한 줄의 구조
- 어디에 올릴까: 내 컴퓨터 · 클라우드 · **배포 플랫폼(Vercel)**, 백엔드는 만들 것인가(Render/Railway) **빌릴 것인가(Supabase)**
- **코드 저장소 → Vercel 자동 배포** 흐름, 그리고 그 흐름을 타면 안 되는 **환경변수(.env)**
- 통제 없는 노출의 대가 — 티빙 개인정보 유출 사례
- **Git(내 컴퓨터의 도구) vs GitHub(인터넷의 저장소)**: init → commit → push → 말로 배포하기

**실습·과제** — Git·GitHub·Vercel로 내 소개 페이지를 실제 주소로 배포하고, 배포 퀴즈로 자가 점검.

**이걸 할 수 있으면 통과**

- [ ] commit과 push의 차이를 설명하고 직접 할 수 있다
- [ ] `.env`가 왜 저장소에 올라가면 안 되는지 사례를 들어 설명할 수 있다
- [ ] 내 소개 페이지가 실제 인터넷 주소로 열린다
