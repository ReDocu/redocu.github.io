확장 · Pac-Man — 유령 AI와 상태머신

확장 · 8번째 게임 · 게임 Pac-Man · 씬 클래스 PacManContent : public IGameContent 처음 배우는 기술 — 성격별 타게팅 AI(4유령) 유한상태기계(FSM) 전역 모드 스케줄(scatter/chase) greedy 격자 경로선택·역방향 금지 방향 버퍼링 파워모드·연쇄 점수 데이터 주도 맵 로딩 예상 학습 시간 — 5–7시간

책의 마지막 게임이다. Battle City의 적 탱크는 "랜덤 + 기지 편향"이라는 소박한 AI였다. Pac-Man은 그것을 한 단계 끌어올린다 — 유령 넷이 각자 다른 성격으로 다른 목표를 계산하고(직접 추적·앞질러 가로막기·협공·변덕), 상태머신으로 흩어졌다 몰려왔다 도망친다. "적 AI란 무엇인가"를 가장 정교하게 다루는 이 게임으로, 우리는 이 책이 쌓아 온 격자·타일·상태·경로탐색을 모두 한자리에 모은다. 그리고 그 끝에서, 변수 하나부터 여기까지 온 두 여정을 함께 돌아본다.

이 단계의 학습 목표

이 단계를 마치면 다음을 할 수 있다.

  1. 유령 넷의 성격별 타게팅(Blinky/Pinky/Inky/Clyde)이 어떻게 다른 목표 타일을 계산하는지 설명하고 구현할 수 있다.
  2. 유령의 유한상태기계(FSM) — HOUSE/SCATTER/CHASE/FRIGHTENED/EATEN — 를 전이 규칙과 함께 짤 수 있다.
  3. 전역 모드 스케줄(scatter↔︎chase 교대)이 개별 유령을 어떻게 모는지 안다.
  4. 교차로에서 목표에 가장 가까운 방향을 greedy 선택하고, 역방향을 금지할 수 있다.
  5. 방향 버퍼링(wantDir)으로 팩맨의 코너링 손맛을 만들 수 있다.
  6. 데이터 주도 맵, 파워모드·연쇄 점수를 구현하고, Maze의 BFS로 AI를 개선하는 방향을 안다.

왜 Pac-Man이 마지막인가

이 책의 게임들은 AI를 조금씩 키워 왔다. Dino의 장애물은 그냥 다가왔고, Road Fighter의 적은 차선을 오갔으며, Galaga의 편대는 정해진 궤도로 행진했고, Battle City의 탱크는 랜덤 40:60으로 기지 쪽을 향했다. 점점 똑똑해졌지만, 어느 것도 "생각하는 적"은 아니었다.

Pac-Man의 유령은 다르다. 이들은 각자 목표를 계산하고, 그 목표를 향해 길을 고른다. 그리고 넷의 성격이 판이하다.

  • Blinky(빨강) 는 팩맨을 곧장 쫓는다.
  • Pinky(분홍) 는 팩맨의 진행 방향 4칸 앞을 노려 가로막는다.
  • Inky(하늘) 는 Blinky의 위치를 반사해 협공한다.
  • Clyde(주황) 는 멀면 쫓고 가까우면 도망치는 변덕쟁이다.

이 네 성격이 만드는 것이 팩맨 특유의 심리전 — 앞은 Pinky가 막고 뒤는 Blinky가 쫓고 옆에서 Inky가 끼어드는 포위망이다. 그리고 이 모든 행동은 하나의 유한상태기계가 지배한다: 유령은 흩어져 코너로 갔다가(SCATTER), 몰려와 쫓다가(CHASE), 파워펠릿에 놀라 도망치고(FRIGHTENED), 먹히면 눈만 남아 집으로 돌아간다(EATEN).

"AI = 목표 선정 + 경로 선택 + 상태 전이" — 게임 AI의 세 기둥을, 이 게임이 가장 또렷하게 보여 준다. 그래서 이 책의 마지막에 둔다. 여기서 쓰는 격자·타일 좌표(Tetris·Maze·Battle City), 상태머신(전 게임), 경로탐색(Maze BFS)이 모두 앞에서 배운 것들이다 — Pac-Man은 새 기법의 나열이 아니라, 배운 것으로 만드는 지능이다.

Battle City와의 대조. Battle City의 AI는 "랜덤 + 기지 편향"(그리디의 가장 단순한 형태)이었다. Pac-Man은 성격별로 다른 목표 + 상태 전이로 한 차원 올라선다. 그리고 GDD가 남긴 폴리시 — Maze의 BFS로 목표까지 최단 다음 칸을 계산하면 벽을 우회하는 더 똑똑한 유령이 된다 — 이 이 책의 두 축(6단계 경로탐색 · 적 AI)을 잇는 마지막 매듭이다.

개념 1 · 데이터 주도 미로 — 코드가 아니라 데이터로

팩맨의 미로는 코드로 그리지 않는다. 문자 배열로 그려 두고, 파싱해서 타일 격자로 읽는다. 미로 모양을 바꾸고 싶으면 코드가 아니라 데이터를 고친다.

// 미로를 '그림'으로 authored — 보이는 그대로가 곧 데이터
static const char* MAZE[MAZE_H] = {
    "###################",
    "#........#........#",
    "#o##.###.#.###.##o#",   // o = 파워펠릿
    "#.................#",
    "#.##.#.#####.#.##.#",
    "#....#...#...#....#",
    // ... (P = 팩맨 시작, G = 유령 집, - = 게이트) ...
};

LoadMaze() 가 이 문자들을 _ETile 격자로 옮긴다 — #→벽, .→펠릿, o→파워펠릿, -→게이트. 그러면서 펠릿 수를 세고(pelletCount), 팩맨·유령 시작 위치를 찾는다.

void PacManContent::LoadMaze()
{
    pelletCount = 0;
    for (int y = 0; y < MAZE_H; y++)
        for (int x = 0; x < MAZE_W; x++) {
            char c = MAZE[y][x];
            switch (c) {
            case '#': tile[y][x] = _ETile::WALL;  break;
            case '.': tile[y][x] = _ETile::PELLET; pelletCount++; break;   // 펠릿 카운트
            case 'o': tile[y][x] = _ETile::POWER;  break;
            case '-': tile[y][x] = _ETile::DOOR;   break;
            case 'P': startX = x; startY = y; tile[y][x] = _ETile::EMPTY; break;  // 시작 위치 추출
            default:  tile[y][x] = _ETile::EMPTY;  break;
            }
        }
}

"보이는 그대로가 데이터" 인 이 방식의 장점은 크다 — 미로를 눈으로 그리며 만들 수 있고, 스테이지마다 다른 미로를 데이터만 바꿔 넣을 수 있다. 1부 F 모듈의 파일 I/O를 붙이면 미로를 외부 파일에서 읽는 맵 에디터로도 자란다. 코드와 데이터를 분리하는 이 발상은 큰 게임일수록 중요해진다(레벨 디자이너가 코드 없이 맵을 만든다).

펠릿이 곧 승리 조건. 펠릿은 별도 리스트가 아니라 타일에 저장된다(_ETile::PELLET). 먹으면 타일을 EMPTY 로 바꾸고 pelletCount--. pelletCount == 0 이 되면 스테이지 클리어다. 오브젝트를 따로 관리하지 않고 맵 자체가 상태를 담는 것 — Battle City의 파괴 지형과 같은 발상이다.

개념 2 · 방향 버퍼링 — 팩맨의 코너링 손맛

팩맨은 격자를 한 칸씩 이동하지만, 조작은 부드러워야 한다. 코너를 돌기 직전에 방향키를 미리 눌러도 제때 꺾여야 한다. 이를 위해 입력은 즉시 "원하는 방향"에 저장하고, 실제 회전은 이동 틱에 벽이 아닐 때만 한다.

void PacManContent::UpdatePlayer()
{
    // 입력은 매 프레임 wantDir 에 예약만 (아직 안 꺾음)
    if (INPUT->OnKeyStay('W') || INPUT->OnKeyStay(VK_UP))    wantDir = _EDir::UP;
    if (INPUT->OnKeyStay('D') || INPUT->OnKeyStay(VK_RIGHT)) wantDir = _EDir::RIGHT;
    if (INPUT->OnKeyStay('S') || INPUT->OnKeyStay(VK_DOWN))  wantDir = _EDir::DOWN;
    if (INPUT->OnKeyStay('A') || INPUT->OnKeyStay(VK_LEFT))  wantDir = _EDir::LEFT;

    if (TIMER->GetTickTimer(pacMove))       // 이동 틱에만
    {
        int wx, wy; DirDelta(wantDir, wx, wy);
        if (Passable(WrapX(px + wx), py + wy, false))   // 원하는 방향이 뚫려 있으면
            pdir = wantDir;                             // 그제서야 회전

        int dx, dy; DirDelta(pdir, dx, dy);
        if (Passable(WrapX(px + dx), py + dy, false)) { px = WrapX(px + dx); py = py + dy; }
    }
}

핵심은 wantDir(원하는 방향)과 pdir(실제 진행 방향)을 분리한 것이다. 매 프레임 키를 wantDir 에 담아 두고, 이동 틱마다 "지금 wantDir 로 꺾을 수 있나?"를 확인해 가능하면 회전한다. 그래서 코너를 조금 일찍 눌러도, 팩맨이 그 지점에 닿는 순간 매끄럽게 꺾인다. 입력을 "의도"로 저장해 두고 적절한 때에 반영하는 이 패턴은, Tetris의 DAS·Dino의 점프 감각과 한 갈래다 — "입력받는 주기와 반영하는 주기는 다르다"(0단계).

WrapX 는 좌우 끝 터널 워프 — x가 미로 밖으로 나가면 반대편으로 순환시킨다(팩맨 미로의 좌우 통로).

개념 3 · 유한상태기계 — 유령의 다섯 상태

유령의 모든 행동은 상태(state) 가 결정한다. 다섯 상태와 그 사이 전이가 유령 AI의 뼈대다.

enum class _EGhostState { HOUSE, SCATTER, CHASE, FRIGHTENED, EATEN };
  HOUSE ──(대기시간 경과)──▶ SCATTER ◀──(전역 스케줄)──▶ CHASE
    ▲                            │  파워펠릿                 │
    │                            ▼                           │
 (집 도착)                  FRIGHTENED ──(팩맨에 먹힘)──▶ EATEN(눈) ──▶ HOUSE
  • HOUSE — 유령 집에서 대기(releaseTimer). 시간이 지나면 게이트로 나가 SCATTER/CHASE로.
  • SCATTER — 각자 자기 코너로 흩어진다(잠깐의 숨통).
  • CHASE — 성격별 목표로 팩맨을 몰아간다(개념 4).
  • FRIGHTENED — 파워펠릿에 놀라 도망(느려지고 교차로에서 랜덤).
  • EATEN — 먹혀서 눈만 남아 집으로 빠르게 복귀 → 부활.

이 상태는 두 힘이 몬다. 첫째, 전역 모드 스케줄 — 정해진 시간표로 SCATTER와 CHASE를 교대시킨다. 둘째, 파워펠릿 — 전체를 FRIGHTENED로 뒤집는다.

void PacManContent::UpdateGhosts()
{
    if (powerTimer <= 0) {
        modeTimer--;
        if (modeTimer <= 0) {                     // 전역 스케줄 교대
            chaseMode = !chaseMode;
            modeTimer = chaseMode ? CHASE_FRAMES : SCATTER_FRAMES;
            for (int i = 0; i < 4; i++)
                if (SCATTER 또는 CHASE 상태) {
                    ghosts[i].state = chaseMode ? CHASE : SCATTER;
                    ghosts[i].dir = Reverse(ghosts[i].dir);   // ★ 모드 전환 시 방향 반전
                }
        }
    } else {
        powerTimer--;
        if (powerTimer == 0) { /* FRIGHTENED → 원래 모드로 복귀 */ }
    }
    // ... 상태별 이동 속도로 각 유령 이동 ...
}

두 가지가 원작 규칙의 정수다. 모드가 바뀌는 순간 유령은 진행 방향을 반전한다(Reverse) — 쫓다가 갑자기 돌아서는 그 움직임이 팩맨의 리듬을 만든다. 그리고 상태마다 이동 속도가 다르다 — 프라이트는 느리게(GHOST_FRIGHT), 눈은 빠르게(GHOST_EATEN), 집에서는 또 다르게. 하나의 상태 값이 "무엇을 목표로, 얼마나 빨리, 어떻게 고를지"를 통째로 정한다. 이것이 FSM의 힘 — 복잡한 행동을 몇 개의 상태와 전이로 정리한다.

개념 4 · 성격별 타게팅 — 넷이 다른 목표를 노린다 ⭐

이 게임의 심장이다. 네 유령은 CHASE 상태에서 각자 다른 목표 타일을 계산한다. 목표만 다를 뿐, 그 목표로 가는 방법(개념 5)은 똑같다 — 그래서 "성격"이 목표 계산 하나에 응축된다.

void PacManContent::GhostTarget(const Ghost& g, int& tx, int& ty)
{
    if (g.state == EATEN || g.state == HOUSE) { tx = EXIT_X; ty = EXIT_Y; return; }  // 집으로
    if (g.state == SCATTER)                   { tx = g.scatterX; ty = g.scatterY; return; }  // 코너로

    // ── CHASE: 성격별 목표 ──
    int dx, dy; DirDelta(pdir, dx, dy);        // 팩맨의 진행 방향
    switch (g.who) {
    case BLINKY:                               // 직접 추적
        tx = px;  ty = py;
        break;
    case PINKY:                                // 팩맨 4칸 앞 (가로막기)
        tx = px + 4 * dx;  ty = py + 4 * dy;
        break;
    case INKY: {                               // Blinky를 반사한 협공 지점
        int ax = px + 2 * dx, ay = py + 2 * dy;         // 팩맨 2칸 앞
        int bx = ghosts[0].x, by = ghosts[0].y;        // Blinky 위치
        tx = ax + (ax - bx);  ty = ay + (ay - by);     // 그 점을 Blinky 기준으로 반사
        break;
    }
    default: {                                 // CLYDE: 멀면 추적, 가까우면 도망
        int man = abs(g.x - px) + abs(g.y - py);       // 맨해튼 거리 (Maze A*의 그것)
        if (man > 8) { tx = px; ty = py; }             // 8칸 밖 → 직접 추적
        else         { tx = g.scatterX; ty = g.scatterY; }  // 8칸 안 → 자기 코너로 후퇴
        break;
    }
    }
}

네 공식을 음미하자.

  • Blinky = (px, py). 팩맨 그 자리. 가장 단순하고 가장 무섭다 — 등 뒤에서 끈질기게.
  • Pinky = 팩맨의 4칸 앞. 팩맨이 가는 곳을 미리 노려 길목을 막는다. 그래서 Blinky가 뒤에서 밀고 Pinky가 앞에서 기다린다.
  • Inky = 팩맨 2칸 앞을 잡고, Blinky 위치를 기준으로 반사한 점. a + (a - b) — 벡터 반사다. Blinky와 팩맨의 상대 위치에 따라 목표가 요동쳐, 예측하기 가장 어려운 협공을 만든다.
  • Clyde = 8칸 밖이면 Blinky처럼 추적, 8칸 안이면 자기 코너로 후퇴. 다가왔다 물러섰다 하는 변덕. 맨해튼 거리(abs(dx)+abs(dy))는 6단계 Maze의 A* 휴리스틱과 같은 계산이다.

성격이 곧 목표 함수다. 이동 코드는 넷이 완전히 똑같고, 오직 GhostTarget 이 주는 목표 좌표만 다르다. 이것이 AI 설계의 우아한 통찰 — 행동의 다양성을 "무엇을 목표로 삼는가"라는 한 점에 모으면, 나머지(길 찾기)는 공유할 수 있다.

개념 5 · greedy 격자 경로선택 — 목표에 가까운 방향으로

목표 타일이 정해지면, 유령은 교차로에서 목표에 가장 가까워지는 방향을 고른다. 단, 왔던 길(역방향)로는 못 돌아간다(원작 규칙). 최단경로를 완벽히 계산(BFS)하는 대신, 매 칸에서 당장 가장 가까워 보이는 이웃을 고르는 — 그리디(greedy) 방식이다.

void PacManContent::ChooseGhostDir(Ghost& g)
{
    int tgtX, tgtY; GhostTarget(g, tgtX, tgtY);   // 이 유령의 목표
    _EDir rev = Reverse(g.dir);                    // 역방향(금지)

    // 갈 수 있는 방향 수집 (역방향 제외, 벽 제외)
    _EDir cand[4]; int n = 0;
    _EDir order[4] = { UP, LEFT, DOWN, RIGHT };    // 동점 시 우선순위 (원작 규칙)
    for (_EDir d : order) {
        if (d == rev) continue;
        int dx, dy; DirDelta(d, dx, dy);
        if (Passable(WrapX(g.x + dx), g.y + dy, true)) cand[n++] = d;
    }
    if (n == 0) { g.dir = rev; return; }           // 막다른 길 → 어쩔 수 없이 역방향

    if (g.state == FRIGHTENED) { g.dir = cand[rand() % n]; return; }  // 겁먹으면 랜덤

    // 목표까지 유클리드 거리가 최소인 방향을 고른다 (greedy)
    _EDir best = cand[0]; long bestD = 1L << 29;
    for (int k = 0; k < n; k++) {
        int dx, dy; DirDelta(cand[k], dx, dy);
        int nx = g.x + dx, ny = g.y + dy;
        long dd = (long)(nx-tgtX)*(nx-tgtX) + (long)(ny-tgtY)*(ny-tgtY);  // 거리²
        if (dd < bestD) { bestD = dd; best = cand[k]; }
    }
    g.dir = best;
}

세 규칙이 원작의 유령을 만든다.

  • 역방향 금지. 왔던 길로 못 돌아가니(d == rev 제외), 유령은 앞으로만 흐른다. 그래서 직선 통로에선 계속 직진하고, 교차로에서만 방향을 고른다. (예외: 모드 전환 순간의 강제 반전은 개념 3.)
  • 거리² 최소 greedy. 각 후보 방향의 다음 칸이 목표와 얼마나 가까운지 거리 제곱으로 재, 가장 가까운 방향을 고른다. 완벽한 최단이 아니라 "지금 가장 가까워 보이는" 선택 — 그래서 벽에 막혀 빙 돌기도 한다. 그 불완전함이 원작 팩맨의 맛이다.
  • 동점 우선순위 UP > LEFT > DOWN > RIGHT. 거리가 같은 후보가 여럿이면 이 순서로 고른다(원작의 유명한 규칙). 그리고 FRIGHTENED면 아예 랜덤으로 골라 도망친다.

greedy vs BFS — Battle City와 Maze가 만나는 곳. 이 greedy는 Battle City 적 AI의 "기지 편향"과 같은 발상(당장 가까운 쪽)이다. 벽을 우회하지 못하는 한계도 같다. 여기에 6단계 Maze의 BFS를 붙이면(GDD 폴리시), 유령이 목표까지 최단경로의 다음 칸을 계산해 벽을 돌아 정확히 추적한다. "적 AI에 경로탐색을 재사용한다" — 이 책이 6단계에서 예고하고 7단계에서 언급한 그 매듭이, 여기서 완성된다.

개념 6 · 파워모드 — 사냥꾼이 사냥감이 되다

파워펠릿을 먹으면 판이 뒤집힌다 — 전체 유령이 FRIGHTENED가 되어 도망치고, 팩맨이 그들을 먹을 수 있다.

void PacManContent::SetFrightened()
{
    powerTimer = PowerFrames();                 // 파워 지속(스테이지 오를수록 짧아짐)
    eatChain = 0;                               // 연쇄 점수 초기화
    for (int i = 0; i < 4; i++)
        if (SCATTER 또는 CHASE 상태) {
            ghosts[i].state = FRIGHTENED;
            ghosts[i].dir = Reverse(ghosts[i].dir);   // 놀라서 돌아선다
        }
}

먹을수록 점수가 배가되는 연쇄 점수가 파워모드의 묘미다.

// 프라이트 유령을 먹으면 (HandleCollisions에서):
score += 200 << eatChain;    // 200 → 400 → 800 → 1600
eatChain++;
g.state = EATEN;             // 눈만 남아 집으로

200 << eatChain — 비트 시프트로 200, 400, 800, 1600을 만든다(한 번 먹을 때마다 2배). 한 파워모드 동안 넷을 다 먹으면 200+400+800+1600 = 3000점. 그래서 파워펠릿을 언제 먹고 어떻게 넷을 몰아 잡을지가 고득점의 열쇠다 — 쫓기던 팩맨이 사냥꾼이 되는 이 반전이, 게임의 심리전을 완성한다.

PowerFrames() 는 스테이지가 오를수록 짧아진다(480 - (stage-1)*60, 최소 120). 후반부엔 반격의 시간이 짧아 긴장이 는다 — Road Fighter·Galaga에서 본 "스테이지 = 난이도 다이얼"이 여기서도.

코드 워크스루 — UpdateGhosts 한 프레임

유령 AI가 한 프레임에서 어떻게 도는지 따라가자.

실제 파일: Main/Content/PacManContent.cpp — UpdateGhosts · ChooseGhostDir · GhostTarget

void PacManContent::UpdateGhosts()
{
    // ① 전역 모드: 파워 중이 아니면 scatter↔chase 시간표를 돌리고,
    //    파워 중이면 powerTimer를 깎다가 0에서 FRIGHTENED 해제 (개념 3)
    if (powerTimer <= 0) { /* modeTimer 교대 + 방향 반전 */ }
    else                 { /* powerTimer-- ; 0이면 원래 모드로 */ }

    // ② 상태별 이동 속도 결정 (프라이트 느리게, 눈 빠르게)
    bool tN = TIMER->GetTickTimer(GHOST_MOVE);
    bool tF = TIMER->GetTickTimer(GHOST_FRIGHT);
    bool tE = TIMER->GetTickTimer(GHOST_EATEN);

    for (int i = 0; i < 4; i++) {
        Ghost& g = ghosts[i];
        bool moveNow = (g.state == FRIGHTENED) ? tF
                     : (g.state == EATEN)      ? tE : tN;      // 상태가 속도를 정한다
        if (!moveNow) continue;

        ChooseGhostDir(g);                    // ③ 목표 계산(개념 4) + greedy 방향(개념 5)
        int dx, dy; DirDelta(g.dir, dx, dy);
        int nx = WrapX(g.x + dx), ny = g.y + dy;
        if (Passable(nx, ny, true)) { g.x = nx; g.y = ny; }   // ④ 이동

        // ⑤ EATEN이 집 입구에 도착 → 부활
        if (g.state == EATEN && g.x == EXIT_X && g.y == EXIT_Y)
            g.state = chaseMode ? CHASE : SCATTER;
    }
}

한 프레임의 인과가 선명하다 — 전역 모드가 각 유령의 상태를 정하고(①), 상태가 이동 속도를 정하고(②), 상태가 목표를 정하고(③, GhostTarget), 목표가 방향을 정하고(③, greedy), 방향이 이동을 만든다(④). 하나의 state 값이 위에서 아래로 흐르며 유령의 모든 것을 결정한다. AI가 "if 문 더미"가 아니라 "상태 → 목표 → 경로"의 깔끔한 파이프라인이 되는 것 — 그것이 FSM + 타게팅 설계의 결실이다.

렌더는 이 상태를 그린다 — 성격색 Ω(NORMAL), 파랑 Ω(FRIGHTENED, 종료 직전 흰 깜빡), 눈 ◎(EATEN). 팩맨은 진행 방향으로 입을 벌린 ▶◀▲▼ 와 닫힌 ● 를 토글해 그 유명한 "냠냠"을 만든다.

그림으로 정리 — 넷의 목표와 상태 전이

📊 웹 교육문서에서는 네 유령의 타게팅과 FSM 전이가 애니메이션(V-P2-9)으로 재생된다. 책에서는 아래 도식으로 대신한다.

성격별 타게팅 — 같은 팩맨, 네 개의 목표

                     Pinky ◎ (팩맨 4칸 앞 — 길목 차단)
                        ↑
        Blinky ─────▶ ●팩맨(→)         Clyde: 멀면 ●추적 / 가까우면 ↙코너
        (직접 추적)      ↘
                        Inky ◎ (Blinky를 반사한 협공 지점)
   행동 코드는 넷이 동일. 다른 건 GhostTarget 이 주는 목표 좌표 하나뿐.

유령 FSM — 상태 하나가 모든 것을 정한다

     ┌──────────── 전역 스케줄(scatter↔chase) ────────────┐
     ▼                                                    │
  HOUSE ──▶ SCATTER ◀──────────────────────────▶ CHASE ──┘
              │  (파워펠릿: 전체 FRIGHTENED)         │
              └──────────▶ FRIGHTENED ──(먹힘)──▶ EATEN ──▶ HOUSE
   state → 이동속도 · 목표(GhostTarget) · 방향선택(greedy/random)

한 문장으로: 성격은 목표 함수 하나에 응축되고, 유한상태기계가 그 목표와 속도와 선택을 지휘한다 — 이것이 "생각하는 적"의 설계다.

알려진 이슈에서 배우기 (as-built)

Pac-Man의 AI도 원작에 충실한 MVP다. 그 단순화가 오히려 AI 설계의 선택지를 가르친다.

이슈 as-built 상태 무엇을 배우나
greedy 경로선택 교차로에서 거리² 최소 (BFS 아님) 벽을 우회 못 함(원작도 그렇다). Maze BFS로 최단 추적 가능(개념 5)
터널·집 규칙 단순화 워프·게이트를 좌표 조건으로 원작의 세밀한 속도/영역 규칙은 폴리시
Inky 반사 근사 팩맨 2칸 앞 + Blinky 반사 원작의 미묘함을 벡터 하나로 근사
연쇄 점수 200<<n 비트 시프트로 배가 간결하지만 4마리 상한 가정
하이스코어 세션 한정 멤버 변수 파일 저장(1부 F)은 향후 확장

"greedy vs BFS"가 다시 눈에 띈다. 이 책을 다 읽은 당신은, 6단계 Maze의 ShortestPath(BFS)를 여기 GhostTarget 다음에 붙여 유령이 벽을 돌아 팩맨을 정확히 추적하게 만들 수 있다. 배운 알고리즘을 다른 게임에 옮겨 붙이는 것 — 그것이 이 책이 준비한 마지막 연습이자, 개발자가 된다는 것의 의미다.

해설 펼치기

A1. 다른 것은 GhostTarget 이 계산하는 목표 타일 하나뿐이다(Blinky=팩맨, Pinky=4칸 앞, Inky=반사, Clyde=거리 조건). 같은 것은 그 목표로 가는 방법 — 교차로에서 목표에 가장 가까운 방향을 greedy로 고르는 ChooseGhostDir 를 넷이 공유한다. "성격 = 목표 함수".

A2. 팩맨의 진행 방향 4칸 앞을 노리면, 팩맨이 가려는 길목을 미리 막을 수 있다. Blinky가 뒤에서 쫓는 동안 Pinky가 앞에서 기다리는 협공이 되어, 팩맨을 포위한다.

A3. 역방향을 허용하면 유령이 교차로에서 왔다갔다 진동하거나 비겁하게 뒤로 빠질 수 있다. 금지하면 앞으로만 흐르는 자연스러운 추격이 된다. 예외는 모드가 전환되는 순간(SCATTER↔︎CHASE, 파워펠릿) — 이때만 강제로 방향을 반전한다(원작 규칙).

A4. 유령의 상태(state) 다. state 하나가 이동 속도(FRIGHTENED 느리게/EATEN 빠르게), 목표(GhostTarget 이 상태로 분기), 방향 선택 방식(FRIGHTENED면 랜덤, 아니면 greedy)을 모두 정한다. FSM의 핵심 — 상태가 행동 전체를 지휘한다.

A5. greedy는 "지금 당장 가장 가까운 방향"만 보므로 벽을 우회하지 못한다(막다른 곳으로 몰리거나 빙 돈다). 6단계 Maze의 BFS(ShortestPath) 로 목표까지 최단경로의 다음 칸을 계산하면, 벽을 돌아 정확히 추적하는 더 똑똑한 유령이 된다.

실습 미션 — 유령 타게팅과 추적 ghost.c ⭐

이 책의 마지막 미션이다. Pac-Man AI의 심장인 성격별 목표 계산 + greedy 방향 선택을 손으로 구현한다.

드릴 문법 — 좌표 계산(타게팅) · 방향 후보 순회 · 거리² 비교(greedy) · 역방향 금지 · 조건 분기 하는 것 — 팩맨 위치·방향과 유령 위치가 주어졌을 때, 네 유령의 목표를 계산하고, 한 유령의 다음 방향을 greedy로 고른다. 클리어 조건 — Blinky/Pinky/Clyde의 목표가 공식대로 나오고, 벽을 피해 목표에 가까워지는 방향을 고른다. 규모 — 약 55줄, 파일 1개. 표준 C만. 격자·거리·조건 — 이 책이 배운 것들의 총정리.

문제 정의

핵심은 두 단계다. ① 성격에 따라 목표 타일을 계산하고(Blinky=팩맨, Pinky=4칸 앞, Clyde=거리 조건), ② 그 목표에 가장 가까워지는 방향을 역방향 제외하고 고른다. 완벽한 최단(BFS)이 아니라 "지금 가장 가까운" greedy — 그래서 짧고 명료하다.

힌트

힌트 1 — 성격별 목표 
/* 팩맨 (px,py) 진행방향 (dx,dy), 유령 gx,gy */
void target(char who, int* tx, int* ty) {
    if (who == 'B') { *tx = px; *ty = py; }                 /* Blinky 직접 */
    else if (who == 'P') { *tx = px + 4*dx; *ty = py + 4*dy; } /* Pinky 4칸 앞 */
    else { int man = abs(gx-px)+abs(gy-py);                  /* Clyde 거리 조건 */
           if (man > 8) { *tx = px; *ty = py; }
           else         { *tx = CLYDE_CORNER_X; *ty = CLYDE_CORNER_Y; } }
}
힌트 2 — greedy 방향 (역방향 제외) 
int DX[4]={0,1,0,-1}, DY[4]={-1,0,1,0};   /* 북 동 남 서 */
int best=-1; long bestD=1L<<30;
for (int d=0; d<4; d++) {
    if (d == (curDir+2)%4) continue;      /* 역방향 금지 */
    int nx=gx+DX[d], ny=gy+DY[d];
    if (maze[ny][nx]=='#') continue;      /* 벽 제외 */
    long dd=(long)(nx-tx)*(nx-tx)+(long)(ny-ty)*(ny-ty);
    if (dd<bestD){bestD=dd;best=d;}
}

#define _CRT_SECURE_NO_WARNINGS
#include <stdio.h>
#include <stdlib.h>

#define H 7
#define W 11
char maze[H][W+1] = {
    "###########",
    "#.........#",
    "#.###.###.#",
    "#....B....#",   /* B = 유령 시작 위치 표시(참고용) */
    "#.###.###.#",
    "#....P....#",   /* P = 팩맨 위치 표시 */
    "###########",
};

/* 방향: 0북 1동 2남 3서 */
int DX[4] = { 0, 1, 0, -1 };
int DY[4] = { -1, 0, 1, 0 };
const char* DNAME[4] = { "북", "동", "남", "서" };

int px = 5, py = 5, pdir = 1;        /* 팩맨: (5,5), 동쪽으로 */
int gx = 5, gy = 3, gdir = 3;        /* 유령: (5,3), 서쪽으로 */

void target(char who, int* tx, int* ty) {
    int dx = DX[pdir], dy = DY[pdir];
    if (who == 'B') { *tx = px; *ty = py; }                       /* Blinky: 직접 */
    else if (who == 'P') { *tx = px + 4*dx; *ty = py + 4*dy; }    /* Pinky: 4칸 앞 */
    else {                                                        /* Clyde: 8칸 조건 */
        int man = abs(gx - px) + abs(gy - py);
        if (man > 8) { *tx = px; *ty = py; }
        else         { *tx = 1;  *ty = H - 2; }                   /* 좌하단 코너로 */
    }
}

int greedyDir(int tx, int ty) {
    int best = gdir; long bestD = 1L << 30;
    for (int d = 0; d < 4; d++) {
        if (d == (gdir + 2) % 4) continue;          /* 역방향 금지 */
        int nx = gx + DX[d], ny = gy + DY[d];
        if (ny < 0 || ny >= H || nx < 0 || nx >= W) continue;
        if (maze[ny][nx] == '#') continue;          /* 벽 제외 */
        long dd = (long)(nx-tx)*(nx-tx) + (long)(ny-ty)*(ny-ty);
        if (dd < bestD) { bestD = dd; best = d; }
    }
    return best;
}

int main(void) {
    int tx, ty;
    printf("팩맨 (%d,%d) %s향 / 유령 (%d,%d)\n\n", px, py, DNAME[pdir], gx, gy);

    char who[3] = { 'B', 'P', 'C' };
    const char* name[3] = { "Blinky(직접)", "Pinky(4칸앞)", "Clyde(변덕)" };
    for (int i = 0; i < 3; i++) {
        target(who[i], &tx, &ty);
        int d = greedyDir(tx, ty);
        printf("  %-14s 목표(%d,%d) → 다음 방향: %s\n", name[i], tx, ty, DNAME[d]);
    }
    return 0;
}

빌드: cl ghost.c (MSVC) 또는 gcc ghost.c -o ghost

함정/체크포인트

  • 목표만 바꾸면 성격이 바뀐다. greedyDir 는 셋이 똑같이 쓰고, target 만 다르다 — 개념 4의 "성격 = 목표 함수"를 코드로 확인하자.
  • 역방향 (gdir+2)%4. 네 방향에서 반대는 인덱스 +2(mod 4). 이걸 빼먹으면 유령이 왔다갔다 진동한다.
  • 거리는 제곱으로 비교. 제곱근을 안 구해도 대소 비교엔 지장 없다(더 빠르다). 6단계 A*의 맨해튼 거리와 짝을 이루는, 격자 거리의 또 다른 형태.

확장 과제

  • Inky(팩맨 2칸 앞을 Blinky로 반사: a + (a-b))를 추가해 넷을 완성하라.
  • greedy 대신 6단계 bfs.c 를 붙여, 벽을 우회해 목표로 가는 방향을 계산해 보라 — 그것이 "똑똑한 유령"이다.

Pac-Man으로 우리는 "생각하는 적"을 만들었다. 성격이 목표를 정하고, 상태머신이 행동을 지휘하며, 격자 위에서 팩맨과 유령이 쫓고 쫓긴다. 이 책의 여덟 번째 게임이자 마지막 게임 — 앞에서 배운 격자·타일·상태·거리·경로탐색이 모두 여기 모였다.

책을 닫으며 — 변수 하나에서 여기까지

기억하는가. 이 책은 1부에서 int money = 5000; 한 줄로, 2부에서 main.cpp 의 게임 루프 다섯 줄로 시작했다. 그 작은 시작에서 —

  • 1부 에서 값·계산·분기(A)로 회사를 세우고, 반복·함수(B)로 게임을 돌리고, 배열·문자열(C)로 사람과 이름을 채우고, 구조체·포인터(D)로 뼈대를 갖추고, 동적 메모리·랜덤(E)으로 회사를 키우고 시장을 살리고, 파일·모듈·캡스톤(F)으로 게임 회사 운영 게임 v1.0 을 출시했다.
  • 2부 에서 엔진(0)으로 실시간의 심장을 짓고, Snake(자료구조)·Tetris(알고리즘)·Dino(물리)·Road Fighter(시스템)로 살아있는 게임을 쌓았으며, 그 사이 Tycoon 이식으로 1부의 C를 2부의 C++로 다시 지었고, Galaga(성능)·Maze(알고리즘 심화)로 엔지니어링에 눈떴으며, Battle City(종합 캡스톤)로 그 모두를 수렴시키고, 마지막 Pac-Man(AI)으로 "생각하는 적"까지 만들었다.

문법과 기술을 배우는 순서가, 그대로 무언가를 완성해 가는 순서였다. 그래서 우리는 한 번도 "이걸 어디에 쓰지?"라고 묻지 않았다. 배운 즉시 그것이 게임의 살이 되었으니까. 그리고 1부에서 C로 만든 게임을 2부에서 C++로 다시 지으며, 두 여정이 사실 하나의 길이었음을 확인했다.

이 책이 되풀이해 말한 것들을 마지막으로 새겨 두자.

  • 불편을 먼저 겪고 도구를 쥔다. 병렬 배열의 고통 뒤에 구조체를, new/delete 남발 뒤에 오브젝트 풀을, malloc/realloc 뒤에 std::vector 를 배웠다. 왜 필요한지 몸으로 안 도구는 잊히지 않는다.
  • 동작하는 단순함을 먼저, 최적화는 측정한 뒤에. 선형 탐색도, 그리디 AI도 — "지금 충분한가"를 판단하는 것이 성능을 다루는 태도였다.
  • 완성된 것에도 언제나 성장의 여지가 있다. 각 게임의 as-built 이슈는 결함이 아니라 다음 문이었다. Pac-Man의 greedy에 Maze의 BFS를 붙이는 것처럼, 배운 것을 다른 곳에 옮겨 붙이는 것이 다음 걸음이다.

여기서 멈추지 않아도 된다. 유령에게 BFS를 달아 똑똑하게 만들고, 각 게임에 1부의 파일 I/O로 최고 점수를 저장하고, Docs/FUTURE_GAMES.md 의 Breakout·Sokoban에 도전하고, 당신만의 게임을 이 엔진 위에 새로 얹을 수 있다. 이 책은 끝이 아니라 출발점이다.

당신은 변수 하나에서 시작해, 두 개의 강을 건너, 여덟 게임과 하나의 회사를 만들었다. 이제 당신의 손에는 값·분기·반복·데이터·메모리·알고리즘·엔진·AI — 무언가를 만드는 데 필요한 뼈대가 모두 쥐어져 있다. 다음에 무엇을 만들지는, 당신의 몫이다.

끝까지 와 주어 고맙다. 이제, 당신의 게임을 만들 차례다.
