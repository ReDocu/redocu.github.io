5단계 · Galaga — 오브젝트 풀링

단계 5 ⭐ · 게임 Galaga · 씬 클래스 GalagaContent : public IGameContent 처음 배우는 기술 — 오브젝트 풀(Pool<T,N>) active 플래그·슬롯 재사용 게임 루프 내 new/delete 0회 제네릭 템플릿 대량 오브젝트 관리 편대(파생 좌표) 풀 × 풀 충돌 측정 후 최적화 예상 학습 시간 — 5–7시간

지금까지 우리는 오브젝트를 순진하게 다뤘다 — Snake는 마디마다 new/delete, Dino와 Road Fighter는 매 프레임 벡터를 통째로 다시 만들었다. 오브젝트가 적을 땐 문제없었다. 하지만 갤러그는 다르다. 총알·적·폭발이 초당 수십 개씩 생겼다 사라진다. 매 프레임 힙에서 잡고 놓기를 반복하면 그게 병목이 된다. 이 단계의 배움 주제는 딱 하나 — 오브젝트 풀(Object Pool). 고정 크기 배열을 미리 잡아 두고, active 플래그로 켜고 끄며 슬롯을 재사용한다. 게임 루프 안에서 new도 delete도 재할당도 0회. "동작하는 코드"에서 "성능을 의식하는 코드" 로 올라서는, 엔지니어링이 성숙하는 지점이다.

이 단계의 학습 목표

이 단계를 마치면 다음을 할 수 있다.

  1. 앞선 게임들의 "매 프레임 할당/재생성"이 왜 대량 오브젝트에서 병목이 되는지 설명할 수 있다.
  2. 제네릭 오브젝트 풀 Pool<T, N> 을 직접 만들고, Spawn/Kill/Clear 의 역할을 안다.
  3. 풀의 사용 패턴(스폰·순회·소멸)에서 왜 힙 할당이 0회인지 설명할 수 있다.
  4. 여러 종류의 단명 오브젝트(총알·적·폭발)를 각각의 풀로 관리할 수 있다.
  5. 적 편대를 슬롯 좌표에서 파생시켜 유지하고, 풀 × 풀 충돌을 처리할 수 있다.
  6. Spawn 의 선형 탐색(O(N))을 프리 리스트(O(1)) 로 바꾸는 "측정 후 최적화"의 방향을 안다.

왜 지금, 오브젝트 풀인가

이 책의 게임들이 오브젝트를 어떻게 다뤘는지 돌아보자.

  • Snake — 몸통 마디마다 new Snake(...), 리셋마다 delete. 아이템 먹을 때마다 힙 할당.
  • Dino · Road Fighter — 장애물·엔티티를 std::vector 에 담고, 화면 밖으로 나가면 매 프레임 새 벡터를 만들어 통째로 교체(kept.push_back).

이 방식들의 공통점은 "필요할 때 만들고, 필요 없으면 버린다" 는 것이다. 직관적이고, 오브젝트가 적을 땐 완벽하게 잘 돈다. 그런데 갤러그를 보자 — 플레이어 총알, 적 총알(강한 적은 3발씩!), 적 편대, 격추 폭발… 초당 수십 개의 오브젝트가 생겼다 사라진다. 이때 매 프레임 new/delete 하거나 벡터를 통째로 복사하면:

  • 할당 비용 — 힙에서 메모리를 잡고 놓는 것은 공짜가 아니다. 초당 수백 번이면 쌓인다.
  • 메모리 단편화 — 잡고 놓기를 반복하면 힙이 조각난다.
  • 예측 불가능한 비용 — 할당이 언제 얼마나 걸릴지 모른다. 실시간 게임에 치명적.

해법이 오브젝트 풀이다. 게임 시작 때 "총알은 최대 32개"라고 미리 배열을 잡아 두고, 총알을 쏠 때 새로 만드는 게 아니라 빈 슬롯을 빌려 쓴다. 총알이 사라지면 슬롯을 지우는 게 아니라 "비었다"고 표시만 한다. 그 슬롯은 다음 총알이 재사용한다. 게임 루프 안에서 힙을 건드리는 일이 아예 없다.

커리큘럼의 대조 학습. Road Fighter의 "매 프레임 벡터 재생성"을 기억하는가? 그 방식의 한계를 정면으로 드러내고 풀로 고치는 것 — 이것이 갤러그가 Road Fighter 바로 다음에 오는 이유다. "동작하는 코드"를 먼저 겪은 뒤에 "성능을 의식하는 코드"로 올라선다. 불편을 몸으로 안 사람만이 해법의 가치를 안다(1부 병렬 배열 → 구조체와 같은 정신).

개념 1 · 문제 다시 보기 — 매 프레임 할당의 비용

Road Fighter가 화면 밖 엔티티를 지우던 코드를 다시 보자.

// Road Fighter — 매 프레임, 벡터를 새로 만들어 교체
std::vector<Entity> kept;
for (size_t i = 0; i < entities.size(); i++)
    if (entities[i].y <= ROAD_BOTTOM)
        kept.push_back(entities[i]);   // 살아남은 것만 새 벡터로 복사
entities = kept;                       // 통째로 교체 (기존 벡터 파괴)

이 코드는 매 프레임 새 벡터를 만들고, 살아남은 엔티티를 전부 복사하고, 옛 벡터를 버린다. 엔티티가 10개면 별것 아니다. 하지만 총알·적·폭발이 100개, 200개가 되면? 매 프레임 200번 복사 + 힙 할당/해제가 초당 60번 반복된다. 이것이 실시간 게임에서 프레임 드랍(끊김) 의 흔한 원인이다.

핵심 통찰 — 단명 오브젝트(짧게 살다 죽는 것)를 매번 새로 만들고 버리는 것은 낭비다. 총알은 다 똑같이 생겼다. 죽은 총알의 메모리를 반납했다가 새 총알을 위해 다시 잡을 이유가 없다. 그 자리를 그냥 다시 쓰면 된다.

개념 2 · 오브젝트 풀 Pool<T, N> — 미리 잡고 재사용

오브젝트 풀은 놀랄 만큼 단순하다. 고정 크기 배열 하나와, 각 칸이 쓰이는 중인지 표시하는 active 배열 하나. 그게 전부다. 이걸 어떤 타입에든 쓸 수 있게 템플릿(0단계 개념 2)으로 만든다.

// 제네릭 오브젝트 풀. 고정 크기 N, active 플래그로 재사용.
template <typename T, int N>
struct Pool
{
    T    items[N];      // 실제 오브젝트들 (미리 N개 잡혀 있음)
    bool active[N];     // 각 칸이 쓰이는 중인가?

    Pool()       { for (int i = 0; i < N; i++) active[i] = false; }   // 처음엔 다 비었음
    void Clear() { for (int i = 0; i < N; i++) active[i] = false; }   // 전부 비우기

    int Spawn()          // 빈 슬롯을 빌린다. 없으면 -1.
    {
        for (int i = 0; i < N; i++)
            if (!active[i]) { active[i] = true; return i; }   // 첫 빈 칸을 켜고 인덱스 반환
        return -1;                                            // 다 찼음
    }
    void Kill(int i) { active[i] = false; }   // ★ 해제가 아니라 '비었음' 표시만!
};

세 함수의 의미를 새기자.

  • Spawn() — 새 오브젝트를 "만드는" 게 아니라, 빈 슬롯을 찾아 켜고 그 번호를 준다. new 가 없다. 이미 있는 배열 칸을 빌려주는 것뿐이다.
  • Kill(i) — 오브젝트를 "지우는" 게 아니라, active[i] = false 로 "이 칸 비었다"고 표시만 한다. delete 가 없다. 메모리는 그대로 남아 다음 Spawn 이 재사용한다.
  • Clear() — 재시작·스테이지 전환 때 전부 비운다. 역시 배열을 버리지 않는다.

items[N] 는 Pool 이 생길 때 딱 한 번 잡히고, 게임 내내 그 자리에 있다. 게임 루프 안에서는 이 배열을 절대 다시 할당하지 않는다. 총알이 몇천 발 나가도 배열은 N칸 그대로다 — 슬롯을 돌려 쓸 뿐이다.

왜 템플릿인가. 총알(Bullet)도, 적(Enemy)도, 폭발(Fx)도 똑같은 풀 로직이 필요하다. Pool<Bullet, 32>, Pool<Enemy, 40>, Pool<Fx, 32> — 타입과 크기만 바꿔 하나의 풀 코드를 재사용한다. 1부에서 같은 코드를 함수로 묶었듯, 여기선 같은 자료구조를 템플릿으로 묶는다. 이 Pool<T,N> 은 갤러그뿐 아니라 캡스톤 Battle City도 공유한다(FrameWork/Object/Pool.h).

개념 3 · 풀 사용 패턴 — 스폰·순회·소멸에 할당이 없다

풀을 실제로 쓰는 모습을 보면, "할당 0회"가 어떻게 실현되는지 보인다. 갤러그는 네 개의 풀을 멤버로 둔다.

Pool<Bullet, MAX_PBULLET> pBullets;   // 플레이어 총알 (32)
Pool<Bullet, MAX_EBULLET> eBullets;   // 적 총알 (64)
Pool<Enemy,  MAX_ENEMY>   enemies;    // 적 (40)
Pool<Fx,     MAX_FX>      fxs;         // 폭발 (32)

스폰 — Spawn() 으로 슬롯을 빌리고, 그 칸을 채운다.

if (INPUT->OnKeyStay(VK_SPACE) && TIMER->GetTickTimer(FIRE_DELAY))   // 발사 쿨다운
{
    int i = pBullets.Spawn();          // 빈 슬롯을 빌린다
    if (i >= 0)                        // 풀이 안 찼으면
        pBullets.items[i] = { px, PLAYER_Y - 1, 0.0f, -PBULLET_SPEED };  // 그 칸을 채움
}

순회 — active 인 칸만 갱신하고, 화면을 벗어나면 Kill.

for (int k = 0; k < MAX_PBULLET; k++)
{
    if (!pBullets.active[k]) continue;              // 비었으면 건너뜀
    pBullets.items[k].y += pBullets.items[k].vy;    // 위로 이동
    if (pBullets.items[k].y < HUD_Y + 1)            // 화면 위로 나갔으면
        pBullets.Kill(k);                           // 슬롯 반납(표시만)
}

이 두 조각을 보라 — 총알을 쏘는 것도(Spawn), 사라지는 것도(Kill), 이동시키는 것도(순회) 어디에도 new/delete/push_back 이 없다. 총알이 사라진 슬롯은 active[k]=false 가 되어, 다음 발사 때 Spawn() 이 바로 그 칸을 재사용한다. 32칸을 무한히 돌려 쓴다.

재시작·스테이지 전환도 마찬가지다.

// 스테이지 클리어 → 다음 스테이지
pBullets.Clear();  eBullets.Clear();  fxs.Clear();   // 전부 비우기 (재할당 아님!)
SpawnStage(stage);                                    // 새 편대를 풀에 채움

Clear() 는 active 를 전부 false 로 만들 뿐, 배열을 버리거나 다시 잡지 않는다. 풀의 배열은 게임이 끝날 때까지 한 번 잡은 그대로다.

개념 4 · 편대 — 슬롯 좌표에서 위치를 파생시키다

적 편대는 스페이스 인베이더식으로 좌우로 행진하다 벽에 닿으면 한 칸 내려온다. 40마리의 좌표를 하나하나 관리하면 복잡하다. 그래서 각 적은 편대 내 슬롯 (col, row) 만 저장하고, 실제 화면 위치는 편대 상태에서 계산(파생) 한다.

struct Enemy { _EEnemyType type; int col, row, hp; };   // 절대 좌표가 없다!

// 편대 상태
float formX;    // 0번 열의 x
int   formY;    // 지금까지 내려온 행 수

int GalagaContent::EnemyX(const Enemy& e) const { return (int)(formX+0.5f) + e.col * SPACING_X; }
int GalagaContent::EnemyY(const Enemy& e) const { return FORM_TOP + formY + e.row * SPACING_Y; }

formX 하나만 움직이면 40마리가 통째로 움직인다 — 각자의 위치는 formX + col*간격 으로 그때그때 계산되니까. 행진은 편대 상태만 갱신한다.

void GalagaContent::UpdateEnemies()
{
    if (TIMER->GetTickTimer(FormTick()))   // 스테이지별 행진 속도
    {
        float next = formX + formDir;
        int leftX = (int)(next + 0.5f);
        int rightX = leftX + (formCols - 1) * SPACING_X;

        if (leftX <= PLAY_LEFT || rightX >= PLAY_RIGHT) {   // 벽에 닿으면
            formDir = -formDir;    // 방향 반전
            formY += 1;            // 한 칸 하강
        } else {
            formX = next;          // 아니면 그냥 옆으로
        }
    }
    // 공격: 살아있는 적 중 랜덤 하나가 발사 ...
}

"상태 하나로 여럿을 제어" 하는 이 발상은 강력하다. 40마리의 x를 각각 갱신하는 대신, formX 하나만 바꾼다. 벽 판정도 편대의 좌·우 끝(0번 열과 마지막 열)만 보면 된다. 데이터를 파생 가능한 최소 상태로 줄이는 것 — 좋은 자료구조 설계의 한 축이다.

공격은 풀에서 살아있는 적을 모아 랜덤으로 하나 골라 쏜다.

int aliveIdx[MAX_ENEMY], n = 0;
for (int i = 0; i < MAX_ENEMY; i++)
    if (enemies.active[i]) aliveIdx[n++] = i;      // 살아있는 적 인덱스 수집

if (n > 0) {
    const Enemy& e = enemies.items[aliveIdx[rand() % n]];   // 랜덤 하나
    // 그 적 위치에서 eBullets.Spawn()으로 아래로 발사 (강한 적은 3방향)
}

개념 5 · 풀 × 풀 충돌 — 활성 슬롯끼리 검사

충돌은 풀과 풀을 겹쳐 검사한다. 내 총알(풀) × 적(풀), 적 총알(풀) × 나, 적(풀) × 나. active 인 것끼리만 좌표를 비교한다.

// 내 총알 → 적
for (int k = 0; k < MAX_PBULLET; k++)
{
    if (!pBullets.active[k]) continue;
    int bx = (int)(pBullets.items[k].x + 0.5f);
    int by = (int)(pBullets.items[k].y + 0.5f);

    for (int i = 0; i < MAX_ENEMY; i++)
    {
        if (!enemies.active[i]) continue;
        if (bx == EnemyX(enemies.items[i]) && by == EnemyY(enemies.items[i]))   // 같은 칸?
        {
            enemies.items[i].hp--;
            pBullets.Kill(k);                       // 총알 소멸(슬롯 반납)
            if (enemies.items[i].hp <= 0) {
                SpawnFx(...);                        // 폭발 풀에서 스폰
                score += ScoreOf(enemies.items[i].type);
                enemies.Kill(i);                     // 적 소멸(슬롯 반납)
                aliveCount--;
            }
            break;   // 이 총알은 이미 소멸, 다음 총알로
        }
    }
}

충돌에서도 Kill 만 부른다 — 총알도 적도 슬롯을 반납할 뿐, 아무것도 delete 되지 않는다. 격추 즉시 SpawnFx 로 폭발 풀에서 이펙트를 스폰하고, 그 폭발도 6프레임 뒤 Kill 된다(개념 4의 무적 타이머와 같은 "수명" 발상). 모든 오브젝트가 풀 위에서 태어나고, 풀 위에서 죽는다.

비용을 의식하기. 이 충돌은 활성 총알 수 × 활성 적 수 만큼 비교한다(이중 루프). 총알 32 × 적 40 = 최대 1280번. 60fps면 초당 76,800번. 콘솔 게임엔 충분하지만, 오브젝트가 수백을 넘으면 공간 분할(격자로 나눠 가까운 것만 검사)로 줄인다 — 그건 폴리시의 몫이다. "지금 충분한가"를 늘 의식하는 것이 성능을 다루는 태도다.

개념 6 · 측정 후 최적화 — 선형 탐색에서 O(1)로

우리 Spawn() 을 다시 보자.

int Spawn()
{
    for (int i = 0; i < N; i++)              // 처음부터 훑으며
        if (!active[i]) { active[i] = true; return i; }   // 첫 빈 칸을 찾는다
    return -1;
}

이건 선형 탐색 — 최악의 경우 N칸을 다 훑는다(O(N)). 총알을 쏠 때마다 32칸을 처음부터 확인할 수 있다. 이걸 프리 리스트(free list) 로 바꾸면 O(1)이 된다: 비어 있는 슬롯의 번호들을 따로 스택에 쌓아 두고, Spawn 은 스택에서 하나 꺼내고 Kill 은 스택에 다시 넣는다. 빈 칸을 "찾을" 필요 없이 "받아" 온다.

그런데 MVP는 일부러 선형 탐색으로 시작한다. 왜? 갤러그 규모(풀 크기 32~64)에선 선형 탐색이 전혀 문제가 안 되기 때문이다. 초당 76,800번 비교? 현대 CPU엔 순식간이다. 문제가 되지도 않는 것을 미리 최적화하는 것은 시간 낭비이자 버그의 씨앗이다.

이것이 "측정 후 최적화(measure, then optimize)" 원칙이다 — 먼저 단순하고 정확하게 만들고, 실제로 느린 곳을 측정한 뒤에 그 지점만 고친다. 갤러그의 교훈은 풀 자체뿐 아니라, "언제 최적화할지 판단하는 것" 이기도 하다.

도널드 크누스의 유명한 말 — "섣부른 최적화는 만악의 근원이다." 선형 탐색 Spawn 은 "충분히 빠른 단순한 코드"의 예다. 프리 리스트는 "정말 필요할 때"의 개선책으로 GDD가 폴리시로 남겨 뒀다. 동작하는 단순함을 먼저, 측정된 최적화를 나중에.

코드 워크스루 — OnInGameUpdate, 네 풀이 도는 한 프레임

갤러그의 한 프레임은 네 개의 풀을 차례로 갱신하고, 풀끼리 충돌시키는 파이프라인이다.

실제 파일: Main/Content/GalagaContent.cpp · FrameWork/Object/Pool.h

void GalagaContent::OnInGameUpdate()
{
    // ... ESC · PAUSE · STAGECLEAR · GAMEOVER 처리 ...

    // ── PLAYING ──
    UpdatePlayer();       // ① 이동 · 발사(pBullets.Spawn) · 무적
    UpdateEnemies();      // ② 편대 행진(formX) · 공격(eBullets.Spawn)
    UpdateBullets();      // ③ pBullets · eBullets 이동 · 화면 밖 Kill
    UpdateFx();           // ④ 폭발 프레임++ · 수명 다하면 Kill
    HandleCollisions();   // ⑤ 풀 × 풀 충돌 → hp--, Kill, 점수, FX

    if (aliveCount <= 0) {                      // 적 전멸 → 스테이지 클리어
        score += 1000;
        gameState = _EGameState::STAGECLEAR;
    }
    if (EnemiesBreached())                      // 편대가 플레이어 줄까지 내려옴 → 패배
        gameState = _EGameState::GAMEOVER;
}

다섯 단계 전부가 풀 연산이다 — Spawn(①②), 순회 후 Kill(③④), 충돌 후 Kill(⑤). 이 프레임 어디에도 힙 할당이 없다. aliveCount 로 전멸을 즉시 알고(적을 Kill 할 때마다 감소), EnemiesBreached() 로 편대가 바닥에 닿았는지 본다.

SpawnStage 는 풀을 채우는 유일한 곳이다 — 편대 크기만큼 enemies.Spawn() 을 불러 슬롯을 켜고 (col, row, type, hp) 를 채운다. 이것도 new 가 아니라 미리 잡힌 배열의 슬롯을 켜는 것이다.

for (int r = 0; r < formRows; r++)
    for (int c = 0; c < formCols; c++) {
        int i = enemies.Spawn();
        if (i < 0) continue;
        enemies.items[i] = { /* type */, c, r, /* hp */ };
        aliveCount++;
    }

렌더도 풀 순회다 — 각 풀을 한 번씩 훑어 active 인 것만 그린다(적 ▼, 총알 |/!, 폭발 * × +). 활성 개수에 상한이 있으니 최악 렌더 비용도 예측 가능하다 — 풀의 또 다른 이점이다.

그림으로 정리 — 재할당 vs 슬롯 재사용

📊 웹 교육문서에서는 벡터 재생성과 풀 슬롯 재사용이 나란히 애니메이션(V-P2-5)으로 비교된다. 책에서는 아래 도식으로 대신한다.

두 방식의 비교 — 총알이 죽고 새로 생길 때

  [Road Fighter 방식] 매 프레임 새 벡터 (힙 할당 반복)
    entities ─┐  kept 새로 만듦 → 살아남은 것 복사 → 옛 벡터 파괴 → 교체
              └─ 초당 60번 × 오브젝트 수만큼 복사·할당

  [Galaga 풀] 미리 잡은 배열, active 플래그만 토글 (할당 0회)
    items:   [총알A][총알B][ ---- ][총알C][ ---- ] ...  ← 배열은 그대로
    active:  [  1  ][  1  ][  0   ][  1  ][  0   ]
                            ▲                ▲
                     Kill 로 0 표시    Spawn 이 이 빈 칸을 재사용
    → new/delete 없이 슬롯을 무한히 돌려 쓴다

풀의 세 연산

  Spawn():  active 에서 첫 0 을 찾아 1 로 → 그 index 반환 (칸을 빌림)
  Kill(i):  active[i] = 0            → 슬롯 반납 (메모리는 그대로)
  Clear():  active 전부 0            → 재시작 (배열은 안 버림)

한 문장으로: 오브젝트 풀은 "만들고 버리기"를 "빌리고 돌려주기"로 바꾼다 — 한 번 잡은 배열을 게임 내내 재사용해, 게임 루프에서 힙을 지운다.

알려진 이슈에서 배우기 (as-built)

갤러그의 풀은 학습용 최소 구현이다. 의도적으로 단순하게 둔 부분들이, 오히려 "다음에 무엇을 개선할지"를 가르친다.

이슈 as-built 상태 무엇을 배우나
Spawn 선형 탐색 O(N) — 매번 처음부터 빈 칸 탐색 규모가 작아 충분. 프리 리스트 O(1) 는 측정 후(개념 6)
충돌 O(총알×적) 이중 루프 전수 비교 수백 개 넘으면 공간 분할. 지금은 충분
댕글링 인덱스 Kill 후에도 그 index로 접근 가능 슬롯 번호가 재사용되면 옛 참조가 엉뚱한 걸 가리킴 — 세대 인덱스로 방어(심화)
풀 넘침 = 조용한 실패 Spawn() 이 -1이면 그냥 발사 안 됨 상한을 넘는 스폰은 무시됨(용량 설계가 중요)
정수 좌표 충돌 float 위치를 반올림해 "같은 칸" 비교 빠른 총알이 한 프레임에 적을 건너뛸 수 있음(터널링)

"선형 탐색"과 "전수 비교"를 일부러 남겨 둔 것에 주목하자. 이것은 게으름이 아니라 판단이다 — 갤러그 규모에선 이 단순함이 옳고, 최적화는 병목이 측정된 뒤에 한다(개념 6). 반면 "댕글링 인덱스", "터널링"은 규모와 무관한 잠재 함정이라, 오브젝트 풀을 실무에 쓸 때 알아 둬야 한다. 무엇이 "지금 충분한 단순함"이고 무엇이 "언젠가 물릴 함정"인지 구별하는 눈 — 그것이 이 단계의 진짜 배움이다.

체크포인트 — 스스로 점검하기

Q1. Snake의 new/delete, Road Fighter의 벡터 재생성이 갤러그에서 문제가 되는 이유는? 오브젝트 풀은 이를 어떻게 해결하나?

Q2. Pool::Kill(i) 는 delete 를 하지 않고 active[i] = false 만 한다. 이렇게 해도 되는 이유와, 그 슬롯은 이후 어떻게 되나?

Q3. 게임 루프 안에서 총알을 수천 발 쏴도 힙 할당이 0회인 이유를 Spawn/Kill 로 설명하라.

Q4. 적 40마리를 각자 좌표로 저장하지 않고 (col, row) 슬롯만 저장한다. formX 하나로 편대 전체를 움직일 수 있는 이유는?

Q5. Spawn() 의 선형 탐색(O(N))을 프리 리스트로 바꾸면 O(1)이 된다. 그런데 MVP가 일부러 선형 탐색으로 시작하는 이유는?

해설 펼치기

A1. 총알·적·폭발이 초당 수십 개 생겼다 사라지는데, 매번 new/delete(힙 할당/해제)나 벡터 통째 복사를 하면 그 비용이 쌓여 프레임 드랍(끊김)과 메모리 단편화를 부른다. 오브젝트 풀은 고정 배열을 미리 잡아 두고 슬롯을 재사용해, 게임 루프 안에서 힙 할당을 0회로 만든다.

A2. 풀의 items[N] 배열은 이미 잡혀 있어서, 슬롯을 "지울" 필요가 없다. active[i]=false 는 "이 칸 비었다"는 표시일 뿐, 메모리는 그대로 남는다. 그 슬롯은 다음 Spawn() 이 재사용한다 — 빌려 줬다 돌려받는 것이다.

A3. Spawn() 은 새로 만드는 게 아니라 미리 잡힌 배열의 빈 칸을 켜서 번호를 준다(new 없음). Kill(i) 는 그 칸을 비었다고 표시만 한다(delete 없음). 그래서 총알이 몇천 발이 오가도 배열은 N칸 그대로고, 슬롯만 돌려 쓴다 — 힙을 건드리지 않는다.

A4. 각 적의 실제 위치를 저장하지 않고 계산(파생) 하기 때문이다: EnemyX = formX + col*간격. formX 하나만 바꾸면 모든 적의 파생 위치가 함께 바뀐다. 40개의 x를 각각 갱신할 필요 없이 편대 상태 하나만 옮기면 된다.

A5. 갤러그 규모(풀 32~64)에선 선형 탐색이 전혀 병목이 아니라서다. 문제되지 않는 것을 미리 최적화하는 것은 시간 낭비이자 버그의 씨앗이다("섣부른 최적화"). 먼저 단순·정확하게 만들고, 실제로 느린 곳을 측정한 뒤 그 지점만 고친다("측정 후 최적화").

실습 미션 — 총알 풀 만들기 pool.c ⭐

Road Fighter의 미션이 "엔티티 시스템"이었다면, 이 미션은 오브젝트 풀 그 자체를 손으로 만든다. 고정 배열 + active 플래그로, new/delete 없이 총알을 무한히 돌려 쓰는 것을 눈으로 확인한다.

드릴 문법 — 고정 배열 + active 플래그 · Spawn(빈 슬롯 찾기) · Kill(표시만) · 순회 · 힙 할당 0회로 슬롯 재사용 만드는 것 — 8칸짜리 총알 풀. 매 턴 총알을 발사(스폰)하고 이동시키며, 화면 밖 총알의 슬롯이 재사용되는 것을 관찰한다. 클리어 조건 — 수십 발을 쏴도 배열은 8칸 그대로, 슬롯이 재사용된다. 규모 — 약 50줄, 파일 1개. 표준 C만(템플릿 없이 struct 배열로 개념 구현).

문제 정의

C에는 템플릿이 없으니, Pool<T,N> 대신 Bullet 구조체 배열 + active 플래그로 개념을 그대로 구현한다. Spawn() 은 빈 칸을 찾아 켜고, Kill() 은 표시만 끈다. 매 턴 총알이 위로 이동하다 화면 밖으로 나가면 그 슬롯이 비고, 다음 발사가 바로 그 칸을 재사용한다. 총 발사 수를 세어, 배열 크기(8)를 훨씬 넘겨도 배열은 그대로임을 확인하는 게 핵심이다.

목표 / 클리어 조건

  1. Bullet pool[8] 과 int active[8] (또는 struct에 active 필드)로 풀을 만든다.
  2. Spawn() — 첫 번째 빈 슬롯을 켜고 인덱스 반환(없으면 -1).
  3. Kill(i) — active[i] = 0 (메모리 안 건드림).
  4. 매 턴: 발사 요청 수만큼 Spawn, active 총알을 위로 이동, 화면 밖은 Kill.
  5. 슬롯 상태(어느 칸이 활성인지)와 누적 발사 수를 매 턴 출력.

힌트

힌트 1 — Spawn / Kill 
#define POOL 8
typedef struct { int y; int active; } Bullet;
Bullet pool[POOL];

int Spawn(void) {
    for (int i = 0; i < POOL; i++)
        if (!pool[i].active) { pool[i].active = 1; return i; }  /* 첫 빈 칸 */
    return -1;                                                  /* 다 참 */
}
void Kill(int i) { pool[i].active = 0; }   /* 표시만 (free 없음) */
힌트 2 — 순회 이동 + 슬롯 재사용 관찰 
for (int i = 0; i < POOL; i++) {
    if (!pool[i].active) continue;
    pool[i].y--;                       /* 위로 이동 */
    if (pool[i].y < 0) Kill(i);        /* 화면 밖 → 슬롯 반납 */
}

슬롯 맵을 출력하면(active[i] 를 #/. 로) 죽은 칸이 다음 스폰에 재사용되는 걸 볼 수 있다.

정답 코드

정답 펼치기 — pool.c 
#define _CRT_SECURE_NO_WARNINGS
#include <stdio.h>

#define POOL   8
#define TOP_Y  10      /* 발사 높이 */

typedef struct { int y; int active; } Bullet;

Bullet pool[POOL];
int totalSpawns = 0;   /* 누적 발사 수 (배열 크기와 비교용) */

int Spawn(void)
{
    for (int i = 0; i < POOL; i++)
        if (!pool[i].active) { pool[i].active = 1; pool[i].y = TOP_Y; totalSpawns++; return i; }
    return -1;          /* 풀이 가득 참 → 발사 무시 */
}
void Kill(int i) { pool[i].active = 0; }   /* 해제 아님, 표시만 */

void PrintPool(void)
{
    printf("  슬롯: ");
    for (int i = 0; i < POOL; i++)
        printf("%c", pool[i].active ? '#' : '.');
    printf("   (활성 y: ");
    for (int i = 0; i < POOL; i++)
        if (pool[i].active) printf("%d ", pool[i].y);
    printf(")\n");
}

int main(void)
{
    int turn, fire;

    for (int i = 0; i < POOL; i++) pool[i].active = 0;   /* Clear */

    printf("=== 총알 풀 (%d칸, new/delete 0회) ===\n", POOL);
    for (turn = 1; turn <= 20; turn++)
    {
        printf("\n[턴 %d] 발사 수(0~%d, -1 종료): ", turn, POOL);
        if (scanf("%d", &fire) != 1) break;
        if (fire < 0) break;

        for (int f = 0; f < fire; f++)
            if (Spawn() < 0) { printf("  ! 풀이 가득 참 — 발사 무시\n"); break; }

        /* 순회 이동 + 화면 밖 제거 (슬롯 재사용) */
        for (int i = 0; i < POOL; i++) {
            if (!pool[i].active) continue;
            pool[i].y--;
            if (pool[i].y < 0) Kill(i);
        }

        PrintPool();
        printf("  누적 발사 %d회 — 그런데 배열은 여전히 %d칸 (재사용!)\n", totalSpawns, POOL);
    }
    return 0;
}

빌드: cl pool.c (MSVC) 또는 gcc pool.c -o pool

함정/체크포인트

  • Kill 은 free 가 아니다. 슬롯을 "비었다"고 표시만 한다. 그래서 배열은 절대 줄거나 늘지 않는다 — 이것이 풀의 핵심이다. 누적 발사가 수십이 되어도 배열은 8칸임을 눈으로 확인하자.
  • 풀이 가득 차면 Spawn 이 -1. 8칸이 다 활성인데 또 쏘면 발사가 무시된다. 상한이 있다는 것 — 용량(N)을 게임에 맞게 정하는 게 설계다.
  • 슬롯 재사용 관찰. 슬롯 맵(#/.)을 보면, 위로 나가 죽은 칸(.)이 다음 발사 때 다시 # 이 된다. 그 자리가 재사용되는 것 — new 없이.

확장 과제

  • Spawn 을 프리 리스트(빈 인덱스 스택)로 바꿔 O(1)로 만들어 보자(개념 6). 선형 탐색과 결과가 같은지 확인.
  • 총알에 type 을 추가하면, Road Fighter의 엔티티 시스템 + 풀이 결합된다 — 그게 바로 캡스톤 Battle City다.

갤러그로 우리는 "동작하는 코드"에서 "성능을 의식하는 코드" 로 올라섰다. 초당 수십 개의 총알이 오가도, 미리 잡은 풀 위에서 슬롯을 돌려 쓰며 힙을 건드리지 않는다. 다음 6단계 Maze는 방향을 튼다 — 성능이 아니라 알고리즘 그 자체를 주제로, 미로를 만드는 다섯 가지 생성 알고리즘과 길을 찾는 네 가지 탐색 알고리즘을 다룬다. 그리고 거기서 익힌 생성·탐색이, 마지막 캡스톤 Battle City에서 이 풀과 만난다.
